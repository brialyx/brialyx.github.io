Event logging

Pourquoi logger les événements ?

📔
Analogie — journal de bord avion Logs réseau = boîte noire avion. Enregistre TOUT (démarrages, erreurs, alertes). En cas incident : Logs = seule source vérité reconstruire timeline. Sans logs = impossible troubleshooter.

Logging = Enregistrement événements système/réseau (démarrages, erreurs, authentifications, changements config). Indispensable troubleshooting, audit, sécurité.

// Problèmes sans logging

  • ❌ Troubleshooting impossible : Switch crashé 03:00 AM → pourquoi ? Aucune trace
  • ❌ Pas audit trail : Qui changé config firewall hier ? Mystère
  • ❌ Sécurité compromise : Intrusion non détectée (pas logs accès)
  • ❌ Compliance violations : PCI-DSS, HIPAA exigent logs centralisés

// Types événements loggés

  • System events : Boot, shutdown, reboot device
  • Interfaces : Link up/down, errors, speed changes
  • Authentification : Login/logout (SSH, console), échecs authentification
  • Configuration : Changements config (qui, quand, quoi)
  • Routing : Changements tables routage (OSPF neighbor down, BGP session reset)
  • Sécurité : ACL denies, VPN connections, intrusion attempts
  • Hardware : Température élevée, fan failure, power supply issues
📝
Logs = données forensiques CRITIQUES Intrusion réseau ? Logs = seule preuve reconstruire attaque. Compliance audits ? Logs = preuves obligatoires. Conservation logs 90-365 jours minimum (réglementations).
Protocole standard

Syslog — Protocole logging

Syslog = protocole standard logging messages système/réseau (RFC 5424). Universel (Cisco, Juniper, Linux, Windows...).

// Architecture syslog

  • Syslog Client : Équipements générant logs (switches, routeurs, firewalls, serveurs)
  • Syslog Server : Serveur centralisé recevant/stockant logs (syslog-ng, rsyslog, Graylog)
  • Transport : UDP 514 (défaut, pas fiable) ou TCP 514/6514 (fiable, TLS sécurisé)
[Switch Cisco] → Syslog message (UDP 514)
[Router Juniper] → Syslog message (UDP 514)
[Firewall Palo Alto] → Syslog message (UDP 514)
↓
[Syslog Server 10.1.1.100]
↓
[Stockage logs centralisé]

→ Tous équipements envoient logs vers serveur unique
→ Centralisation = recherche/analyse facilitées

// UDP vs TCP syslog

UDP 514 (défaut)

  • ✅ Léger, pas overhead
  • ⚠️ Pas garanti livraison (logs perdus possible si congestion)
  • Usage : Environnements bande passante limitée

TCP 514/6514

  • ✅ Livraison garantie (acknowledgments)
  • ✅ TLS possible (port 6514, chiffrement logs sensibles)
  • ⚠️ Overhead TCP
  • Usage : Logs critiques (sécurité, compliance)
Severity levels

8 niveaux de sévérité (0-7)

Syslog définit 8 niveaux sévérité (0 = plus critique, 7 = moins critique).

NiveauNomDescriptionExemple
0 Emergency Système inutilisable (crash total) Device crash, kernel panic
1 Alert Action immédiate requise Hardware failure critique (PSU down)
2 Critical Condition critique Température CPU excessive
3 Error Erreur fonctionnelle Interface down, routing protocol error
4 Warning Avertissement (pas erreur) Utilisation mémoire >80%, config deprecated
5 Notice Normal mais significatif Interface up, OSPF neighbor established
6 Informational Messages informatifs Login réussi, config saved
7 Debug Debug détaillé (développeurs) Packets détails, protocole debug

// Filtrage par sévérité

Configurer niveau minimum logging → évite spam logs non critiques.

Config Cisco IOS :

logging trap warnings

→ Envoie vers syslog server uniquement niveaux 0-4
→ (Emergency, Alert, Critical, Error, Warning)
→ Ignore niveaux 5-7 (Notice, Informational, Debug)

Production typique : logging trap warnings ou errors
Debug jamais en production (flood logs)
🎯
Question TRÈS fréquente examen "Niveau syslog le PLUS critique ?"

Réponse : 0 (Emergency)

"Niveau syslog debug ?"
Réponse : 7 (Debug)

Mémo : 0 = pire (Emergency), 7 = moins grave (Debug)
Structure

Format message syslog

// Structure message (RFC 5424)

<Priority> Timestamp Hostname Process[PID]: Message

Exemple :
<189> Feb 26 14:35:12 SW-CORE-01 %SYS-5-CONFIG_I: Configured from console by admin on vty0

Décomposition :
- <189> : Priority (Facility 23 × 8 + Severity 5 = 189)
- Feb 26 14:35:12 : Timestamp
- SW-CORE-01 : Hostname device source
- %SYS-5-CONFIG_I : Facility-Severity-Mnemonic
- Configured from console... : Message texte

// Priority (Facility + Severity)

Priority = Facility × 8 + Severity

Facility = Catégorie source message (0-23).

  • 0 = kernel
  • 1 = user-level
  • 4 = auth/security
  • 16 = local0 (custom)
  • 23 = local7 (custom)

Calcul : Facility 23 (local7) + Severity 5 (Notice) = 23 × 8 + 5 = 189

// Timestamp

Problème : Équipements réseau souvent pas horloge synchronisée → timestamps inexacts.

Solution : NTP (Network Time Protocol) synchronise horloges devices.

Config NTP + Syslog timestamps :

ntp server 10.1.1.50
clock timezone EST -5
service timestamps log datetime msec localtime show-timezone

→ Synchro NTP vers 10.1.1.50
→ Timestamps logs précis millisecondes + timezone
⏰
NTP OBLIGATOIRE pour logs fiables Logs sans NTP = timestamps erronés → impossible corréler événements multi-devices. Troubleshooting incident 03:15 AM impossible si timestamps +/- 2h différence.
Syslog server

Centralisation logs

Syslog server = Serveur dédié recevant/stockant logs TOUS devices réseau. Centralisation CRITIQUE.

// Avantages centralisation

  • ✅ Recherche unifiée : 1 lieu chercher logs tous devices
  • ✅ Corrélation événements : Switch + Router + Firewall logs ensemble → détecte patterns
  • ✅ Stockage long-terme : Devices RAM limitée (100-1000 logs max), serveur illimité
  • ✅ Backup automatique : Logs perdus si device crashe SAUF si centralisés
  • ✅ Compliance : Audits exigent logs centralisés 90-365 jours
  • ✅ Analyse/Alerting : SIEM consomme logs centralisés

// Solutions syslog server

rsyslog (Linux)

  • Open-source, inclus Linux
  • Très rapide, léger
  • Filtres puissants
  • Export MySQL, PostgreSQL, Elasticsearch

syslog-ng

  • Open-source, avancé
  • Parsing/filtrage complexe
  • Corrélation événements

Graylog

  • Open-source, moderne
  • Web interface puissante
  • Search Elasticsearch-based
  • Dashboards, alerting

Splunk

  • Commercial (très cher)
  • Leader marché entreprise
  • Analytics avancées
  • Machine learning anomaly detection

// Configuration syslog client

Cisco IOS config syslog :

logging host 10.1.1.100
logging trap warnings
logging facility local7
logging source-interface GigabitEthernet0/0

→ Envoie logs vers syslog server 10.1.1.100
→ Niveau minimum : warnings (0-4)
→ Facility local7
→ Source IP = Gi0/0 (identifie device)
💾
Rétention logs : 90-365 jours minimum Compliance (PCI-DSS, HIPAA) exige logs 90-365 jours. Forensics incidents nécessite historique. Stockage disque pas cher → conserver 1 an standard.
Logging destinations

Où envoyer les logs ?

Équipements réseau peuvent logger vers multiples destinations simultanément.

// Destinations communes

1. Console (console 0)

  • Port console physique
  • Temps réel (troubleshooting direct)
  • ⚠️ Perdu après déconnexion
  • Config : logging console

2. Buffer (RAM)

  • Stockage RAM device (circulaire, 4096 bytes défaut)
  • Commande : show logging
  • ⚠️ Perdu si reboot
  • Config : logging buffered 16384

3. Terminal/VTY (SSH/Telnet)

  • Affiche logs session SSH active
  • Commande : terminal monitor
  • Config : logging monitor

4. Syslog Server

  • ✅ Centralisation (RECOMMANDÉ)
  • Stockage permanent
  • Config : logging host 10.1.1.100

5. SNMP Traps

  • Convertit logs en SNMP traps
  • Intégration monitoring SNMP
  • Config : logging trap
