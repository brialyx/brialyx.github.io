Routing table

Route manquante — Missing route

Route manquante = Router pas entrée routing table pour destination → packets dropped.

// Principe routing (rappel)

Router reçoit packet destination 10.2.0.50

1. Check routing table : Route vers 10.2.0.0/24 ?
2. Match found → Forward interface
3. No match → Drop packet (no route)

// Symptômes route manquante

  • Ping timeout : Destination unreachable
  • Traceroute stops : Last hop = router sans route
  • ICMP error : "Destination network unreachable" (if router sends ICMP reply)
  • Selective connectivity : Some networks accessible, others not

// Diagnostic route manquante

User report : Can't reach server 10.2.0.50

Step 1 : Traceroute from source
C:\> tracert 10.2.0.50

1    1 ms    192.168.1.1 (gateway)
2    5 ms    10.0.0.1 (router-core)
3    * * *    Request timed out

→ Stops at 10.0.0.1 (router-core) → No route forward

Step 2 : Check router routing table
Router-Core# show ip route

C    10.0.0.0/24 is directly connected, Gi0/0
C    192.168.1.0/24 is directly connected, Gi0/1
S    0.0.0.0/0 via 203.0.113.1 (default route Internet)

→ No route to 10.2.0.0/24 (missing!)
→ Packets to 10.2.0.50 dropped

// Résolution route manquante

Option 1 : Add static route

Router-Core(config)# ip route 10.2.0.0 255.255.255.0 10.0.0.2

→ Add route : 10.2.0.0/24 via next-hop 10.0.0.2

Verify :
Router-Core# show ip route
S    10.2.0.0/24 [1/0] via 10.0.0.2

Test :
Router-Core# ping 10.2.0.50
→ Success ✅

Option 2 : Enable dynamic routing

Router-Core(config)# router ospf 1
Router-Core(config-router)# network 10.0.0.0 0.0.0.255 area 0
Router-Core(config-router)# network 192.168.1.0 0.0.0.255 area 0

→ OSPF advertises routes automatically
→ Learns 10.2.0.0/24 from neighbor routers
🎯
Question fréquente examen "Traceroute stops at router. Ping destination unreachable. MOST likely cause ?"
A) Cable defect
B) Missing route
C) DNS failure
D) Firewall blocking

Réponse : B) Missing route
Traceroute stops router = no route forward (routing table incomplete).
Misconfiguration

Route incorrecte — Incorrect route

Route incorrecte = Route exists MAIS pointe wrong next-hop/interface → packets sent wrong direction.

// Scénario route incorrecte

Network topology :

10.2.0.0/24 (remote network) ← Router-B (10.0.0.2)
                                   ↑
                            Router-Core (10.0.0.1)
                                   ↓
                           192.168.1.0/24 (local)

INCORRECT route config :
Router-Core(config)# ip route 10.2.0.0 255.255.255.0 10.0.0.99

→ Next-hop 10.0.0.99 doesn't exist (typo - should be .2)
→ Packets to 10.2.0.0/24 sent nowhere → Dropped

// Symptômes route incorrecte

  • Route exists : show ip route shows entry (misleading)
  • Connectivity fails : Ping timeout despite route
  • Traceroute strange path : Packets routed unexpected direction
  • Intermittent connectivity : Works if failover path exists, fails primary

// Diagnostic route incorrecte

Router-Core# show ip route 10.2.0.0

Routing entry for 10.2.0.0/24
  Known via "static", distance 1, metric 0
  Routing Descriptor Blocks:
  * 10.0.0.99, from 0.0.0.0

→ Next-hop 10.0.0.99 (verify exists ?)

Router-Core# ping 10.0.0.99
→ Timeout (next-hop unreachable)

→ Route exists BUT next-hop invalid = incorrect route

// Résolution route incorrecte

Step 1 : Remove incorrect route
Router-Core(config)# no ip route 10.2.0.0 255.255.255.0 10.0.0.99

Step 2 : Add correct route
Router-Core(config)# ip route 10.2.0.0 255.255.255.0 10.0.0.2

Step 3 : Verify
Router-Core# show ip route 10.2.0.0
  * 10.0.0.2 ✅

Router-Core# ping 10.2.0.50
→ Success ✅
Loop prevention

Routing Loop — Packets circulent infiniment

Routing loop = 2+ routers forward packet back-and-forth infinitely → TTL expires → packet dropped.

// Principe routing loop

Routing loop scenario :

Router-A : Route 10.2.0.0/24 via Router-B
Router-B : Route 10.2.0.0/24 via Router-A (misconfiguration)

Packet to 10.2.0.50 :
1. PC → Router-A
2. Router-A → Router-B (follows route)
3. Router-B → Router-A (follows route - LOOP!)
4. Router-A → Router-B
5. ... (infinite loop until TTL = 0)

→ Packet TTL decrements each hop
→ TTL reaches 0 → Packet dropped
→ ICMP "Time exceeded" sent source

// TTL prevents infinite loops

IP header TTL (Time To Live) :

Initial TTL : 64 (Linux), 128 (Windows), 255 (Cisco)
Each hop (router) : TTL - 1
TTL = 0 ? → Drop packet, send ICMP Time Exceeded

→ Prevents packets looping forever
→ Maximum hops = initial TTL value

// Symptômes routing loop

  • Ping fails : "Time exceeded" (TTL expired)
  • Traceroute shows loop : Same routers repeat
  • High CPU routers : Processing looping packets
  • Network congestion : Bandwidth consumed looping traffic

// Diagnostic routing loop

C:\> tracert 10.2.0.50

1    1 ms    192.168.1.1 (Router-A)
2    2 ms    10.0.0.1 (Router-B)
3    3 ms    192.168.1.1 (Router-A again - LOOP!)
4    4 ms    10.0.0.1 (Router-B again - LOOP!)
5    5 ms    192.168.1.1
...
30   * * *    Request timed out (TTL expired)

→ Same routers repeat = routing loop detected

// Causes routing loops

  • Static routes misconfigured : Manual error (Router-A → B, Router-B → A)
  • Dynamic routing protocol issue : Split horizon disabled, convergence slow
  • Redistribution misconfigured : Routes redistributed back original protocol
  • Default route pointing wrong : Default 0.0.0.0/0 loops back

// Résolution routing loop

Step 1 : Identify loop routers
→ Traceroute shows repeating hops

Step 2 : Check routing tables both routers
Router-A# show ip route 10.2.0.0
→ via Router-B

Router-B# show ip route 10.2.0.0
→ via Router-A (WRONG - creates loop)

Step 3 : Fix incorrect route
Router-B(config)# no ip route 10.2.0.0 255.255.255.0 Router-A
Router-B(config)# ip route 10.2.0.0 255.255.255.0

→ Break loop by correcting route
⚠️
Routing loop ≠ Switching loop (Layer 2) Routing loop (L3) : IP packets, routers, TTL prevents infinite.
Switching loop (L2) : Broadcast frames, switches, STP prevents.
Routing loops self-limiting (TTL), switching loops catastrophic (broadcast storm).
Silent discard

Black Hole — Packets disparaissent silencieusement

Black hole = Route exists, packets forwarded, MAIS destination silently discards (no ICMP error).

// Scénario black hole

Topology :
PC → Router-A → Router-B → [Black Hole]

Router-B configured :
ip route 10.2.0.0 255.255.255.0 Null0

→ Route to Null0 = discard interface
→ Packets sent Null0 = silently dropped (no error)

// Causes black holes

  • Route to Null0 : Intentional discard (summarization, security) OR misconfiguration
  • Firewall silent drop : Firewall blocks, no ICMP reply
  • Next-hop down : Route points downed interface, no alternative
  • Asymmetric routing : Forward path works, return path fails (covered next)

// Symptômes black hole

  • Ping timeout : No response (not "unreachable")
  • Traceroute shows path : Reaches certain point, then silence
  • No ICMP error : Unlike missing route (gets "unreachable")
  • One-way communication : Outbound works, inbound fails (OR vice versa)

// Diagnostic black hole

C:\> tracert 10.2.0.50

1    1 ms    192.168.1.1 (Router-A)
2    5 ms    10.0.0.1 (Router-B)
3    * * *    Request timed out (silence - black hole)

→ Reaches Router-B, then nothing

Router-B# show ip route 10.2.0.0
S    10.2.0.0/24 is directly connected, Null0

→ Route to Null0 = black hole found

// Résolution black hole

Router-B(config)# no ip route 10.2.0.0 255.255.255.0 Null0
Router-B(config)# ip route 10.2.0.0 255.255.255.0

→ Remove Null0 route, add correct route
Path asymmetry

Asymmetric Routing — Chemins différents aller/retour

Asymmetric routing = Outbound path ≠ Inbound path. Can cause problems stateful devices (firewalls, NAT).

// Principe asymmetric routing

Network topology :

PC-A → Router-1 → Router-Core → Server (outbound path)
PC-A ← Router-2 ← Router-Core ← Server (inbound path)

→ Outbound via Router-1
→ Inbound via Router-2
→ Asymmetric (different paths)

// Problèmes asymmetric routing

1. Stateful firewall drops

Firewall on Router-1 :
→ Sees outbound SYN (PC → Server) → Creates state entry
→ Expects inbound SYN-ACK via Router-1

BUT SYN-ACK returns via Router-2 (asymmetric)
→ Firewall Router-1 never sees return traffic
→ State entry times out → Connection fails

2. NAT translation broken

NAT on Router-1 :
→ Outbound : 192.168.1.50 → 203.0.113.10 (translated)
→ Creates NAT table entry

Inbound returns via Router-2 (no NAT)
→ Destination 203.0.113.10 → Router-2 can't translate (no table)
→ Packet dropped or misrouted

// Symptômes asymmetric routing

  • Connection timeouts : TCP handshake fails (SYN sent, SYN-ACK lost)
  • Intermittent connectivity : Works sometimes (symmetric), fails others (asymmetric)
  • Firewall logs : "Out-of-state packet" dropped
  • Traceroute different paths : Forward vs reverse show different hops

// Diagnostic asymmetric routing

Traceroute forward :
C:\> tracert 10.2.0.50
1 → 192.168.1.1 (Router-1)
2 → 10.0.0.1 (Router-Core)
3 → 10.2.0.50 (Server)

Traceroute reverse (from server) :
Server# traceroute 192.168.1.50
1 → 10.0.0.1 (Router-Core)
2 → 192.168.1.2 (Router-2 - different!)
3 → 192.168.1.50 (PC)

→ Asymmetric detected (outbound Router-1, inbound Router-2)

// Solutions asymmetric routing

  • Fix routing : Make paths symmetric (adjust metrics, prefer same path)
  • Disable stateful inspection : Firewall allow asymmetric (if safe)
  • Clustered firewalls : Firewalls share state (both paths see traffic)
  • Policy-based routing : Force symmetric paths (source-based routing)
🔀
Asymmetric routing = not always bad Simple routing (no firewalls/NAT) : Asymmetric OK (packets reach destination).
Stateful devices : Asymmetric problematic (state not synchronized).
Modern networks : Often asymmetric (Internet multiple paths), design accordingly.
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : Routing loop = infinite (pas de fin)
FAUX. Routing loop = self-limiting (TTL expires).

TTL (Time To Live) decrements each hop.
TTL = 0 → Packet dropped, ICMP Time Exceeded sent.

Maximum loop iterations = initial TTL (64, 128, 255).
Examen : Routing loops problematic BUT not infinite (TTL protection).
❌ Piège #2 : Black hole = route manquante
DIFFÉRENT. Black hole = route EXISTS, packets discarded silently.

Missing route : No entry routing table → ICMP "unreachable" (usually).
Black hole : Route to Null0 OR silent drop → No ICMP error.

Symptom différence : Missing route = "unreachable", Black hole = timeout silence.
Examen : Black hole harder troubleshoot (no error messages).
❌ Piège #3 : Asymmetric routing = always problem
NUANCE. Asymmetric routing = problematic IF stateful devices.

No firewall/NAT : Asymmetric OK (packets reach destination both directions).
Stateful firewall/NAT : Asymmetric breaks (state not synchronized paths).

Internet routing : Often asymmetric (multiple ISPs, optimized paths).
Examen : Context matters (stateful devices present ?).
Résumé

🔑 Ce que tu dois retenir absolument

→Route manquante : No entry routing table. Symptom: Traceroute stops, unreachable. Fix: Add static route OR enable dynamic routing protocol
→Route incorrecte : Route exists BUT wrong next-hop. Symptom: Route shows, connectivity fails. Diagnostic: Ping next-hop (unreachable?). Fix: Correct next-hop address
→Routing loop : Routers forward packet back-and-forth. TTL expires (self-limiting). Symptom: Traceroute repeats hops, "Time exceeded". Fix: Correct routes break loop
→Black hole : Route to Null0 OR silent discard. No ICMP error. Symptom: Timeout silence (not "unreachable"). Diagnostic: show ip route (Null0?). Fix: Remove Null0 route
→Asymmetric routing : Outbound path ≠ inbound path. Problems: Stateful firewall, NAT. Symptom: Connection fails, firewall drops. Fix: Symmetric paths, clustered firewalls, disable stateful if safe
→Commands : show ip route (table), show ip route X.X.X.X (specific), ping, traceroute/tracert, show ip interface brief (status). Static route: ip route network mask next-hop
