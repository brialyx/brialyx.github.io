Concepts

DoS vs DDoS — Déni de service

🚪
Analogie — restaurant submergé DoS : 1 personne appelle restaurant 1000 fois, ligne occupée → vrais clients pas réserver (single attacker).
DDoS : 10,000 personnes appellent simultanément → lignes saturées, impossible prendre réservations (distributed attack).

// DoS (Denial of Service)

Définition : Attaque 1 source → rendre service/réseau indisponible users légitimes.

Objectif : Saturer ressources (bande passante, CPU, mémoire, connexions) → crash ou lenteur extrême.

Caractéristiques :

  • Source unique (1 attaquant, 1 machine)
  • Plus facile bloquer (blacklist IP source)
  • Impact limité (bande passante 1 machine = ~1 Gbps max)

// DDoS (Distributed Denial of Service)

Définition : Attaque MULTIPLE sources coordonnées → amplifier impact.

Architecture botnet :

[Attacker Command & Control]
        │
        ├──> [Bot 1] ─┐
        ├──> [Bot 2] ─┤
        ├──> [Bot 3] ─┼──> Flood traffic ──> [Target Server]
        ├──> [Bot ...] ┤
        └──> [Bot 10,000] ─┘

→ Botnet = armée machines compromises (IoT, PCs malware)
→ Attaquant contrôle via C2 (Command & Control)
→ Volume combiné = 100+ Gbps (cible submergée)

Caractéristiques DDoS :

  • Sources multiples (10K-1M+ machines)
  • Difficile bloquer (blacklist 100K IPs = impraticable)
  • Impact massif (volume agrégé 1+ Tbps possible)
  • Mirai botnet (2016) : 600+ Gbps attaque Dyn DNS
📊
Tendances DDoS modernes Record DDoS : 3.47 Tbps (2024, Microsoft Azure).
Botnets IoT : Caméras, routers mal sécurisés = zombies parfaits (millions devices).
DDoS-for-hire : Services "stresser/booter" louent botnets ($10-$50/attaque).
Network layer

Attaques volumétriques — Saturation bande passante

Attaques volumétriques = Saturer bande passante réseau cible (Gbps/Tbps trafic flood).

// 1. UDP Flood

Principe : Envoyer masse paquets UDP vers ports cible.

Workflow :

  • Attaquant envoie UDP flood port 53 (DNS), 123 (NTP), ports aléatoires
  • Cible processus chaque paquet (check port, répond ICMP unreachable si fermé)
  • Volume trafic sature lien réseau (1+ Gbps)
  • Users légitimes = pas bande passante disponible

Caractéristique : UDP = connectionless (pas handshake, easy flood).

// 2. ICMP Flood (Ping Flood)

Principe : Submerger cible paquets ICMP Echo Request (ping).

Impact : Cible répond chaque ping (ICMP Reply) → CPU + bande passante consommés.

Mitigation : Rate-limit ICMP ou bloquer ICMP externe (firewall).

// 3. Ping of Death

Principe : Envoyer paquet ICMP > taille maximum autorisée (>65,535 bytes).

Technique : Fragmenter paquet géant → réassemblage cible = buffer overflow → crash.

État actuel : Obsolète (OS modernes patchés depuis 1990s) MAIS examen mentionne encore (histoire).

// 4. Smurf Attack

Principe : Amplification ICMP via broadcast.

Workflow :

  1. Attaquant envoie ICMP Echo Request broadcast address (192.168.1.255)
  2. Source IP = spoofé (IP victime)
  3. TOUS hosts subnet répondent ICMP Reply → victime (amplification 100×)
  4. Victime submergée réponses

Mitigation : Désactiver IP directed-broadcast routers (standard moderne).

Cisco IOS mitigation Smurf :

interface GigabitEthernet0/0
  no ip directed-broadcast

→ Bloque paquets broadcast externes
→ Empêche subnet être amplificateur Smurf
🎯
Question TRÈS fréquente examen "Attaque ICMP amplification via broadcast ?"

Réponse : Smurf attack

"Paquet ICMP > 65,535 bytes causant crash ?"
Réponse : Ping of Death
Reflection/Amplification

Attaques amplification — DNS, NTP

Amplification attack = Exploiter protocoles réflexifs (réponse > requête) → amplifier volume attaque.

// Principe amplification

  1. IP Spoofing : Attaquant spoof source IP (IP victime)
  2. Requête petite : Envoyer requête minuscule serveur public (ex: DNS query 60 bytes)
  3. Réponse massive : Serveur répond requête → victime (ex: DNS response 4000 bytes)
  4. Amplification : 60 bytes requête → 4000 bytes réponse = facteur 60×

// DNS Amplification

Cible : Serveurs DNS récursifs ouverts (open resolvers).

Workflow :

1. Attaquant query DNS : "ANY isc.org" (requête ALL records)
   → Source IP = spoofée (victime 203.0.113.50)
   → Destination = DNS resolver 8.8.8.8
   → Taille requête = 60 bytes

2. DNS resolver répond → 203.0.113.50 (victime)
   → Réponse ALL records = 4000+ bytes
   → Amplification = 60× - 100×

3. Attaquant répète 100,000× via botnet
   → Victime reçoit Gbps trafic DNS

Facteur amplification DNS : 28× - 60× typique (jusqu'à 100× avec réponses massives).

Mitigation :

  • DNS servers : Désactiver récursion externe (limiter trusted IPs)
  • Rate limiting : Limiter requêtes/sec par source IP
  • Response Rate Limiting (RRL) : RFC 7534, limite réponses identiques

// NTP Amplification

Cible : Serveurs NTP publics (UDP 123).

Commande exploitée : monlist (commande NTP retourne 600 dernières IPs clients).

Workflow :

1. Attaquant envoie : monlist request (8 bytes)
   → Source IP = spoofée (victime)
   → Destination = NTP server 206.x.x.x

2. NTP server répond liste 600 IPs (468 bytes)
   → Facteur amplification = 58×

3. Répétition via botnet → Tbps possible

Facteur amplification NTP : 58× - 200× (record).

Mitigation :

  • NTP servers : Désactiver monlist (commande obsolète, disabled NTPv4.2.7+)
  • Firewall : Bloquer NTP externe si pas nécessaire

TCP attacks

SYN Flood — Épuisement connexions

SYN Flood = Exploiter 3-way handshake TCP → saturer table connexions serveur.

// Rappel 3-way handshake normal

[Client] ──SYN──> [Server]
[Client] <──SYN-ACK─ [Server]
[Client] ──ACK──> [Server]
→ Connexion établie

// Attaque SYN Flood

1. Attaquant envoie masse SYN packets
  → Source IP = spoofées (IPs aléatoires)
  → Destination = Serveur cible
  → Volume = 10,000+ SYN/sec

2. Serveur répond SYN-ACK chaque requête
  → Alloue ressources (mémoire, entry table connexions)
  → Attend ACK final (timeout 75 sec défaut)

3. ACK jamais arrive (IPs spoofées invalides)
  → Connexions restent état "SYN_RECEIVED" (half-open)
  → Table connexions PLEINE (100K+ half-open)

4. Serveur épuise ressources
  → Plus d'entrées table disponibles
  → Refuse connexions légitimes (DoS réussi)

// Mitigation SYN Flood

SYN Cookies

  • Pas allocation ressources AVANT ACK final reçu
  • Encode infos connexion dans sequence number SYN-ACK
  • Vérifie ACK retour contient bon sequence → alors alloue
  • ✅ Défense efficace, activée OS modernes

Rate Limiting

  • Limiter SYN packets/sec acceptés
  • Exemple : Max 1000 SYN/sec par source IP

Timeout réduction

  • Réduire SYN_RECEIVED timeout (75 sec → 10 sec)
  • Libère ressources plus vite

Firewall / IPS

  • Proxy SYN (firewall complète handshake avant forward backend)
  • Détection pattern flood → blacklist source IPs
Layer 7 attacks

Attaques Application Layer — HTTP Flood, Slowloris

Attaques Layer 7 = Ciblent application (HTTP, DNS queries) vs réseau (volumétrique). Plus sophistiquées, difficiles détecter.

// HTTP Flood (Layer 7 DDoS)

Principe : Envoyer masse requêtes HTTP légitimes → épuiser ressources serveur web.

Caractéristiques :

  • Requêtes = valides (GET /index.php, POST /search)
  • Pas détectable simple firewall (traffic looks normal)
  • Cible pages coûteuses (database queries, calculs lourds)
  • Volume nécessaire = moins élevé que volumétrique (100 req/sec peut suffire)

Exemple :

GET /search.php?query=complexquery HTTP/1.1

→ search.php exécute requête DB complexe (5 sec CPU)
→ 100 requêtes simultanées = 500 sec CPU nécessaires
→ Serveur épuisé (queue requests, timeouts users)

Mitigation :

  • WAF (Web Application Firewall) : Rate-limit per IP, CAPTCHA challenges
  • CDN : Cloudflare, Akamai absorbent trafic (cache + filtre)
  • Bot detection : Distinguer humans vs bots (behavior analysis)

// Slowloris

Principe : Maintenir connexions HTTP ouvertes longtemps → épuiser slots connexions serveur.

Workflow :

  1. Attaquant ouvre connexion HTTP (SYN handshake complet)
  2. Envoie headers HTTP LENTEMENT (1 byte toutes 10 sec)
  3. Jamais complete requête (garde connexion pending)
  4. Répète 1000× → Tous slots connexions serveur occupés
  5. Users légitimes = refusés (no slots available)
Slowloris attack :

GET / HTTP/1.1
Host: target.com
[wait 10 seconds]
User-Agent: Mozilla
[wait 10 seconds]
Accept: */*
[wait 10 seconds]
...
[never send final \r\n\r\n]

→ Connexion reste ouverte indéfiniment
→ Serveur attend completion headers (timeout 300 sec défaut)

Mitigation :

  • Timeout agressif : Réduire header timeout (300 sec → 10 sec)
  • Reverse proxy : Nginx, HAProxy buffer requests (forward backend seulement si complets)
  • Connection limits : Limiter connexions simultanées per IP
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : DDoS = toujours volumétrique
FAUX. DDoS peut être volumétrique OU application layer.

Volumétrique : UDP flood, ICMP flood (Gbps bande passante)
Application : HTTP flood, Slowloris (requêtes valides, épuise serveur)

Layer 7 attacks = plus sophistiqués, difficiles mitiger
❌ Piège #2 : Smurf attack utilise TCP
FAUX. Smurf attack = ICMP (pas TCP).

Smurf : ICMP Echo Request broadcast → amplification réponses
SYN Flood : TCP SYN packets → épuiser table connexions

Protocols différents, attaques différentes
❌ Piège #3 : Amplification = toujours DNS
INCOMPLET. Amplification possible multiples protocoles.

DNS (28-100×), NTP (58-200×), SNMP (6-10×), CharGEN (358×), SSDP (30×)

Tout protocole UDP réponse > requête = amplification potential
Résumé

🔑 Ce que tu dois retenir absolument

→DoS vs DDoS : DoS = 1 source, DDoS = distributed (botnet 10K+ machines). DDoS volume = Gbps-Tbps, difficile bloquer (sources multiples)
→Volumétrique : UDP flood, ICMP flood, Ping of Death (paquet >65K bytes). Smurf = ICMP broadcast amplification (mitigation : no ip directed-broadcast)
→Amplification : DNS (28-100× ANY query), NTP (58-200× monlist), SNMP, CharGEN. IP spoofing + requête petite → réponse massive victime
→SYN Flood : Masse SYN packets spoofés → serveur alloue ressources, attend ACK jamais arrive → table connexions pleine. Mitigation : SYN cookies, rate limiting
→Application Layer : HTTP flood (requêtes valides épuisent serveur), Slowloris (connexions slow incomplete, occupent slots). Mitigation : WAF, CDN, timeouts réduits
