Framework

AAA — Authentication, Authorization, Accounting

🎫
Analogie — entrée concert Authentication = Vérifier billet (qui es-tu ?). Authorization = Billet VIP vs général (que peux-tu faire ?). Accounting = Scanner billet (tracking qui entre, quand, combien temps reste).

AAA = Framework sécurité réseau contrôlant accès ressources. 3 composants.

// 1. Authentication (Authentification)

Question : "Qui es-tu ?"

Définition : Vérifier identité user/device.

Méthodes :

  • Username/Password : Basique
  • Certificates (PKI) : X.509 certificates
  • Biométrie : Empreinte digitale, reconnaissance faciale
  • MFA : Multi-Factor Authentication (password + code SMS)

// 2. Authorization (Autorisation)

Question : "Que peux-tu faire ?"

Définition : Déterminer permissions user APRÈS authentification réussie.

Exemples :

  • Admin : Accès full config (show + configure)
  • Network operator : Read-only (show commands uniquement)
  • Help desk : Reset passwords, pas modifier config
  • VLAN assignment : User VLAN 10 (data), contractor VLAN 40 (guest)

// 3. Accounting (Comptabilité)

Question : "Qu'as-tu fait ?"

Définition : Tracking/logging actions users.

Données collectées :

  • Login/logout times
  • Commandes exécutées (audit trail)
  • Bande passante consommée
  • Ressources accédées
  • Échecs authentification (tentatives intrusion)

Usage accounting :

  • Forensics (qui changé config hier 03:00 ?)
  • Compliance audits (PCI-DSS, HIPAA)
  • Billing (ISP facture usage)
🔐
AAA = fondation sécurité réseau Authentication seule = insuffisant (qui a fait quoi ?). Authorization = principe least privilege. Accounting = audit trail compliance. AAA complet = sécurité + traçabilité.
Protocol

RADIUS — Remote Authentication Dial-In User Service

RADIUS = Protocole AAA client-server centralisé (RFC 2865). Standard industry authentification réseau.

// Architecture RADIUS

[User] ──> [NAS: Switch/Router/AP] ──> [RADIUS Server]
                                              │
                                            [User DB]

→ NAS (Network Access Server) = RADIUS client
→ RADIUS server = Base users centralisée
→ Users DB = AD, LDAP, local database

// Workflow RADIUS

  1. User login switch (SSH ou 802.1X)
  2. Switch (NAS) envoie credentials → RADIUS server
  3. RADIUS server vérifie username/password DB
  4. RADIUS répond : Access-Accept (OK) ou Access-Reject (NOK)
  5. Si Accept : RADIUS envoie attributs authorization (VLAN, ACL, permissions)
  6. Switch applique policies, autorise accès
  7. Accounting : RADIUS logs session (start/stop, commandes, duration)

// Ports & Protocole

UDP 1812 : Authentication & Authorization

UDP 1813 : Accounting

Legacy ports : 1645 (auth), 1646 (acct) — obsolète, pas utiliser

UDP = Léger, pas overhead TCP. Timeout/retransmit si no response.

// Chiffrement RADIUS

Shared secret : Password partagé NAS ↔ RADIUS server.

Password user : Chiffré MD5 (pas texte clair).

Reste message : Texte clair (pas chiffré complet).

⚠️ Sécurité RADIUS faible : Attributes pas chiffrés → RADIUS over IPsec/TLS recommandé (RadSec).

// RADIUS Attributes

RADIUS transmet infos via attributes (AVPs - Attribute-Value Pairs).

Attributes courants :

  • User-Name : Username
  • User-Password : Password (chiffré MD5)
  • NAS-IP-Address : IP switch/router
  • Tunnel-Type / Tunnel-Private-Group-ID : VLAN assignment dynamique
  • Filter-Id : ACL à appliquer
  • Session-Timeout : Durée session max
🎯
Question TRÈS fréquente examen "Protocole authentification centralisée, ports UDP 1812/1813 ?"

Réponse : RADIUS

"Port RADIUS authentication ?"
Réponse : UDP 1812
Cisco protocol

TACACS+ — Terminal Access Controller Access-Control System Plus

TACACS+ = Protocole AAA Cisco propriétaire. Alternative RADIUS, focus device administration (switches/routers management).

// TACACS+ vs RADIUS

CritèreRADIUSTACACS+
VendorStandard (multi-vendor)Cisco propriétaire
ProtocoleUDP 1812/1813TCP 49
AAA⚠️ Authentication + Authorization combinés✅ AAA séparés (granulaire)
Chiffrement⚠️ Password seul (MD5)✅ Payload COMPLET chiffré
Usage principalNetwork access (802.1X, VPN, WiFi)Device administration (SSH switches/routers)
Command authorization❌ Non✅ Oui (per-command)

// Avantages TACACS+

1. Chiffrement complet : Payload entier chiffré (pas juste password).

2. AAA séparés : Authentication, Authorization, Accounting = processus indépendants → granularité fine.

3. Command authorization : Contrôler QUELLES commandes user peut exécuter.

Exemple command authorization :

User "helpdesk" :
- Autorisé : show commands (show run, show ip route)
- Refusé : configure terminal, reload, copy

→ TACACS+ vérifie CHAQUE commande exécutée
→ Autorise/refuse basé policy server
→ Accounting logs toutes commandes

// TCP vs UDP

TACACS+ = TCP 49 (vs RADIUS UDP).

Avantages TCP :

  • ✅ Fiabilité (acknowledgments, retransmission)
  • ✅ Pas timeout issues (UDP RADIUS timeout problématique)

// Usage TACACS+

Device administration SSH : Admin SSH switch → TACACS+ authentification → command authorization.

Privilege levels : Cisco privilege 0-15 mappés TACACS+ policies.

🔒
Quand utiliser RADIUS vs TACACS+ ? RADIUS : Network access (802.1X ports, VPN users, WiFi), multi-vendor
TACACS+ : Device management (SSH admin switches/routers), command authorization granulaire, Cisco environment
Port-based NAC

802.1X — Port-Based Network Access Control

802.1X = Standard IEEE authentification port-based. Empêche accès réseau AVANT authentification réussie.

// Architecture 802.1X

3 composants :

1. Supplicant : Client demandant accès (PC, laptop, téléphone).

2. Authenticator : Switch/AP contrôlant port (proxy RADIUS).

3. Authentication Server : RADIUS server vérifiant credentials.

[Supplicant: Laptop] ─────> [Authenticator: Switch] ─────> [Auth Server: RADIUS]
     EAP (802.1X)                         RADIUS

Workflow :
1. Laptop branché port switch
2. Port = état UNAUTHORIZED (bloqué)
3. Laptop envoie EAP-Start
4. Switch demande identity (EAP-Request-Identity)
5. Laptop répond username (EAP-Response-Identity)
6. Switch forward credentials → RADIUS (RADIUS Access-Request)
7. RADIUS vérifie → Access-Accept ou Access-Reject
8. Si Accept : Port → AUTHORIZED (accès réseau)
   Si Reject : Port reste UNAUTHORIZED (bloqué)

// EAP (Extensible Authentication Protocol)

EAP = Framework authentification 802.1X. Multiples méthodes EAP.

EAP methods courants :

EAP-TLS

  • Utilise certificates PKI (client + server)
  • ✅ Sécurité maximale (mutual authentication)
  • ⚠️ Complexité (PKI infrastructure nécessaire)

EAP-TTLS / PEAP

  • Tunnel TLS sécurisé (server certificate uniquement)
  • Inside tunnel : Username/password (MSCHAPv2)
  • ✅ Balance sécurité/simplicité (password + TLS)
  • Usage : Corporate WiFi

EAP-FAST (Cisco)

  • Pas certificate (PAC - Protected Access Credential)
  • ✅ Simple déploiement
  • Cisco propriétaire

// Usage 802.1X

  • Wired networks : Ports switches (empêche rogue devices)
  • WiFi WPA2/WPA3-Enterprise : SSID corporate (vs WPA2-PSK shared password)
  • Dynamic VLAN assignment : RADIUS attribue VLAN basé user/device
🔐
802.1X = sécurité enterprise (pas home) Home WiFi = WPA2-PSK (shared password simple).
Enterprise = WPA2/WPA3-Enterprise (802.1X + RADIUS + EAP).
802.1X empêche rogue devices (laptop non autorisé branché switch = bloqué).
Access control

NAC — Network Access Control

NAC (Network Access Control) = Solution sécurité vérifiant conformité devices AVANT accès réseau (posture assessment).

// Principe NAC

Question : Device conforme policies sécurité ?

Checks NAC :

  • Antivirus : Installé ? Up-to-date ?
  • OS patches : Système jour ?
  • Firewall : Activé ?
  • Encryption : Disque chiffré ?
  • Device type : Corporate laptop vs BYOD ?

// Workflow NAC

  1. Device branché réseau (ou connexion WiFi)
  2. 802.1X authentication
  3. NAC posture assessment : Agent scanne device compliance
  4. Policy evaluation :
    - Compliant → Full network access (VLAN production)
    - Non-compliant → Quarantine VLAN (accès limité update servers)
    - Unknown/Guest → Guest VLAN (Internet only)
  5. Device remediation si nécessaire (install patches, enable firewall)
  6. Re-assessment → Full access si compliant

// Modes NAC

Agent-based : Software agent installé device (deep inspection).

Agentless : Scan réseau passif (BYOD, guests, IoT devices).

// Solutions NAC

Cisco ISE (Identity Services Engine) : Leader marché NAC enterprise.

Aruba ClearPass : NAC multi-vendor (HP).

ForeScout : NAC agentless (visibility IoT/OT devices).

Microsoft NAP : Network Access Protection (Windows-centric, deprecated).

Exemple NAC policy :

Compliant device :
- Antivirus = updated
- OS patches = current
- Firewall = enabled
→ Access : VLAN 10 (Production, full access)

Non-compliant device :
- Antivirus = outdated (3 months)
→ Access : VLAN 99 (Quarantine, update servers only)
→ Remediation : Update antivirus → Re-scan

Guest device :
- Unknown device (BYOD, visitor)
→ Access : VLAN 40 (Guest, Internet only, no internal)
🛡️
NAC = défense Zero Trust "Trust but verify" obsolète → "Never trust, always verify".
NAC vérifie TOUS devices (internal employees included).
Empêche malware spread (device compromis isolé quarantine immédiatement).

INCOMPLET. 802.1X = framework port-based access.

- EAP (authentification method)
- RADIUS (AAA server backend)

802.1X seul ≠ suffit → besoin EAP + RADIUS
