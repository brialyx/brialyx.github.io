High Availability

Haute disponibilité — Qu'est-ce que c'est ?

🏥
Analogie — hôpital vs cabinet médecin Cabinet médecin : Ouvert 9h-17h, 1 médecin. Médecin malade = cabinet fermé (pas HA). Hôpital : Ouvert 24/7, équipes multiples, équipements redondants. 1 médecin malade = hôpital fonctionne quand même (HA).

Haute disponibilité (HA - High Availability) = Architecture système minimisant downtime, maximisant uptime. Service accessible TOUJOURS (ou presque).

// Principes HA

  • Redondance : Composants multiples (2 routeurs, 2 liens WAN, 2 alimentations)
  • Élimination SPOF : Single Point of Failure (1 routeur = SPOF, 2 routeurs = pas SPOF)
  • Failover automatique : Basculement automatique si panne (secondes-minutes)
  • Monitoring : Détection pannes rapide
  • Maintenance sans downtime : Upgrade/patch sans arrêter service

Métriques fiabilité

MTBF et MTTR

// MTBF (Mean Time Between Failures)

Temps MOYEN entre pannes. Mesure fiabilité composant.

Formule :

MTBF = Temps total opérationnel / Nombre pannes

Exemple :
Disque dur fonctionne 50,000 heures avant panne
MTBF = 50,000 heures (≈ 5.7 ans)

→ Plus MTBF élevé = Plus fiable

Usage : Fabricants spécifient MTBF produits (disques, switches, alimentations).

Exemple : Switch Cisco MTBF 200,000 heures = Panne attendue tous les 22 ans (statistiquement).

// MTTR (Mean Time To Repair)

Temps MOYEN réparation. Combien temps restaurer service après panne ?

Formule :

MTTR = Temps total downtime / Nombre pannes

Exemple :
3 pannes année :
- Panne 1 : Réparation 2 heures
- Panne 2 : Réparation 4 heures
- Panne 3 : Réparation 1 heure

MTTR = (2 + 4 + 1) / 3 = 2.33 heures

→ Plus MTTR faible = Meilleur (réparation rapide)

Facteurs MTTR :

  • Temps détection panne (monitoring)
  • Temps diagnostic (troubleshooting)
  • Disponibilité pièces rechange (stock)
  • Compétence techniciens
  • Documentation/procédures

// Disponibilité calculée

Availability = MTBF / (MTBF + MTTR)

Exemple calcul :

Serveur :
MTBF = 8,760 heures (1 an)
MTTR = 2 heures

Availability = 8760 / (8760 + 2)
= 8760 / 8762
= 0.9997 = 99.97%

→ Uptime attendu : 99.97%
📈
Améliorer disponibilité Augmenter MTBF : Équipements qualité, maintenance préventive
Réduire MTTR : Monitoring proactif, pièces rechange stock, procédures claires, failover automatique
Disaster Recovery

RTO et RPO

// RTO (Recovery Time Objective)

Temps MAXIMUM tolérable avant restauration service. Combien temps downtime acceptable ?

Exemple :

  • E-commerce : RTO = 1 heure (perte revenus énorme si down >1h)
  • Système comptabilité interne : RTO = 24 heures (moins critique)
  • Site web vitrine : RTO = 4 heures (impact modéré)

RTO dicte architecture :

  • RTO 1 heure → Failover automatique requis (HSRP, cluster actif/passif)
  • RTO 24 heures → Backup/restore manuel acceptable

// RPO (Recovery Point Objective)

Perte données MAXIMUM tolérable. Combien données perdues acceptable ?

Exemple :

  • Base données financières : RPO = 0 (zéro perte données acceptable)
  • CRM : RPO = 1 heure (perte 1h données transactions OK)
  • Logs système : RPO = 24 heures (perte 1 jour logs acceptable)

RPO dicte stratégie backup :

  • RPO 0 → Réplication synchrone temps réel (coûteux)
  • RPO 1 heure → Backup incrémental toutes les heures
  • RPO 24 heures → Backup quotidien suffisant

// RTO vs RPO — Différence

Timeline incident :

[12:00] Dernier backup
[14:00] ← INCIDENT (serveur crash)
↓
[Downtime...]
↓
[16:00] Service restauré

RTO = 2 heures (14:00→16:00 downtime)
RPO = 2 heures (12:00→14:00 données perdues)

→ RTO = Combien temps down ?
→ RPO = Combien données perdues ?
💰
RTO/RPO faibles = coûts élevés RTO 5 min + RPO 0 = Architecture redondante complexe + réplication temps réel = $$$$
RTO 24h + RPO 24h = Backup daily simple = $

Balance : Exigences business vs budget
Service Level

SLA — Service Level Agreement

SLA (Service Level Agreement) = Contrat garantissant niveau service minimum. ISP, cloud providers, fournisseurs services garantissent uptime.

// Contenu SLA typique

  • Uptime garanti : 99.9%, 99.99%, 99.999%
  • Pénalités : Remboursement si SLA non respecté
  • Fenêtres maintenance : Downtime planifié exclu SLA
  • Support : Temps réponse incidents (4h, 24h, etc.)
  • Exclusions : Force majeure, attaques DDoS, etc.

// Exemple SLA cloud (AWS EC2)

AWS EC2 SLA :

Uptime garanti région : 99.99%

Pénalités :
- Uptime 99.0-99.99% : Crédit 10%
- Uptime 95.0-99.0% : Crédit 25%
- Uptime <95% : Crédit 100%

→ Si AWS downtime >52 min/mois → Client reçoit crédit service

// SLO vs SLA vs SLI

SLA (Service Level Agreement) : Contrat avec pénalités.

SLO (Service Level Objective) : Objectif interne équipe (plus strict que SLA).

SLI (Service Level Indicator) : Métrique mesurée (uptime réel, latence...).

Exemple :

SLA : Garantie client 99.9% uptime
SLO : Objectif équipe interne 99.95% uptime (buffer)
SLI : Mesure réelle mois dernier 99.97% uptime

→ SLI > SLO > SLA = OK (marge sécurité)
Calculs pratiques

Calculs uptime — Formules examen

// Calcul downtime autorisé

Formule : Downtime = Période × (1 - Uptime%)

Exemple 1 : SLA 99.9% annuel

Downtime max/an = 365 jours × (1 - 0.999)
= 365 × 0.001
= 0.365 jours
= 8.76 heures/an

Exemple 2 : SLA 99.99% mensuel

Downtime max/mois = 30 jours × (1 - 0.9999)
= 30 × 0.0001
= 0.003 jours
= 4.32 minutes/mois

// Calcul uptime depuis pannes

Formule : Uptime% = (Temps total - Downtime) / Temps total × 100

Exemple : Service down 10 heures sur année

Temps total = 365 × 24 = 8760 heures/an
Downtime = 10 heures

Uptime% = (8760 - 10) / 8760 × 100
= 8750 / 8760 × 100
= 99.886%

⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : MTBF plus élevé = pire
FAUX. MTBF élevé = MEILLEUR.

MTBF 10,000h = Panne tous les 10,000h (médiocre)
MTBF 100,000h = Panne tous les 100,000h (excellent)

Plus MTBF grand = Plus fiable
❌ Piège #2 : RTO = perte données
CONFUSION. RTO ≠ perte données.

RTO : Temps downtime maximum tolérable
RPO : Perte données maximum tolérable

RTO = Time to recover, RPO = Point in time recover
❌ Piège #3 : 99% uptime = bon SLA
MÉDIOCRE. 99% uptime = 3.65 jours downtime/an.

Services critiques : 99.9% minimum (9h/an)
Haute dispo : 99.99% ("four nines", 1h/an)
Mission-critical : 99.999% ("five nines", 5 min/an)

99% inacceptable production moderne
Résumé

🔑 Ce que tu dois retenir absolument

→MTBF : Mean Time Between Failures (temps moyen entre pannes). Plus MTBF élevé = Plus fiable. MTTR : Mean Time To Repair (temps moyen réparation). Plus MTTR faible = Meilleur
→RTO : Recovery Time Objective (downtime max tolérable). Dicte architecture failover. RPO : Recovery Point Objective (perte données max). Dicte fréquence backup
→Uptime : 99.9% = 8.76h/an, 99.99% = 52 min/an, 99.999% ("five nines") = 5.26 min/an. Calcul : Downtime = Période × (1 - Uptime%)
→SLA : Service Level Agreement (contrat uptime + pénalités). SLO : Objectif interne (>SLA). SLI : Mesure réelle
→HA principes : Redondance composants, élimination SPOF, failover automatique, monitoring proactif. Availability = MTBF / (MTBF + MTTR)
