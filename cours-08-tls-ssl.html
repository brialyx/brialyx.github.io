Protocol overview

SSL vs TLS — Protocoles chiffrement

🔒
Analogie — conversation téléphone crypté HTTP = conversation téléphone standard (écoutable). HTTPS = conversation téléphone crypté militaire (chiffré end-to-end). TLS/SSL = protocole chiffrement sécurise conversation.

SSL (Secure Sockets Layer) = Protocole chiffrement legacy (1995-2015), développé Netscape.

TLS (Transport Layer Security) = Successeur SSL, standard IETF (1999-présent).

⚠️ Terminologie courante : "SSL" utilisé générique (ex: "SSL certificate") MAIS techniquement TLS moderne. SSL protocol = mort 2015.

// Objectifs TLS/SSL

  • Confidentiality : Chiffrement données transit (eavesdropping impossible)
  • Integrity : Détection modifications (tampering → MAC verification fail)
  • Authentication : Vérification identité server (certificat PKI)
🌐
TLS = backbone sécurité Internet HTTPS (HTTP over TLS), SMTPS (email), FTPS, VPN, APIs → TOUT repose TLS.
70%+ trafic web = HTTPS (Google Chrome force 2024).
TLS = infrastructure critique invisible.
Connection establishment

TLS 1.2 Handshake — Établissement connexion

TLS Handshake = Négociation initiale client ↔ server avant données chiffrées. Objectif : Authentifier server, négocier cipher suite, établir session keys.

// Workflow TLS 1.2 (6 étapes, 2 RTT)

1. Client Hello
Client → Server :
- TLS version supported (1.2, 1.3)
- Cipher suites list (AES-GCM, ChaCha20)
- Random nonce (client_random)
- Extensions (SNI, ALPN)

2. Server Hello
Server → Client :
- TLS version selected (1.2)
- Cipher suite chosen (TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256)
- Random nonce (server_random)
- Certificate (server cert + chain)
- Server Key Exchange (ECDHE public key)
- Server Hello Done

3. Client Key Exchange
Client → Server :
- Verify certificate (PKI chain, signature, expiration)
- Generate pre-master secret
- Encrypt pre-master with server public key (RSA) OU
- Generate ECDHE public key → send server
- Change Cipher Spec (switch to encrypted)
- Finished (encrypted, MAC verify)

4. Server Finished
Server → Client :
- Derive session keys (master_secret from pre-master + randoms)
- Change Cipher Spec
- Finished (encrypted, MAC verify)

5. Application Data
→ Connexion établie (symmetric encryption AES-GCM)
→ HTTP requests/responses chiffrés

// Détails clés handshake

Cipher Suite negotiation

  • Client propose liste cipher suites (ordre préférence)
  • Server choisit premier supporté (TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256)
  • Format : KeyExchange_Authentication_WITH_Encryption_MAC

Session keys derivation

  • Pre-master secret (48 bytes random)
  • Master secret = PRF(pre-master, client_random, server_random)
  • Session keys = PRF(master_secret) → Encryption key + MAC key (client/server)

RTT (Round Trip Time)

  • TLS 1.2 = 2 RTT avant application data (latency notable)
  • Handshake + data = 3 RTT total (TCP handshake 1 RTT + TLS 2 RTT)
🎯
Question fréquente examen "Première étape TLS handshake envoyée client ?"
Réponse : Client Hello

"Message server contenant certificat PKI ?"
Réponse : Server Hello (+ Certificate message)
Modern TLS

TLS 1.3 — Améliorations performance & sécurité

TLS 1.3 (RFC 8446, 2018) = Révision majeure TLS. Plus rapide, plus sécurisé, simplifié.

// Différences TLS 1.2 vs 1.3

1. Handshake rapide (1-RTT)

TLS 1.2 : 2-RTT
Client → Server : ClientHello
Server → Client : ServerHello + Certificate + KeyExchange
Client → Server : KeyExchange + Finished
Server → Client : Finished
→ Application Data (after 2 RTT)

TLS 1.3 : 1-RTT
Client → Server : ClientHello + KeyShare (guess cipher)
Server → Client : ServerHello + Certificate + KeyShare + Finished
→ Application Data possible immediately (1 RTT)

2. 0-RTT Resumption

  • Connexions suivantes = 0-RTT (application data FIRST packet)
  • Client envoie encrypted data immédiatement (PSK from previous session)
  • ⚠️ Replay attack risk (mitigation required application-level)

3. Cipher suites simplifiées

  • TLS 1.2 : 37 cipher suites (many weak)
  • TLS 1.3 : 5 cipher suites uniquement (all strong)
  • Obligatoire : AEAD (Authenticated Encryption with Associated Data)
  • Exemples : TLS_AES_128_GCM_SHA256, TLS_CHACHA20_POLY1305_SHA256

4. Algorithmes faibles retirés

  • ❌ RSA key exchange (pas forward secrecy)
  • ❌ Static DH (pas forward secrecy)
  • ❌ RC4, 3DES, MD5, SHA-1 (vulnérables)
  • ✅ Obligatoire : ECDHE ou DHE (forward secrecy)

5. Handshake chiffré

  • TLS 1.2 : Certificate + extensions = plaintext (metadata leak)
  • TLS 1.3 : Tout après ServerHello chiffré (including certificate)
  • ✅ Privacy amélioration (SNI toujours plaintext, encrypted SNI in progress)

Handshake RTT2-RTT1-RTT (0-RTT resumption)
Cipher suites37 options (many weak)5 options (all strong AEAD)

⚡
TLS 1.3 = recommandé 2024+ Browsers modernes : TLS 1.3 par défaut (Chrome, Firefox, Safari).
Servers : Nginx, Apache supportent TLS 1.3.
Migration : Enable TLS 1.3, garde TLS 1.2 fallback (compatibility).
Cryptographic algorithms

Cipher Suites — Algorithmes cryptographiques

Cipher Suite = Ensemble algorithmes crypto utilisés connexion TLS (key exchange, authentication, encryption, MAC).

// Format Cipher Suite (TLS 1.2)

TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256

  • TLS : Protocol
  • ECDHE : Key Exchange (Elliptic Curve Diffie-Hellman Ephemeral)
  • RSA : Authentication (certificat server RSA)
  • AES_128_GCM : Encryption (AES 128-bit Galois/Counter Mode)
  • SHA256 : MAC/Hash (SHA-256)

// Composants cipher suite

Key Exchange

  • RSA : Chiffre pre-master avec server public key (⚠️ pas forward secrecy)
  • DHE : Diffie-Hellman Ephemeral (✅ forward secrecy)
  • ECDHE : Elliptic Curve DHE (✅ forward secrecy, faster)

Authentication

  • RSA : Certificat RSA (2048-4096 bits)
  • ECDSA : Certificat ECC (256-384 bits, faster)

Encryption

  • AES-128-GCM : AES 128-bit Galois/Counter Mode (AEAD)
  • AES-256-GCM : AES 256-bit (stronger)
  • ChaCha20-Poly1305 : Stream cipher (mobile optimized, Google/Cloudflare)

MAC (Message Authentication Code)

  • SHA256 : Hash function integrity
  • SHA384 : Stronger hash
  • ⚠️ GCM/Poly1305 = AEAD (integrated MAC, pas séparé)

// Forward Secrecy

Définition : Compromission private key server PAS compromet sessions passées.

Sans Forward Secrecy (RSA key exchange) :

  • Attaquant capture trafic chiffré (2020)
  • Attaquant compromet server private key (2024)
  • Attaquant déchiffre TOUS trafics 2020-2024 (catastrophe)

Avec Forward Secrecy (ECDHE) :

  • Chaque session = ephemeral keys (jetables)
  • Compromission server key 2024 → UNIQUEMENT sessions 2024+ affectées
  • Trafics 2020-2023 = sécurisés (keys destroyed)
🔐
Forward Secrecy = obligatoire moderne TLS 1.3 : Forward secrecy REQUIRED (ECDHE/DHE mandatory).
TLS 1.2 : Préférer cipher suites ECDHE (avoid RSA key exchange).
Configuration : Disable RSA key exchange cipher suites.
Application protocols

HTTPS & SNI — HTTP sécurisé

// HTTPS (HTTP Secure)

HTTPS = HTTP over TLS (port 443). Chiffre tout trafic web (requests, responses, headers, cookies).

Workflow HTTPS :

  1. Browser connexion TCP port 443
  2. TLS handshake (certificate verification, session keys)
  3. HTTP requests/responses chiffrés TLS
  4. Browser affiche padlock 🔒 (indicates HTTPS)

Avantages HTTPS :

  • ✅ Confidentiality : Passwords, credit cards, cookies chiffrés
  • ✅ Integrity : Tampering détecté (MitM impossible)
  • ✅ Authentication : Certificat prouve identité server
  • ✅ SEO : Google ranking boost HTTPS sites (2014+)
  • ✅ Features : HTTP/2, Service Workers = HTTPS-only

// SNI (Server Name Indication)

Problème résolu : 1 IP multiple domains (virtual hosting) → quel certificat envoyer ?

Scénario sans SNI :

Server 192.0.2.100 hosts :
- example.com
- test.com
- demo.org

Client connects 192.0.2.100:443
→ Server doesn't know which domain client wants
→ Cannot send correct certificate (3 domains = 3 certs)
→ Certificate mismatch error

Solution SNI :

Client → Server (TLS ClientHello) :
Extension: SNI = "example.com"

Server :
→ Reads SNI extension
→ Knows client wants "example.com"
→ Sends correct certificate (example.com cert)
→ TLS handshake succeeds

SNI = extension TLS (RFC 6066) :

  • Client envoie hostname plaintext ClientHello (server_name extension)
  • Server sélectionne certificat approprié
  • ✅ Permet virtual hosting HTTPS (1 IP, multiples domains)
  • ⚠️ SNI plaintext = privacy leak (ISP voit hostname visité)

Encrypted SNI (ESNI → ECH) :

  • Chiffre SNI (privacy amélioration)
  • Draft standard (adoption progressive Cloudflare, Firefox)
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : SSL = TLS (synonymes exacts)
FAUX techniquement. SSL = protocole obsolète (SSL 2.0/3.0 deprecated).
TLS = successeur moderne.

MAIS usage courant : "SSL certificate" = terme générique (techniquement TLS cert).
Examen : Accepte "SSL/TLS" interchangeable contexte général.
Précis : TLS 1.2/1.3 = standard actuel (pas SSL)
❌ Piège #2 : TLS handshake = 1 RTT toujours
FAUX. Dépend version TLS.

TLS 1.2 : 2-RTT handshake (legacy)
TLS 1.3 : 1-RTT handshake (moderne)
TLS 1.3 resumption : 0-RTT (connexions suivantes)

Performance : TLS 1.3 = 30-40% faster
❌ Piège #3 : Cipher suite = encryption algorithm uniquement
INCOMPLET. Cipher suite = ENSEMBLE algorithmes :

1. Key Exchange (ECDHE, DHE, RSA)
2. Authentication (RSA, ECDSA)
3. Encryption (AES-GCM, ChaCha20)
4. MAC/Hash (SHA256, SHA384)

Exemple : TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256
