Configuration error

Duplex Mismatch — Problème #1 production

Duplex Mismatch = 2 devices connexion Ethernet configurés duplex modes DIFFÉRENTS (un Full-duplex, autre Half-duplex).

📞
Analogie — conversation téléphonique Full-duplex = conversation normale (parler + écouter simultanément).
Half-duplex = walkie-talkie (parler OU écouter, pas simultanément).
Mismatch = Personne A pense phone normal, Personne B pense walkie-talkie → Confusion totale.

// Half-Duplex vs Full-Duplex (rappel)

Half-Duplex :
→ One direction at a time (send OR receive)
→ Shared medium (collisions possible)
→ CSMA/CD enabled (collision detection)
→ Performance ~50-60% theoretical max
→ Legacy (hubs, old NICs)

Full-Duplex :
→ Both directions simultaneously (send AND receive)
→ Dedicated TX/RX paths (no collisions)
→ CSMA/CD disabled (unnecessary)
→ Performance ~95%+ theoretical max
→ Modern standard (switches)

// Pourquoi mismatch problématique ?

Comportement asymétrique : Device Full-duplex transmet librement (pas collision detection). Device Half-duplex détecte "collisions" (overlap TX/RX simultané) → retransmissions → performance dégradée.

⚠️
Duplex mismatch = erreur configuration #1 Cause principale : Manual config incohérente OU auto-negotiation failed.
Symptômes frustrants : Link UP (green LED) MAIS performance terrible.
"Internet lent" user reports → Souvent duplex mismatch root cause.
Symptoms

Symptômes Duplex Mismatch — Signature caractéristique

// Symptôme #1 : Late Collisions

Late collisions = RED FLAG #1 duplex mismatch (90%+ cause).

Scénario :

Switch port Gi0/1 : Full-duplex
Server NIC : Half-duplex

Server transmits frame → Half-duplex logic (listen first, CSMA/CD)
Switch transmits simultaneously → Full-duplex allows (no collision detection)
Server detects collision LATE (after 64 bytes) → Late collision

Switch# show interfaces Gi0/1
→ 850 late collisions (incrementing rapidly)

// Symptôme #2 : Performance dégradée

Throughput terrible : Link 1 Gbps négocié, MAIS débit réel 10-50 Mbps (90-95% perte performance).

  • File transfer lent (MB/s au lieu GB/s)
  • Ping latency élevée (50-200 ms au lieu <1 ms)
  • Application timeouts (database queries, web requests)

// Symptôme #3 : CRC Errors

CRC errors : Retransmissions multiples → frames corruption.

Switch# show interfaces Gi0/1
→ 1250 input errors, 1100 CRC, 150 frame
→ Errors incrementing 20-50/min

// Symptôme #4 : Runts

Runts (frames < 64 bytes) : Collision fragments (half-duplex side).

// Symptôme Trinity Duplex Mismatch

🚩 Signature classique :

1. Late collisions (hundreds-thousands)
2. CRC errors (incrementing rapidly)
3. Poor throughput (90% perte performance)

→ Diagnosis = Duplex mismatch (99% probability)
🎯
Question TRÈS fréquente examen "Network interface shows late collisions + CRC errors + slow performance. MOST likely cause ?"
A) Cable defect
B) Duplex mismatch
C) Broadcast storm
D) DNS failure

Réponse : B) Duplex mismatch
Trinity symptoms = signature duplex mismatch.
Root causes

Causes Duplex Mismatch — Comment arrive ?

// Cause #1 : Manual configuration incohérente

Scénario : Admin configure duplex manually, oublie configure other end OU configure wrong value.

Example erreur :

Switch port Gi0/1 :
interface GigabitEthernet0/1
  duplex full
  speed 1000

Server NIC : Auto-negotiation (default)
→ Auto-neg one side + Manual other side = Auto-neg FAILS
→ Server falls back half-duplex (IEEE 802.3 standard fallback)

→ Result : Switch Full ↔ Server Half = MISMATCH

// Cause #2 : Auto-negotiation failure

Auto-negotiation = IEEE 802.3u standard négocier duplex + speed automatiquement. Requires BOTH ends support.

Failure scenarios :

  • One side manual, other auto (auto-neg disabled)
  • Faulty NIC (corrupted auto-neg signals)
  • Bad cable (signal degradation, incomplete negotiation)
  • Legacy device (no auto-neg support)

// Cause #3 : Speed/Duplex fallback incorrect

IEEE 802.3 fallback rules : Si auto-negotiation fails →

Fallback logic :

1. Detect link speed (sensing electrical signals)
   → 10/100/1000 Mbps

2. Assume half-duplex (conservative safe default)
   → Half-duplex = no collisions if wrong
   → Full-duplex assumed wrongly = collisions

Example :
Switch manual Full, 1000 Mbps
Server auto-neg FAILS → Detects 1000 Mbps → Assumes Half-duplex
→ Mismatch

// Cause #4 : Configuration drift

Drift over time : Network changes incrementally (NIC replacement, switch upgrade) → duplex settings pas synchronized.

  • Server NIC replaced → New NIC defaults auto, old config manual
  • Switch firmware update → Duplex settings reset default
  • Cable swap → Different quality cable affects auto-neg
Auto-negotiation

Auto-Negotiation — Résolution automatique

// Principe Auto-Negotiation

Auto-negotiation (IEEE 802.3u) = Fast Link Pulses (FLP) échangés between devices négocier optimal speed + duplex.

Auto-neg workflow :

1. Link established (physical layer UP)

2. Devices exchange FLP (Fast Link Pulses)
   → Advertise capabilities (10/100/1000, Half/Full)

3. Negotiate highest common capability
   → Priority order : 1000Full > 1000Half > 100Full > 100Half > 10Full > 10Half

4. Configure both ends identically
   → Consistent duplex + speed

Example :
Device A : 10/100/1000, Half/Full
Device B : 10/100, Half/Full
→ Negotiate : 100 Mbps Full-duplex (highest common)

// Best Practices Auto-Negotiation

✅ RECOMMANDÉ : Auto-negotiation BOTH ends

  • Default modern equipment (switches, NICs)
  • Prevents mismatch (both negotiate same)
  • Adaptive (cable quality, distance adjustments)
  • Works 99%+ scenarios

⚠️ ACCEPTABLE : Manual BOTH ends (matching)

  • Troubleshooting auto-neg issues
  • Legacy devices (no auto-neg support)
  • CRITICAL : Configure BOTH ends identically
  • Document configuration (avoid drift)

❌ JAMAIS : Manual one end, Auto other end

  • Auto-neg FAILS (requires both participate)
  • Auto side falls back half-duplex → Mismatch
  • Cause #1 duplex mismatch production
✅
Best practice = Auto-negotiation everywhere Modern networks : Enable auto-neg ALL devices (switches, servers, workstations).
Exception rare : Legacy equipment (force manual, document thoroughly).
Cisco recommendation : "speed auto" + "duplex auto" (default config).
Troubleshooting

Diagnostic & Résolution — Step-by-step

// Step 1 : Identifier symptômes

User report : "Server connexion très lente"

Check link status :
→ LED NIC : GREEN (link UP)
→ LED switch : GREEN (link UP)

→ Link established MAIS performance poor = suspect duplex mismatch

// Step 2 : Vérifier duplex settings

Switch (Cisco) :
Switch# show interfaces GigabitEthernet0/5
→ Full-duplex, 1000Mb/s, media type is 10/100/1000BaseTX

Server (Linux) :
$ ethtool eth0
Speed: 1000Mb/s
Duplex: Half

→ MISMATCH FOUND : Switch Full ↔ Server Half

// Step 3 : Vérifier interface errors

Switch# show interfaces Gi0/5

1523 input errors, 1450 CRC, 73 frame
2847 late collisions
852 output errors

→ Trinity symptoms confirmed (CRC + late collisions + errors)

// Step 4 : Résolution

Option A : Enable auto-negotiation BOTH ends (RECOMMANDÉ)

Switch :
Switch(config)# interface GigabitEthernet0/5
Switch(config-if)# speed auto
Switch(config-if)# duplex auto

Server (Linux) :
$ sudo ethtool -s eth0 autoneg on

→ Both auto-negotiate → Consistent Full-duplex 1000 Mbps ✅

Option B : Manual configuration BOTH ends (si auto-neg fails)

Switch :
Switch(config-if)# speed 1000
Switch(config-if)# duplex full

Server :
$ sudo ethtool -s eth0 speed 1000 duplex full autoneg off

→ Both manual SAME settings → Consistent ✅

// Step 5 : Vérifier résolution

Switch# show interfaces Gi0/5 | include duplex
→ Full-duplex, 1000Mb/s

Server :
$ ethtool eth0
Speed: 1000Mb/s
Duplex: Full

→ Settings match ✅

Wait 5 minutes, check errors :
Switch# show interfaces Gi0/5
→ Late collisions : 0 (no increment)
→ CRC errors : 0 new

Test throughput :
$ iperf3 -c server
→ 950 Mbps (near line-rate, excellent)

→ Problem RESOLVED ✅
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : Link UP = duplex correct
FAUX. Link UP (LED green) ≠ duplex correct.

Duplex mismatch = link established (Layer 1 OK), MAIS performance terrible (Layer 2 errors).
LED green = "Physical layer UP", pas "Configuration optimal".

Symptômes : Link UP + slow performance = investigate duplex/speed.
❌ Piège #2 : Auto-neg one side = OK si other manual same value
FAUX. Manual one side + Auto other = Auto-neg FAILS.

Auto-neg requires BOTH participate (exchange FLP).
Si one manual → Auto side pas receive FLP → Falls back half-duplex.

Correct : BOTH auto OU BOTH manual (never mixed).
❌ Piège #3 : Late collisions = cable problem
RARE. Late collisions = 90%+ duplex mismatch.

Cable trop long (>100m) peut causer late collisions, MAIS rare modern.
Duplex mismatch = cause #1 (configuration error common).

Examen : Late collisions symptoms → Check duplex FIRST (pas cable).
