Fake AP

Evil Twin — AP jumeau malveillant

👯
Analogie — faux distributeur ATM Attaquant installe faux ATM identique banque vraie (même logo, même look). Clients insèrent carte → ATM clone carte + PIN. Evil Twin WiFi = pareil : faux AP copie SSID légitime.

Evil Twin = AP malveillant mimant AP légitime (même SSID) → users connectent croyant réseau officiel.

// Principe Evil Twin

  1. Attaquant créé AP malveillant :
    - SSID identique AP légitime (ex: "Starbucks_WiFi")
    - Signal puissant (plus fort que AP légitime si possible)
  2. Users auto-connect :
    - Laptop/phone reconnait SSID connu → connexion automatique
    - Ou user sélectionne manuellement (croit AP légitime)
  3. Trafic user passe attaquant :
    - MitM complet (passwords, cookies, données)
    - Captive portal fake (phishing credentials WiFi/email)
  4. Attaquant relay trafic Internet :
    - Fonctionne normalement → user pas soupçonne
    - Ou bloque complètement (DoS)
Scénario café :

AP légitime :
SSID: "CoffeeShop_Free"
MAC: aa:bb:cc:dd:ee:ff
Signal: -60 dBm

Evil Twin (attaquant table voisine) :
SSID: "CoffeeShop_Free" (identique)
MAC: 11:22:33:44:55:66 (différent)
Signal: -45 dBm (plus fort, laptop préfère)

→ Laptop auto-connect Evil Twin
→ Captive portal : "Enter email/password login"
→ User entre credentials → stolen

// Détection Evil Twin

  • Multiple SSIDs identiques : WIDS (Wireless IDS) alerte duplicates
  • MAC address inconnu : Whitelist APs légitimes
  • Signal anormal : AP légitime -60 dBm, soudain -30 dBm suspect

// Mitigation Evil Twin

  • WPA2/WPA3-Enterprise : 802.1X authentification (certificats) empêche Evil Twin
  • VPN : Chiffre trafic même si Evil Twin (attaquant voit ciphertext)
  • User awareness : Vérifier SSID exact, https:// sites sensibles
  • WIPS : Wireless IPS détecte/bloque rogue APs
🎯
Question fréquente examen "AP malveillant SSID identique AP légitime capturer credentials users ?"

Réponse : Evil Twin attack
WiFi disruption

Deauthentication Attack

Deauthentication attack = Envoyer frames deauth forcent clients déconnecter AP → DoS WiFi ou facilitent Evil Twin.

// Principe Deauth

802.11 Management Frames : Frames contrôle WiFi (association, deauth) PAS chiffrés (même WPA2).

Workflow attaque :

  1. Attaquant sniff WiFi → identifie AP + clients connectés
  2. Attaquant craft deauth frames :
    - Source MAC = AP
    - Destination MAC = Client
    - Reason code : "Leaving network"
  3. Attaquant broadcast deauth → TOUS clients
  4. Clients reçoivent deauth → déconnectent immédiatement
  5. Clients tentent reconnecter → re-deauth → boucle
$ aireplay-ng -0 0 -a AA:BB:CC:DD:EE:FF wlan0mon

-0 0 : Deauth attack, 0 = continuous
-a : Target AP MAC

→ Envoie deauth frames continuously
→ Tous clients AP déconnectés
→ WiFi DoS complet

// Usages Deauth attack

  • DoS WiFi : Empêcher users connecter (nuisance)
  • Faciliter Evil Twin : Deauth AP légitime → users reconnectent Evil Twin
  • Capture WPA handshake : Force client reconnecter → capture 4-way handshake → crack PSK offline

// Mitigation Deauth

  • 802.11w (PMF - Protected Management Frames) :
    - Chiffre management frames (deauth, disassoc)
    - WPA3 obligatoire, WPA2 optionnel
    - ✅ Défense efficace
  • WIPS : Détecte flood deauth frames → alerte
📡
Deauth = trivial exécuter Outils : aircrack-ng suite (aireplay-ng), mdk3, scapy.
Attaque très courante (script kiddies).
WPA3 + PMF = OBLIGATOIRE réseaux modernes.
PIN brute-force

WPS Attack — WiFi Protected Setup

WPS = Feature simplifiant connexion WiFi (bouton push ou PIN). Vulnérabilité : PIN brute-force facile.

// WPS fonctionnement

Méthodes WPS :

  • Push Button : Appuyer bouton physique AP + device → connexion auto
  • PIN Entry : Entrer PIN 8 chiffres (imprimé label AP ou généré device)

// Vulnérabilité WPS PIN

Design flaw WPS PIN :

  • PIN = 8 chiffres MAIS validé 2 parties (4 + 4)
  • AP répond si première moitié (4 digits) correcte
  • Attaquant brute-force séparément :
    - 10^4 = 10,000 combinaisons première moitié
    - 10^4 = 10,000 combinaisons seconde moitié
    - Total = 20,000 max (vs 10^8 = 100M si validé ensemble)
  • Brute-force 20,000 PINs = 2-10 heures (faisable)
$ reaver -i wlan0mon -b AA:BB:CC:DD:EE:FF -vv

→ Brute-force WPS PIN
→ 2-10 hours average
→ Success → Recovers WPA PSK (password WiFi complet)

[+] WPS PIN: 12345670
[+] WPA PSK: MySecurePassword123

→ Même WPA2-AES compromis si WPS enabled

// Mitigation WPS

  • Disable WPS : TOUJOURS désactiver production (feature convenance ≠ sécurité)
  • WPS lockout : Certains APs lockent après X tentatives failed (rate limiting)
  • ⚠️ Certains APs "disable WPS" interface MAIS reste actif → vérifier scanner (wash)
WiFi mapping

Wardriving

Wardriving = Conduire véhicule équipé laptop/smartphone + GPS → mapper réseaux WiFi (SSIDs, encryption, locations).

// Équipement Wardriving

  • Laptop + WiFi adapter haute sensibilité (antenne externe)
  • GPS (géolocaliser APs trouvés)
  • Software : Kismet, inSSIDer, WiFi Analyzer, Wigle
  • Véhicule (voiture, vélo, drone)

// Informations collectées

  • SSID : Nom réseau
  • BSSID : MAC address AP
  • Channel : Fréquence (1-11, 36-165)
  • Encryption : Open, WEP, WPA2, WPA3
  • Signal strength : RSSI (dBm)
  • GPS coordinates : Latitude/longitude AP

// Objectifs Wardriving

  • Légitime : Mapping coverage WiFi (researchers, hobbyists)
  • Malveillant :
    - Identifier réseaux faibles (WEP, Open, WPS enabled)
    - Cibler attaques (Evil Twin, crack WPA)
    - Geolocation businesses (competitor intel)

// Databases publiques

Wigle.net : Database crowdsourcée 1B+ APs WiFi mappés worldwide (GPS locations publiques).

🗺️
Wardriving = légal MAIS écouter trafic illégal Scanner WiFi (passive) = légal majorité juridictions.
Connecter AP sans autorisation = illégal.
Intercepter trafic = illégal (Wiretap laws).
Unauthorized AP

Rogue AP — Point accès non autorisé

Rogue AP = AP non autorisé connecté réseau corporate (intentionnel malveillant OU accidentel employee).

// Scénarios Rogue AP

1. Malveillant (attaquant externe)

  • Attaquant installe AP caché (placard, faux plafond)
  • Connecté switch réseau corporate
  • Backdoor permanent (accès remote réseau interne)

2. Accidentel (employee bien intentionné)

  • Employee branche AP personnel bureau (WiFi coverage faible)
  • Pas configuration sécurité (Open ou WPA2-PSK faible)
  • Créé vulnérabilité involontaire
Exemple Rogue AP accidentel :

Employee marketing : "WiFi bureau faible signal"
→ Achète AP consumer (TP-Link, $30)
→ Branche port Ethernet bureau
→ Configure SSID "Marketing_WiFi", pas password

Résultat :
→ Bypass firewall corporate (direct Internet)
→ Attaquant parking lot connecte AP Open
→ Accès réseau interne complet

// Risques Rogue AP

  • ❌ Bypass security controls (firewall, IDS/IPS)
  • ❌ Accès non autorisé réseau interne
  • ❌ Data exfiltration
  • ❌ Compliance violations (PCI-DSS, HIPAA)

// Détection Rogue AP

  • WIDS/WIPS : Scan continuous APs → alerte unknown BSSIDs
  • NAC (Network Access Control) : Détecte devices non autorisés plugged réseau
  • Physical surveys : Warwalking bureaux régulier (chercher APs cachés)
  • MAC address audits : Whitelist APs autorisés

// Mitigation Rogue AP

  • Policy : Interdire APs personnels (disciplinary action)
  • Port security : 802.1X authentication ports Ethernet (empêche plug rogue AP)
  • WIPS : Containment automatique (jamming rogue AP)
  • Employee training : Sensibilisation risques
Auto-connect exploit

Karma Attack

Karma attack = Exploiter WiFi auto-connect devices → AP malveillant répond TOUS probe requests.

// Principe WiFi probe requests

Comportement normal device :

  1. Laptop/phone cherche réseaux connus (preferred network list)
  2. Envoie probe requests : "Starbucks_WiFi ici ?" "HomeWiFi ici ?"
  3. Si AP matching répond → auto-connect

// Karma Attack workflow

  1. Attaquant setup AP Karma :
    - Mode promiscuous (écoute TOUS probe requests)
    - Répond "Oui !" à CHAQUE SSID demandé
  2. Victim phone probe :
    "HomeWiFi ici ?" "Office_Corporate ici ?" "Starbucks_WiFi ici ?"
  3. Karma AP répond TOUS :
    "Oui HomeWiFi c'est moi !"
    "Oui Office_Corporate c'est moi !"
    "Oui Starbucks_WiFi c'est moi !"
  4. Device auto-connect premier match :
    → Connexion AP malveillant
    → Trafic MitM complet
Karma attack demo :

Victim phone probe requests :
→ "HomeWiFi"
→ "Starbucks_WiFi"
→ "Airport_Free"

Karma AP responds ALL :
[Probe Response] SSID: HomeWiFi
[Probe Response] SSID: Starbucks_WiFi
[Probe Response] SSID: Airport_Free

→ Phone auto-connects "HomeWiFi" (thinks home network)
→ Actually attacker AP → MitM

// Révèle preferred networks

Bonus attaquant : Probe requests révèlent historique WiFi victim (homes, cafés, hôtels visités).

// Mitigation Karma

  • Disable WiFi auto-connect : Manuel connect only (iOS/Android settings)
  • Forget old networks : Nettoyer preferred network list régulièrement
  • VPN : Chiffre trafic même si connecté AP malveillant
  • WPA2/WPA3-Enterprise : Certificate validation empêche Karma (mutual auth)
📱
Smartphones = vulnérables Karma par défaut iOS/Android auto-connect réseaux connus.
Preferred network list = historique complet voyages/lieux.
Désactiver WiFi public si pas utilisé (drain batterie + sécurité).
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : Evil Twin = casser chiffrement WPA2
FAUX. Evil Twin = copier SSID légitime (pas casser crypto).

Evil Twin peut être Open (pas chiffrement) OU WPA2 (fake password).
Users connectent volontairement (croient AP légitime).

Pas besoin crack WPA2 → users donnent credentials
❌ Piège #2 : Deauth attack = casser WPA handshake
INCOMPLET. Deauth = forcer déconnexion (pas crack).

Deauth UTILISÉ capturer handshake (étape 1).
Crack handshake = étape séparée (offline brute-force).

Deauth seul = DoS (pas crack password)
❌ Piège #3 : Rogue AP = toujours malveillant
FAUX. Rogue AP peut être accidentel (employee bien intentionné).

Malveillant : Attaquant installe backdoor
Accidentel : Employee branche AP personnel (WiFi faible)

Risque identique (bypass security) même si accidentel
Résumé

🔑 Ce que tu dois retenir absolument

→Evil Twin : AP malveillant SSID identique AP légitime. Users auto-connect → MitM. Captive portal phishing. Mitigation : WPA2-Enterprise, VPN, WIPS
→Deauth attack : Management frames 802.11 falsifiés forcent déconnexion. DoS WiFi ou facilite Evil Twin/crack handshake. Mitigation : 802.11w PMF (WPA3 obligatoire)
→WPS attack : Brute-force PIN 8 chiffres (20K combinaisons, 2-10h). Récupère WPA PSK complet. Mitigation : DISABLE WPS (toujours)
→Wardriving : Mapper WiFi (SSID, encryption, GPS) via véhicule. Légal scan, illégal intercept. Wigle.net database 1B+ APs
→Rogue AP : AP non autorisé réseau (malveillant ou accidentel employee). Bypass firewall. Mitigation : WIDS/WIPS, 802.1X ports, policy
→Karma attack : AP répond TOUS probe requests (fake every SSID). Devices auto-connect. Révèle preferred networks. Mitigation : Disable auto-connect
