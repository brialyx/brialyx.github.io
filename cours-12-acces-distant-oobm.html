Management access

In-Band vs Out-of-Band Management

🚪
Analogie — portes bâtiment In-band = porte principale (accès normal, utilisée quotidiennement). Out-of-band = porte secours arrière (utilisée urgences, fonctionne même si porte principale bloquée). Réseau down → In-band impossible, Out-of-band sauve.

// In-Band Management

Définition : Accès management via réseau production DATA (même réseau que users).

Méthodes in-band :

  • SSH via IP management (ex: ssh admin@192.168.1.10)
  • HTTPS web interface (GUI switch/router)
  • Telnet (legacy, non sécurisé)
  • SNMP management

Avantages :

  • ✅ Simple (pas infra séparée)
  • ✅ Accès remote facile (Internet/VPN)
  • ✅ Économique (réutilise réseau existant)

Problème CRITIQUE :

  • ❌ Réseau data down = management impossible
  • ❌ Configuration erronée casse accès (lockout)
  • ❌ Vulnérable attaques (même réseau users)

// Out-of-Band Management (OOBM)

Définition : Accès management via réseau SÉPARÉ dédié (indépendant réseau production).

Méthodes out-of-band :

  • Console port physique (câble RS-232/RJ45)
  • Management VLAN isolé (réseau dédié OOBM)
  • Terminal server (accès console centralisé)
  • KVM over IP (accès serveur BIOS-level)
  • Modem dial-up (legacy backup)

Avantages :

  • ✅ Fonctionne même si réseau data crashé
  • ✅ Sécurité accrue (réseau séparé)
  • ✅ Troubleshooting réseau data possible (pas lockout)

Inconvénients :

  • ⚠️ Coût (infrastructure dédiée)
  • ⚠️ Complexité (câblage supplémentaire)
🚨
OOBM = OBLIGATOIRE environnements critiques Production datacenter SANS out-of-band = suicide. Réseau data crash → admins lockout complet → downtime prolongé. OOBM = assurance accès TOUJOURS possible.
Physical access

Console Port — Accès physique direct

Console port = Port série physique équipement réseau (switch/router/firewall) permettant accès CLI direct (pas réseau).

// Caractéristiques console port

  • Type connecteur : RJ45 (Cisco standard) ou DB9 RS-232 (legacy)
  • Câble : Rollover cable (Cisco) bleu typiquement, ou USB-to-serial
  • Vitesse : 9600 baud défaut (configurable)
  • Protocole : RS-232 série (pas Ethernet)
  • Accès : CLI full (même si réseau down complet)

// Usage console port

  • Initial setup : Configurer device neuf (pas encore IP)
  • Password recovery : Reset password oublié (rommon mode)
  • Troubleshooting : Réseau data inaccessible → console seule option
  • Firmware upgrade : Accès garanti pendant upgrade (si fail, console récupère)

// Connexion console (Windows/Mac/Linux)

Windows (PuTTY) :
1. Brancher câble console (USB-to-serial adapter)
2. Device Manager → Ports → Noter COM port (ex: COM3)
3. PuTTY → Serial → COM3, Speed 9600
4. Connect

Mac/Linux (screen) :
$ ls /dev/tty.*   # Identifier port (ex: /dev/tty.usbserial)
$ screen /dev/tty.usbserial 9600

→ Accès CLI switch/router

// Sécurité console port

Problème : Console = accès physique total (bypass authentification réseau).

Protection :

  • ✅ Password console : line console 0 → password XXX → login
  • ✅ Datacenter physique sécurisé : Serrures, badges, caméras
  • ✅ Logging console access : Audit qui accède console
🎯
Question fréquente examen "Méthode accès switch SANS réseau fonctionnel ?"

Réponse : Console port (câble série direct)
Centralized console access

Terminal Server — Console centralisé

Terminal server = Device centralisé connectant multiples console ports équipements → accès remote console via réseau (IP).

// Architecture terminal server

[Admin laptop] ─── SSH ───> [Terminal Server 192.168.99.100]
                                 │
                          Console cables (rollover)
                                 │
         ├───────┬──────┬──────┬──────┤
       [SW1]  [SW2]  [RTR1] [FW1] [SRV1]
    Console Console Console Console Console

→ Terminal server = hub console centralisé
→ 1 connexion SSH admin → accès TOUS consoles
→ Pas besoin câble physique chaque device

// Fonctionnement

  1. Admin SSH terminal server (ex: ssh admin@192.168.99.100)
  2. Menu présente ports disponibles (Port 1 = SW1, Port 2 = SW2...)
  3. Admin sélectionne port → session console device
  4. Admin travaille CLI device (comme câble direct)
  5. Déconnexion → Retour menu terminal server

// Avantages terminal server

  • ✅ Accès remote console : Pas besoin présence physique datacenter
  • ✅ Centralisé : 1 connexion → tous devices
  • ✅ Out-of-band : Réseau OOBM dédié → fonctionne même si data network down
  • ✅ Logging : Audit complet accès console (qui, quand, quel device)
  • ✅ 24/7 remote hands : Admins travaillent home/bureau (pas déplacement datacenter nuit)

// Solutions terminal server

Opengear : Leader marché (terminal servers dédiés 16-48 ports).

Cisco 2900 série : Routeur Cisco async ports (fonction terminal server).

Lantronix : Terminal servers économiques.

Raritan : Solutions enterprise.

🔌
Terminal server = standard datacenters modernes Environnements 50+ devices = terminal server OBLIGATOIRE (évite multiplier câbles console). Remote troubleshooting 03:00 AM = possible sans déplacement datacenter.
Bastion host

Jump Server / Bastion Host

Jump server (Bastion host) = Serveur intermédiaire sécurisé servant point d'accès UNIQUE vers infrastructure critique. Admins connectent jump server → puis devices cibles.

// Concept jump server

[Internet/Remote admin] ──> [Jump Server] ──> [Production devices]
                             │                    │
                             │                    ├─ Switches
                             │                    ├─ Routers
                       DMZ                    ├─ Firewalls
                    Hardened                   └─ Servers
                    2FA/MFA
                    Audit logs

→ Jump server = seul point entrée autorisé
→ Production devices = PAS accès direct Internet/remote
→ Firewall bloque tout sauf jump server

// Fonctionnement

  1. Admin SSH jump server (ex: ssh admin@jumpserver.company.com)
  2. Authentification forte (2FA/MFA requis)
  3. Depuis jump server, admin SSH device cible (ssh switch01)
  4. Travail sur device cible
  5. Toutes actions loggées (audit complet)

// Sécurité jump server

  • ✅ Hardened OS : Système minimal, patché, sécurisé
  • ✅ 2FA/MFA obligatoire : Pas juste password
  • ✅ Whitelist IPs : Seules IPs admin autorisées
  • ✅ Logging complet : Toutes commandes enregistrées (audit trail)
  • ✅ Session recording : Vidéo sessions admin (forensics)
  • ✅ Pas outils admin : Environnement restreint (pas compiler, pas éditeurs complexes)
  • ✅ Time-limited access : Sessions auto-disconnect après timeout

// Jump server vs VPN

CritèreVPN directJump server
AccèsDirect devices productionVia jump intermédiaire
Sécurité⚠️ Laptop admin compromis = réseau exposé✅ Jump hardened, monitoring
Audit⚠️ Difficile tracer actions précises✅ Logging centralisé complet
Contrôle⚠️ Admin accès large✅ Accès contrôlé granulaire
🛡️
Jump server = best practice sécurité Accès admin production JAMAIS direct Internet/VPN. Jump server = choke point sécurité (1 point fortifier vs N devices exposés). Compliance (PCI-DSS, SOX) souvent exige bastion hosts.
Server management

KVM over IP et IPMI

// KVM over IP

KVM = Keyboard, Video, Mouse. KVM over IP = Accès remote console serveur (clavier/écran/souris) via réseau, comme si physiquement devant serveur.

Usage :

  • Accès BIOS/UEFI (pré-boot, avant OS démarre)
  • Installation OS (monter ISO remote)
  • Troubleshooting boot failures (kernel panic, BSOD)
  • Firmware updates serveurs

Fonctionnalités :

  • Console graphique complète (pas juste CLI)
  • Virtual media (monter CD/DVD/USB ISO remote)
  • Power control (power on/off/reboot)
  • Fonctionne MÊME si OS crashé (hardware-level)

// IPMI (Intelligent Platform Management Interface)

IPMI = Standard hardware management serveurs (Dell iDRAC, HP iLO, Supermicro IPMI). Inclut KVM over IP + monitoring hardware.

Fonctionnalités IPMI :

  • Remote KVM : Console graphique full
  • Power management : On/off/reboot serveur
  • Hardware monitoring : Température CPU, fan speed, voltages
  • Alertes hardware : Email si CPU overheat, fan failure
  • Virtual media : Boot ISO remote
  • Serial-over-LAN (SOL) : Console série via réseau

// Exemples IPMI vendors

Dell iDRAC (integrated Dell Remote Access Controller)

HP iLO (integrated Lights-Out)

Supermicro IPMI

Lenovo XClarity

// Sécurité IPMI

CRITIQUE : IPMI = accès hardware complet → vulnérabilité = catastrophe.

  • ✅ Réseau OOBM dédié : JAMAIS exposer IPMI réseau production/Internet
  • ✅ Passwords forts : Changer password défaut IMMÉDIATEMENT
  • ✅ Firmware updates : IPMI vulnérabilités fréquentes → patcher régulièrement
  • ✅ Firewall : Restreindre accès IPs admin uniquement
  • ⚠️ Vulnérabilités connues : IPMI historique vulns (IPMI 1.5 = very insecure, utiliser 2.0)
🔐
IPMI exposé Internet = hack garanti Scanners Internet cherchent IPMI ports exposés (UDP 623). Password défaut "admin/admin" = compromission 5 minutes. IPMI TOUJOURS réseau OOBM isolé + firewall strict.
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : In-band = sécurisé car SSH
INCOMPLET. In-band SSH sécurisé SI réseau fonctionne.

Problème in-band : Réseau data down = accès impossible (lockout).
SSH sécurise transmission MAIS pas résout problème disponibilité.

Production critique → OOBM obligatoire (backup in-band)
❌ Piège #2 : Jump server = VPN
FAUX. Jump server ≠ VPN (différent).

VPN : Tunnel chiffré → accès direct devices
Jump server : Bastion intermédiaire → accès via jump uniquement

Jump server = contrôle/audit supérieur (choke point sécurité)
❌ Piège #3 : Console port nécessite réseau fonctionnel
FAUX. Console port = connexion série physique (pas réseau).

Console fonctionne MÊME si :
- Réseau data complètement down
- Device pas configuré IP
- Firmware crashé

Console = dernier recours accès garanti
Résumé

🔑 Ce que tu dois retenir absolument

→In-band : Management via réseau data production (SSH/HTTPS). Simple MAIS lockout si réseau down. Out-of-band (OOBM) : Réseau séparé dédié (console, terminal server). Fonctionne TOUJOURS
→Console port : Port série physique (RJ45 rollover cable, 9600 baud). Accès CLI sans réseau. Usage : initial setup, password recovery, troubleshooting critique
→Terminal server : Hub console centralisé (1 connexion SSH → tous devices consoles). Remote console access via réseau OOBM. Standard datacenters modernes
→Jump server (Bastion) : Point accès unique sécurisé vers production. Admins → Jump → Devices (pas direct). Hardened, 2FA, logging complet. Best practice sécurité
→KVM over IP / IPMI : Accès remote console serveur (clavier/écran). Fonctionne pré-boot (BIOS). Dell iDRAC, HP iLO. CRITIQUE : IPMI réseau OOBM isolé (pas Internet)
