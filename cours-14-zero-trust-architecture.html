Modern security model

Zero Trust — Never trust, always verify

🏛️
Analogie — aéroport sécurité Perimeter security (legacy) = porte château. Inside = trusted, outside = untrusted.
Zero Trust = aéroport moderne. Badge check entrée + boarding gate + passeport chaque étape. JAMAIS trust implicite même si déjà inside.

Zero Trust = Modèle sécurité assumant breach (compromission inévitable). Aucune confiance implicite, vérification continue chaque accès.

// Évolution modèle sécurité

Perimeter Security (Legacy)

  • Principe : "Castle and moat" (château + douves)
  • Inside network = TRUSTED (no verification)
  • Outside network = UNTRUSTED (firewall blocks)
  • ⚠️ Problème : Breach perimeter = attacker free roam internal (lateral movement facile)

Zero Trust (Modern)

  • Principe : "Never trust, always verify"
  • AUCUNE zone trusted (internal = potentiellement compromis)
  • Chaque accès vérifié (authentication + authorization continuous)
  • ✅ Avantage : Breach localisée (microsegmentation limite propagation)
Perimeter model (old) :

Employee laptop inside network → Access ALL internal resources
No re-authentication, no verification
Malware on laptop → Spreads entire network

Zero Trust model (new) :

Employee laptop → Request access database
→ Verify identity (MFA)
→ Verify device posture (patched ? AV running ?)
→ Verify authorization (user allowed database ?)
→ Grant minimum access (read-only specific tables)
→ Re-verify periodically (session timeout, continuous auth)

Malware on laptop → Isolated (microsegmentation blocks lateral movement)

// Principes fondamentaux Zero Trust

1. Verify explicitly

  • Toujours authenticate + authorize (jamais assume trust)
  • Use ALL available data : User identity, device health, location, behavior

2. Least privilege access

  • Just-in-time (JIT), just-enough-access (JEA)
  • Minimum permissions nécessaires task spécifique
  • Time-bound access (expire automatiquement)

3. Assume breach

  • Design assuming attacker already inside
  • Minimize blast radius (microsegmentation)
  • Encrypt everything, verify everything, log everything
🔐
Zero Trust = réponse cloud/remote work Perimeter dissout (employees home, SaaS apps, mobile devices).
Traditional VPN = extend perimeter (still trust inside).
Zero Trust = no perimeter, verify every access regardless location.
Google BeyondCorp = first large-scale Zero Trust implementation (2014).
Network isolation

Microsegmentation — Granular isolation

Microsegmentation = Diviser réseau en zones minuscules isolées (workload-level). Chaque workload = segment propre avec policies dédiées.

// Segmentation traditionnelle vs Microsegmentation

CritèreSegmentation traditionnelleMicrosegmentation
Granularité Subnets, VLANs (coarse) Per-workload, per-app (fine)
Périmètre Network-centric (VLAN boundaries) Workload-centric (follows app)
Policies IP-based ACLs (static) Identity-based (dynamic)
Lateral movement ⚠️ Easy (within VLAN = trusted) ✅ Blocked (each workload isolated)
Scalabilité ⚠️ Limited (VLAN sprawl, 4094 max) ✅ Unlimited (software-defined)

// Exemple microsegmentation

Traditional segmentation :

VLAN 10 (Web Tier) : 10 web servers
→ All web servers communicate freely (same VLAN = trusted)
→ Malware web-server-1 → Spreads web-server-2,3,4...

Microsegmentation :

Each web server = isolated segment
Web-server-1 → Allowed : Load balancer inbound, Database outbound
Web-server-1 → Denied : Web-server-2,3,4... (lateral blocked)

Malware web-server-1 → ISOLATED (cannot spread peers)
→ Only impacts web-server-1 (blast radius minimized)

// Implémentation microsegmentation

1. Software-Defined Segmentation

  • Policies attached workloads (tags, labels), pas IPs
  • Dynamic (workload moves, policies follow)
  • Exemples : VMware NSX, Cisco ACI, Illumio

2. Host-based Firewalls

  • Firewall chaque server/container
  • Centrally managed (group policies)
  • Exemples : Windows Firewall + GPO, iptables, cloud security groups

3. Service Mesh (Cloud-native)

  • Proxy sidecar chaque microservice
  • mTLS encryption + authorization policies
  • Exemples : Istio, Linkerd, Consul

// Bénéfices microsegmentation

  • ✅ Limit lateral movement : Attacker breach 1 workload ≠ propagation
  • ✅ Compliance : PCI-DSS, HIPAA exigent isolation cardholder/patient data
  • ✅ Visibility : Logs every connection attempt (forensics, anomaly detection)
  • ✅ Reduced attack surface : Each workload minimal exposure
Identity-centric security

Identity-Based Access — Who, not where

Identity-Based Access = Policies basées identité user/device (WHO), pas network location (WHERE).

// Shift paradigm : Location → Identity

Traditional (location-based) :

IF source IP = 10.0.1.0/24 (internal network)
→ THEN allow access database

Problem : IP spoofing, stolen laptop with valid IP = access granted

Zero Trust (identity-based) :

IF user = john@company.com
AND device = managed corporate laptop
AND MFA passed
AND device compliant (AV updated, patched)
AND user role = "Data Analyst"
→ THEN allow read-only access database tables X,Y (not Z)

No location checks → Works office, home, coffee shop

// Composants Identity-Based Access

1. Strong Authentication

  • MFA mandatory (something you know + have + are)
  • Passwordless (FIDO2, biometrics)
  • Continuous authentication (behavior analysis, risk-based)

2. Device Trust

  • Device registration (corporate-managed vs BYOD)
  • Posture assessment (OS patched, AV running, encryption enabled)
  • Device certificates (mutual TLS)

3. Contextual Authorization

  • Risk scoring : User + Device + Location + Time + Behavior
  • Adaptive policies : High risk = deny or require step-up auth
  • Example : Login unusual country = MFA + manager approval

4. Least Privilege (Dynamic)

  • Just-In-Time (JIT) : Permissions granted on-demand, expire automatically
  • Just-Enough-Access (JEA) : Minimum rights specific task
  • Example : Developer needs production DB access 1 hour → Auto-revoked

// Technologies Identity-Based Access

  • IAM (Identity and Access Management) : Okta, Azure AD, Ping Identity
  • SSO (Single Sign-On) : SAML, OAuth 2.0, OpenID Connect
  • NAC (Network Access Control) : 802.1X, posture checks (Cours O13)
  • CASB (Cloud Access Security Broker) : Monitor/control SaaS apps
🔑
Identity = new perimeter Legacy : Network perimeter (firewall edge).
Zero Trust : Identity perimeter (user/device verification).
"The network is hostile" → Trust identity, not location.
Works anywhere : Office, home, airport, cloud.
Cloud-native security

SASE / SSE — Secure Access Service Edge

// SASE (Secure Access Service Edge)

SASE = Convergence networking + security cloud-delivered. Gartner 2019.

Composants SASE :

  • Networking : SD-WAN (Software-Defined WAN), WAN optimization
  • Security : ZTNA, SWG, CASB, FWaaS, DLP
  • Delivered : Single cloud platform (pas appliances on-prem)
Traditional architecture :

Branch office → MPLS → Datacenter firewall → Internet
Remote user → VPN → Datacenter firewall → SaaS apps

Problems :
- Backhauling (traffic détour datacenter = latency)
- On-prem appliances (expensive, complex)
- VPN = extend perimeter (trust after connect)

SASE architecture :

Branch / Remote user → Nearest SASE PoP (cloud edge)
→ Security stack (ZTNA, SWG, FWaaS) applied edge
→ Direct Internet / SaaS (optimal path)

Benefits :
- No backhauling (low latency)
- Cloud-native (no hardware)
- Zero Trust integrated (verify every access)

// SSE (Security Service Edge)

SSE = Security subset SASE (networking components excluded). Gartner 2021.

SSE = SASE - SD-WAN

Composants SSE :

  • ZTNA (Zero Trust Network Access) : Remplace VPN, identity-based
  • SWG (Secure Web Gateway) : Web filtering, malware protection
  • CASB (Cloud Access Security Broker) : SaaS apps visibility/control
  • FWaaS (Firewall as a Service) : Cloud firewall
  • DLP (Data Loss Prevention) : Prevent data exfiltration
CritèreSASESSE
Scope Networking + Security Security only
Components SD-WAN + ZTNA + SWG + CASB + FWaaS ZTNA + SWG + CASB + FWaaS (no SD-WAN)
Use case Replace MPLS + VPN + security stack Secure access users → cloud/SaaS
Target Full WAN transformation Security transformation (keep existing WAN)

// ZTNA (Zero Trust Network Access)

ZTNA = VPN replacement Zero Trust. Application-level access (pas network-level).

VPN (traditional) :

User connects VPN → Full network access (10.0.0.0/8)
→ Can access ALL internal resources (lateral movement easy)
→ Trust after authentication (no continuous verification)

ZTNA (Zero Trust) :

User requests app1.company.com
→ Verify identity (MFA)
→ Verify device (posture check)
→ Authorize (user allowed this app ?)
→ Grant access app1 ONLY (not network)
→ App-level access (cannot see other apps, cannot ping internal)
→ Re-verify continuously

// Vendors SASE/SSE

  • Zscaler : Leader SSE (ZPA = ZTNA, ZIA = SWG)
  • Netskope : CASB + SSE platform
  • Palo Alto Prisma SASE : Full SASE (SD-WAN + security)
  • Cloudflare Zero Trust : ZTNA + SWG global network
  • Cisco Secure Access : SASE (Umbrella + Duo + Meraki)
☁️
SASE/SSE = future networking security Remote work + cloud apps = perimeter mort.
SASE/SSE = security follows users (not datacenter-centric).
Gartner : 60% enterprises adopt SASE by 2025 (2021 prediction).
Zero Trust + Cloud delivery = modern standard.
Implementation

Zero Trust — Stratégie déploiement

// Zero Trust = journey (pas destination)

Implémentation graduelle : Pas "big bang" migration. Phases progressives.

Phase 1 : Visibility

  • Map all resources (apps, data, users, devices)
  • Monitor traffic flows (who accesses what ?)
  • Identify critical assets (crown jewels)

Phase 2 : Identity & MFA

  • Deploy SSO (Single Sign-On)
  • Enforce MFA everywhere (especially admin accounts)
  • Implement conditional access (risk-based policies)

Phase 3 : Microsegmentation

  • Start critical assets (databases, financial systems)
  • Isolate workloads (block lateral movement)
  • Expand progressively (entire environment)

Phase 4 : ZTNA Deployment

  • Replace VPN with ZTNA (pilot group first)
  • Application-level access (not network)
  • Device posture checks (NAC integration)

Phase 5 : Continuous Monitoring

  • SIEM/SOAR integration (logs, alerts, automated response)
  • User behavior analytics (detect anomalies)
  • Regular audits (access reviews, policy updates)

// Challenges implémentation

  • ⚠️ Legacy apps : Pas supportent modern auth (workarounds needed)
  • ⚠️ User friction : MFA fatigue, re-authentication (balance security/UX)
  • ⚠️ Complexity : Multiple tools/vendors (integration challenges)
  • ⚠️ Cost : Cloud services, licensing, professional services
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : Zero Trust = VPN replacement uniquement
INCOMPLET. Zero Trust = architecture globale, pas juste VPN replacement.

Zero Trust inclut : Identity verification, microsegmentation, least privilege, continuous monitoring, device trust, encryption everywhere.

ZTNA = 1 composant Zero Trust (VPN replacement), pas définition complète.
❌ Piège #2 : SASE = SSE (synonymes)
FAUX. SSE = subset SASE.

SASE : Networking (SD-WAN) + Security (ZTNA, SWG, CASB, FWaaS)
SSE : Security only (ZTNA, SWG, CASB, FWaaS) - no SD-WAN

SSE ⊂ SASE (SSE = security part of SASE)
❌ Piège #3 : Microsegmentation = VLANs
FAUX. Microsegmentation ≠ VLANs (plus granulaire).

VLANs : Network-level (subnet isolation, ~100s segments max)
Microsegmentation : Workload-level (per-app, per-VM, thousands segments)

Microsegmentation = software-defined, identity-based, dynamic policies.
VLANs = hardware-based, IP-based, static.
