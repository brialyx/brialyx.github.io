Troubleshooting methodology

Méthodologie structurée — Pourquoi ?

Dépannage ad-hoc = Approche aléatoire (essayer solutions random jusqu'à succès). Inefficace, risque casser davantage.

Méthodologie CompTIA = Processus structuré 7 étapes. Systématique, reproductible, efficace.

🔬
Analogie — méthode scientifique Troubleshooting réseau = méthode scientifique.
Observer symptômes → Hypothèse cause → Tester hypothèse → Confirmer ou rejeter → Documenter résultats.
Pas deviner aveuglément, mais déduction logique.

// Bénéfices méthodologie structurée

  • ✅ Efficacité : Résolution rapide (pas perte temps solutions inutiles)
  • ✅ Reproductibilité : Process standardisé (différents techniciens même approche)
  • ✅ Documentation : Historique problèmes (future reference, knowledge base)
  • ✅ Minimise risque : Pas modifications hasardeuses (changements contrôlés)
  • ✅ Conformité : Change management, audit trail (ITIL, ISO standards)
⚠️
Erreur #1 techniciens juniors : Skip steps "Je sais problème → je fixe directement" = DANGER.
Sauter étapes = risque diagnostic erroné, casser autre chose, pas documentation.
Examen CompTIA : Toujours suivre 7 étapes DANS ORDRE.
CompTIA 7-step process

Les 7 étapes — Détails

ÉTAPE 1
Identifier le problème
Objectif : Comprendre symptômes, scope, impact.

Actions :
• Gather information (interviewer user : "Que s'est-il passé ?")
• Duplicate problem (reproduire erreur si possible)
• Question users (symptômes précis, quand démarré, changements récents ?)
• Determine scope (1 user ? Tout département ? Tout site ?)

Exemple questions :
- "Internet ne marche pas" → "Aucun site ? Sites spécifiques ? Email fonctionne ?"
- "Depuis quand ?" → "Ce matin ? Après reboot ? Après mise à jour ?"
- "Changements récents ?" → "Nouveau logiciel ? Config réseau modifiée ?"
ÉTAPE 2
Établir théorie cause probable
Objectif : Hypothèses causes possibles (du plus probable au moins probable).

Actions :
• Question obvious (câble débranché ? Alimentation OK ? Erreurs simples d'abord)
• Consider multiple approaches (Top-down, Bottom-up, Divide-and-conquer - Cours TR02)
• List probable causes (ordre probabilité décroissante)

Exemple théories :
Symptôme : PC pas ping gateway
Théorie 1 : Câble défectueux (probable - vérifier LED switch)
Théorie 2 : Config IP incorrecte (moins probable - fonctionnait hier)
Théorie 3 : Switch port disabled (peu probable - autres PCs OK même switch)
ÉTAPE 3
Tester théorie déterminer cause
Objectif : Valider ou invalider hypothèses via tests.

Actions :
• Test théorie la plus probable FIRST
• IF confirmed → Proceed Étape 4
• IF NOT confirmed → Établir nouvelle théorie (retour Étape 2), tester théorie suivante

Exemple tests :
Théorie 1 : Câble défectueux
→ Test : Remplacer câble → Toujours pas ping
→ Théorie invalidée → Passer théorie 2

Théorie 2 : Config IP incorrecte
→ Test : ipconfig /all → IP 169.254.x.x (APIPA)
→ DHCP failed → Théorie confirmée ✅
ÉTAPE 4
Établir plan d'action résoudre problème
Objectif : Définir steps résolution AVANT implémenter (pas improviser).

Actions :
• Identify effects (solution impact quoi ? Downtime ? Users affectés ?)
• Plan timing (maintenance window ? Business hours éviter ?)
• Prepare rollback (comment revenir arrière si problème ?)
• Get approval (change management - si modification production)

Exemple plan :
Cause : DHCP server down
Plan :
1. Vérifier DHCP service stopped (check logs)
2. Restart DHCP service
3. IF pas fix → Reboot DHCP server
4. Verify lease distribution (test client DHCP request)
Impact : 5 min downtime new DHCP leases (existing leases OK)
ÉTAPE 5
Implémenter solution ou escalader
Objectif : Exécuter plan OU escalader si hors compétences/permissions.

Actions :
• Implement solution (suivre plan Étape 4)
• OR escalate (si nécessite expertise senior, vendor support, permissions management)

Escalation nécessaire si :
- Hors compétences (configuration routeur BGP complexe, jamais fait)
- Hors permissions (modifications firewall production = senior engineer only)
- Vendor support requis (bug firmware switch, TAC case needed)
- Impact critique (changement risqué production, CTO approval needed)
ÉTAPE 6
Vérifier fonctionnalité complète système
Objectif : Confirmer problème résolu ET pas créé nouveaux problèmes.

Actions :
• Verify full system functionality (pas juste symptôme initial - TOUT fonctionne ?)
• Test related systems (solution pas cassé autre chose ?)
• Implement preventive measures (empêcher récurrence future)

Exemple vérification :
Fix : DHCP server restarted
Verify :
✅ Client obtient IP DHCP (problème initial résolu)
✅ Client ping gateway (connectivity OK)
✅ Client accède Internet (end-to-end OK)
✅ DNS résolution fonctionne (related service OK)
✅ Autres clients DHCP OK (pas impact négatif)

Preventive : Monitor DHCP service (alert si down future)
ÉTAPE 7
Documenter conclusions, actions, résultats
Objectif : Knowledge base, audit trail, future reference.

Actions :
• Document findings (symptômes, cause root, solution)
• Document actions taken (steps exactes exécutées)
• Document outcomes (résultat, lessons learned)
• Update ticketing system (close ticket avec details)

Exemple documentation :
Ticket #12345 : Users VLAN 10 pas accès Internet
Symptômes : 50 users VLAN 10 rapportent "pas Internet" 9h30 AM
Cause : DHCP server service crashed (Windows Event Log error 1000)
Solution : Restarted DHCP service → Functional 9h45 AM
Preventive : Configured DHCP service auto-restart on failure
Root cause : Server RAM full (memory leak application tierce) → Reboot hebdomadaire schedulé

Question TRÈS fréquente examen "Première étape troubleshooting CompTIA ?"
Réponse : Identifier le problème (Étape 1)

"Après tester théorie confirmée, étape suivante ?"
Réponse : Établir plan d'action (Étape 4)

Examen teste ORDRE étapes. Mémoriser séquence 1→2→3→4→5→6→7.
Practical example

Scénario complet — Application 7 étapes

Problème rapporté : "Les utilisateurs bâtiment B ne peuvent pas accéder serveur fichiers."

ÉTAPE 1 : Identifier
→ Question users : "Tous users bâtiment B ? Depuis quand ?"
→ Response : "Oui tous (~30 users). Depuis 8h ce matin."
→ Duplicate : Laptop admin bâtiment B → Confirme pas accès \\fileserver\shared
→ Scope : Bâtiment B uniquement (Bâtiment A OK accès fileserver)

ÉTAPE 2 : Théorie
Théorie 1 : Fileserver down (❌ peu probable - Bât A accède OK)
Théorie 2 : Réseau bâtiment B problème (✅ probable - tous users Bât B affected)
Théorie 3 : Switch bâtiment B défaillant (✅ très probable - symptôme soudain tous users)

ÉTAPE 3 : Tester
Test théorie 3 :
→ Ping gateway depuis PC Bât B : Success (L3 OK)
→ Ping fileserver IP : Timeout (pas reach fileserver)
→ Check switch Bât B uplink : LED port uplink OFF
→ Théorie confirmée : Uplink switch Bât B → Core switch DOWN

ÉTAPE 4 : Plan d'action
1. Vérifier câble uplink (physical damage ?)
2. Test port uplink switch (port disabled ? Error ?)
3. Si port failed → Basculer uplink port backup
Impact : 5 min downtime Bât B pendant changement uplink
Rollback : Rebrancher uplink port original si backup failed

ÉTAPE 5 : Implémenter
→ Débrancher câble uplink port Gi0/24
→ Brancher câble uplink port backup Gi0/23
→ LED port Gi0/23 GREEN (link UP)
→ Solution implémentée

ÉTAPE 6 : Vérifier
✅ PC Bât B ping fileserver : Success
✅ Access \\fileserver\shared : OK
✅ Test 5 users différents Bât B : All OK
✅ Bât A toujours OK : No impact
Preventive : RMA port Gi0/24 failed (vendor support case)

ÉTAPE 7 : Documenter
Ticket #98765 : Bâtiment B pas accès fileserver
Cause : Switch Bât B uplink port Gi0/24 hardware failure (LED OFF, no link)
Solution : Failover uplink port backup Gi0/23
Downtime : 5 minutes (8h15-8h20 AM)
Follow-up : Port Gi0/24 RMA case #ABC123 vendor support
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : Implémenter solution AVANT établir plan
FAUX. Ordre étapes STRICT examen.

Étape 3 : Tester théorie → Cause confirmée
Étape 4 : Plan d'action (planifier solution)
Étape 5 : Implémenter (exécuter plan)

Jamais sauter Étape 4 (plan) → Aller direct Étape 5 (implement).
Examen : "Après confirmer cause, next step ?" → Réponse = Plan (pas implement).
❌ Piège #2 : Documentation optionnelle (Étape 7 skip)
FAUX. Étape 7 = OBLIGATOIRE.

"Problème résolu → job done" = INCOMPLET.
Documentation = partie intégrante process (knowledge base, audit, compliance).

Examen : Toujours inclure Étape 7 (documenter) même si question pas mentionne explicitement.
❌ Piège #3 : Vérifier symptôme initial uniquement (Étape 6)
INCOMPLET. Étape 6 = Verify FULL system functionality.

Pas juste : "Problème initial fixed ?"
Mais : "TOUT système fonctionne ? Solution pas créé nouveaux problèmes ?"

Exemple : Fix DHCP → Vérifier DHCP + DNS + Internet + Email (end-to-end).
Examen : "Verify full system functionality" = test complet, pas juste symptôme.
Résumé

🔑 Ce que tu dois retenir absolument

→7 Étapes ORDRE STRICT : 1.Identifier → 2.Théorie → 3.Tester → 4.Plan → 5.Implémenter → 6.Vérifier → 7.Documenter. Jamais skip, jamais inverser ordre
→Étape 1 (Identify) : Gather info, duplicate problem, question users, determine scope. Comprendre symptômes AVANT agir
→Étape 2 (Theory) : Hypothèses causes probables (most probable first). Question obvious. Multiple approaches (Top-down/Bottom-up/Divide-conquer)
→Étape 3 (Test) : Valider théories via tests. Si confirmée → Étape 4. Si invalidée → Nouvelle théorie (retour Étape 2)
→Étape 4 (Plan) : Définir steps résolution, identify effects, plan timing, prepare rollback, get approval. AVANT implémenter
→Étape 5 (Implement) : Exécuter plan OU escalader (hors compétences/permissions). Suivre plan Étape 4
→Étape 6 (Verify) : Full system functionality (pas juste symptôme). Test related systems. Implement preventive measures
→Étape 7 (Document) : Findings, actions, outcomes. Knowledge base, ticketing system. OBLIGATOIRE (pas optionnel)
