Révolution réseau

SDN — Définition et concept

🧠
Analogie — système nerveux Réseau traditionnel = chaque équipement (switch/routeur) a son propre cerveau → décisions locales, config manuelle chacun.
SDN = cerveau central (contrôleur) commande tous équipements → cohérence, programmabilité, automatisation.

SDN (Software-Defined Networking) = architecture réseau séparant plan de contrôle (décisions) du plan de données (forwarding). Contrôle centralisé programmable.

// Problème réseaux traditionnels

  • ❌ Config manuelle : SSH chaque switch/routeur individuellement
  • ❌ Rigidité : Changement réseau = reconfigurer équipements un par un
  • ❌ Vendor lock-in : CLI propriétaires (Cisco IOS vs Juniper Junos)
  • ❌ Pas programmable : Automatisation difficile
  • ❌ Lenteur déploiement : Semaines/mois ajuster réseau

// Solution SDN

  • ✅ Contrôle centralisé : Contrôleur SDN = cerveau unique
  • ✅ Programmabilité : APIs (REST, Python)
  • ✅ Automatisation : Provisioning réseau en minutes
  • ✅ Vendor-neutral : Protocoles standards (OpenFlow)
  • ✅ Visibilité globale : Vue centralisée topologie
Découplage

Control Plane vs Data Plane — Séparation fondamentale

// Data Plane (Plan de Données)

Forwarding : Déplace paquets physiquement (switching, routage).

Fonctions :

  • Lookup table forwarding (MAC, IP)
  • Envoi paquets sur bons ports
  • QoS (marquage, queuing)
  • ACLs (filtrage rapide hardware)

Hardware : ASIC (Application-Specific Integrated Circuit) — ultra-rapide (wire-speed)

// Control Plane (Plan de Contrôle)

Décisions : Construit tables forwarding (comment router/switcher).

Fonctions :

  • Protocoles routage (OSPF, BGP) — calculs routes
  • Spanning Tree (topologie sans boucles)
  • Construction table MAC (apprentissage switches)
  • Construction table ARP

Hardware : CPU (logiciel) — plus lent que data plane MAIS intelligence

// Réseau Traditionnel vs SDN

Traditionnel : Control + Data planes couplés dans même équipement.

[Switch/Routeur traditionnel]
┌──────────────────────┐
│ Control Plane (CPU) │ ← Décisions routage locales
│ ↓ │
│ Data Plane (ASIC) │ ← Forwarding paquets
└──────────────────────┘

→ Chaque équipement décide indépendamment
→ Config manuelle chacun

SDN : Control plane découplé + centralisé.

[Contrôleur SDN — Control Plane centralisé]
↓ OpenFlow / API
┌──────────────────────┐
│ Switch SDN │
│ Data Plane UNIQUEMENT│ ← Forwarding selon instructions contrôleur
└──────────────────────┘

→ Contrôleur décide pour TOUS switches
→ Switches = "dumb forwarding" rapide
🎯
Question TRÈS fréquente examen "Quelle couche SDN prend décisions routage ?"

Réponse : Control Plane (contrôleur SDN)

Data Plane = exécution (forwarding paquets)
Couches

Architecture SDN — 3 couches

// Application Layer (Couche Application)

Apps réseau business utilisant SDN.

Exemples : Load balancing, firewalling, monitoring, QoS, traffic engineering

// Control Layer (Contrôleur SDN)

Cerveau central. Gère topologie, calcule routes, pousse règles switches.

Exemples contrôleurs : OpenDaylight, ONOS, Cisco DNA Center, VMware NSX

// Infrastructure Layer (Switches SDN)

Équipements data plane exécutant instructions contrôleur.

Switches OpenFlow : Hardware supportant protocole OpenFlow

// Interfaces

Northbound API : Apps ↔ Contrôleur (REST, Python)

Southbound API : Contrôleur ↔ Switches (OpenFlow, NETCONF, OVSDB)

[Application Layer — Apps réseau]
↕ Northbound API (REST/Python)
[Control Layer — Contrôleur SDN]
↕ Southbound API (OpenFlow)
[Infrastructure Layer — Switches SDN]
↓
[Trafic réseau physique]
Protocole

OpenFlow — Protocole Southbound standard

OpenFlow = protocole communication contrôleur SDN ↔ switches. Standard ouvert (Open Networking Foundation).

// Fonctionnement

  1. Switch reçoit paquet (nouveau flux)
  2. Pas de règle correspondante dans flow table → envoie au contrôleur
  3. Contrôleur décide action (forward, drop, modify)
  4. Contrôleur pousse règle flow dans switch via OpenFlow
  5. Paquets suivants même flux : Switch forward directement (hardware speed)

// Flow Table

Table dans switch contenant règles forwarding.

Flow Entry structure :
[Match Fields] [Actions] [Counters]

Exemple :
Match: IP_src=10.0.1.5, IP_dst=10.0.2.10
Action: Forward to Port 3
Counter: 1500 packets, 1.2 MB

→ Tous paquets 10.0.1.5→10.0.2.10 envoyés port 3

// Avantages OpenFlow

  • ✅ Standard ouvert (pas vendor-specific)
  • ✅ Flexible (match sur Layer 2-4 fields)
  • ✅ Programmable (contrôleur modifie flow tables dynamiquement)
  • ✅ Granulaire (flow-by-flow vs réseau entier)
Virtualisation

NFV — Network Functions Virtualization

NFV = virtualisation fonctions réseau. Remplace appliances hardware dédiés par VMs/containers software.

// Traditionnel vs NFV

Traditionnel : Appliances hardware dédiés.

  • Firewall physique (Palo Alto PA-5220)
  • Load balancer hardware (F5 BIG-IP)
  • Routeur physique (Cisco ASR)
  • ⚠️ Coût élevé (CAPEX hardware)
  • ⚠️ Pas flexible (hardware fixe)

NFV : Fonctions network = VMs/containers sur serveurs x86 standard.

  • Firewall virtuel (VM pfsense, container)
  • Load balancer software (HAProxy VM)
  • Routeur virtuel (VyOS, Cisco CSR1000v)
  • ✅ Coût réduit (serveurs commodity)
  • ✅ Flexible (spin up/down dynamique)
  • ✅ Scaling rapide

// NFV vs SDN

SDN : Comment contrôler réseau (separation control/data planes)

NFV : Où exécuter fonctions réseau (hardware dédié → VMs/containers)

Complémentaires : SDN + NFV souvent utilisés ensemble (SDN contrôle NFV VMs)

🚀
NFV = révolution Telcos/ISPs Opérateurs réseau remplacent hardware propriétaire ($$$$) par VMs sur serveurs commodity. Économies massives + agilité déploiement nouveaux services.
Bénéfices

Avantages SDN en datacenter

  • ✅ Agilité : Provisioning réseau minutes (vs semaines traditionnel)
  • ✅ Automatisation : APIs programmables (Python, Ansible)
  • ✅ Visibilité globale : Contrôleur voit topologie complète
  • ✅ Optimisation trafic : Routage intelligent dynamique
  • ✅ Multi-tenancy : Isolation réseaux virtuels (VXLANs)
  • ✅ CAPEX réduit : Switches "dumb" moins chers
  • ✅ Innovation : Nouvelles apps réseau sans hardware

// Cas d'usage datacenter

Micro-segmentation : Isolation fine VMs (sécurité zero-trust)

Load balancing dynamique : Ajuster flux temps réel selon charge serveurs

Network slicing : Réseaux virtuels isolés (dev/test/prod)

Disaster recovery : Failover automatique entre sites

⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : SDN = pas besoin switches
FAUX. SDN utilise switches MAIS switches "dumb" (data plane uniquement).

Control plane externalisé → contrôleur.
Switches SDN toujours nécessaires forwarding paquets
❌ Piège #2 : NFV = SDN
DIFFÉRENTS MAIS COMPLÉMENTAIRES.

SDN : Architecture réseau (control/data planes séparés)
NFV : Virtualisation fonctions réseau (firewall/LB en VMs)

Souvent combinés en production
❌ Piège #3 : OpenFlow = contrôleur SDN
NON. OpenFlow = protocole communication.

Contrôleur SDN = software (OpenDaylight, ONOS)
OpenFlow = protocole southbound utilisé PAR contrôleur
Résumé

🔑 Ce que tu dois retenir absolument

→SDN : Sépare control plane (décisions, contrôleur central) et data plane (forwarding, switches). Programmable, automatisation, agilité
→Architecture 3 couches : Application (apps réseau), Control (contrôleur SDN), Infrastructure (switches). Northbound API (apps↔contrôleur), Southbound API (contrôleur↔switches)
→OpenFlow : Protocole southbound standard, flow tables dans switches, contrôleur pousse règles dynamiquement
→NFV : Virtualisation fonctions réseau (firewall/LB en VMs vs hardware). Différent SDN MAIS complémentaires
