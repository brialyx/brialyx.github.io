Contraintes temps réel

VoIP — Exigences qualité CRITIQUES

📞
Analogie — conversation téléphonique Conversation téléphone = temps réel. Retard 500ms → conversation impossible (chevauche paroles). Email = pas temps réel, délai 5 secondes acceptable. VoIP = ultra-sensible latence/jitter.

VoIP (Voice over IP) = téléphonie sur réseau IP. Trafic temps réel extrêmement sensible latence, jitter, perte paquets.

// 3 métriques CRITIQUES

1. Latence (Delay)

  • Définition : Temps paquet A → B (one-way delay)
  • Cible : < 150ms (recommandation ITU-T G.114)
  • Impact : >150ms = latence perceptible, >300ms = conversation difficile
  • Causes : Propagation (distance), queuing (congestion), serialization (slow links)

2. Jitter (Variation latence)

  • Définition : Variation délai entre paquets (paquet 1 : 20ms, paquet 2 : 80ms → jitter 60ms)
  • Cible : < 30ms (idéalement <20ms)
  • Impact : Jitter élevé = voix hachée, coupures, qualité dégradée
  • Solution : Jitter buffer (buffer réception compense variations)

3. Perte de paquets (Packet Loss)

  • Cible : < 1% perte paquets
  • Impact : >1% = qualité voix dégradée, >3% = inacceptable
  • Remarque : VoIP utilise UDP (pas TCP) → pas retransmission paquets perdus

Classification trafic

Marquage VoIP — CoS et DSCP

VoIP = 2 types trafic distincts nécessitant marquage QoS différent.

// 1. Signalisation (Call Control)

Protocoles : SIP, H.323, SCCP (Skinny)

Fonction : Établir/terminer appels, négocier paramètres

Caractéristiques : Peu fréquent, latence acceptable ~1 seconde

Marquage :

  • CoS 3 (Layer 2 — 802.1p)
  • DSCP 24 (CS3 — Class Selector 3, Layer 3)

// 2. Media (RTP Audio)

Protocole : RTP (Real-Time Transport Protocol)

Fonction : Transport voix numérisée temps réel

Caractéristiques : Flux continu, latence CRITIQUE <150ms

Marquage :

  • CoS 5 (Layer 2 — 802.1p)
  • DSCP 46 (EF — Expedited Forwarding, Layer 3)

// Tableau marquage VoIP

TraficProtocoleCoS (L2)DSCP (L3)Priorité
Signalisation SIP, H.323 CoS 3 DSCP 24 (CS3) ⚠️ Moyenne
Media (RTP) RTP CoS 5 DSCP 46 (EF) ✅ MAXIMALE
Appel téléphonique IP — flux réseau :

1. SIP INVITE (TCP 5060)
→ Marqué DSCP 24 (CS3) / CoS 3
→ Établir appel

2. RTP stream (UDP 16384-32767)
→ Marqué DSCP 46 (EF) / CoS 5
→ Voix temps réel (priorité MAX)

3. SIP BYE
→ Marqué DSCP 24 (CS3) / CoS 3
→ Terminer appel
📌
DSCP EF (Expedited Forwarding) = PHB VoIP DSCP 46 (EF) = PHB (Per-Hop Behavior) dédié VoIP. Garantit : faible latence, faible jitter, faible perte. Standard RFC 3246 pour trafic voix.
Protocoles VoIP

SIP et RTP — Signalisation et Media

// SIP (Session Initiation Protocol)

Rôle : Signalisation (établir/modifier/terminer sessions multimédia).

Port : TCP/UDP 5060 (non-sécurisé), TCP 5061 (TLS sécurisé)

Fonctions :

  • Localiser endpoint (user@domain.com)
  • Négocier paramètres appel (codec, bande passante)
  • Établir session (INVITE)
  • Modifier session (RE-INVITE)
  • Terminer session (BYE)

Messages SIP principaux :

  • INVITE : Initier appel
  • ACK : Confirmer réception
  • BYE : Terminer appel
  • CANCEL : Annuler appel en cours
  • REGISTER : Enregistrer endpoint auprès serveur SIP

// RTP (Real-Time Transport Protocol)

Rôle : Transport media temps réel (audio/vidéo).

Port : UDP 16384-32767 (dynamique, pairs pour RTP, impairs pour RTCP)

Caractéristiques :

  • UDP (pas TCP) : Pas retransmission → latence faible
  • Sequence numbers : Détecte paquets perdus, ordre
  • Timestamps : Reconstitue timing playback (jitter buffer)
  • SSRC (Synchronization Source) : Identifie source RTP

// RTCP (RTP Control Protocol)

Compagnon RTP : Feedback qualité session (statistics, packet loss, jitter).

Port : RTP port + 1 (exemple : RTP 16384 → RTCP 16385)

Flux appel VoIP typique :

[Téléphone A] ─────────────→ [Téléphone B]

1. SIP INVITE (TCP 5060) → Veux appeler B
2. SIP 100 Trying ← Recherche B...
3. SIP 180 Ringing ← B sonne
4. SIP 200 OK ← B décroche
5. SIP ACK → Confirmation

-- Conversation établie --
6. RTP audio bidirectionnel (UDP dynamique)
A → B : UDP 16384 (RTP) + 16385 (RTCP)
B → A : UDP 20000 (RTP) + 20001 (RTCP)

-- Fin appel --
7. SIP BYE → A raccroche
8. SIP 200 OK ← Confirmation terminaison
🔊
SIP = signalisation, RTP = voix SIP : Contrôle (appeler, raccrocher) — TCP/UDP 5060
RTP : Audio temps réel — UDP dynamique (16384-32767)
Firewall doit autoriser BOTH pour VoIP fonctionnel
Files d'attente

Priority Queuing — Mécanismes QoS

Switches/routeurs utilisent files d'attente prioritaires pour traiter VoIP avant autre trafic.

// Strict Priority Queue (PQ)

Principe : Queue haute priorité TOUJOURS vidée en premier.

4 queues sur interface sortie :

Queue 4 (Strict Priority) : CoS 5-7 (VoIP, contrôle)
Queue 3 : CoS 3-4 (signalisation, vidéo)
Queue 2 : CoS 2
Queue 1 (Best Effort) : CoS 0-1 (web, email)

Traitement :
1. Vider Queue 4 complètement
2. Si Queue 4 vide → vider Queue 3
3. Si Queue 3 vide → vider Queue 2
4. Si Queue 2 vide → vider Queue 1

→ VoIP (Queue 4) jamais en attente → latence minimale

Avantage : Latence VoIP minimale (queue vidée immédiatement)

Risque : Starvation autres queues si VoIP sature lien (besoin policing)

// CBWFQ (Class-Based Weighted Fair Queuing)

Principe : Alloue bande passante garantie par classe + strict priority pour VoIP.

Policy-map exemple (lien 10 Mbps) :

class VOICE (DSCP EF)
 priority 2 Mbps ← Strict priority, max 2 Mbps

class VIDEO (DSCP AF41)
 bandwidth 3 Mbps ← Garantie minimum 3 Mbps

class DATA (default)
 bandwidth 4 Mbps ← Garantie minimum 4 Mbps
 fair-queue

→ VoIP strict priority (max 2 Mbps protège autres classes)
→ Vidéo garantie 3 Mbps minimum
→ Data garantie 4 Mbps minimum

// Policing vs Shaping

Policing : Drop paquets excédant rate (hard limit)

  • Exemple : VoIP max 2 Mbps → si >2 Mbps, drop excédent immédiatement
  • Pas buffer, drop instantané

Shaping : Buffer paquets excédant rate (smooth traffic)

  • Exemple : Trafic >10 Mbps → buffer excédent, envoyer progressivement
  • Ajoute latence (buffer) → pas recommandé VoIP
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : Latence VoIP = 300ms acceptable
FAUX. Latence cible VoIP = <150ms.

300ms = maximum absolu (qualité médiocre).
150ms = cible qualité correcte (ITU-T G.114)
❌ Piège #2 : VoIP utilise TCP (fiable)
FAUX. VoIP media (RTP) utilise UDP.

UDP = pas retransmission → latence faible.
TCP retransmissions ajouteraient latence inacceptable.
SIP peut utiliser TCP OU UDP (signalisation, latence OK)
❌ Piège #3 : Jitter = latence
DIFFÉRENT.

Latence : Délai absolu (A → B, 100ms)
Jitter : Variation délai (paquet 1: 50ms, paquet 2: 150ms → jitter 100ms)

Jitter <30ms VoIP, latence <150ms
Résumé

🔑 Ce que tu dois retenir absolument

→Exigences VoIP : Latence <150ms, Jitter <30ms, Perte <1%. VoIP = UDP (RTP), pas TCP
→Marquage : Signalisation SIP = CoS 3 / DSCP 24 (CS3). Media RTP = CoS 5 / DSCP 46 (EF)
→Protocoles : SIP (TCP/UDP 5060) = signalisation appels. RTP (UDP 16384-32767) = transport voix temps réel
→Queuing : Strict Priority Queue = VoIP vidée en premier (latence min). CBWFQ = bande passante garantie + priority VoIP
