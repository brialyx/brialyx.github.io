Définition

Cloud Computing — Définition NIST

☁️
Analogie — électricité Cloud = comme électricité. Tu ne construis pas ta propre centrale électrique chez toi → tu consommes depuis réseau public (fournisseur). Pareil avec IT : pas besoin datacenter maison → consommes compute/storage depuis AWS/Azure/GCP.

Cloud Computing (NIST) = accès réseau ubiquitaire, pratique et à la demande à un pool partagé de ressources informatiques configurables (réseaux, serveurs, stockage, applications, services) pouvant être rapidement provisionnées et libérées avec effort minimal ou interaction fournisseur.

// 5 caractéristiques essentielles NIST

  • On-demand self-service : Provisionner ressources automatiquement (sans appeler support)
  • Broad network access : Accessible partout via réseau (Internet)
  • Resource pooling : Ressources mutualisées multi-tenants (partage infrastructure)
  • Rapid elasticity : Scaling up/down automatique selon demande
  • Measured service : Facturation à l'usage (pay-as-you-go, pas CAPEX fixe)
Modèles de service

IaaS / PaaS / SaaS — 3 niveaux abstraction

// IaaS (Infrastructure as a Service)

Loue infrastructure : Serveurs virtuels, stockage, réseau. Client gère OS, middleware, apps.

Exemples : AWS EC2, Azure VMs, Google Compute Engine

Responsabilité client :

  • OS (Windows/Linux)
  • Middleware (runtime, database)
  • Applications
  • Données
  • Sécurité applicative

Responsabilité fournisseur : Hardware physique, virtualisation, réseau datacenter

Cas d'usage : Héberger VMs custom, migrations lift-and-shift, contrôle total OS

// PaaS (Platform as a Service)

Loue plateforme dev/déploiement : Runtime, middleware, OS gérés. Client code uniquement apps.

Exemples : AWS Elastic Beanstalk, Azure App Service, Google App Engine, Heroku

Responsabilité client :

  • Code application
  • Données
  • Config app

Responsabilité fournisseur : OS, runtime (Python, Node.js, Java), scaling, patching

Cas d'usage : Développement rapide web apps, pas gérer infra, focus code business

// SaaS (Software as a Service)

Loue application complète : Logiciel prêt à l'emploi, accessible via navigateur/API.

Exemples : Office 365, Gmail, Salesforce, Dropbox, Slack, Zoom

Responsabilité client :

  • Données utilisateur
  • Configuration utilisateur (permissions, settings)

Responsabilité fournisseur : TOUT (infrastructure, OS, middleware, app, sécurité, disponibilité)

Cas d'usage : Utiliser logiciel sans installation, accès partout, zéro maintenance

Modèles déploiement

Public / Privé / Hybride / Multi-Cloud

// Cloud Public

Infrastructure partagée gérée par fournisseur tiers (AWS, Azure, GCP). Multi-tenants.

Avantages :

  • ✅ Coût faible (OPEX, pas CAPEX)
  • ✅ Scaling illimité
  • ✅ Pas maintenance hardware
  • ✅ Disponibilité globale (datacenters worldwide)

Inconvénients :

  • ⚠️ Données hors site (compliance possibles)
  • ⚠️ Moins contrôle infra physique
  • ⚠️ Partage ressources (noisy neighbors possible)

// Cloud Privé

Infrastructure dédiée une seule organisation. Peut être on-premise ou hébergé.

Avantages :

  • ✅ Contrôle total (sécurité, compliance)
  • ✅ Performances dédiées (pas noisy neighbors)
  • ✅ Personnalisation complète

Inconvénients :

  • ❌ Coût TRÈS élevé (CAPEX hardware)
  • ❌ Scaling limité (capacité fixe)
  • ❌ Maintenance/gestion interne nécessaire

Technologies : VMware vSphere, OpenStack, Microsoft Azure Stack

// Cloud Hybride

Combinaison public + privé interconnectés. Workloads peuvent bouger entre les deux.

Exemple :

[Cloud Privé on-premise]
- Données sensibles clients (PII, financier)
- Apps legacy critiques

[Cloud Public AWS]
- Site web frontend public
- Dev/Test environments
- Big data analytics (scaling)

→ Interconnexion VPN/Direct Connect
→ Workloads peuvent burst vers public si pic demande

Cas d'usage :

  • Données sensibles on-premise (compliance) + workloads standard cloud public
  • Cloud bursting : Scaling temporaire vers public si privé saturé
  • Disaster recovery : Backup privé → public

// Multi-Cloud

Utilise PLUSIEURS fournisseurs cloud publics (AWS + Azure + GCP). Pas forcément interconnectés.

Raisons multi-cloud :

  • ✅ Éviter vendor lock-in
  • ✅ Best-of-breed services (AWS Lambda + Azure AD + GCP BigQuery)
  • ✅ Résilience (si AWS down → basculer Azure)
  • ✅ Conformité géographique (data residency)
  • ⚠️ Complexité gestion accrue

Sécurité

Modèle de Responsabilité Partagée

Principe fondamental cloud : Sécurité = responsabilité PARTAGÉE entre fournisseur cloud et client.

// Fournisseur Cloud (Security OF the Cloud)

Responsable infrastructure physique/virtuelle :

  • Hardware physique (serveurs, disques, réseau)
  • Datacenters (sécurité physique, climatisation, électricité)
  • Hyperviseur / virtualisation
  • Réseau backbone
  • Disponibilité infrastructure (SLA uptime)

// Client (Security IN the Cloud)

Responsable données, applications, accès :

  • Données (chiffrement, backup)
  • Gestion identités/accès (IAM, MFA)
  • OS (patching si IaaS)
  • Applications (sécurité code)
  • Configuration réseau (Security Groups, firewalls)
  • Compliance données

⚠️
Client TOUJOURS responsable données Même en SaaS, client responsable : données utilisateur, contrôle accès, compliance réglementaire (RGPD, HIPAA). Fournisseur NE PEUT PAS être responsable données métier client.
🎯
Question piège examen "Qui responsable patching OS sur AWS EC2 (IaaS) ?"

Réponse : CLIENT (IaaS = client gère OS)

"Qui responsable patching OS sur Heroku (PaaS) ?"
Réponse : FOURNISSEUR (PaaS = fournisseur gère OS)
Bénéfices

Avantages Cloud Computing

  • ✅ OPEX vs CAPEX : Pay-as-you-go (pas investissement hardware lourd)
  • ✅ Elasticité : Scaling up/down automatique selon demande
  • ✅ Agilité : Provisionner ressources en minutes (pas semaines)
  • ✅ Global reach : Déployer apps worldwide (datacenters multiples)
  • ✅ Disponibilité : SLA 99.9-99.99% (haute redondance)
  • ✅ Disaster recovery : Backup automatisé multi-régions
  • ✅ Innovation rapide : Accès services avancés (AI/ML, BigData)
  • ✅ Pas maintenance : Fournisseur gère hardware/datacenter

Résumé

🔑 Ce que tu dois retenir absolument

→IaaS : Loue VMs (client gère OS+apps). PaaS : Plateforme dev (client code apps). SaaS : App complète (client gère données)
→Public : Partagé multi-tenant, OPEX faible. Privé : Dédié, CAPEX élevé, contrôle total. Hybride : Public+Privé interconnectés
→Multi-Cloud : Plusieurs fournisseurs publics (AWS+Azure+GCP), évite lock-in, complexité gestion
→Responsabilité partagée : Fournisseur = infra (OF cloud), Client = données/apps/accès (IN cloud). Client TOUJOURS responsable données
