Demilitarized Zone

DMZ — Zone démilitarisée

🏰
Analogie — château médiéval Internet = wilderness (danger). Internal network = château (précieux, protégé). DMZ = marché village entre murailles externes/internes. Public accède marché (web server) MAIS pas château (database interne).

DMZ (Demilitarized Zone) = Sous-réseau isolé hébergeant services publics (web, email, DNS) accessible Internet MAIS séparé réseau interne.

// Principe DMZ

Problème résolu : Services publics (web server) doivent être accessibles Internet MAIS pas exposer réseau interne.

Architecture DMZ (3 zones) :

[Internet] ──> [Firewall External] ──> [DMZ]
                                                 │
                             Web Server (203.0.113.10)
                             Mail Server (203.0.113.11)
                             DNS Server (203.0.113.12)
                                                 │
                             [Firewall Internal]
                                                 │
                         [Internal Network]
                         Database (10.0.1.50)
                         File Server (10.0.1.51)
                         Employees (10.0.1.0/24)

// Règles firewall DMZ

External Firewall (Internet → DMZ)

  • Permit HTTP/HTTPS (80/443) → Web Server DMZ
  • Permit SMTP (25) → Mail Server DMZ
  • Permit DNS (53) → DNS Server DMZ
  • Deny all else

Internal Firewall (DMZ → Internal)

  • Permit Web Server → Database (specific queries only, port 3306)
  • Deny DMZ → Internal (default, très restrictif)

Internal → DMZ

  • Permit Internal → DMZ (admin access SSH, updates)

// Avantages DMZ

  • ✅ Isolation : Web server compromis ≠ accès internal network
  • ✅ Exposition contrôlée : Services publics Internet accessible, internal protégé
  • ✅ Defense in depth : 2 firewalls (external + internal)
  • ✅ Compliance : PCI-DSS exige DMZ web servers (cardholder data separated)
🎯
Question fréquente examen "Sous-réseau isolé hébergeant services publics (web, email) accessible Internet ?"
Réponse : DMZ (Demilitarized Zone)

"Nombre firewalls minimum architecture DMZ ?"
Réponse : 2 (external + internal)
Network isolation

Segmentation réseau — Divide and conquer

Segmentation = Diviser réseau en zones isolées (subnets, VLANs) → limiter propagation attaques, appliquer policies différentes.

// Objectifs segmentation

  • Isolation : Malware VLAN Finance ≠ propagation VLAN HR
  • Least privilege : Accès minimum nécessaire (guests pas accès servers)
  • Performance : Broadcast domains séparés (moins collisions)
  • Compliance : PCI-DSS exige cardholder data isolated

// Types segmentation

1. Segmentation physique

  • Réseaux séparés physiquement (câbles, switches différents)
  • ✅ Isolation maximale (air gap)
  • ⚠️ Coût élevé, complexité management

2. Segmentation logique (VLANs)

  • VLANs séparent broadcast domains (Layer 2)
  • ✅ Flexibilité, économique (1 switch physique, multiples VLANs)
  • Exemple : VLAN 10 Data, VLAN 20 Voice, VLAN 99 Management

3. Segmentation subnets (Layer 3)

  • Subnets IP différents (10.1.0.0/24, 10.2.0.0/24)
  • Routage entre subnets = firewall/ACL control point

// Exemple segmentation enterprise

VLAN 10 (Data) : 10.0.10.0/24
→ Employees workstations, printers
→ Access : Internal resources, Internet

VLAN 20 (Voice) : 10.0.20.0/24
→ VoIP phones
→ QoS priority, isolated traffic

VLAN 30 (Servers) : 10.0.30.0/24
→ File servers, app servers, databases
→ Access : VLAN 10 → VLAN 30 (permit specific ports)

VLAN 40 (Guest) : 192.168.1.0/24
→ Visitors WiFi
→ Access : Internet only (deny internal)

VLAN 99 (Management) : 10.0.99.0/24
→ Switch/router management interfaces
→ Access : Admin workstations only (highly restricted)
🔒
Segmentation = défense essentielle Flat network (tout même subnet) = malware spread instant.
Segmented network = attaquant bloqué firewall entre VLANs.
Modern standard : Microsegmentation (Zero Trust, chaque workload isolé).
Trust boundaries

Zones de confiance — Trust zones

Zones de confiance = Classifier réseau en zones trust levels (Internet untrusted, Internal trusted, DMZ semi-trusted). Policies différentes chaque zone.

// Hiérarchie trust zones (typique)

ZoneTrust levelExemplesPolicies
Internet (Untrusted) ❌ Zero trust Public Internet Deny all (sauf permit explicit services DMZ)
DMZ (Semi-trusted) ⚠️ Low trust Web servers, mail servers Permit public access, deny internal access
Internal (Trusted) ✅ Medium trust Employee workstations, printers Permit internal communication, Internet filtered
Secure (High trust) ✅ High trust Databases, financial systems, HR data Highly restricted (MFA, logging, encryption)
Management (Critical) ✅ Maximum trust Admin workstations, jump servers Admin-only, 2FA, session recording

// Règles trafic entre zones

Principe général : Trafic lower trust → higher trust = deny default. Higher trust → lower trust = permit contrôlé.

Internet → Internal : DENY ALL
Internet → DMZ : PERMIT (HTTP/HTTPS, SMTP, DNS)
DMZ → Internal : DENY (sauf specific DB queries web server)
Internal → Internet : PERMIT (filtered, proxy, IDS)
Internal → DMZ : PERMIT (admin SSH, updates)
Internal → Secure : PERMIT (authenticated, authorized users)

// Zero Trust model

Evolution moderne : "Never trust, always verify". Pas zones trusted/untrusted → TOUT traffic vérifié (même internal).

  • Assume breach (internal = potentiellement compromis)
  • Microsegmentation (chaque workload isolé)
  • Continuous authentication (pas trust après login)
  • Least privilege access (minimum rights)
Access principle

Principe du moindre privilège — Least Privilege

Least Privilege = Users/systems ont minimum permissions nécessaires accomplir tâches (rien de plus).

// Principe fondamental

Question : User besoin-t-il VRAIMENT admin rights ?

❌ BAD (excessive privileges) :
Employee marketing = Domain Admin
→ Peut : Delete servers, modify AD, access all files
→ Risque : Account compromis = attacker full control

✅ GOOD (least privilege) :
Employee marketing = Standard User
→ Peut : Access marketing files, use office apps
→ Risque : Account compromis = limited damage (marketing data only)

// Applications least privilege

1. User accounts

  • Employees = standard users (pas admin local)
  • Admin tasks = separate admin account (elevation UAC)
  • Service accounts = minimum rights (read-only si possible)

2. Network access

  • VLANs restrictifs (guests pas accès internal)
  • ACLs deny default (permit explicit necessaire)
  • Firewall rules spécifiques (deny broad "any any")

3. File permissions

  • Read-only par défaut (write si nécessaire)
  • Shared folders = permissions granulaires (group-based)
  • Sensitive data = encryption + restricted access

4. Application permissions

  • Database : Users SELECT only (pas INSERT/DELETE/DROP)
  • APIs : Rate limiting, scope restrictions (OAuth scopes)

// Bénéfices least privilege

  • ✅ Limit blast radius : Compromission 1 account ≠ catastrophe totale
  • ✅ Reduce attack surface : Moins permissions = moins exploitation possible
  • ✅ Compliance : PCI-DSS, HIPAA, SOX exigent least privilege
  • ✅ Audit trail : Permissions restrictives = easier track qui accède quoi
⚠️
Privilege creep = problème courant Users accumulent permissions au fil temps (changements rôles, projets).
Jamais nettoyage = everyone admin eventually.
Solution : Regular access reviews (quarterly), automated deprovisioning.
Layered security

Defense in Depth — Défense en profondeur

Defense in Depth = Multiples couches sécurité (layers). Attaquant bypass 1 layer → bloqué layer suivante.

🧅
Analogie — oignon sécurité Single layer security = porte château. Porte forcée = attacker inside (game over).
Defense in depth = oignon multiples couches (murailles, douves, gardes, portes, coffre-fort). Bypass 1 couche ≠ victoire → encore 5 layers défense.

// Layers défense (7 couches)

1. Policies & Procedures
→ Security awareness training, acceptable use policy, incident response

2. Physical Security
→ Datacenter locks, badge access, cameras, guards

3. Perimeter Security
→ Firewall, IDS/IPS, VPN, DMZ

4. Network Security
→ VLANs, ACLs, 802.1X, NAC

5. Host Security
→ Antivirus, host firewall, patch management, hardening

6. Application Security
→ Input validation, authentication, secure coding, WAF

7. Data Security
→ Encryption at-rest/in-transit, DLP, backups, access controls

// Exemple defense in depth

Scénario : Attaquant tente accéder database customer credit cards.

Layer 1 : Perimeter
→ Attacker phishing email → Employee clicks
→ Firewall blocks C2 callback (IPS signature match)
→ ❌ Attack stopped Layer 1

IF Layer 1 bypassed (0-day C2) :
Layer 2 : Network
→ Malware lateral movement → Blocked VLAN segmentation
→ ❌ Attack stopped Layer 2

IF Layer 2 bypassed (VLAN hopping exploit) :
Layer 3 : Host
→ Malware execute → Antivirus detection + quarantine
→ ❌ Attack stopped Layer 3

IF Layer 3 bypassed (AV evasion) :
Layer 4 : Application
→ SQL injection attempt → WAF blocks malicious query
→ ❌ Attack stopped Layer 4

IF Layer 4 bypassed (WAF blind spot) :
Layer 5 : Data
→ Database access → Credit cards ENCRYPTED (TDE)
→ Attacker gets ciphertext (useless without key)
→ ❌ Attack mitigated Layer 5

// Principe clé

"No single point of failure" : 1 control échoue ≠ breach. Redundancy défense.

  • ✅ Multiple independent controls (fail-safe)
  • ✅ Attacker doit bypass TOUS layers (exponentially harder)
  • ✅ Detection opportunities multiples (logs, alerts chaque layer)
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : DMZ = zone totalement sécurisée
FAUX. DMZ = semi-trusted (pas trusted).

DMZ expose Internet → risque compromission élevé.
DMZ isolée internal → compromission DMZ ≠ accès internal.

DMZ = buffer zone (sacrificial, acceptable compromise)
❌ Piège #2 : Least privilege = aucun privilège
INCOMPLET. Least privilege = minimum NÉCESSAIRE (pas zero).

User besoin access marketing files → GET permissions marketing folder.
User PAS besoin admin rights → DON'T get admin.

Balance : Functional (user peut travailler) + Secure (pas excessive)
❌ Piège #3 : Defense in depth = firewall + antivirus
INCOMPLET. Defense in depth = MULTIPLES layers (7+ categories).

Pas juste 2 tools → Policies, Physical, Perimeter, Network, Host, App, Data.
Chaque layer = multiples controls (firewall + IPS + VPN, etc.).

Comprehensive approach (pas minimal 2 products)
