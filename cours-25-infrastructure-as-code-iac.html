Révolution ops

IaC — Infrastructure as Code

📜
Analogie — recette cuisine Config manuelle = Chef impro (résultat varie, pas reproductible).
IaC = Recette écrite précise (même résultat toujours, n'importe qui peut reproduire).

Infrastructure as Code (IaC) = Gérer infrastructure (réseaux, serveurs, configs) via fichiers code versionnés, plutôt que config manuelle GUI/CLI.

// Problème config manuelle

  • ❌ Pas reproductible : SSH switch 1, oublie commande switch 2 → incohérence
  • ❌ Pas tracé : Qui a changé VLAN 10 hier ? Mystère
  • ❌ Erreurs humaines : Typo IP → réseau cassé
  • ❌ Lent : Configurer 50 switches = 50× SSH manuel
  • ❌ Pas testable : Changement prod direct = risqué

// Solution IaC

  • ✅ Code source : Config = fichiers texte (YAML, HCL)
  • ✅ Version control : Git → historique complet, rollback facile
  • ✅ Reproductible : Même code → même résultat (idempotent)
  • ✅ Automatisation : 1 commande → configure 1000 devices
  • ✅ Testing : Test changements lab avant prod
  • ✅ Documentation : Code = documentation infrastructure
🚀
IaC = DevOps/NetDevOps standard 2024 : Entreprises modernes TOUTES utilisent IaC. Config manuelle = legacy/petites structures. IaC = agilité, fiabilité, scalabilité.
Configuration management

Ansible — Automatisation agentless

Ansible = outil IaC configuration management. Automatise configuration/déploiement serveurs, réseaux, apps.

// Caractéristiques

  • Agentless : Pas besoin installer agent devices cibles (SSH/WinRM)
  • YAML : Syntaxe simple, human-readable
  • Idempotent : Exécuter 10× = même résultat que 1×
  • Push model : Contrôleur pousse configs vers devices
  • Modules : +3000 modules (Cisco IOS, Juniper, AWS, Azure...)

// Composants Ansible

Inventory : Fichier listant devices à gérer (IPs, groupes)

Playbook : Fichier YAML décrivant tâches à exécuter

Module : Code exécutant action spécifique (ios_config, nxos_vlan...)

Task : Unité travail (configurer VLAN, créer user...)

Role : Ensemble playbooks réutilisable (ex: "firewall_hardening")

// Exemple Playbook réseau

configure_vlans.yml

---
- name: Configurer VLANs sur switches Cisco
hosts: datacenter_switches
gather_facts: no

tasks:
- name: Créer VLAN 10 (Data)
cisco.ios.ios_vlans:
config:
- vlan_id: 10
name: Data_VLAN
state: merged

- name: Créer VLAN 20 (Voice)
cisco.ios.ios_vlans:
config:
- vlan_id: 20
name: Voice_VLAN
state: merged

- name: Assigner ports au VLAN 10
cisco.ios.ios_l2_interfaces:
config:
- name: GigabitEthernet0/1
mode: access
access:
vlan: 10
state: merged

Exécution :

$ ansible-playbook configure_vlans.yml

→ Ansible SSH vers TOUS switches groupe "datacenter_switches"
→ Exécute tâches (créer VLANs, assigner ports)
→ Idempotent : Si VLAN 10 existe déjà → skip (pas erreur)

// Cas d'usage réseau

  • Configuration initiale switches/routeurs (day-0)
  • Déploiement changements config (VLANs, ACLs, routes)
  • Backup configs automatisé (nightly)
  • Compliance checking (auditer configs vs baseline)
  • Rollback configs (restaurer version antérieure Git)
📚
Ansible = parfait débutants IaC YAML simple, agentless (juste SSH), courbe apprentissage douce. Standard automation réseau.
Infrastructure provisioning

Terraform — Infrastructure orchestration

Terraform = outil IaC provisioning infrastructure. Crée/gère ressources cloud (VPCs, VMs, réseaux) via code déclaratif.

// Caractéristiques

  • Declarative : Décris état final désiré (pas étapes)
  • HCL : HashiCorp Configuration Language (ou JSON)
  • Multi-cloud : AWS, Azure, GCP, VMware (1 outil → tous clouds)
  • State management : Fichier state.tf suit ressources créées
  • Plan/Apply : Preview changements avant exécution

// Workflow Terraform

  1. Write : Écrire config .tf (VPC, subnets, instances...)
  2. Plan : terraform plan → preview changements
  3. Apply : terraform apply → créer ressources
  4. State : terraform.tfstate enregistre ressources créées
  5. Destroy : terraform destroy → supprimer tout

// Exemple Terraform réseau

vpc.tf (AWS VPC + subnets)

provider "aws" {
region = "us-east-1"
}

resource "aws_vpc" "main" {
cidr_block = "10.0.0.0/16"
tags = {
Name = "Production-VPC"
}
}

resource "aws_subnet" "public" {
vpc_id = aws_vpc.main.id
cidr_block = "10.0.1.0/24"
availability_zone = "us-east-1a"
tags = {
Name = "Public-Subnet"
}
}

resource "aws_subnet" "private" {
vpc_id = aws_vpc.main.id
cidr_block = "10.0.2.0/24"
availability_zone = "us-east-1a"
tags = {
Name = "Private-Subnet"
}
}

Exécution :

$ terraform plan
→ Preview : Créer 1 VPC, 2 subnets

$ terraform apply
→ Créer ressources AWS réelles
→ State sauvegardé terraform.tfstate

$ terraform destroy
→ Supprimer VPC + subnets AWS

// Cas d'usage réseau

  • Provisioning VPCs cloud (AWS, Azure, GCP)
  • Création subnets, Security Groups, route tables
  • Déploiement infrastructure multi-cloud reproductible
  • Disaster recovery (rebuild infra depuis code)
  • Environnements éphémères (dev/test)
Comparaison

Ansible vs Terraform — Cas d'usage

// Différences clés

CritèreAnsibleTerraform
FocusConfiguration managementInfrastructure provisioning
TypeImperative (tasks)Declarative (state)
StatePas de state fileState file (terraform.tfstate)
SyntaxeYAML (simple)HCL (spécialisé infra)
AgentAgentless (SSH)Agentless (APIs)
ForteConfig devices existantsCréer/détruire ressources
Usage réseauConfigs switches/routeursVPCs/subnets cloud

// Quand utiliser quoi ?

Ansible :

  • Configurer équipements réseau existants (switches, routeurs, firewalls)
  • Déployer apps sur serveurs
  • Orchestration tasks multi-étapes (backup → config → test)
  • Compliance checking configs

Terraform :

  • Créer infrastructure cloud (VPCs, VMs, Load Balancers)
  • Provisioning multi-cloud (AWS + Azure + GCP)
  • Infrastructure éphémère (créer → utiliser → destroy)
  • State management important (track ressources)

Combinaison (meilleure pratique) :

1. Terraform : Créer VPC AWS + subnets + EC2 instances
2. Ansible : Configurer instances EC2 (install apps, users, firewall)

→ Terraform = provisioning (créer)
→ Ansible = configuration (configurer)
🔧
Terraform + Ansible = stack complet Terraform provision infra, Ansible configure. Standard DevOps moderne. Pas opposés → complémentaires.
Bénéfices

Avantages IaC — Reproductibilité et version control

// Reproductibilité

Idempotence : Exécuter code 10× = même résultat que 1×.

  • ✅ Environnements identiques (dev = prod)
  • ✅ Disaster recovery rapide (rebuild depuis code)
  • ✅ Scaling horizontal (50 serveurs identiques)

// Version Control (Git)

Code IaC = fichiers texte → Git/GitHub.

  • ✅ Historique complet : Qui changé quoi quand
  • ✅ Rollback facile : Revenir version antérieure (git revert)
  • ✅ Collaboration : Pull requests, reviews, merge
  • ✅ Branching : Test changements branch séparé
  • ✅ Audit trail : Compliance (PCI-DSS, SOC2)

// Testing

IaC = code → testable.

  • ✅ Dry-run : Ansible --check, Terraform plan (preview sans exécuter)
  • ✅ Environnements test : Deploy lab avant prod
  • ✅ CI/CD pipeline : Tests automatisés chaque commit

// Documentation vivante

Code = documentation auto-updated.

  • ✅ Config actuelle = fichiers Git (toujours à jour)
  • ✅ Pas doc Word obsolète

// Declarative vs Imperative

Imperative (Ansible) : Décris comment faire (étapes).

tasks:
- Créer VLAN 10
- Nommer VLAN 10 "Data"
- Assigner port Gi0/1 au VLAN 10

Declarative (Terraform) : Décris état final désiré.

resource "aws_vpc" "main" {
cidr_block = "10.0.0.0/16"
}

→ Terraform calcule steps nécessaires automatiquement
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : IaC = uniquement cloud
FAUX. IaC applicable partout : cloud, on-premise, réseaux physiques, VMs.

Ansible configure switches Cisco on-premise.
Terraform provision VMware vSphere.
❌ Piège #2 : Terraform = configuration management
NON. Terraform = provisioning (créer ressources).

Ansible = configuration management (configurer devices).
Terraform créer VPC, Ansible configure apps dedans
❌ Piège #3 : Ansible nécessite agents sur devices
FAUX. Ansible = agentless.

SSH (Linux/réseau) ou WinRM (Windows) → pas installation agent.
Puppet/Chef = agents requis (différent)
Résumé

🔑 Ce que tu dois retenir absolument

→IaC : Infra = code (YAML/HCL), version control Git, reproductible, testable, automatisé (vs config manuelle)
→Ansible : Configuration management, agentless (SSH), YAML, imperative, config devices existants (switches, serveurs)
→Terraform : Infrastructure provisioning, HCL, declarative, state file, créer/détruire ressources (VPCs cloud, VMs)
→Avantages : Reproductibilité (idempotent), version control (Git historique/rollback), testing (dry-run), documentation vivante
→Combinaison : Terraform provision (créer), Ansible configure (configurer) → stack complet DevOps
