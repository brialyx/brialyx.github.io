Name resolution failure

Pas de résolution nom → IP — DNS failure

Pas de résolution DNS = Client cannot resolve hostnames (google.com) → IP addresses → connections fail.

// Principe DNS (rappel Cours F27)

User : Browse google.com

1. PC query DNS server : "What's IP for google.com ?"
2. DNS server replies : "142.250.185.46"
3. PC connects : 142.250.185.46 (Google)

DNS failure :
→ Step 1 fails (no DNS server, timeout, wrong server)
→ Can't get IP address
→ Connection impossible

// Symptômes DNS failure

  • Can't browse websites : google.com, facebook.com timeout
  • Ping hostname fails : "ping google.com" → "could not find host"
  • Ping IP works : "ping 8.8.8.8" → Success ✅ (proves Internet OK)
  • Email fails : Can't resolve mail.company.com
  • Error message : "DNS server not responding" OR "Cannot resolve hostname"

// Causes DNS failure

1. DNS server down/unreachable

PC configured : DNS 192.168.1.100
DNS server : Offline (service stopped, server crashed)

→ Query timeout
→ No resolution

2. No DNS server configured

C:\> ipconfig /all

DNS Servers : (none)

→ No DNS server = no resolution

3. Firewall blocking DNS (port 53)

DNS uses : UDP/TCP port 53
Firewall blocks : port 53 outbound

→ Queries can't reach DNS server
→ Timeout

4. Network connectivity problem

  • Wrong gateway (can't reach DNS server different subnet)
  • Routing problem (DNS server unreachable)
  • Cable unplugged (no network at all)

// Diagnostic DNS failure

Step 1 : Verify DNS configured
C:\> ipconfig /all

DNS Servers : 192.168.1.100
              8.8.8.8 (backup)

→ DNS servers configured ✅

Step 2 : Test DNS resolution
C:\> nslookup google.com

DNS request timed out.
timeout was 2 seconds.
Server: UnKnown
Address: 192.168.1.100

*** Request to UnKnown timed-out

→ DNS server 192.168.1.100 not responding

Step 3 : Test alternative DNS
C:\> nslookup google.com 8.8.8.8
Server: dns.google
Address: 8.8.8.8

Non-authoritative answer:
Name: google.com
Addresses: 142.250.185.46

→ Google DNS (8.8.8.8) works ✅
→ Problem = local DNS server 192.168.1.100

Step 4 : Ping DNS server
C:\> ping 192.168.1.100
→ Timeout = DNS server unreachable (network problem)
→ Reply = DNS server reachable (service problem)

// Résolution DNS failure

  • Restart DNS service : Server-side (if service down)
  • Change DNS server : Use public DNS (8.8.8.8, 1.1.1.1) temporarily
  • Fix network connectivity : Routing, gateway, VLAN
  • Check firewall : Allow UDP/TCP 53 outbound
  • Flush DNS cache : Client-side ipconfig /flushdns
🎯
Question CLASSIQUE examen "User can ping 8.8.8.8 but NOT google.com. MOST likely problem ?"
A) Cable defect
B) DNS server failure
C) Gateway problem
D) NAT misconfigured

Réponse : B) DNS server failure
IP ping works, hostname fails = DNS resolution problem.
Incorrect configuration

Mauvais serveur DNS — Wrong DNS server

Mauvais DNS server = Client configured incorrect DNS server IP → queries sent wrong destination → timeouts OR wrong answers.

// Scénarios mauvais DNS server

1. Typo DNS IP address

Correct DNS : 192.168.1.100
Configured : 192.168.1.101 (typo - wrong server)

→ 192.168.1.101 : not a DNS server (OR nonexistent)
→ Queries timeout

2. DNS server external network (unreachable)

PC network : 192.168.1.0/24
Configured DNS : 10.0.0.100 (different network)
No route to 10.0.0.0/24

→ DNS server unreachable
→ Queries fail

3. Internal DNS for Internet queries (split DNS problem)

Internal DNS : 192.168.1.100 (company.local only)
PC configured : ONLY 192.168.1.100

Query google.com → Internal DNS
Internal DNS : "Don't know google.com" (no forwarder configured)
→ Resolution fails

Fix : Add forwarder to public DNS (8.8.8.8) OR configure both:
Primary : 192.168.1.100 (internal)
Secondary : 8.8.8.8 (Internet)

// Diagnostic wrong DNS server

C:\> ipconfig /all
DNS Servers : 192.168.1.101

Verify DNS server exists :
C:\> ping 192.168.1.101
→ Timeout = server doesn't exist/unreachable

Check correct DNS :
→ Ask admin : "What's DNS server this network ?"
→ Check other working PCs : ipconfig /all

Working PC shows : DNS 192.168.1.100 (different!)
→ Wrong server identified : .101 should be .100

// Résolution wrong DNS server

Windows :
Network Adapter Properties → TCP/IPv4
→ Change DNS server : 192.168.1.100 (correct)
→ Add alternate : 8.8.8.8 (backup public DNS)

OR DHCP :
→ Enable "Obtain DNS server automatically"
→ DHCP provides correct DNS

Verify fix :
C:\> ipconfig /all
DNS Servers : 192.168.1.100 ✅

C:\> nslookup google.com
→ Success ✅
Security threat

Cache DNS empoisonné — DNS cache poisoning

DNS cache poisoning = Attacker inserts fake DNS records → client gets wrong IP → redirected malicious site.

// Principe DNS cache poisoning

Normal DNS query :
PC → DNS server : "bank.com ?"
DNS server → Authoritative DNS : "bank.com ?"
Authoritative DNS : "203.0.113.50"
DNS server caches : bank.com → 203.0.113.50
PC receives : 203.0.113.50 ✅

Cache poisoning attack :
Attacker spoofs DNS response (before legitimate reply)
Fake response : bank.com → 198.51.100.10 (attacker's site)
DNS server caches : bank.com → 198.51.100.10 (poisoned!)

All future queries :
PC → DNS : "bank.com ?"
DNS (poisoned cache) : "198.51.100.10"
PC connects : Attacker's fake bank site (phishing)

// Symptômes cache poisoning

  • Wrong website loads : bank.com → suspicious site (phishing)
  • SSL certificate mismatch : Browser warning "Certificate name doesn't match"
  • Intermittent : Some users affected (poisoned cache), others not
  • nslookup shows wrong IP : Expected 203.0.113.50, gets 198.51.100.10

// Diagnostic cache poisoning

Test resolution :
C:\> nslookup bank.com
Server: dns.company.local
Address: 192.168.1.100

Name: bank.com
Address: 198.51.100.10 (wrong!)

Verify against public DNS :
C:\> nslookup bank.com 8.8.8.8
Server: dns.google
Address: 8.8.8.8

Name: bank.com
Address: 203.0.113.50 (correct)

→ Mismatch = local DNS cache poisoned

// Résolution cache poisoning

Immediate fix :
DNS server : Flush cache (clear poisoned entries)

Windows DNS Server :
> dnscmd /clearcache

Linux BIND :
$ sudo rndc flush

Prevention :
→ DNSSEC (DNS Security Extensions) : Validate responses cryptographically
→ Randomize source ports (harder spoof responses)
→ Update DNS software (patch vulnerabilities)
→ Monitor unusual DNS traffic
⚠️
Cache poisoning = serious security threat Redirects users malicious sites (phishing, malware).
User thinks legitimate site (URL correct, fake site looks real).
DNSSEC = defense (validates DNS responses authentic).
Time To Live

TTL trop court / trop long — TTL misconfiguration

TTL (Time To Live) = Durée (secondes) DNS record cached. Affects performance, propagation changes.

// Principe DNS TTL

DNS record with TTL :
example.com. 3600 IN A 203.0.113.10
                ↑
                TTL 3600 seconds (1 hour)

Caching behavior :
1. DNS server queries authoritative
2. Gets response + TTL 3600
3. Caches record 1 hour
4. Future queries (within 1h) → Use cache (fast)
5. After 1h → Cache expires → Re-query authoritative

// Problème TTL trop court (< 5 minutes)

Impact :

  • Excessive DNS queries : Cache expires rapidly → constant re-queries
  • DNS server load : Authoritative server overwhelmed
  • Slow performance : Frequent queries add latency
  • Higher bandwidth : DNS traffic increased
Example TTL trop court :
example.com 60 IN A 203.0.113.10
TTL 60 seconds (1 minute)

→ Cache expires every minute
→ 1000 users → 1000 queries/minute (instead 1/hour if TTL 3600)
→ DNS server overloaded

// Problème TTL trop long (> 24 hours)

Impact :

  • Slow DNS propagation : Changes take DAYS propagate
  • Stale cached data : Users get old IP (server moved)
  • Difficult migrations : Can't switch servers quickly
  • Disaster recovery delay : Failover slow (cached old IP)
Example TTL trop long :
example.com 86400 IN A 203.0.113.10
TTL 86400 seconds (24 hours)

Admin changes IP : 203.0.113.10 → 203.0.113.20

Problem :
→ Cached records still valid 24h
→ Users continue accessing 203.0.113.10 (old IP) until cache expires
→ Migration takes 24 hours complete (unacceptable downtime)

// TTL Best Practices

ScenarioRecommended TTLRaison
Normal operation 1-4 hours (3600-14400) Balance performance vs flexibility
Before DNS change 5 minutes (300) Fast propagation upcoming migration
During migration 5 minutes (300) Quick failback if problems
After migration stable 1 hour (3600) Return normal (reduce queries)
High-availability critical 30 seconds - 5 minutes Fast failover required

// Diagnostic TTL problems

Check current TTL :
C:\> nslookup -type=A example.com

Non-authoritative answer:
Name: example.com
Address: 203.0.113.10
TTL: 60 (too short!)

OR :

Linux/Mac :
$ dig example.com

;; ANSWER SECTION:
example.com. 60 IN A 203.0.113.10
              ↑
              TTL seconds

// Résolution TTL incorrect

Authoritative DNS server (BIND example) :
example.com. IN SOA ns1.example.com. admin.example.com. (
    2024030101 ; Serial
    7200 ; Refresh
    3600 ; Retry
    3600 ; Expire
    600 ) ; Negative TTL

example.com. 3600 IN A 203.0.113.10
              ↑
              Change to appropriate value (1 hour = 3600)
⏱️
TTL strategy migrations 1 week before : Lower TTL 5 minutes (allow fast changes).
During migration : Keep TTL 5 minutes (quick rollback if issues).
1 week after stable : Raise TTL 1 hour (reduce load).
This minimizes propagation delay while maintaining flexibility.
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : DNS problem = can't ping anything
FAUX. DNS problem = can ping IPs, can't resolve hostnames.

Test : ping 8.8.8.8 (works) vs ping google.com (fails).
If both fail → Network problem (not DNS specific).

DNS resolves names→IPs. Ping IP directly bypasses DNS.
Examen : "IP ping works, hostname fails" = DNS problem signature.
❌ Piège #2 : ipconfig /flushdns fixes DNS server down
FAUX. ipconfig /flushdns = clears local cache only.

Doesn't fix : DNS server down, wrong DNS configured, network unreachable.
Fixes : Stale cached entries, local cache corruption.

DNS server down requires : Server restart, change DNS server, fix network.
Examen : Flush cache = client-side fix (not server problems).
❌ Piège #3 : TTL = how long before DNS record deleted
FAUX. TTL = how long record cached, not deleted.

TTL expires → Cache entry removed → Re-query authoritative (NOT deleted from authoritative).
Authoritative DNS : Records permanent (until admin changes).
Caching DNS : Records temporary (TTL duration).

Examen : TTL controls caching duration, not record lifespan authoritative.
Résumé

🔑 Ce que tu dois retenir absolument

→Pas de résolution : Ping IP works, hostname fails. Causes: DNS server down, no DNS configured, firewall port 53, network unreachable. Test: nslookup, try 8.8.8.8. Fix: Restart DNS, change server, fix network
→Mauvais serveur DNS : Wrong IP configured (typo, unreachable). Internal DNS no forwarder (can't resolve Internet). Test: nslookup, compare working PCs. Fix: Correct DNS IP, add forwarder/backup 8.8.8.8
→Cache empoisonné : Attacker spoofs DNS → wrong IP cached → phishing redirect. Symptom: Wrong site, SSL mismatch. Test: Compare local vs public DNS (8.8.8.8). Fix: Flush cache, enable DNSSEC
→TTL trop court : Excessive queries, DNS load, slow. TTL < 5 min problematic. TTL trop long : Slow propagation, stale data, difficult migrations. TTL > 24h problematic. Best: 1-4h normal, 5 min migrations
→Commands : ipconfig /all (DNS configured?), nslookup hostname (resolution test), nslookup hostname 8.8.8.8 (test public), ipconfig /flushdns (clear local cache), dig (Linux TTL check)
→Diagnostic workflow : 1) Ping IP (8.8.8.8 works?), 2) Ping hostname (google.com fails?), 3) Check DNS config (ipconfig /all), 4) Test DNS (nslookup), 5) Try public DNS (8.8.8.8), 6) Fix root cause
