Avant de commencer

C'est quoi le Default Gateway — la porte de sortie du réseau local

🚪
Analogie — la porte de sortie d'un immeuble Imagine que ton réseau local = ton immeuble. Pour parler à un voisin du même étage (même réseau) → tu marches direct jusqu'à sa porte. Pour parler à quelqu'un dans un autre immeuble (autre réseau) → tu dois passer par la porte de sortie de l'immeuble (le gateway), qui te mène au monde extérieur.

Le Default Gateway (passerelle par défaut) est l'adresse IP du routeur qui permet à un hôte de communiquer avec des appareils situés en dehors de son sous-réseau local.

Rôle simple :

  • Destination locale (même réseau) → communication directe
  • Destination distante (autre réseau) → envoyer au gateway, qui route vers la destination
🎯
Pour l'examen Network+ CompTIA teste massivement : comment l'hôte décide local vs distant, symptômes d'un gateway manquant, et diagnostic de problèmes de routage. Maîtrise le processus de décision — c'est fondamental.
Vocabulaire essentiel

Les mots clés de ce cours

Default Gateway (Passerelle par défaut)
Adresse IP du routeur qui permet de sortir du réseau local vers d'autres réseaux. Généralement le premier IP utilisable du sous-réseau (ex: 192.168.1.1).
Réseau local (Local Network)
Ensemble des IPs appartenant au même sous-réseau. Déterminé par l'IP + masque de sous-réseau.
Réseau distant (Remote Network)
Toute destination IP en dehors du sous-réseau local. Nécessite un routeur (gateway) pour y accéder.
Table de routage (Routing Table)
Table interne de l'hôte qui contient les routes connues. Contient au minimum : route locale (même réseau) + route par défaut (0.0.0.0/0 → gateway).
Processus de décision

Comment l'hôte décide : local ou gateway ?

Quand un hôte veut envoyer un paquet, il doit répondre à UNE question : "La destination est-elle dans mon réseau local ou dans un réseau distant ?"

// L'algorithme de décision (CRUCIAL)

🧮 Processus étape par étape
Étape 1 : Récupérer les infos de l'hôte
- Mon IP : 192.168.1.10
- Mon masque : 255.255.255.0
- Mon gateway : 192.168.1.1

Étape 2 : Appliquer le masque sur mon IP
192.168.1.10 AND 255.255.255.0 = 192.168.1.0 ← mon réseau

Étape 3 : Appliquer le MÊME masque sur l'IP destination
Exemple destination : 192.168.1.50
192.168.1.50 AND 255.255.255.0 = 192.168.1.0 ← réseau de la destination

Étape 4 : Comparer
- Mon réseau = 192.168.1.0
- Réseau destination = 192.168.1.0
- Conclusion : MÊME réseau → communication directe ✅

Si différent :
Destination : 10.5.2.100
10.5.2.100 AND 255.255.255.0 = 10.5.2.0
- Mon réseau = 192.168.1.0
- Réseau destination = 10.5.2.0
- Conclusion : RÉSEAUX DIFFÉRENTS → envoyer au gateway (192.168.1.1) 🚪

// Exemple concret pas à pas

Configuration de l'hôte :

  • IP : 192.168.10.25
  • Masque : 255.255.255.0
  • Gateway : 192.168.10.1

Scénario 1 : Ping 192.168.10.30

  1. Calcul : 192.168.10.25 AND 255.255.255.0 = 192.168.10.0 (mon réseau)
  2. Calcul : 192.168.10.30 AND 255.255.255.0 = 192.168.10.0 (réseau destination)
  3. Même réseau ! → Envoyer directement à 192.168.10.30 (pas besoin du gateway)
  4. L'hôte fait une requête ARP pour obtenir la MAC de .30, puis envoie le paquet

Scénario 2 : Ping 8.8.8.8 (Google DNS)

  1. Calcul : 192.168.10.25 AND 255.255.255.0 = 192.168.10.0 (mon réseau)
  2. Calcul : 8.8.8.8 AND 255.255.255.0 = 8.8.8.0 (réseau destination)
  3. Réseaux différents ! → Envoyer au gateway (192.168.10.1)
  4. L'hôte fait une requête ARP pour la MAC du gateway (.1), puis envoie le paquet au gateway
  5. Le gateway route le paquet vers Internet
🔑
Règle d'OR L'hôte applique TOUJOURS son propre masque à la destination pour décider. Il ne demande pas au gateway "est-ce que tu connais cette IP ?". Il décide seul en comparant les réseaux.
Conséquences

Sans gateway = pas de communication inter-réseau

Que se passe-t-il si le gateway n'est PAS configuré ?

🚫
Symptômes d'un gateway manquant ou incorrect Réseau local fonctionne : Tu peux pinguer d'autres PC du même réseau (192.168.1.x ↔ 192.168.1.y) ✅

Internet ne fonctionne PAS : Tu ne peux pas pinguer 8.8.8.8, ni accéder à des sites web ❌

Pourquoi ? L'hôte n'a aucune idée où envoyer les paquets pour les réseaux distants. Il essaie quand même de les envoyer en broadcast sur le réseau local → échec.

// Diagnostic étape par étape

Test 1 : Ping loopback

C:\> ping 127.0.0.1
✅ Réponse reçue → Pile TCP/IP fonctionne

Test 2 : Ping mon IP

C:\> ping 192.168.1.10
✅ Réponse reçue → Interface réseau fonctionne

Test 3 : Ping un PC local (même réseau)

C:\> ping 192.168.1.50
✅ Réponse reçue → Réseau local fonctionne

Test 4 : Ping le gateway

C:\> ping 192.168.1.1
❌ Délai d'attente dépassé → Gateway inaccessible !

Test 5 : Ping Internet (8.8.8.8)

C:\> ping 8.8.8.8
❌ Délai d'attente dépassé → Pas d'accès Internet

Diagnostic : Gateway manquant ou incorrect. Solutions :

  1. Vérifier la config IP : ipconfig /all (Windows) ou ip addr (Linux)
  2. Gateway défini ? Si non → le configurer
  3. Gateway correct ? Doit être dans le même sous-réseau que l'hôte
  4. Gateway accessible ? Pinguer le gateway
  5. Câble branché ? VLAN correct ?
Erreurs courantes

Gateway mal configuré — les pièges

// Erreur 1 : Gateway dans un autre sous-réseau

Configuration INCORRECTE :

  • IP hôte : 192.168.1.10
  • Masque : 255.255.255.0
  • Gateway : 192.168.2.1 ❌ (autre réseau !)

Problème : Le gateway 192.168.2.1 n'est PAS dans le réseau local (192.168.1.0/24). L'hôte ne pourra JAMAIS l'atteindre directement → pas de communication possible.

Symptôme : Impossible de pinguer le gateway. Internet ne fonctionne pas.

// Erreur 2 : Gateway = adresse réseau ou broadcast

Configurations INCORRECTES :

  • Gateway : 192.168.1.0 ❌ (adresse réseau)
  • Gateway : 192.168.1.255 ❌ (broadcast)

Ces adresses ne peuvent PAS être assignées à un appareil → gateway invalide.

// Erreur 3 : Pas de gateway du tout

Configuration :

  • IP : 192.168.1.10
  • Masque : 255.255.255.0
  • Gateway : (vide) ❌

Conséquence : Réseau local fonctionne, mais AUCUN accès à l'extérieur (Internet, autres sous-réseaux).

🎯
Question type examen "Un utilisateur peut pinguer d'autres PC du bureau mais pas Internet. Quelle est la cause probable ?"

Réponse : Gateway manquant, incorrect, ou inaccessible. Le réseau local fonctionne (même sous-réseau) mais pas l'accès distant.
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : L'hôte demande au gateway
Beaucoup pensent que l'hôte "demande au gateway s'il connaît la destination". FAUX.

L'hôte décide SEUL en comparant les réseaux (IP dst AND masque). Si réseaux différents → il envoie au gateway sans demander. Le gateway n'est consulté qu'APRÈS la décision.
❌ Piège #2 : Gateway peut être n'importe où
FAUX. Le gateway DOIT être dans le même sous-réseau que l'hôte.

Si hôte = 192.168.1.10/24 → gateway DOIT être 192.168.1.x (entre .1 et .254)

Un gateway 192.168.2.1 ne fonctionnera jamais car l'hôte ne peut pas l'atteindre directement.
❌ Piège #3 : Gateway manquant = rien ne fonctionne
FAUX. Sans gateway, le réseau LOCAL fonctionne parfaitement.

Fonctionne : Communication entre PC du même réseau ✅
Ne fonctionne PAS : Internet, autres sous-réseaux ❌

C'est un piège fréquent : "Réseau local OK mais pas Internet" → vérifier le gateway.
❌ Piège #4 : ARP pour la destination distante
Beaucoup pensent que l'hôte fait ARP pour l'IP distante (ex: 8.8.8.8). FAUX.

Pour une destination distante, l'hôte fait ARP pour obtenir la MAC du GATEWAY, pas la MAC de la destination finale.

Le paquet IP contient : IP dst = 8.8.8.8, mais trame Ethernet : MAC dst = MAC du gateway.
🎯 Questions types

Ce que CompTIA peut te poser

Question type #1 : Diagnostic réseau
"Un utilisateur (IP: 192.168.1.50/24) peut communiquer avec d'autres PC du bureau mais pas accéder à Internet. Quelle est la première chose à vérifier ?"

Réponse : Vérifier que le default gateway est configuré et accessible (ping 192.168.1.1).
Question type #2 : Configuration invalide
"Configuration : IP = 192.168.5.100, Masque = 255.255.255.0, Gateway = 192.168.10.1. Quel est le problème ?"

Réponse : Le gateway (192.168.10.1) n'est PAS dans le même sous-réseau que l'hôte (192.168.5.0/24). Gateway invalide → pas d'accès Internet.
Question type #3 : Processus de décision
"Un PC (192.168.1.10/24, gateway 192.168.1.1) veut pinguer 192.168.1.200. L'hôte envoie-t-il le paquet au gateway ?"

Réponse : Non. 192.168.1.200 est dans le même réseau (192.168.1.0/24) → communication directe, pas besoin du gateway.
Question type #4 : ARP destination distante
"Un PC veut envoyer un paquet vers 8.8.8.8 (Internet). Pour quelle IP va-t-il faire une requête ARP ?"

Réponse : Pour l'IP du gateway (ex: 192.168.1.1), pas pour 8.8.8.8. L'hôte a besoin de la MAC du gateway pour lui transmettre le paquet.
