Protocole monitoring

SNMP — Simple Network Management Protocol

📡
Analogie — capteurs santé SNMP = capteurs santé équipements réseau. Manager (monitoring server) = médecin, Agents (switches/routeurs) = patients avec capteurs (CPU, RAM, bande passante). Manager interroge agents régulièrement, agents signalent anomalies immédiatement.

SNMP (Simple Network Management Protocol) = protocole standard monitoring/gestion équipements réseau. Collecte métriques (CPU, RAM, interfaces, erreurs), configuration à distance, alertes.

// Architecture SNMP

3 composants :

1. SNMP Manager (NMS - Network Management System)

  • Serveur monitoring (PRTG, Zabbix, LibreNMS)
  • Interroge agents (polling)
  • Reçoit alertes (traps)
  • Affiche dashboards, graphiques

2. SNMP Agent

  • Software intégré équipements (switches, routeurs, serveurs)
  • Collecte métriques locales
  • Répond requêtes Manager
  • Envoie traps (alertes) au Manager

3. MIB (Management Information Base)

  • Base données structure arborescente
  • Définit objets monitorables (CPU, RAM, interfaces...)
  • Chaque objet = OID unique (Object Identifier)

// Workflow SNMP

[SNMP Manager — PRTG]
↓ SNMP GET (UDP 161)
[Switch Agent] → Répond CPU 45%, Gi0/1 utilisation 60%
↑ SNMP TRAP (UDP 162)
[Switch Agent] → Alerte : Interface Gi0/2 down !

→ Polling : Manager interroge régulièrement
→ Trap : Agent signale événement important immédiatement
⚠️
Ports SNMP standards UDP 161 : Requêtes Manager → Agent (GET, SET)
UDP 162 : Traps Agent → Manager (alertes)
UDP (pas TCP) = performances (pas acknowledgments)
Structure données

OID et MIB — Identifiants objets

// OID (Object Identifier)

OID = identifiant numérique hiérarchique unique objet SNMP. Notation pointée (ex: 1.3.6.1.2.1.1.1.0).

Structure hiérarchique :

1 (iso)
└─ 3 (org)
└─ 6 (dod)
└─ 1 (internet)
├─ 2 (mgmt)
│  └─ 1 (mib-2)
│     ├─ 1 (system)
│     │  ├─ 1 (sysDescr) → 1.3.6.1.2.1.1.1.0
│     │  ├─ 3 (sysUpTime) → 1.3.6.1.2.1.1.3.0
│     │  └─ 5 (sysName) → 1.3.6.1.2.1.1.5.0
│     ├─ 2 (interfaces)
│     │  └─ 2.1.10 (ifInOctets) → Bytes entrants interface
│     └─ 25 (host)
└─ 4 (private)
└─ 1 (enterprises)
├─ 9 (Cisco)
├─ 2636 (Juniper)
└─ 311 (Microsoft)

// Exemples OIDs courants

OIDDescription
1.3.6.1.2.1.1.1.0sysDescr (description device)
1.3.6.1.2.1.1.3.0sysUpTime (uptime depuis boot)
1.3.6.1.2.1.1.5.0sysName (hostname)
1.3.6.1.2.1.2.2.1.10.XifInOctets (bytes IN interface X)
1.3.6.1.2.1.2.2.1.16.XifOutOctets (bytes OUT interface X)
1.3.6.1.2.1.25.3.3.1.2hrProcessorLoad (CPU usage)

// MIB (Management Information Base)

MIB = fichier texte définissant structure arborescente OIDs + types données + descriptions.

MIBs standards :

  • MIB-II (RFC 1213) : MIB standard universel (system, interfaces, IP, TCP, UDP)
  • IF-MIB : Interfaces réseau détaillées
  • HOST-RESOURCES-MIB : CPU, RAM, disques

MIBs vendor-specific :

  • Cisco MIBs (.1.3.6.1.4.1.9.*)
  • Juniper MIBs (.1.3.6.1.4.1.2636.*)
  • HP MIBs, Dell MIBs, etc.

Utilité MIBs : Manager charge MIBs → traduit OID numérique en nom lisible (1.3.6.1.2.1.1.5.0 = sysName).

🎯
Question fréquente examen "Identifiant numérique unique objet SNMP ?"

Réponse : OID (Object Identifier)

"Base données définissant objets monitorables SNMP ?"
Réponse : MIB (Management Information Base)
Modes collecte

Polling vs Traps

// Polling (Interrogation active)

Manager interroge Agent régulièrement (ex: toutes les 5 minutes).

Workflow :

  1. Manager envoie SNMP GET request (UDP 161)
  2. Agent répond valeur OID demandée
  3. Manager stocke data, affiche graphique
  4. Répète intervalles réguliers

Commandes SNMP polling :

  • GET : Lire 1 OID (ex: CPU usage)
  • GET-NEXT : Lire OID suivant dans arbre (parcourir MIB)
  • GET-BULK : Lire multiples OIDs à la fois (plus efficace, v2c/v3 uniquement)

Avantages polling :

  • ✅ Historique continu (graphiques tendances)
  • ✅ Détecte dégradation progressive (CPU monte lentement)
  • ✅ Contrôlé par Manager (pas surcharge Agents)

Inconvénients polling :

  • ⚠️ Bande passante (polling 100 devices toutes les 5 min)
  • ⚠️ Latence détection (événement entre 2 polls = pas détecté immédiatement)

// Traps (Alertes push)

Agent envoie notification Manager quand événement important (interface down, CPU >90%, etc.).

Workflow :

  1. Événement critique sur Agent (interface down)
  2. Agent envoie SNMP TRAP (UDP 162) immédiatement au Manager
  3. Manager reçoit alerte, notifie admins

Types traps :

  • coldStart : Device redémarré
  • linkDown : Interface physique down
  • linkUp : Interface up
  • authenticationFailure : Tentative accès SNMP invalide
  • Custom traps : Vendor-specific (Cisco, Juniper...)

Avantages traps :

  • ✅ Notification instantanée (pas attendre prochain poll)
  • ✅ Bande passante économisée (pas polling constant)
  • ✅ Événements critiques prioritaires

Inconvénients traps :

  • ⚠️ UDP = pas garanti livraison (trap perdu possible)
  • ⚠️ Pas historique continu (uniquement événements ponctuels)

// Polling + Traps = Best practice

Configuration optimale : Combiner les deux.

  • Polling : Métriques continues (CPU, RAM, bande passante) → graphiques tendances
  • Traps : Événements critiques (interface down, reboot) → alertes immédiates
🔄
Polling vs Traps — Complémentaires Polling = monitoring continu santé générale. Traps = sirène alarme événements critiques. Les deux ensemble = visibilité complète réseau.
Évolution protocole

SNMP v1 / v2c / v3 — Versions

// SNMP v1 (1988)

Version originale. Simple, largement supporté, MAIS sécurité nulle.

Caractéristiques :

  • Authentification : Community string (mot de passe texte clair)
  • Chiffrement : ❌ Aucun
  • Opérations : GET, GET-NEXT, SET, TRAP
  • Community strings :
    - Public (lecture seule, par défaut)
    - Private (lecture/écriture)

Problème sécurité : Community string = texte clair → Wireshark capture = compromise totale.

// SNMP v2c (1993)

Améliorations performances, sécurité toujours faible.

Nouveautés :

  • GET-BULK : Récupérer multiples OIDs efficacement
  • Inform : Trap avec acknowledgment (fiable)
  • 64-bit counters : Support interfaces >4 Gbps
  • Sécurité : ❌ Toujours community strings texte clair

Usage : v2c = plus courant que v1 (performances), MAIS sécurité identique (faible).

// SNMP v3 (2002)

Révolution sécurité. Authentification forte + chiffrement.

Sécurité v3 (USM - User-based Security Model) :

3 niveaux sécurité :

1. noAuthNoPriv

  • Pas authentification, pas chiffrement
  • Username uniquement (équivalent community string)
  • ⚠️ Pas sécurisé (comme v1/v2c)

2. authNoPriv

  • ✅ Authentification : HMAC-MD5 ou HMAC-SHA (hash password)
  • ❌ Pas chiffrement données
  • ⚠️ Wireshark voit données, MAIS pas modifier (intégrité garantie)

3. authPriv (RECOMMANDÉ)

  • ✅ Authentification : HMAC-SHA (SHA-256/512 preferred)
  • ✅ Chiffrement : DES, 3DES, AES-128/192/256
  • ✅ Sécurité complète (confidentialité + intégrité + authentification)

// Configuration SNMPv3 exemple

Cisco IOS config SNMPv3 authPriv :

snmp-server group ADMINS v3 priv
snmp-server user john ADMINS v3 auth sha MyAuthPass123 priv aes 256 MyPrivPass456

→ User "john"
→ Authentification : SHA + password "MyAuthPass123"
→ Chiffrement : AES-256 + password "MyPrivPass456"
→ Sécurité maximale (authPriv)

// Comparaison versions

VersionAuthentificationChiffrementSécuritéUsage
v1Community string❌ Aucun❌ NulleLegacy
v2cCommunity string❌ Aucun❌ NullePerformances (GET-BULK)

🔒
SNMPv3 authPriv = OBLIGATOIRE production v1/v2c = sécurité nulle (community strings texte clair). Environnements sensibles : v3 authPriv AES-256 MINIMUM. Wireshark capture v1/v2c = compromise complète.
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : SNMP utilise TCP
FAUX. SNMP utilise UDP (pas TCP).

UDP 161 : Manager → Agent (GET, SET)
UDP 162 : Agent → Manager (TRAP)

UDP = performances (pas overhead TCP acknowledgments)
❌ Piège #2 : SNMPv2c ajoute sécurité vs v1
FAUX. v2c = même sécurité nulle que v1.

v2c améliore performances (GET-BULK, 64-bit counters) MAIS sécurité identique (community strings texte clair).

Sécurité vraie = SNMPv3 authPriv uniquement
❌ Piège #3 : Traps garantissent livraison
FAUX. Traps = UDP → pas garanti livraison.

Trap perdu possible (congestion réseau, firewall).

Solution : Inform (SNMPv2c/v3) = trap avec acknowledgment → fiable
Résumé

🔑 Ce que tu dois retenir absolument

→SNMP : Protocole monitoring réseau. Manager (NMS) interroge Agents (équipements). UDP 161 (GET/SET), UDP 162 (TRAP)
→OID : Identifiant numérique unique objet (ex: 1.3.6.1.2.1.1.5.0 = sysName). MIB : Base données définissant structure OIDs
→Polling : Manager interroge Agent régulièrement (historique continu). Traps : Agent alerte Manager événement critique (instantané). Combiner les deux
→v1/v2c : Community strings texte clair, sécurité nulle. v3 : authPriv = authentification (HMAC-SHA) + chiffrement (AES-256) → Production OBLIGATOIRE
→Commandes : GET (1 OID), GET-BULK (multiples OIDs v2c/v3), SET (modifier config), TRAP (alerte), Inform (trap avec ACK)
