Packet analyzer

Wireshark — Introduction

Wireshark = Analyseur de paquets réseau (packet sniffer). Capture trafic en temps réel, analyse protocoles, troubleshooting réseau.

// Principe fonctionnement

1. Promiscuous mode : NIC capture ALL packets (not just addressed to it)
2. Capture : Wireshark reads packets from NIC
3. Decode : Parse protocols (Ethernet, IP, TCP, HTTP, etc.)
4. Display : Show packets human-readable format
5. Filter : Display specific traffic (by IP, port, protocol)
6. Analyze : Statistics, follow streams, export data

// Installation

  • Windows/macOS/Linux : Download wireshark.org (free, open source)
  • Requires admin/root : Promiscuous mode = privileged operation
  • WinPcap/Npcap : Windows driver required (packet capture)

// Interface Wireshark

3 panels principaux :

1. Packet List (top) :
   → Summary each packet (time, source, dest, protocol, info)
   → One line per packet

2. Packet Details (middle) :
   → Protocol layers expandable (Ethernet → IP → TCP → HTTP)
   → Header fields detailed

3. Packet Bytes (bottom) :
   → Raw hexadecimal data
   → ASCII representation
⚠️
Wireshark = powerful tool, use responsibly Captures ALL traffic (including passwords if unencrypted).
Corporate networks : Get permission before packet capture.
Legal/ethical : Only capture traffic you're authorized to monitor.
Filtering traffic

Filtres de capture — Capture Filters

Capture filters = Applied BEFORE capture (limit what's captured). Syntax BPF (Berkeley Packet Filter).

// Capture filter vs Display filter

TypeWhen appliedPurposeSyntax
Capture filter Before capture Limit what's captured (save disk/memory) BPF (tcpdump)
Display filter After capture Show subset captured packets Wireshark syntax

// Capture filter examples

Host-based :
host 192.168.1.100
→ Capture traffic to/from 192.168.1.100

src host 192.168.1.50
→ Only traffic FROM 192.168.1.50

dst host 8.8.8.8
→ Only traffic TO 8.8.8.8

Port-based :
port 80
→ HTTP traffic (port 80)

port 53
→ DNS traffic

portrange 1000-2000
→ Ports 1000 to 2000

Protocol-based :
tcp
→ TCP traffic only

udp
→ UDP traffic only

icmp
→ ICMP (ping) traffic

Combinations (AND, OR, NOT) :
host 192.168.1.100 and port 443
→ Traffic to/from .100 on port 443 (HTTPS)

tcp port 80 or tcp port 443
→ HTTP OR HTTPS

not broadcast and not multicast
→ Exclude broadcast/multicast (cleaner capture)

// Setting capture filter

1. Wireshark → Capture → Options
2. Select interface (eth0, Wi-Fi, etc.)
3. Enter capture filter field : port 80
4. Start capture

→ Only HTTP packets captured ✅
💡
Capture filter = performance optimization Busy network : Capture ALL = millions packets (disk/memory).
Capture filter : Only relevant traffic (focused troubleshooting).
Example : "port 53" = DNS only (ignore other traffic).
Post-capture filtering

Filtres d'affichage — Display Filters

Display filters = Applied AFTER capture (show subset packets). More flexible syntax than capture filters.

// Display filter examples

IP-based :
ip.addr == 192.168.1.100
→ Traffic to/from 192.168.1.100

ip.src == 192.168.1.50
→ Source IP = .50

ip.dst == 8.8.8.8
→ Destination IP = 8.8.8.8

Protocol-based :
tcp
→ TCP packets only

dns
→ DNS traffic

http
→ HTTP traffic

tls
→ TLS/SSL traffic

Port-based :
tcp.port == 443
→ TCP port 443 (HTTPS)

udp.port == 53
→ UDP port 53 (DNS)

tcp.srcport == 80
→ Source port 80

TCP flags :
tcp.flags.syn == 1
→ SYN packets (connection initiation)

tcp.flags.reset == 1
→ RST packets (connection reset)

tcp.flags == 0x012
→ SYN+ACK packets (handshake response)

HTTP-specific :
http.request.method == "GET"
→ HTTP GET requests

http.response.code == 404
→ HTTP 404 errors

http.host contains "google.com"
→ Requests to google.com

Combinations :
ip.addr == 192.168.1.100 && tcp.port == 443
→ .100 HTTPS traffic

dns || http
→ DNS OR HTTP

!(arp || icmp || dns)
→ Exclude ARP, ICMP, DNS (show other traffic)

// Using display filters

1. Capture packets (no filter = capture ALL)
2. Filter toolbar (top) : Enter display filter
   Example : tcp.port == 443
3. Press Enter → Only matching packets displayed
4. Clear filter : X button (show all again)
Packet analysis

Lecture d'une capture — Reading packets

// TCP handshake example

3-way handshake :

Packet 1 : SYN
Source : 192.168.1.50:49678
Dest : 8.8.8.8:443
Flags : SYN
Seq : 0 (relative)
→ Client initiates connection

Packet 2 : SYN+ACK
Source : 8.8.8.8:443
Dest : 192.168.1.50:49678
Flags : SYN, ACK
Seq : 0, Ack : 1
→ Server acknowledges, sends SYN

Packet 3 : ACK
Source : 192.168.1.50:49678
Dest : 8.8.8.8:443
Flags : ACK
Seq : 1, Ack : 1
→ Client acknowledges → Connection established ✅

// DNS query example

Packet 1 : DNS Query
Source : 192.168.1.50:54321 (random port)
Dest : 8.8.8.8:53 (DNS)
Protocol : DNS
Query : google.com A record
→ Client asks IP for google.com

Packet 2 : DNS Response
Source : 8.8.8.8:53
Dest : 192.168.1.50:54321
Protocol : DNS
Answer : google.com → 142.250.185.46
→ DNS server responds with IP ✅

// HTTP request/response

Packet X : HTTP GET
GET /index.html HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
→ Client requests page

Packet Y : HTTP 200 OK
HTTP/1.1 200 OK
Content-Type: text/html

Troubleshooting

Identification d'anomalies — Detecting problems

// Anomaly #1 : Retransmissions excessives

Display filter :
tcp.analysis.retransmission

What it shows :
→ Packets retransmitted (original packet lost)
→ Red/black background in packet list

Cause :
→ Packet loss (congestion, bad cable, interference)
→ High retransmissions = network problem ❌

Normal : <1% retransmissions
Problem : >5% retransmissions

// Anomaly #2 : TCP resets (RST)

Display filter :
tcp.flags.reset == 1

Causes RST :
→ Firewall blocking connection
→ Application crash (closes connection abruptly)
→ Port closed (connection refused)
→ Network error

Symptom : Connection drops unexpectedly

// Anomaly #3 : Duplicate ACKs

Display filter :
tcp.analysis.duplicate_ack

What it means :
→ Receiver sends multiple ACKs same sequence number
→ Indicates : Packet(s) missing (out-of-order delivery)

Cause : Packet loss, reordering (network problem)

// Anomaly #4 : DNS failures

Display filter :
dns.flags.rcode != 0

Response codes :
0 = No error (success)
1 = Format error
2 = Server failure
3 = Name error (NXDOMAIN - domain doesn't exist)
5 = Refused

→ NXDOMAIN = typo in domain OR domain doesn't exist

// Anomaly #5 : HTTP errors

Display filter :
http.response.code >= 400

Common errors :
400 = Bad Request (client error)
401 = Unauthorized (authentication required)
403 = Forbidden (access denied)
404 = Not Found (page doesn't exist)
500 = Internal Server Error (server problem)
503 = Service Unavailable (server overloaded)

// Anomaly #6 : ARP spoofing

Display filter :
arp

Check :
Same IP address → Multiple different MAC addresses
→ ARP poisoning attack (man-in-the-middle) ❌

Normal : 1 IP = 1 MAC (consistent)
🎯
Question examen : "High retransmissions in Wireshark. MOST likely cause ?" A) DNS failure
B) Packet loss (congestion, bad cable)
C) Firewall blocking
D) Wrong IP address

Réponse : B) Packet loss
Retransmissions = packets lost in transit (need resend).
Analysis tools

Statistiques Wireshark — Built-in analysis

// Statistics menu

1. Protocol Hierarchy

Statistics → Protocol Hierarchy

Shows :
→ Percentage each protocol (Ethernet, IP, TCP, HTTP, etc.)
→ Bytes transmitted per protocol

Useful : Identify what's consuming bandwidth

2. Conversations

Statistics → Conversations

Shows :
→ All IP conversations (who talking to whom)
→ Bytes transferred each conversation
→ Sort by bytes (find top talkers)

Useful : Identify bandwidth hogs

3. I/O Graph

Statistics → I/O Graph

Shows :
→ Traffic over time (packets/sec, bytes/sec)
→ Visualize traffic patterns (spikes = events)

Useful : Identify when problem occurred (time correlation)

4. Expert Information

Analyze → Expert Information

Shows :
→ Errors (red) : Serious problems (RST, malformed packets)
→ Warnings (yellow) : Potential issues (retransmissions, duplicate ACKs)
→ Notes (cyan) : Informational (TCP window updates)

Useful : Quick overview all problems in capture
