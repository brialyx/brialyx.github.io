Delay

Latence — Network latency (RTT)

Latence (Latency) = Délai transmission packet source → destination → source (Round-Trip Time RTT). Mesurée en millisecondes (ms).

// Composants latence

  • Propagation delay : Temps signal travel cable (vitesse lumière, distance physique)
  • Transmission delay : Temps push bits onto wire (dépend taille packet, link speed)
  • Processing delay : Temps router/switch process packet (routing table lookup, ACLs)
  • Queuing delay : Temps packet waits in queue (congestion, buffer)

// Latence acceptable (rappel Cours TR15)

ApplicationLatency targetImpact if exceeded
VoIP <150 ms Voice delay, echo, talk-over
Video conferencing <150 ms Lag, out-of-sync audio/video
Gaming (FPS) <50 ms Input lag, unplayable
Web browsing <300 ms Pages load slowly (noticeable)
File transfer <1000 ms Slow but functional (TCP retransmit)

// Mesure latence

ping : Mesure RTT (Round-Trip Time)

C:\> ping 8.8.8.8

Reply from 8.8.8.8: bytes=32 time=15ms TTL=54
Reply from 8.8.8.8: bytes=32 time=14ms TTL=54
Reply from 8.8.8.8: bytes=32 time=16ms TTL=54

Statistics:
Minimum = 14ms, Maximum = 16ms, Average = 15ms

→ Latency = 15ms (excellent) ✅

High latency example :
Reply from server: time=250ms TTL=54
→ Latency = 250ms (poor for VoIP) ❌

// Causes high latency

  • Distance : Geographic distance (physics limit - speed of light)
  • Congestion : Network saturated → queuing delay
  • Satellite links : Geostationary satellite ~500-700 ms RTT
  • Too many hops : Each router adds processing delay (1-10 ms/hop)
  • Slow links : Low-speed WAN (T1 1.5 Mbps adds transmission delay)
💭
Analogie : Latency = delivery time Package NYC → Los Angeles :
Latency LOW (air express) : 1 day delivery.
Latency HIGH (ground shipping) : 5 days delivery.
Both deliver same package (bandwidth), but speed different (latency).
Capacity vs Speed

Bande passante vs Débit — Bandwidth vs Throughput

// Définitions

Bandwidth (Bande passante) = Capacité maximale théorique link (Mbps, Gbps). Mesure : Combien de données CAN be transferred per second (potential).

Throughput (Débit) = Vitesse effective réelle transfer (Mbps, Gbps). Mesure : Combien de données ACTUALLY transferred per second (actual).

// Bandwidth vs Throughput comparison

Example :

Link : 1 Gbps Ethernet (bandwidth = 1000 Mbps theoretical)

Actual throughput test :
File transfer : 850 Mbps (real speed)

→ Bandwidth : 1000 Mbps (capacity)
→ Throughput : 850 Mbps (actual - 85% efficiency)

Difference caused by :
→ Protocol overhead (TCP headers, ACKs)
→ Network congestion
→ Packet loss (retransmissions)
→ Processing limitations (CPU, disk speed)

// Factors reducing throughput

  • Protocol overhead : Headers (Ethernet 18 bytes, IP 20, TCP 20 = 58 bytes minimum)
  • Packet loss : TCP retransmissions reduce effective speed
  • Congestion : Queuing delays slow transfer
  • TCP window size : Small window limits throughput high-latency links
  • Application limits : Server/client CPU, disk I/O bottleneck
💡
Rule of thumb : Throughput = 70-95% bandwidth Ideal conditions (low latency, no loss) : 90-95%.
Real-world (Internet, congestion) : 70-85%.
If throughput << bandwidth : Investigate bottleneck.
Latency variation

Jitter (Gigue) — Latency inconsistency

Jitter (Gigue) = Variation latency packet-to-packet. Mesure inconsistency delay (ms). Critical real-time applications (VoIP, video).

// Principe jitter

Consistent latency (LOW jitter) :
Packet 1 : 20 ms
Packet 2 : 21 ms
Packet 3 : 20 ms
Packet 4 : 22 ms
Packet 5 : 20 ms

Average : 20.6 ms
Jitter : ~1-2 ms (consistent) ✅

Variable latency (HIGH jitter) :
Packet 1 : 20 ms
Packet 2 : 85 ms (spike!)
Packet 3 : 25 ms
Packet 4 : 120 ms (spike!)
Packet 5 : 30 ms

Average : 56 ms
Jitter : ~50-100 ms (highly variable) ❌

// Impact jitter

VoIP / Video conferencing

  • Audio choppy : Packets arrive irregular intervals → gaps, stuttering
  • Video freeze/skip : Frames delayed unevenly
  • Buffer underrun : Jitter buffer empty (packet late) → silence/freeze

Jitter buffer = Buffer absorb latency variation (smooth delivery). But excessive jitter overwhelms buffer.

// Jitter targets

ApplicationMax acceptable jitter
VoIP<30 ms
Video conferencing<30 ms
Streaming video<50 ms (buffering compensates)
File transferN/A (jitter doesn't matter)

// Causes jitter

  • Congestion : Variable queuing delays (traffic bursts)
  • Route changes : Packets take different paths (asymmetric routing)
  • Wireless interference : Wi-Fi retransmissions cause delay spikes
  • Prioritization issues : VoIP traffic not prioritized (QoS missing)

// Mesure jitter

ping continuous + analysis :
C:\> ping 8.8.8.8 -t

time=20ms
time=22ms
time=85ms (spike)
time=21ms
time=120ms (spike)

→ High jitter detected (variance >50 ms)

Specialized tools :
→ VoIP test tools (measure jitter directly)
→ iperf -u (UDP test shows jitter statistics)
→ Wireshark (analyze packet timing)

// Solutions jitter

  • QoS (Quality of Service) : Prioritize VoIP/video traffic (consistent queuing)
  • Increase bandwidth : Reduce congestion (less queuing delay variation)
  • Traffic shaping : Smooth traffic bursts (consistent flow)
  • Upgrade network : Faster switches/routers (lower processing delay)
  • Dedicated circuit : VoIP-only link (no competing traffic)
🎯
Question fréquente examen "VoIP calls choppy, audio cuts out. Average latency 30 ms (acceptable). MOST likely cause ?"
A) High latency
B) Low bandwidth
C) High jitter
D) Packet loss

Réponse : C) High jitter
Latency acceptable (30 ms), BUT jitter (variation) causes choppy audio.
Jitter ≠ Average latency (jitter = inconsistency).
Dropped packets

Perte de paquets — Packet loss

Perte de paquets (Packet loss) = Percentage packets dropped in transit. Causes retransmissions, slow speeds, application failures (voir Cours TR15 rappel).

// Packet loss mesure

C:\> ping 8.8.8.8 -n 100

Packets: Sent = 100, Received = 95, Lost = 5 (5% loss)

Acceptable loss :
0-1% : Excellent (normal network noise)
1-2.5% : Acceptable (noticeable VoIP/video quality reduction)
2.5-5% : Poor (applications slow, retransmissions frequent)
>5% : Bad (serious problem, investigate immediately)

// Impact packet loss

TCP applications (web, email, file transfer)

  • Retransmissions (automatic recovery)
  • Reduced throughput (waiting retransmit adds delay)
  • Slower speeds (TCP backs off, reduces window size)

UDP applications (VoIP, video, gaming)

  • No retransmission : Lost packet = lost forever
  • Audio gaps : Missing voice packets → silence/distortion
  • Video artifacts : Missing frames → pixelation, freezing
  • Gaming lag : Player position updates lost → teleporting

// Causes packet loss

  • Congestion : Link saturated → buffers full → tail drop
  • Bad cable : Physical damage → CRC errors → dropped frames
  • Interface errors : CRC, runts, giants (voir Cours TR04)
  • Wireless interference : Wi-Fi collisions, weak signal
  • Routing loops : TTL expires (voir Cours TR10)
  • Firewall/ACL : Blocking traffic (intentional drop)

// Diagnostic packet loss

Step 1 : Measure loss :
C:\> ping 8.8.8.8 -n 100
→ Loss % ?

Step 2 : Locate where loss occurs :
C:\> tracert 8.8.8.8

1    1 ms    192.168.1.1 — 0% loss
2    10 ms   10.0.0.1 (ISP) — 15% loss
3    15 ms   8.8.8.8 — 15% loss

→ Loss starts hop 2 (ISP router) = ISP problem

Step 3 : Check interface errors :
Router# show interfaces GigabitEthernet0/0

CRC errors : 1,234 (incrementing)
Input errors : 2,456

→ Physical problem (cable, connector)

// Solutions packet loss

  • Congestion : Upgrade bandwidth, enable QoS, reduce traffic
  • Cable problems : Replace cable, fix connectors, test with cable tester
  • Wireless : Reduce interference, change channel, improve signal
  • Interface errors : Fix duplex mismatch, replace NIC/cable
Bandwidth testing

iperf — Mesure throughput/bandwidth

iperf = Network performance testing tool. Mesure throughput, bandwidth, jitter, packet loss. Client/server model.

// Installation

Linux :
$ sudo apt install iperf3

Windows :
Download : iperf.fr (iperf3.exe)

macOS :
$ brew install iperf3

// Usage iperf (TCP throughput test)

Server (listener) :
$ iperf3 -s
Server listening on 5201

Client (sender) :
$ iperf3 -c 192.168.1.100

Output example :
[ ID] Interval         Transfer    Bitrate
[ 5] 0.00-10.00 sec 1.09 GBytes 938 Mbits/sec

→ Throughput = 938 Mbps (on 1 Gbps link = 93.8% efficiency) ✅

Test duration :
$ iperf3 -c 192.168.1.100 -t 30 (30 seconds test)

Reverse test (server → client) :
$ iperf3 -c 192.168.1.100 -R

// Usage iperf (UDP test - jitter/loss)

Server :
$ iperf3 -s

Client (UDP mode, 100 Mbps target) :
$ iperf3 -c 192.168.1.100 -u -b 100M

Output example :
[ ID] Interval         Transfer    Bitrate         Jitter    Lost/Total
[ 5] 0.00-10.00 sec 119 MBytes 100 Mbits/sec 0.5 ms 0/8500 (0%)

→ Jitter = 0.5 ms (excellent) ✅
→ Packet loss = 0% (perfect) ✅

High jitter/loss example :
[ 5] 0.00-10.00 sec 119 MBytes 100 Mbits/sec 85 ms 425/8500 (5%)

→ Jitter = 85 ms (very high) ❌
→ Packet loss = 5% (poor) ❌

// Common iperf options

OptionDescriptionExample
-sServer mode (listener)iperf3 -s
-c HOSTClient mode (connect to HOST)iperf3 -c 192.168.1.100
-uUDP mode (default TCP)iperf3 -c 192.168.1.100 -u
-b RATEBandwidth target (UDP)iperf3 -u -b 100M
-t SECTest duration secondsiperf3 -c HOST -t 30
-RReverse (server → client)iperf3 -c HOST -R
-P NUMParallel streamsiperf3 -c HOST -P 10

// Interpretation results

TCP test (throughput)

  • Good : Throughput 80-95% link capacity (normal overhead)
  • Poor : Throughput <50% → Investigate (congestion, errors, CPU limit)

UDP test (jitter/loss)

  • Jitter : <10 ms excellent, 10-30 ms acceptable VoIP, >50 ms poor
  • Loss : <1% excellent, 1-2.5% acceptable, >5% poor
🔧
iperf = essential performance testing Isolates network performance (no disk/application variables).
Tests max throughput link can achieve (baseline).
Compare : iperf result vs real application speed (identify bottleneck).
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : Latency = Bandwidth (confusion)
FAUX. Latency ≠ Bandwidth (separate concepts).

Latency : Delay (ms) — how long packet takes arrive.
Bandwidth : Capacity (Mbps) — how much data per second.

Can have : High bandwidth (1 Gbps) + high latency (satellite 500 ms).
OR : Low bandwidth (10 Mbps) + low latency (fiber 5 ms).

Examen : "Slow downloads" = bandwidth. "Delay/lag" = latency.
❌ Piège #2 : Jitter = Average latency high
FAUX. Jitter = variation latency, NOT average.

Can have : Low average latency (30 ms) BUT high jitter (spikes 20-200 ms).
Jitter problem : Inconsistency (some packets fast, some slow).

VoIP choppy : Often jitter (not latency average).
Examen : Average latency OK, audio choppy = suspect jitter.
❌ Piège #3 : Throughput always equals bandwidth
FAUX. Throughput ≤ Bandwidth (always).

Bandwidth : Theoretical maximum (1 Gbps link capacity).
Throughput : Actual measured speed (850 Mbps real transfer).

Difference : Protocol overhead, packet loss, congestion, application limits.
Normal : Throughput 70-95% bandwidth.
Examen : If throughput << bandwidth → Investigate bottleneck.
Résumé

🔑 Ce que tu dois retenir absolument

→Latence : Delay RTT (ms). Components: propagation, transmission, processing, queuing. VoIP <150 ms, gaming <50 ms. Causes: distance, congestion, satellite, hops. Measure: ping (average RTT)
→Bandwidth vs Throughput : Bandwidth = capacity théorique (Mbps). Throughput = vitesse réelle (70-95% bandwidth). Factors: protocol overhead, loss, congestion, TCP window. iperf mesure throughput baseline
→Jitter (Gigue) : Variation latency packet-to-packet (ms). VoIP/video <30 ms acceptable. Causes: congestion (variable queuing), route changes, wireless, no QoS. Symptom: choppy audio/video (average latency OK). Solution: QoS
→Perte paquets : 0-1% excellent, >5% serious. TCP: retransmit (slow). UDP: lost forever (gaps audio/video). Causes: congestion, bad cable, CRC errors, wireless. Diagnostic: ping loss %, traceroute locate hop, show interfaces errors
→iperf : Client/server. TCP test: iperf3 -c HOST (throughput Mbps). UDP test: iperf3 -c HOST -u -b 100M (jitter/loss). Options: -t duration, -R reverse, -P parallel. UDP shows jitter ms + loss %. Essential baseline testing
→CRITICAL distinctions : Latency ≠ Bandwidth (delay vs capacity). Jitter ≠ Average latency (variation vs mean). Throughput ≤ Bandwidth (actual vs theoretical). VoIP choppy + latency OK = jitter problem. Examen tests understanding differences
