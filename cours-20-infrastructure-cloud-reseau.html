Réseau cloud isolé

VPC — Virtual Private Cloud

🏢
Analogie — immeuble de bureaux VPC = ton étage privé dans immeuble datacenter cloud. Tu contrôles : qui entre (Security Groups), subdivision pièces (subnets), connexions vers extérieur (Internet Gateway). Autres locataires (VPCs) isolés de toi.

VPC = réseau virtuel privé dans cloud public. Isolation logique ressources dans espace IP dédié.

// Composants VPC

  • CIDR block : Plage IPs privées (ex: 10.0.0.0/16)
  • Subnets : Subdivisions VPC (public/privé)
  • Route tables : Routage trafic entre subnets
  • Internet Gateway : Accès Internet
  • NAT Gateway : Internet sortant pour subnets privés
  • Security Groups : Firewall stateful instances
  • NACLs : Firewall stateless subnets

// Architecture VPC typique

[VPC 10.0.0.0/16]

[Subnet Public 10.0.1.0/24]
- Web servers (instances avec IPs publiques)
- Internet Gateway (accès bidirectionnel Internet)

[Subnet Privé 10.0.2.0/24]
- Database servers (PAS IP publique)
- NAT Gateway (sortant uniquement, updates/patches)

→ Subnets séparés = isolation sécurité
→ Route tables contrôlent communication
Segmentation

Sous-réseaux cloud — Public vs Privé

// Subnet Public

Subnet avec route vers Internet Gateway. Instances peuvent avoir IPs publiques.

Usage :

  • Web servers frontaux (accessible depuis Internet)
  • Load balancers
  • Bastion hosts (jump boxes SSH)

// Subnet Privé

Subnet SANS route Internet Gateway. Instances pas accessibles Internet directement.

Usage :

  • Databases (isolées)
  • Backend application servers
  • Services internes

NAT Gateway : Permet subnets privés accéder Internet (sortant uniquement, updates/patches) SANS être accessibles depuis Internet.

// Multi-AZ

Best practice : Déployer subnets dans multiples Availability Zones (datacenters physiques séparés).

Région us-east-1 (Virginie) :
- AZ-1a : Subnet Public 10.0.1.0/24 + Subnet Privé 10.0.2.0/24
- AZ-1b : Subnet Public 10.0.3.0/24 + Subnet Privé 10.0.4.0/24

→ Si datacenter AZ-1a tombe → apps continuent sur AZ-1b
→ Haute disponibilité
Filtrage instance

Security Groups — Firewall stateful

Security Group = firewall virtuel stateful au niveau instance (VM/container).

// Caractéristiques

  • Stateful : Retours connexions autorisés automatiquement
  • Allow only : Règles PERMIT uniquement (pas DENY explicites)
  • Niveau instance : Attaché à ENI (Elastic Network Interface)
  • Par défaut : Tout entrant bloqué, tout sortant autorisé

// Exemple règles Security Group

INBOUND (Entrant) :
- ALLOW TCP port 80 from 0.0.0.0/0 (HTTP depuis anywhere)
- ALLOW TCP port 443 from 0.0.0.0/0 (HTTPS)
- ALLOW TCP port 22 from 203.0.113.0/24 (SSH depuis IP admin)

OUTBOUND (Sortant) :
- ALLOW ALL (par défaut)

// Security Groups vs NACLs

CritèreSecurity GroupsNACLs (Network ACLs)
NiveauInstance (ENI)Subnet entier
StateStateful (retours auto)Stateless (règles bidirectionnelles)
RèglesALLOW uniquementALLOW + DENY
Ordre règlesToutes évaluéesOrdre numérique (première correspondance)
Par défautDeny all inboundAllow all (modifiable)
🎯
Question type examen "Instance AWS EC2 ne reçoit pas trafic HTTP. Première vérification ?"

Réponse : Security Group (port 80 autorisé entrant ?)

Security Groups = première ligne défense instance-level.
Connexion dédiée

Direct Connect — Lien privé datacenter ↔ Cloud

Direct Connect (AWS) / ExpressRoute (Azure) / Cloud Interconnect (GCP) = connexion réseau privée dédiée entre datacenter on-premise et cloud provider.

// vs Internet public

Internet public : Trafic cloud ↔ on-premise passe Internet (chiffré VPN)

Direct Connect : Liaison fibre directe privée (pas Internet)

// Avantages Direct Connect

  • ✅ Bande passante dédiée (1-100 Gbps)
  • ✅ Latence faible/prévisible (~10ms vs ~30-50ms Internet)
  • ✅ Pas congestion Internet (lien privé)
  • ✅ Coûts transfert data réduits (vs Internet egress)
  • ✅ Compliance (certaines régulations interdisent data sur Internet public)
  • ⚠️ Setup long (semaines/mois provisioning)
  • ⚠️ Coût élevé (port fee mensuel + circuit)

// Architecture

[Datacenter on-premise]
↓ Fibre optique dédiée (1-100 Gbps)
[Point présence AWS Direct Connect]
↓ Liaison privée
[VPC AWS]

→ Pas Internet (privé bout-en-bout)
→ Virtual interfaces (VIFs) pour VLANs séparés

// Cas d'usage

  • Migrations massives données vers cloud (TB/PB)
  • Applications hybrides (latence critique)
  • Sauvegardes continues datacenter → cloud
  • Compliance (finance, santé) interdisant Internet public
Connexion chiffrée

VPN Gateway — Tunnel IPsec vers cloud

VPN Gateway = point terminaison VPN côté cloud. Établit tunnels IPsec chiffrés sur Internet public.

// vs Direct Connect

CritèreVPN Gateway (IPsec)Direct Connect
TransportInternet publicLiaison privée dédiée
Bande passante~1-2 Gbps max1-100 Gbps
Latence⚠️ Variable (Internet)✅ Faible/prévisible
Setup✅ Minutes (logiciel)⚠️ Semaines/mois
Coût💰 Faible (~$0.05/h)💰💰💰 Élevé ($1000s/mois)
UsagePME, backup Direct ConnectEntreprises, haute perf

// Architectures

Site-to-Site VPN : On-premise ↔ VPC cloud

[Router on-premise] → IPsec tunnel → [VPN Gateway VPC]
→ Accès ressources VPC depuis datacenter
→ Alternative économique Direct Connect

Client VPN : Remote users ↔ VPC

[Laptop employé] → VPN client → [Client VPN endpoint VPC]
→ Télétravail accès ressources cloud privées
✅
VPN + Direct Connect = Best Practice Hybrid : Direct Connect primaire (perf) + VPN backup (si DC tombe). Redondance maximale.
Interconnexion VPCs

VPC Peering — Connexion privée VPC ↔ VPC

VPC Peering = connexion réseau privée entre 2 VPCs. Communication directe sans passer Internet.

// Caractéristiques

  • Connexion 1-to-1 (VPC A ↔ VPC B)
  • Pas de transitivité : A↔B + B↔C ≠ A↔C automatique
  • CIDRs ne doivent PAS se chevaucher
  • Même région OU inter-régions (cross-region peering)
  • Même compte OU cross-account

// Architecture

[VPC Production 10.0.0.0/16]
↕ VPC Peering
[VPC Development 172.16.0.0/16]

→ Instances Production accèdent instances Dev directement
→ Trafic privé (pas Internet Gateway)
→ Route tables configurées chaque VPC

// Cas d'usage

  • Séparer environnements (Prod/Dev/Test) avec communication sécurisée
  • Services partagés (Active Directory central accessible depuis multiple VPCs)
  • Multi-account organizations (VPC par département)

// Transit Gateway (alternative)

Pour interconnecter nombreux VPCs (10+) : Transit Gateway = hub central (modèle hub-and-spoke) au lieu de mesh peering complexe.

⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : Security Group = subnet-level
FAUX. Security Group = instance-level (ENI).

NACLs = subnet-level.
Security Groups plus granulaires.
❌ Piège #2 : Direct Connect passe par Internet
FAUX. Direct Connect = liaison privée dédiée (fibre physique).

VPN Gateway = passe Internet (tunnel chiffré)
❌ Piège #3 : VPC Peering = transitif
FAUX. Peering PAS transitif.

A↔B + B↔C ≠ A peut parler à C
Besoin créer peering A↔C explicitement OU Transit Gateway
Résumé

🔑 Ce que tu dois retenir absolument

→VPC : Réseau privé cloud (CIDR block), subnets public (Internet Gateway) vs privé (NAT Gateway), multi-AZ haute dispo
→Security Groups : Firewall stateful instance-level, ALLOW only. NACLs : Stateless subnet-level, ALLOW+DENY
→Direct Connect : Liaison privée dédiée (1-100Gbps, faible latence, cher). VPN Gateway : IPsec sur Internet (économique, setup rapide)
→VPC Peering : Connexion privée VPC↔VPC, pas transitif, CIDRs non-chevauchants. Transit Gateway = hub multi-VPCs
