Flow monitoring

NetFlow & sFlow — Qu'est-ce qu'un flow ?

🚗
Analogie — comptage trafic routier Flow monitoring = caméras autoroute comptant véhicules. Pas enregistrer chaque voiture individuellement (trop lourd), mais statistiques : "100 voitures Paris→Lyon entre 14h-15h, vitesse moyenne 110 km/h". NetFlow = idem pour paquets réseau.

Flow (flux) = séquence paquets unidirectionnels partageant caractéristiques communes (même source, destination, ports, protocole). NetFlow/sFlow = technologies capturant métadonnées flows (pas payload).

// Définition flow (NetFlow)

Flow identifié par 7 champs (7-tuple) :

  • Source IP : 192.168.1.10
  • Destination IP : 203.0.113.50
  • Source Port : 54321
  • Destination Port : 443 (HTTPS)
  • Protocol : TCP (6)
  • ToS (Type of Service) : DSCP marking
  • Input interface : GigabitEthernet0/1

Exemple flow :

Flow #1234567
Src IP: 192.168.1.10:54321
Dst IP: 203.0.113.50:443
Protocol: TCP
Start: 14:05:12
Duration: 45 seconds
Packets: 1,250
Bytes: 1,800,000 (1.8 MB)

→ 1 flow = conversation complète (client 192.168.1.10 télécharge 1.8 MB depuis serveur HTTPS 203.0.113.50)

// Différence SNMP vs Flow monitoring

SNMP : Statistiques interface globales (total bytes IN/OUT, erreurs).

NetFlow : Détails conversations individuelles (qui parle à qui, quels ports, combien data).

🔍
Flow = métadonnées, PAS payload NetFlow/sFlow capturent headers paquets (IPs, ports), PAS contenu (pas DPI). Visibilité trafic SANS inspection profonde. Compliance privacy OK.
Cisco standard

NetFlow — Analyse de flux

NetFlow = technologie Cisco (devenue standard de facto). Capture flows, exporte vers collecteur.

// Architecture NetFlow

  1. Flow Exporter : Routeur/switch Cisco capture flows, agrège statistiques
  2. Flow Export : Envoie records flows vers collecteur (UDP 2055 typiquement)
  3. Flow Collector : Serveur reçoit/stocke flows (nfdump, Elastiflow, SolarWinds)
  4. Flow Analyzer : Visualise données (graphiques, top talkers, alertes)
[Router Cisco] (Flow Exporter)
↓ NetFlow export (UDP 2055)
[Collecteur — nfdump] (Flow Collector)
↓
[Analyzer — Grafana] (Visualisation)

→ Router capture flows trafic traversant
→ Exporte records flows vers collecteur
→ Analyzer affiche top talkers, tendances

// Versions NetFlow

NetFlow v5 (Legacy)

  • Version classique (1996)
  • IPv4 uniquement
  • Champs basiques (7-tuple)
  • Pas extensible
  • Encore largement utilisé

NetFlow v9 (Template-based)

  • Templates flexibles (champs personnalisables)
  • Support IPv6
  • Support MPLS, VLAN tags
  • Base IPFIX standard

IPFIX (IP Flow Information Export)

  • Standard IETF (RFC 7011)
  • Basé NetFlow v9
  • Multi-vendor (Cisco, Juniper, HP...)
  • Recommandé production moderne

// Informations flow NetFlow

Champs capturés par flow :

  • Source/Destination IP + Ports
  • Protocol (TCP/UDP/ICMP)
  • ToS/DSCP (QoS marking)
  • Timestamps (start, end, duration)
  • Packet count (nombre paquets)
  • Byte count (volume données)
  • TCP Flags (SYN, ACK, FIN...)
  • Input/Output interfaces
  • Next-hop IP (routage)
  • AS numbers (BGP)

// Configuration NetFlow (Cisco IOS)

Enable NetFlow sur interface :

interface GigabitEthernet0/1
 ip flow ingress
 ip flow egress
!

Configure export vers collecteur :

ip flow-export version 9
ip flow-export destination 10.1.1.100 2055
ip flow-export source GigabitEthernet0/0

→ Capture flows Gi0/1 (entrant+sortant)
→ Exporte NetFlow v9 vers 10.1.1.100:2055
Sampling standard

sFlow — Sampled Flow

sFlow = standard industrie (RFC 3176) concurrent NetFlow. Développé HP/InMon, multi-vendor.

// Principe sFlow

Sampling (échantillonnage) : Capture 1 paquet sur N (ex: 1/1000).

Workflow :

  1. Switch sample 1 paquet / 1000 paquets
  2. Extrait header paquet (pas payload)
  3. Envoie immédiatement vers collecteur (temps réel)
  4. Collecteur reconstruit statistiques flows

// sFlow vs NetFlow

CritèreNetFlowsFlow
MéthodeAgrégation flows (cache)Sampling paquets (1/N)
Précision✅ 100% paquets capturés⚠️ Estimation (sample)
Overhead CPU⚠️ Moyen-élevé✅ Très faible
Temps réel⚠️ Latence (cache export)✅ Immédiat (streaming)
VendorCisco principalementMulti-vendor (HP, Arista, Juniper)
StandardIPFIX (IETF)sFlow (RFC 3176)
UsageAccounting précis, facturationMonitoring haute perf, DDoS detection

// Sampling rate

Rate 1:1000 : Capture 1 paquet sur 1000.

  • Lien 1 Gbps → ~1,000,000 paquets/sec → 1,000 samples/sec
  • Overhead CPU très faible (1/1000)
  • Estimation statistique suffisante monitoring général

Rate 1:100 : Plus précis, overhead plus élevé.

⚙️
Quand utiliser NetFlow vs sFlow ? NetFlow : Accounting précis, facturation ISP, forensics détaillées (besoin 100% flows)
sFlow : Monitoring performance haute capacité (10/40/100 Gbps), DDoS detection temps réel, overhead CPU critique
Applications

Cas d'usage — Pourquoi flow monitoring ?

// 1. Traffic Accounting (Comptabilité)

Mesurer consommation bande passante par utilisateur, département, application.

Exemples :

  • ISP : Facturation usage clients (95th percentile billing)
  • Entreprise : Chargebacks départements (Finance consomme X GB/mois)
  • Top talkers : Qui consomme plus bande passante ? (serveur backup 500 GB/jour)

// 2. Capacity Planning

Analyser tendances croissance trafic → planifier upgrades liens.

Exemple :

  • Lien WAN HQ→DC 500 Mbps utilisé 80% heures pointe (baseline)
  • Croissance 10%/mois observée NetFlow
  • Projection : Saturation dans 2 mois → upgrade 1 Gbps planifié

// 3. Détection anomalies / Sécurité

Identifier patterns anormaux trafic.

DDoS detection :

  • Baseline : Serveur web reçoit 1000 flows/sec normalement
  • Spike soudain : 100,000 flows/sec depuis 10,000 IPs différentes
  • NetFlow alerte : DDoS probable → mitigation (blackhole routing)

Malware C2 (Command & Control) :

  • PC interne 192.168.1.50 communique 185.x.x.x:8443 (IP russe suspect)
  • Pattern : Connexions régulières toutes les 5 min (beacon)
  • Flow analysis → Investigation malware

Data exfiltration :

  • Laptop employé upload 50 GB vers Dropbox 02:00 AM
  • Baseline : Usage Dropbox normal 100 MB/jour
  • NetFlow alerte : Anomalie volume → Investigation insider threat

// 4. Application visibility

Identifier applications consommant bande passante (classification ports/IPs).

Top applications (flows analysis) :

1. HTTPS (port 443) : 45% trafic total
2. Video streaming (Netflix IPs) : 25%
3. Office 365 (Microsoft IPs) : 15%
4. VoIP (RTP ports 16384-32767) : 8%
5. Backups (port 8888 custom) : 5%
6. Autres : 2%

→ Visibilité apps réelles (pas juste ports génériques)

// 5. Troubleshooting performance

Diagnostiquer lenteurs réseau.

Exemple :

  • Users : "Internet lent ce matin"
  • NetFlow analysis : Backup job non planifié consomme 90% lien WAN
  • Action : Throttle backup → Problème résolu

⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : NetFlow capture payload paquets
FAUX. NetFlow/sFlow = métadonnées uniquement (headers).

Capture : IPs, ports, protocole, volume (bytes/packets)
PAS capture : Contenu emails, fichiers, passwords

DPI (Deep Packet Inspection) ≠ NetFlow
❌ Piège #2 : sFlow plus précis que NetFlow
FAUX. sFlow = sampling (échantillonnage 1/N) → estimation.

NetFlow : 100% paquets capturés → précis
sFlow : 1/1000 paquets → estimation (moins overhead CPU)

sFlow = moins précis MAIS overhead très faible
❌ Piège #3 : NetFlow = protocole temps réel
NUANCE. NetFlow = near real-time (quasi temps réel).

Flows agrégés cache local → export périodique (ex: toutes les 60 sec ou fin flow).
sFlow = plus temps réel (streaming immédiat samples).

NetFlow latence seconds-minutes acceptable monitoring, MAIS pas packet-level real-time
Résumé

🔑 Ce que tu dois retenir absolument

→Flow : Séquence paquets unidirectionnels (7-tuple : src/dst IP+ports, proto, ToS, interface). NetFlow/sFlow = métadonnées flows, PAS payload
→NetFlow : Cisco standard (IPFIX). 100% paquets capturés, agrégation flows, export vers collecteur. Versions v5 (legacy IPv4), v9/IPFIX (template, IPv6)
→sFlow : Standard multi-vendor. Sampling 1/N paquets, overhead CPU très faible, temps réel. Estimation statistique (moins précis NetFlow)
→Cas usage : Traffic accounting/facturation, capacity planning, détection anomalies/DDoS, application visibility, troubleshooting performance
→Collecteurs : nfdump/nfsen (CLI open-source), Elastiflow (ELK stack), PRTG (commercial PME), SolarWinds (entreprise). Export UDP 2055 (NetFlow), 6343 (sFlow)
