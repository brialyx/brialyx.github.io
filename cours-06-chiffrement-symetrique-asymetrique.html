Cryptography basics

Chiffrement — Concepts fondamentaux

🔐
Analogie — coffre-fort Message clair = argent visible. Chiffrement = mettre argent coffre-fort verrouillé. Ciphertext = coffre fermé (personne voit contenu). Clé = seule façon ouvrir coffre récupérer argent original.

Chiffrement (Encryption) = Transformer données lisibles (plaintext) → données illisibles (ciphertext) via algorithme mathématique + clé.

// Terminologie

  • Plaintext : Données originales lisibles (ex: "Hello World")
  • Ciphertext : Données chiffrées illisibles (ex: "4f9a2c8e...")
  • Encryption : Plaintext → Ciphertext (chiffrer)
  • Decryption : Ciphertext → Plaintext (déchiffrer)
  • Key : Secret utilisé algorithme (chiffrer/déchiffrer)
  • Algorithm : Méthode mathématique (AES, RSA, etc.)

// Objectifs chiffrement

Confidentiality : Seules personnes autorisées lisent données.

Integrity : Détecte modifications données (tampering).

Authentication : Vérifie identité sender/receiver.

Non-repudiation : Sender pas nier avoir envoyé (signatures numériques).

// 2 types chiffrement

TypeClésVitesseUsage
Symétrique1 clé (shared secret)✅ RapideDonnées bulk (fichiers, disques)
Asymétrique2 clés (public + private)⚠️ LentKey exchange, signatures, SSL/TLS
🔑
Hybrid cryptography (real world) Production utilise COMBINAISON symétrique + asymétrique :
1. Asymétrique échange clé symétrique (RSA)
2. Symétrique chiffre données bulk (AES)
→ Balance sécurité (asymétrique) + performance (symétrique)
Symmetric encryption

Chiffrement symétrique — Clé partagée

Chiffrement symétrique = Même clé chiffre ET déchiffre données (shared secret key).

// Principe symétrique

Alice veut envoyer message Bob :

1. Alice et Bob partagent clé secrète (K = "MySecret123")
2. Alice chiffre : Plaintext + K → Ciphertext
3. Alice envoie Ciphertext → Bob
4. Bob déchiffre : Ciphertext + K → Plaintext

MÊME clé (K) utilisée encrypt ET decrypt

// Avantages symétrique

  • ✅ Rapide : Algorithmes légers (hardware acceleration AES)
  • ✅ Efficient : Chiffre GB données rapidement (disques, VPN)
  • ✅ Simple : 1 clé gérer (vs 2 asymétrique)

// Problème clé symétrique

Key distribution problem : Comment Alice et Bob partagent clé secrète AVANT communication ?

  • ❌ Email clé = insécure (interceptée)
  • ❌ Meeting physique = impraticable (Alice Tokyo, Bob Paris)
  • ❌ N personnes = N×(N-1)/2 clés (scaling nightmare : 100 personnes = 4950 clés)

Solution : Asymétrique échange clé symétrique (Diffie-Hellman, RSA key exchange).

// Algorithmes symétriques

DES (Data Encryption Standard)

  • Clé : 56 bits (trop court)
  • Block size : 64 bits
  • ⚠️ Obsolète : Brute-force 56 bits = 24h (1998)
  • Legacy systems uniquement

3DES (Triple DES)

  • Applique DES 3× (encrypt-decrypt-encrypt)
  • Clé : 168 bits effective (3 × 56)
  • ⚠️ Deprecated : Lent, remplacé AES
  • NIST retire 2023

AES (Advanced Encryption Standard)

  • Clés : 128, 192, 256 bits
  • Block size : 128 bits
  • ✅ Standard actuel : Fort, rapide, widespread
  • Usage : WiFi WPA2/WPA3, VPN, disk encryption, HTTPS
  • Hardware acceleration : AES-NI (Intel/AMD CPUs)
🎯
Questions TRÈS fréquentes examen "Algorithme chiffrement symétrique moderne recommandé ?"
Réponse : AES (AES-128, AES-256)

"DES obsolète pourquoi ?"
Réponse : Clé 56 bits trop courte (brute-force facile)
Asymmetric encryption

Chiffrement asymétrique — Clés publique/privée

Chiffrement asymétrique = 2 clés mathématiquement liées : publique (public key) chiffre, privée (private key) déchiffre.

// Principe asymétrique

Bob génère paire clés :
- Public key : Partagée TOUT LE MONDE (annuaire public)
- Private key : Secrète, JAMAIS partagée (Bob seul)

Alice veut envoyer message secret Bob :
1. Alice récupère public key Bob (annuaire)
2. Alice chiffre message : Plaintext + Bob_Public → Ciphertext
3. Alice envoie Ciphertext → Bob
4. Bob déchiffre : Ciphertext + Bob_Private → Plaintext

→ SEUL Bob (private key) peut déchiffrer
→ Alice pas besoin clé Bob (utilise public)

// Propriétés clés asymétriques

1. Chiffrement (Confidentiality)

  • Public key chiffre → Private key déchiffre
  • Usage : Envoyer message secret (SSL/TLS, PGP email)

2. Signature numérique (Authentication)

  • Private key signe → Public key vérifie signature
  • Usage : Prouver authenticité sender (non-repudiation)
Signature numérique workflow :

Alice signe document :
1. Alice hash document → Hash (SHA-256)
2. Alice chiffre hash : Hash + Alice_Private → Signature
3. Alice envoie : Document + Signature → Bob

Bob vérifie :
4. Bob hash document reçu → Hash_computed
5. Bob déchiffre signature : Signature + Alice_Public → Hash_original
6. Compare : Hash_computed == Hash_original ?
   → Yes : Document authentique, pas modifié
   → No : Document falsifié ou corrompu

// Avantages asymétrique

  • ✅ Pas key distribution problem : Public key partagée librement
  • ✅ Scaling : N personnes = N paires clés (vs N² symétrique)
  • ✅ Signatures numériques : Non-repudiation possible
  • ✅ Key exchange : Établir shared secret canal insécure

// Inconvénients asymétrique

  • ⚠️ Lent : 100-1000× plus lent symétrique (math complexe)
  • ⚠️ Clés grandes : RSA 2048-4096 bits (vs AES 256 bits)
  • ⚠️ Pas pratique bulk data : Chiffrer 1 GB = heures

// Algorithmes asymétriques

RSA (Rivest-Shamir-Adleman)

  • Clés : 2048 bits (minimum), 4096 bits (recommandé 2024+)
  • Basé : Factorisation nombres premiers (math problème difficile)
  • Usage : SSL/TLS, SSH, PGP, signatures code
  • ✅ Standard industry, widely supported
  • ⚠️ Quantum computers future = threat (Shor's algorithm)

ECC (Elliptic Curve Cryptography)

  • Clés : 256 bits (équivalent sécurité RSA 3072 bits)
  • Basé : Courbes elliptiques (math avancée)
  • ✅ Plus efficient RSA : Clés courtes, calculs rapides
  • Usage : Mobile (moins CPU), IoT, Bitcoin, modern TLS
  • Variantes : ECDSA (signatures), ECDH (key exchange)
⚡
ECC = futur cryptographie 256-bit ECC = même sécurité 3072-bit RSA MAIS :
- Clés 12× plus petites
- Calculs 10× plus rapides
- Moins batterie (mobile/IoT)
Adoption croissante (Apple, Google, Bitcoin).

// Usage hybride (Real World)

HTTPS (SSL/TLS) exemple :

  1. Handshake (Asymétrique) :
    - Client vérifie certificat server (RSA/ECC public key)
    - Client génère session key symétrique random
    - Client chiffre session key : SessionKey + Server_Public → Encrypted_SessionKey
    - Client envoie Encrypted_SessionKey → Server
    - Server déchiffre : Encrypted_SessionKey + Server_Private → SessionKey
    → DEUX parties ont maintenant SessionKey partagée
  2. Data transfer (Symétrique) :
    - Tout trafic HTTPS chiffré AES avec SessionKey
    - Rapide (GB données encrypted efficiently)

→ Balance : Asymétrique sécurise échange clé, Symétrique chiffre données bulk

⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : AES = chiffrement asymétrique
FAUX. AES = chiffrement SYMÉTRIQUE.

Symétrique : AES, DES, 3DES (1 clé)
Asymétrique : RSA, ECC, Diffie-Hellman (2 clés)

Mémo : AES = Advanced Encryption Standard (symmetric)
❌ Piège #2 : Public key = gardée secrète
FAUX. Public key = PARTAGÉE publiquement.

Public key : Partagée tout le monde (annuaire, certificat)
Private key : Secrète, JAMAIS partagée

Public ≠ privé (names hint usage)
❌ Piège #3 : RSA plus sécurisé AES (clés plus grandes)
FAUX. Taille clé ≠ comparaison directe symétrique vs asymétrique.

AES-256 (256 bits) ≈ RSA-15360 bits sécurité équivalente
RSA-2048 ≈ AES-112 sécurité

Math différente → tailles incomparables directement
Résumé

🔑 Ce que tu dois retenir absolument

→Chiffrement : Plaintext → Ciphertext (encrypt), reverse (decrypt). Objectifs : Confidentiality, Integrity, Authentication, Non-repudiation
→Symétrique : 1 clé (shared secret). Rapide, efficient bulk data. Problème key distribution. AES (standard actuel 128/256 bits), DES/3DES (obsolète). Usage : VPN, disk encryption
→Asymétrique : 2 clés (public partagée, private secrète). Lent MAIS résout key distribution. Public chiffre, private déchiffre. Signatures : private signe, public vérifie
→RSA : 2048-4096 bits, factorisation nombres premiers, standard SSL/TLS. ECC : 256 bits = RSA 3072 sécurité, plus efficient (mobile/IoT), futur crypto
→Hybrid crypto : Real world combine. Asymétrique échange session key (RSA/ECC), Symétrique chiffre données (AES). HTTPS exemple : TLS handshake (RSA) + data transfer (AES)
