Remote management

SSH vs Telnet — Accès remote sécurisé

Hardening = Réduire attack surface équipements réseau (switches, routers) via configuration sécurité.

// Telnet (Legacy, insecure)

Telnet = Protocol remote management TCP port 23. ⚠️ TEXTE CLAIR (plaintext).

Telnet session (Wireshark capture) :

→ User types : "enable"
→ Password : "Cisco123"
→ Commands : "show running-config"

Wireshark shows :
Plaintext : enable
Plaintext : Cisco123
Plaintext : show running-config
Plaintext : [entire config visible]

→ Attaquant sniff réseau = credentials + config stolen

// SSH (Secure Shell)

SSH = Protocol remote management chiffré TCP port 22. Remplace Telnet.

Avantages SSH :

  • ✅ Chiffrement : Tout trafic chiffré (credentials, commands, output)
  • ✅ Authentication : Password OU public key (stronger)
  • ✅ Integrity : Détection tampering (HMAC)
  • ✅ Standard : SSH version 2 (SSHv1 deprecated, vulnerabilities)

// Configuration SSH Cisco

1. Set hostname + domain :
hostname Switch1
ip domain-name example.com

2. Generate RSA keys :
crypto key generate rsa
How many bits? 2048 (minimum 1024, recommend 2048)

3. Enable SSH, disable Telnet :
line vty 0 15
  transport input ssh (SSH only)
  login local

username admin secret MySecurePass123

4. Force SSH version 2 :
ip ssh version 2
🎯
Question TRÈS fréquente examen "Protocol remote management chiffré remplace Telnet ?"
Réponse : SSH (Secure Shell, port 22)

"Commande Cisco disable Telnet, enable SSH only ?"
Réponse : transport input ssh
Attack surface reduction

Désactiver services non nécessaires

Principe : Chaque service running = potential vulnerability. Disable TOUT service non utilisé.

// Services couramment disabled

1. HTTP/HTTPS (Web GUI management)

Default Cisco : HTTP server enabled (GUI management)
Risque : HTTP = plaintext, exploits web GUI

Disable :
no ip http server
no ip http secure-server

2. CDP (Cisco Discovery Protocol)

Usage : Discover Cisco neighbors (hostname, IP, capabilities)
Risque : Information leakage (reconnaissance attack)

Disable globally :
no cdp run

Disable per-interface (external facing) :
interface GigabitEthernet0/1
  no cdp enable

3. LLDP (Link Layer Discovery Protocol)

Usage : Standard discovery protocol (multi-vendor)
Risque : Similar CDP (info leakage)

Disable :
no lldp run

4. Finger service

  • Legacy service (user info lookup)
  • Obsolète, vulnerabilities multiples
  • no service finger

5. Unused VTY lines

line vty 0 4
  transport input ssh
  login local

Unused lines (5-15) :
line vty 5 15
  transport input none
  no exec

// Best practices disable

  • ❌ HTTP/HTTPS : Use SSH CLI only (disable GUI)
  • ❌ CDP/LLDP : Disable external interfaces (prevent recon)
  • ❌ Telnet : Always disabled (use SSH)
  • ❌ SNMP v1/v2c : Upgrade SNMPv3 ou disable
  • ✅ Audit : Review show ip sockets, disable services unknown
⚠️
CDP leak = reconnaissance goldmine CDP broadcasts : Hostname, IP addresses, IOS version, device model.
Attaquant port mirroring → CDP packets → full network map.
Always disable CDP external-facing interfaces (Internet, guest VLANs).
Network monitoring

SNMP — Versions sécurisées

SNMP (Simple Network Management Protocol) = Monitoring équipements réseau. UDP ports 161 (agent), 162 (trap).

// SNMP Versions

VersionAuthenticationEncryptionStatut
SNMPv1 Community string (plaintext) ❌ None ⚠️ Obsolète, insecure
SNMPv2c Community string (plaintext) ❌ None ⚠️ Legacy, avoid
SNMPv3 Username + password (hashed) ✅ DES/AES encryption ✅ Modern, secure

// SNMPv1/v2c (Insecure)

Problème : Community string = password plaintext UDP.

SNMPv2c config (BAD) :

snmp-server community public ro
snmp-server community private rw

→ Community strings "public" / "private" = defaults
→ Sent plaintext UDP (sniffable)
→ "rw" (read-write) = attacker modify configs

// SNMPv3 (Secure)

Améliorations SNMPv3 :

  • Authentication : MD5 ou SHA (hashed passwords)
  • Encryption : DES ou AES (confidentiality)
  • Access control : RBAC (views, groups)
SNMPv3 config (GOOD) :

snmp-server group ADMIN v3 priv
snmp-server user snmpadmin ADMIN v3 auth sha MyAuthPass123 priv aes 128 MyPrivPass456

→ Authentication : SHA (hashed password)
→ Encryption : AES-128 (confidential)
→ No plaintext community strings

// Best practices SNMP

  • ✅ Use SNMPv3 : Auth + encryption mandatory
  • ❌ Avoid defaults : "public" / "private" = first attack attempt
  • ✅ Read-only preferred : Monitoring = ro sufficient (pas rw)
  • ✅ ACL restrict : SNMP access management subnet only
  • ✅ Disable si non utilisé : no snmp-server
Legal notices

Bannières — Legal warnings

Bannières = Messages affichés login → avertissement légal accès non autorisé.

// Types bannières Cisco

1. MOTD (Message of the Day)

  • Affiché AVANT login (tous users, même non authentifiés)
  • Usage : Legal warning, acceptable use policy
banner motd #
***********************************************
UNAUTHORIZED ACCESS PROHIBITED
This system is for authorized use only.
All activity is logged and monitored.
Violators will be prosecuted.
***********************************************
#

2. Login banner

  • Affiché APRÈS MOTD, AVANT username prompt
  • Usage : Legal consent, monitoring notice

3. Exec banner

  • Affiché APRÈS login successful
  • Usage : Welcome message, company policy reminder

// Considérations légales bannières

⚠️ NE PAS inclure :

  • ❌ "Welcome" (implies invitation → affaiblit legal case)
  • ❌ Informations système (hostname, IOS version → reconnaissance)
  • ❌ Contact admin (social engineering vector)

✅ INCLURE :

  • ✅ "Unauthorized access prohibited"
  • ✅ "All activity monitored and logged"
  • ✅ "Violators will be prosecuted"
  • ✅ "No expectation of privacy"
⚖️
Bannières = legal protection Prosecution attacker NÉCESSITE preuve "unauthorized access".
Banner "Welcome" = defense argument "I was invited".
Banner "Unauthorized access prohibited" = clear legal ground.
Consulter legal team language exact bannières.
Account management

Comptes locaux — Password policies

// Local accounts vs AAA

Local accounts : Stockés device config (username/password local database).

AAA (RADIUS/TACACS+) : Centralized authentication (preferred enterprise).

Best practice : AAA primary, local account backup emergency (AAA server down).

// Configuration local accounts secure

❌ BAD (weak) :
username admin password cisco
→ Plaintext password config (show run visible)

⚠️ BETTER (encrypted) :
username admin password 7 082048430017
→ Type 7 encryption (weak, reversible Vigenere cipher)

✅ BEST (hashed) :
username admin secret MySecurePass123
→ Type 5 MD5 hash (irreversible, stronger)
→ Config shows : username admin secret 5 $1$xyz$abc...

// Password policies

Enable password encryption :
service password-encryption
→ Encrypt ALL passwords type 7 (better than plaintext)

Minimum password length :
security passwords min-length 10

Login attempt limits :
login block-for 120 attempts 3 within 60
→ Block 120 sec after 3 failed attempts within 60 sec

Session timeout :
line vty 0 15
  exec-timeout 10 0
→ Auto-logout after 10 minutes idle

// Privilege levels

Cisco privilege levels : 0-15 (0 = minimal, 15 = full admin).

Custom privilege levels :

privilege exec level 5 show running-config
privilege exec level 5 show startup-config

username viewer privilege 5 secret ViewPass123

→ User "viewer" : Level 5 (can show configs, but NOT modify)

// Account management best practices

  • ✅ AAA primary : RADIUS/TACACS+ centralized (avec local fallback)
  • ✅ Strong passwords : 10+ chars, complexity, use "secret" (MD5 hash)
  • ✅ Unique accounts : Pas shared "admin" → individual usernames (audit trail)
  • ✅ Disable unused accounts : Remove old accounts (ex-employees)
  • ✅ Session timeouts : Auto-logout idle sessions (10-15 min)
  • ✅ Login limits : Block après failed attempts (brute-force protection)
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : SSH port 23, Telnet port 22
INVERSE.

Telnet : TCP port 23 (plaintext, insecure)
SSH : TCP port 22 (encrypted, secure)

Mémo : SSH = Secure = 22 (pair = sécurisé moderne)
❌ Piège #2 : SNMPv3 = community strings chiffrés
FAUX. SNMPv3 = PAS community strings.

SNMPv1/v2c : Community strings (plaintext)
SNMPv3 : Username + password (auth SHA/MD5, priv AES/DES)

SNMPv3 abandonne concept community strings complètement
❌ Piège #3 : Bannières MOTD = après login
FAUX. MOTD = AVANT login.

MOTD : Displayed BEFORE login (legal warning all users)
Login banner : AFTER MOTD, BEFORE username prompt
Exec banner : AFTER successful login

Ordre : MOTD → Login → [username/password] → Exec
Résumé

🔑 Ce que tu dois retenir absolument

→SSH vs Telnet : SSH port 22 chiffré (secure). Telnet port 23 plaintext (insecure, disable). Config : crypto key generate rsa, transport input ssh, ip ssh version 2
→Disable services : HTTP/HTTPS GUI (no ip http server), CDP/LLDP external interfaces (no cdp enable), Telnet, Finger. Principle : Disable TOUT non utilisé
→SNMP : SNMPv1/v2c = community strings plaintext (insecure). SNMPv3 = username/password auth SHA + encryption AES (secure). Always SNMPv3, ACL restrict, read-only preferred
→Bannières : MOTD before login (legal warning). Pas "Welcome" (affaiblit legal). Include "Unauthorized access prohibited", "All activity monitored"
→Comptes locaux : Use "secret" (MD5 hash) pas "password" (weak). AAA primary, local fallback. Strong passwords 10+ chars, exec-timeout, login block-for (failed attempts), privilege levels
