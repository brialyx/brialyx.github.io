Physical connectivity

Câbles défectueux — Cause #1 problèmes L1

Câbles défectueux = Cause la plus fréquente problèmes Layer 1 (30-40% tickets support).

// Symptômes câble défectueux

  • No link : LED NIC OFF, LED switch port OFF
  • Intermittent connectivity : Link flapping (UP/DOWN cycles)
  • Slow performance : 1 Gbps negocié → throughput 10 Mbps (degraded)
  • Packet loss : Ping 50% loss

// Causes câbles défectueux

1. Physical damage

  • Câble écrasé (door, chair wheels, heavy equipment)
  • Bend radius excessive (courbure trop serrée, internal wires cassés)
  • Connecteur RJ45 cassé (clip broken, pins bent)

2. Poor termination

  • Wires pas complètement insérées RJ45 connector
  • Crimp faible (connector pas bien serti)
  • Wires exposition (jacket pas assez stripped, outer jacket crushed by crimp)

3. Cable degradation

  • Age (10+ years, insulation brittle)
  • Environmental (humidity, temperature extremes)
  • Rodents (rats/mice chew cables datacenter)

// Diagnostic câble défectueux

Step 1 : Visual inspection
→ Check cable physical damage (kinks, cuts, crushed)
→ Check RJ45 connectors (clip intact, pins straight)
→ Check connection seated fully (push firmly both ends)

Step 2 : LED check
→ NIC LED : OFF = no link (cable or port problem)
→ Switch port LED : OFF = no link, AMBER = error/disabled

Step 3 : Swap cable
→ Replace suspect cable known-good cable
→ Link UP ? → Original cable defective ✅
→ Link still DOWN ? → Problem elsewhere (NIC, switch port)

Step 4 : Cable tester (if available)
→ Simple tester : Continuity test (all 8 wires intact ?)
→ Advanced tester : TDR (Time Domain Reflectometer) locate break exact distance
🔌
Cable swap = quickest diagnostic Simple tester continuity utile, MAIS swap cable known-good = fastest.
Toujours avoir spare cables (Cat5e, Cat6, différentes longueurs).
Production : Swap first, test defective cable later (minimize downtime).
Cable standards

Pinout incorrect — Straight-through vs Crossover

Pinout = Ordre wires (pins) RJ45 connector. Incorrect pinout = no connectivity OU degraded.

// T568A vs T568B standards

T568A : Pin 1 → Green/White, Pin 2 → Green, Pin 3 → Orange/White...
T568B : Pin 1 → Orange/White, Pin 2 → Orange, Pin 3 → Green/White...

Différence : Pins 1,2 ↔ 3,6 swapped (TX ↔ RX)

IMPORTANT : T568A vs T568B = both valid standards.
Consistency matters (choose one, use everywhere).

// Straight-Through cable

Définition : MÊME pinout both ends (T568B ↔ T568B OU T568A ↔ T568A).

Usage : Connecter devices DIFFÉRENTS types (PC ↔ Switch, Router ↔ Switch).

Straight-Through pinout :

End A : T568B
End B : T568B

Pin 1 (O/W) → Pin 1 (O/W)
Pin 2 (O) → Pin 2 (O)
Pin 3 (G/W) → Pin 3 (G/W)
...

TX (pins 1,2) End A → TX (pins 1,2) End B
RX (pins 3,6) End A → RX (pins 3,6) End B

// Crossover cable

Définition : DIFFÉRENT pinout ends (T568B ↔ T568A). TX/RX pins swapped.

Usage : Connecter devices MÊME type (PC ↔ PC, Switch ↔ Switch, Router ↔ Router).

Crossover pinout :

End A : T568B
End B : T568A

Pin 1 (O/W) → Pin 3 (G/W)
Pin 2 (O) → Pin 6 (G)
Pin 3 (G/W) → Pin 1 (O/W)
Pin 6 (G) → Pin 2 (O)

Result : TX End A → RX End B (crossed)

// Auto-MDIX (Modern NICs)

Auto-MDIX = Automatic Medium-Dependent Interface Crossover. NIC détecte cable type, auto-adjust TX/RX.

  • ✅ Modern NICs (2005+) : Auto-MDIX enabled → Straight-through OU Crossover = both work
  • ⚠️ Legacy NICs (pre-2005) : Require correct cable type
🎯
Question TRÈS fréquente examen "Cable type connecter PC directement PC (no switch) ?"
Réponse : Crossover

"Cable type connecter PC switch ?"
Réponse : Straight-through

Mémo : SAME device type = Crossover, DIFFERENT device type = Straight-through
(MAIS modern Auto-MDIX = both work, examen assume legacy)

// Symptômes pinout incorrect

  • No link (LED OFF) : Legacy NIC, wrong cable type
  • Link UP mais no data : Partial pinout error (some pins correct, others not)
  • Modern Auto-MDIX : Usually works regardless (but verify if issues)
Electromagnetic interference

EMI — Interférences électromagnétiques

EMI (Electromagnetic Interference) = Signaux électromagnétiques externes perturbent signaux réseau câbles.

// Sources EMI communes

  • Fluorescent lights : Ballasts génèrent EMI (especially old T12 fixtures)
  • Electric motors : HVAC, fans, elevators
  • Power cables : High-voltage AC lines proximité câbles réseau
  • Radio transmitters : Walkie-talkies, cell towers
  • Microwaves : Kitchen appliances (2.4 GHz interfere WiFi also)

// Symptômes EMI

  • Intermittent connectivity : Works sometimes, drops randomly
  • Slow performance : Throughput degraded (retransmissions)
  • Packet loss : Errors spike specific times (ex: lunchtime microwave usage)
  • CRC errors : Frame Check Sequence failures (corrupted data)

// Diagnostic EMI

Pattern recognition :
→ Errors specific times ? (Ex: 12-1 PM lunch = microwave)
→ Errors specific location ? (Near HVAC room, elevator shaft)
→ Multiple cables affected ? (EMI source affects area, not single cable)

Physical inspection :
→ Cable proximity power lines ? (< 12 inches = risk)
→ Cable parallel fluorescent lights ? (run perpendicular if possible)
→ Cable near motors/transformers ?

Test :
→ Relocate cable away suspected EMI source → Errors stop ? ✅
→ Swap cable shielded (STP) → Errors reduce ? ✅

// Mitigation EMI

  • Shielded cables (STP) : Metal foil/braid shield blocks EMI (vs UTP unshielded)
  • Conduit : Metal conduit protects cables (bonus: physical protection)
  • Distance : Keep 12+ inches separation power cables, EMI sources
  • Perpendicular crossing : If must cross power cable, cross 90° (not parallel)
  • Fiber optic : Immune EMI (light signals, no electrical)
⚡
EMI = frustrating troubleshoot Symptômes intermittent, timing unpredictable.
Clé : Pattern recognition (time-based, location-based).
Exemple classique : Errors 12-1 PM daily → Kitchen microwave lunchtime.
Switching loops

Boucles réseau — Broadcast storms

Boucle réseau (Switching loop) = Chemin redondant Layer 2 (switches) sans STP → Broadcast storm catastrophique.

// Cause boucle réseau

Scénario boucle :

Switch A ←→ Switch B (link 1)
  ↓               ↓
  └────────┘ (link 2 - redundant path)

PC-A sends broadcast (ARP request) → Switch A
→ Switch A forwards ALL ports (including both links to Switch B)
→ Switch B receives broadcast BOTH links
→ Switch B forwards ALL ports (including both links back to Switch A)
→ Switch A receives broadcast again → Forwards again → LOOP ∞

→ Broadcast multiplies exponentially (2→4→8→16...)
→ Network saturated seconds (broadcast storm)
→ All devices unresponsive (CPU 100% processing broadcasts)

// Symptômes boucle réseau

  • Network meltdown soudain : Everything stops working instantly
  • Switch CPU 100% : Processing infinite broadcasts
  • MAC address table flapping : Same MAC seen multiple ports (unstable)
  • LED switch ports rapid blinking : Massive traffic flood
  • Timeouts everywhere : Ping, web, email = all timeout

// Spanning Tree Protocol (STP)

STP (802.1D) = Protocol prevents loops. Logically blocks redundant paths, maintains topology loop-free.

STP workflow :

1. Elect Root Bridge (switch lowest Bridge ID)
2. Calculate best path each switch → Root Bridge
3. Block redundant ports (Blocking state, no forwarding)
4. If primary link fails → Blocked port activates (Forwarding state)

→ STP maintains single active path (loop-free)
→ Redundancy available failover (convergence ~30-50 sec)

// Diagnostic boucle réseau

Symptom : Network sudden total failure, all devices timeout

Step 1 : Identify loop
→ Check switch logs : "STP topology change" messages
→ Check MAC address table : Same MAC multiple ports (flapping)
→ Physical inspection : Redundant cables switch ↔ switch ?

Step 2 : Break loop
→ Unplug ONE redundant cable → Network recovers instantly ✅
→ Identify root cause : STP disabled ? New cable added ?

Step 3 : Fix permanent
→ Enable STP all switches (if disabled)
→ Verify STP convergence (show spanning-tree)
→ Remove unauthorized cables (user plugged both ports PC → switch loop)

// Causes boucles production

  • STP disabled : Admin disabled STP (bad idea, legacy reasons)
  • User error : User branche BOTH ports laptop → switch (2 cables)
  • Rogue switch : User branche personal switch (creates loop)
  • Misconfiguration : STP priority incorrect, root bridge wrong
🔁
STP = always enable production Redundancy good (failover) MAIS requires STP prevent loops.
Modern variants : RSTP (Rapid STP, convergence <5 sec), MSTP (Multiple STP).
Never disable STP unless très specific reason + understanding risks.
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : Crossover cable = obsolète (Auto-MDIX everywhere)
NUANCE. Auto-MDIX = modern NICs (2005+), MAIS examen assume legacy.

Examen questions : "Which cable type PC-to-PC ?" → Réponse = Crossover.
Real world : Modern devices Auto-MDIX → Straight-through works aussi.

Examen : Assume NO Auto-MDIX unless stated. Follow classic rules (SAME type = Crossover).
❌ Piège #2 : EMI affect fiber optic cables
FAUX. Fiber optic = immune EMI.

Fiber = light signals (photons), pas electrical signals.
EMI = electromagnetic interference → affects electrical only.

Fiber advantages : Immune EMI, longer distance, higher bandwidth.
Examen : "Solution EMI environment ?" → Fiber optic = correct answer.
❌ Piège #3 : Boucle réseau = problème Layer 3 routing
FAUX. Boucle réseau = Layer 2 switching problem.

Layer 2 : Switches, broadcast frames, STP, MAC address tables.
Layer 3 : Routers, IP packets, routing protocols (TTL prevents routing loops).

Broadcast storm = L2 phenomenon (switches forward broadcasts all ports).
Examen : "Broadcast storm cause ?" → Switching loop (L2), not routing loop (L3).
