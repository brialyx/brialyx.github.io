Duplicate IP

Conflit d'adresses IP — Duplicate address

Conflit IP (IP conflict) = 2 devices même réseau assigned même IP address → connectivity intermittent/failed.

// Symptômes conflit IP

  • Windows : Pop-up "IP address conflict detected"
  • Connectivity intermittent : Works sometimes, fails randomly
  • ARP cache confusion : IP maps 2 different MACs (flapping)
  • Both devices affected : Neither works reliably

// Causes conflit IP

1. Static IP duplicate manual

Admin assigns 192.168.1.100 → Server-A (manual static)
Admin assigns 192.168.1.100 → Server-B (manual static)
→ Conflict (human error, poor documentation)

2. DHCP + Static overlap

DHCP pool : 192.168.1.100 - 192.168.1.200
Static assignment : 192.168.1.150 → Printer (manual)
DHCP server : Assigns 192.168.1.150 → PC-A (automatic)
→ Conflict (DHCP doesn't know about static)

3. Device moved networks (stale IP)

Laptop : 10.0.1.50 (Office A DHCP)
User moves Office B : 10.0.1.50 still cached
Office B DHCP assigns 10.0.1.50 → Desktop
→ Conflict when laptop reconnects

// Diagnostic conflit IP

Windows :
C:\> ipconfig /all
→ IP Address: 192.168.1.100
→ Error message : "Duplicate IP address detected"

Check ARP cache :
C:\> arp -a | findstr 192.168.1.100
192.168.1.100       aa-bb-cc-dd-ee-ff
192.168.1.100       11-22-33-44-55-66 (different MAC = conflict)

Linux :
$ ip addr show
→ Check for duplicate address detection (DAD) messages

// Résolution conflit IP

Step 1 : Identify both devices
→ Use ARP cache (2 MACs same IP)
→ Use MAC address lookup (vendor, switch port)

Step 2 : Resolve conflict
Option A : Change one IP (static reassignment)
Option B : Release/renew DHCP
   C:\> ipconfig /release
   C:\> ipconfig /renew

Step 3 : Prevent recurrence
→ Exclude static IPs from DHCP pool
   DHCP server : Exclude 192.168.1.1 - 192.168.1.50
   Static assignments : Use 192.168.1.1 - .50 only
→ Document static assignments (IP management spreadsheet)
🎯
Question fréquente examen "User reports intermittent connectivity. Error 'IP address conflict'. BEST solution ?"
A) Replace cable
B) Release/renew DHCP
C) Reboot switch
D) Change DNS server

Réponse : B) Release/renew DHCP
IP conflict = duplicate address. Release/renew gets new unique IP.
Automatic private IP

APIPA — 169.254.x.x addresses

APIPA (Automatic Private IP Addressing) = Windows self-assigns IP 169.254.0.1 - 169.254.255.254 si DHCP fails.

// Principe APIPA

DHCP workflow normal :
1. PC boots → Sends DHCP Discover (broadcast)
2. DHCP server → DHCP Offer (192.168.1.100)
3. PC → DHCP Request
4. DHCP server → DHCP Ack
5. PC configured : 192.168.1.100/24 ✅

DHCP failure → APIPA :
1. PC boots → Sends DHCP Discover
2. No response (DHCP server down/unreachable)
3. PC retries multiple times (timeout)
4. PC self-assigns 169.254.x.x (APIPA)
5. PC : 169.254.123.45/16 (link-local only)

// Causes APIPA (169.254.x.x)

  • DHCP server down : Service stopped, server offline
  • No network connectivity : Cable unplugged, switch port down
  • Wrong VLAN : PC VLAN 20, DHCP server VLAN 10 (isolation)
  • DHCP relay missing : Different subnet, no IP helper-address
  • DHCP pool exhausted : All IPs leased, no available addresses

// Symptômes APIPA

C:\> ipconfig

Ethernet adapter Local Area Connection:
   IP Address: 169.254.123.45
   Subnet Mask: 255.255.0.0
   Default Gateway: (blank)

→ Can't access Internet (no gateway, no routing)
→ Can communicate local APIPA devices only (169.254.0.0/16)

// Diagnostic APIPA

Step 1 : Verify APIPA
C:\> ipconfig | findstr 169.254
→ 169.254.x.x = APIPA detected

Step 2 : Check physical
→ Cable plugged ? LED link UP ?
→ Switch port enabled ?

Step 3 : Test DHCP
C:\> ipconfig /release
C:\> ipconfig /renew
→ Error "Unable to contact DHCP server" ? = DHCP down

Step 4 : Verify DHCP server
→ DHCP service running ?
→ DHCP pool available addresses ?
→ DHCP reachable (ping DHCP server IP) ?

// Résolution APIPA

  • Restart DHCP service : Server side fix
  • Fix network connectivity : Cable, VLAN assignment
  • Configure IP helper-address : DHCP relay different subnets
  • Expand DHCP pool : Add more IP addresses (if exhausted)
  • Manual static IP : Temporary workaround (if DHCP unavailable)
⚠️
169.254.x.x = RED FLAG DHCP problem APIPA address = symptom, not solution.
Indicates DHCP failure (server, connectivity, VLAN, relay).
Never acceptable production (troubleshoot root cause).
Subnet misconfiguration

Mauvais masque de sous-réseau — Wrong subnet mask

Mauvais masque = Device configured incorrect subnet mask → cannot communicate expected devices.

// Impact mauvais masque

Network design :
Subnet : 192.168.1.0/24 (255.255.255.0)
Gateway : 192.168.1.1
Devices : 192.168.1.2 - 192.168.1.254

PC misconfigured :
IP : 192.168.1.50
Mask : 255.255.0.0 (WRONG - should be 255.255.255.0)
Gateway : 192.168.1.1

Result :
PC thinks network = 192.168.0.0/16 (65,534 hosts)
→ PC tries direct communication 192.168.x.x (no gateway)
→ ARP requests sent, no responses (devices different subnet)
→ Communication FAILS

// Symptômes mauvais masque

  • Can ping gateway : Gateway same subnet (coincidence)
  • Can't ping other devices : Thinks direct communication (wrong)
  • Can't access Internet : Routing broken (gateway not used)
  • Selective connectivity : Some IPs work, others don't (unpredictable)

// Diagnostic mauvais masque

Verify subnet mask :
C:\> ipconfig

IP Address: 192.168.1.50
Subnet Mask: 255.255.0.0 (suspect - verify network design)
Default Gateway: 192.168.1.1

Compare autre devices :
Working PC :
IP : 192.168.1.51
Mask : 255.255.255.0 (correct)

→ Mismatch detected (50 wrong mask, 51 correct)

// Résolution mauvais masque

Windows :
Network Adapter Properties → TCP/IPv4
→ Change Subnet Mask : 255.255.255.0

OR DHCP :
→ Enable "Obtain IP address automatically"
→ DHCP provides correct mask

Verify fix :
C:\> ipconfig
→ Subnet Mask : 255.255.255.0 ✅
→ Ping other devices → Success ✅
Default gateway

Mauvaise passerelle par défaut — Wrong default gateway

Mauvaise passerelle = Device configured incorrect/missing default gateway → cannot reach devices outside local subnet.

// Rôle default gateway (rappel)

Default Gateway = Router IP (local subnet)

PC : 192.168.1.50/24
Gateway : 192.168.1.1 (router)

Communication :
→ Destination same subnet (192.168.1.x) ? → Direct (ARP, switch)
→ Destination different subnet (8.8.8.8) ? → Send to Gateway (router forwards)

// Problèmes gateway

1. Gateway missing (blank)

PC config :
IP : 192.168.1.50
Mask : 255.255.255.0
Gateway : (blank)

→ Can communicate local subnet (192.168.1.0/24) ✅
→ Cannot reach Internet (no router) ❌

2. Gateway incorrect IP

PC config :
IP : 192.168.1.50
Gateway : 192.168.1.99 (wrong - should be .1)

→ PC sends traffic 192.168.1.99 (nonexistent)
→ No response (gateway doesn't exist)
→ Cannot reach outside subnet ❌

3. Gateway wrong subnet

PC config :
IP : 192.168.1.50/24
Gateway : 192.168.2.1 (different subnet)

→ Gateway not reachable (different network)
→ PC cannot communicate gateway ❌

// Symptômes mauvaise gateway

  • Can ping local devices : Same subnet communication works
  • Can't ping Internet : 8.8.8.8, google.com timeout
  • Can't ping gateway : Gateway IP unreachable (if wrong subnet)
  • Traceroute stops local : No hops beyond local network

// Diagnostic mauvaise gateway

Step 1 : Verify gateway config
C:\> ipconfig

Default Gateway: 192.168.1.99 (verify correct ?)

Step 2 : Ping gateway
C:\> ping 192.168.1.99
→ Timeout = gateway unreachable (wrong IP or down)

Step 3 : Find correct gateway
→ Check DHCP server config (correct gateway ?)
→ Ask network admin (what's gateway this subnet ?)
→ Check autre working PCs (ipconfig gateway)

// Résolution mauvaise gateway

Windows :
Network Adapter Properties → TCP/IPv4
→ Change Default Gateway : 192.168.1.1 (correct)

Verify :
C:\> ping 192.168.1.1 (gateway)
→ Reply ✅
C:\> ping 8.8.8.8 (Internet)
→ Reply ✅
C:\> tracert google.com
→ Hop 1 : 192.168.1.1 (gateway) ✅
🎯
Question classique examen "User can ping local devices, but NOT Internet. MOST likely cause ?"
A) Cable defect
B) Wrong default gateway
C) DNS failure
D) VLAN mismatch

Réponse : B) Wrong default gateway
Local works, Internet fails = gateway problem (routing outside subnet).
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : APIPA = valid IP configuration
FAUX. APIPA = failure indicator, not valid config.

169.254.x.x = DHCP failed (server down, network issue).
APIPA allows local link-local communication ONLY (temporary).
Production : APIPA unacceptable (troubleshoot DHCP).

Examen : 169.254.x.x = DHCP problem (fix DHCP, not accept APIPA).
❌ Piège #2 : Wrong subnet mask = no connectivity at all
NUANCE. Wrong subnet mask = selective connectivity problems.

May still ping gateway (coincidence same calculated subnet).
May reach some devices (within calculated subnet), not others.
Unpredictable behavior (depends specific IPs involved).

Symptom : "Some devices work, others don't" = check subnet mask consistency.
❌ Piège #3 : Missing gateway = can't ping anything
FAUX. Missing gateway = local subnet still works.

No gateway : Can ping devices same subnet (direct communication).
Can't reach : Different subnets, Internet (routing required).

Symptom : "Can ping 192.168.1.x, can't ping Internet" = missing/wrong gateway.
Examen : Differentiate local (no gateway needed) vs remote (gateway required).
Résumé

🔑 Ce que tu dois retenir absolument

→Conflit IP : 2 devices same IP. Symptom: "IP conflict detected", intermittent connectivity. Causes: Static duplicate, DHCP+static overlap. Fix: Release/renew DHCP, exclude statics from pool
→APIPA (169.254.x.x) : Auto-assigned if DHCP fails. Causes: DHCP down, no network, wrong VLAN, no relay, pool exhausted. Symptom: No Internet, no gateway. Fix: Restart DHCP, fix connectivity
→Wrong subnet mask : Device thinks different network size. Symptom: Selective connectivity (some work, others don't). Fix: Correct mask match network design (verify working devices)
→Wrong gateway : Missing, incorrect IP, wrong subnet. Symptom: Local works, Internet/remote fails. Test: Ping gateway, ping 8.8.8.8, tracert. Fix: Configure correct gateway IP (router local subnet)
→Diagnostic commands : ipconfig (Windows), ip addr (Linux), ipconfig /release, ipconfig /renew, ping gateway, ping Internet, tracert/traceroute, arp -a (conflicts)
→Troubleshooting workflow : 1) Check IP (APIPA? Conflict?), 2) Verify mask (match network?), 3) Test gateway (ping? Correct?), 4) Test Internet (8.8.8.8), 5) Fix root cause (DHCP, static, config)
