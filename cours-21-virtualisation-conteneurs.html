Virtualisation

Hyperviseurs — Type 1 vs Type 2

🏗️
Analogie — appartements vs colocations Serveur physique = immeuble entier pour 1 locataire (gaspillage).
VMs = diviser immeuble en appartements séparés (plusieurs locataires, isolation complète).
Hyperviseur = gestionnaire immeuble gérant appartements.

Hyperviseur = logiciel créant/gérant machines virtuelles (VMs). Abstrait hardware physique.

// Type 1 (Bare-Metal)

S'exécute directement sur hardware physique (pas OS hôte).

Architecture :

[VM1 Windows] [VM2 Linux] [VM3 Ubuntu]
↓ ↓ ↓
[Hyperviseur Type 1 — ESXi/Hyper-V]
↓
[Hardware physique — CPU/RAM/Disque]

Exemples :

  • VMware ESXi (vSphere) — leader datacenter
  • Microsoft Hyper-V — intégré Windows Server
  • Citrix XenServer
  • KVM (Linux Kernel-based VM)

Avantages :

  • ✅ Performances maximales (accès direct hardware)
  • ✅ Densité élevée (nombreuses VMs par serveur)
  • ✅ Stabilité (pas OS hôte vulnérable)
  • ✅ Production-grade (datacenters, cloud)

// Type 2 (Hosted)

S'exécute sur OS hôte comme application.

Architecture :

[VM1 Windows] [VM2 Linux]
↓ ↓
[Hyperviseur Type 2 — VirtualBox/VMware Workstation]
↓
[OS Hôte — Windows 10 / macOS]
↓
[Hardware physique]

Exemples :

  • VMware Workstation (Windows/Linux)
  • Oracle VirtualBox (gratuit, cross-platform)
  • Parallels Desktop (macOS)

Avantages :

  • ✅ Simple installation (comme app normale)
  • ✅ Coexistence (OS hôte + VMs simultanés)
  • ✅ Dev/Test (laptop développeurs)
  • ⚠️ Performances moindres (overhead OS hôte)
  • ⚠️ Densité limitée

Réseau virtualisé

vSwitch et vNIC — Réseau virtuel

// vSwitch (Virtual Switch)

Switch software dans hyperviseur. Connecte VMs entre elles + réseau physique.

Fonctions :

  • Switching Layer 2 entre VMs
  • VLANs (isolation VMs)
  • Port groups (policies réseau)
  • Uplink vers switch physique

Types vSwitch :

Standard vSwitch : Basic, local à 1 hôte ESXi

Distributed vSwitch (vDS) : Centralisé, gestion multiple hôtes ESXi (entreprise)

// vNIC (Virtual NIC)

Carte réseau virtuelle dans VM. VM pense qu'elle a vraie NIC physique.

Caractéristiques :

  • MAC address unique (généré hyperviseur)
  • Connectée à vSwitch (port group)
  • Plusieurs vNICs par VM possible
  • Types émulation : E1000, VMXNET3 (VMware)

Conteneurisation

Containers — Docker & Kubernetes

📦
Analogie — appartement vs chambre hôtel VM = appartement complet (cuisine, salle bain, salon) → lourd, isolé.
Container = chambre hôtel (partage plomberie, électricité immeuble) → léger, rapide.

Container = paquet léger contenant app + dépendances. Partage OS kernel hôte (pas OS complet comme VM).

[Container 1 — App Node.js]
[Container 2 — App Python]
[Container 3 — Nginx]
↓
[Container Runtime — Docker Engine]

[OS Hôte — Linux kernel PARTAGÉ]

→ Containers partagent kernel Linux
→ Isolation via namespaces/cgroups

// Docker

Docker = plateforme containerisation dominante.

Composants :

  • Docker Image : Template read-only (app + dépendances)
  • Docker Container : Instance running d'image
  • Dockerfile : Recette construire image
  • Docker Hub : Registry images publiques

// Kubernetes (K8s)

Kubernetes = orchestrateur containers. Gère déploiement/scaling/load balancing containers en production.

Fonctions :

  • Auto-scaling (augmente containers si charge)
  • Self-healing (redémarre containers crashés)
  • Load balancing distribué
  • Rolling updates (déploiements sans downtime)
  • Service discovery

💡
Containers vs VMs — Pas opposés ! Souvent combinés : VMs (isolation forte) contiennent containers (densité élevée).
Exemple : Kubernetes clusters sur VMs cloud (AWS EKS, Azure AKS).
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : Type 2 plus performant que Type 1
FAUX. Type 1 (Bare-Metal) = plus performant.

Type 2 a overhead OS hôte → moins performant.
Type 1 = production, Type 2 = dev/test
❌ Piège #2 : Containers = VMs légères
ARCHITECTURE DIFFÉRENTE.

VM : OS complet, hyperviseur, isolation forte
Container : Partage kernel, isolation process-level

Containers ≠ "VMs light" — technologie fondamentalement différente
❌ Piège #3 : vSwitch = switch physique virtualisé
PAS EXACTEMENT. vSwitch = switch software dans hyperviseur.

Fonctions similaires switch physique MAIS implémentation software.
Connecte VMs localement + uplink vers switch physique
Résumé

🔑 Ce que tu dois retenir absolument

→Type 1 (Bare-Metal) : Direct hardware, performances max, production (ESXi, Hyper-V). Type 2 (Hosted) : Sur OS hôte, dev/test (VirtualBox)
→vSwitch : Switch software hyperviseur, connecte VMs + uplink physique. vNIC : Carte réseau virtuelle VM (MAC unique)
→Containers : Légers (MB, secondes boot), partagent kernel, densité élevée. VMs : Lourdes (GB, minutes boot), OS complet, isolation forte
→Docker : Runtime containers. Kubernetes : Orchestration containers (auto-scaling, self-healing, load balancing)
