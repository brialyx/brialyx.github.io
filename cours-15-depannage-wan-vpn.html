Packet drops

Perte de paquets — Packet loss

Perte de paquets (Packet loss) = Packets dropped en transit → retransmissions → slow applications, timeouts.

// Mesure packet loss

C:\> ping 8.8.8.8 -n 100

Packets: Sent = 100, Received = 95, Lost = 5 (5% loss)

Acceptable loss :
0-1% : Excellent (normal)
1-2.5% : Acceptable (noticeable video/VoIP)
2.5-5% : Poor (applications slow, retransmissions)
>5% : Bad (serious problem, investigate)

// Causes packet loss

1. Congestion link (bandwidth saturated)

WAN link : 100 Mbps
Traffic load : 110 Mbps (peak hours)

→ Link saturated (exceeds capacity)
→ Router buffers full → Drops packets (tail drop)
→ 10% packet loss

2. Interface errors (CRC, collisions)

Router# show interfaces GigabitEthernet0/0

CRC errors : 1,234 (incrementing)
Input errors : 2,456

→ Cable defect, EMI, duplex mismatch
→ Corrupted frames → Dropped

3. Routing loops / TTL expiration

Packets loop routers → TTL expires → Dropped (voir Cours TR10).

4. Firewall/ACL blocking

Firewall rule : Block ICMP (ping)

→ Ping packets dropped → 100% loss
→ Other traffic (TCP/UDP) may work

// Symptômes packet loss

  • Slow applications : TCP retransmissions add latency
  • Video/audio choppy : VoIP, streaming buffer/stutter
  • File transfer slow : Downloads/uploads take longer
  • Intermittent connectivity : Works sometimes, fails randomly

// Diagnostic packet loss

Step 1 : Ping test (measure loss)
C:\> ping 8.8.8.8 -n 100
→ Loss 5% ? = Problem

Step 2 : Traceroute (locate where loss occurs)
C:\> tracert 8.8.8.8

1    1 ms    192.168.1.1 (gateway) — 0% loss
2    10 ms   10.0.0.1 (ISP router) — 15% loss
3    15 ms   203.0.113.1 (ISP core) — 15% loss

→ Loss starts hop 2 (ISP router) = ISP problem

Step 3 : Check interface errors
Router# show interfaces
→ CRC errors incrementing ? = Physical problem

// Résolution packet loss

  • Congestion : Upgrade bandwidth, QoS prioritize traffic, compress data
  • Interface errors : Replace cable, fix duplex, check connectors
  • ISP problem : Contact ISP (provide traceroute showing loss hop)
  • Firewall : Adjust rules (allow required traffic)
🎯
Question fréquente examen "VoIP calls choppy. Ping shows 5% packet loss. BEST tool identify WHERE loss occurs ?"
A) nslookup
B) traceroute/tracert
C) netstat
D) ipconfig

Réponse : B) traceroute/tracert
Traceroute shows loss per hop (identifies problem router/link).
Bandwidth saturation

Congestion WAN — Link congestion

Congestion = Traffic demand exceeds link capacity → buffers full → packet loss, high latency.

// Principe congestion

WAN link capacity : 100 Mbps

Traffic load :
09:00 : 30 Mbps (30% utilization) — Good ✅
12:00 : 70 Mbps (70% utilization) — Moderate ⚠️
15:00 : 105 Mbps (105% utilization) — Congested ❌

→ Exceeds capacity → Queue buffers → Tail drop → Loss

// Causes congestion

  • Insufficient bandwidth : Link too small user count/applications
  • Bulk transfers : Backups, large file downloads saturate link
  • Broadcast storms : Layer 2 loops (voir Cours TR08)
  • DDoS attack : Flood traffic overwhelms link
  • Misconfigured QoS : Low-priority traffic crowds out high-priority

// Symptômes congestion

  • Slow during peak hours : Morning/lunchtime slow, night fast
  • High latency : Ping times increase (10 ms → 200 ms)
  • Packet loss : Buffers overflow → Drops
  • Applications timeout : Web, email, VoIP fail

// Diagnostic congestion

Check interface utilization :
Router# show interfaces GigabitEthernet0/0

5 minute input rate : 95000000 bits/sec (95 Mbps)
5 minute output rate : 98000000 bits/sec (98 Mbps)
Link speed : 100 Mbps

→ Utilization 95-98% (near capacity) = Congestion ❌

Monitor over time (SNMP, NetFlow) :
→ Graph utilization 24 hours
→ Identify peak times (when >80% utilization)

// Résolution congestion

Option 1 : Upgrade bandwidth

Current : 100 Mbps WAN link
Upgrade : 1 Gbps WAN link

→ 10x capacity increase
→ Solves congestion (if budget permits)

Option 2 : QoS (Quality of Service)

Priority classes :
1. VoIP (highest) — Guaranteed 20% bandwidth
2. Business apps (high) — 40%
3. Web browsing (medium) — 30%
4. Backups (low) — 10% (can be dropped if congestion)

→ Critical traffic prioritized
→ Less important traffic throttled/dropped first

Option 3 : Traffic shaping / Policing

  • Shaping : Buffer excess traffic (smooth bursts)
  • Policing : Drop excess traffic (enforce limits)
  • Limit non-business traffic (YouTube, social media)

Option 4 : Schedule bulk transfers

Backups : Run overnight (2:00 AM - 6:00 AM)
→ Avoid peak business hours
→ Utilize unused bandwidth off-hours
Round-trip time

Latence élevée — High latency

Latence (Latency) = Délai transmission packet source → destination (round-trip time RTT). High latency → slow applications.

// Latency targets

ApplicationAcceptable LatencyImpact if exceeded
VoIP <150 ms Voice delay noticeable, echo
Video conferencing <150 ms Lag, out-of-sync audio/video
Gaming <50 ms Input lag, unplayable FPS
Web browsing <300 ms Pages load slowly
File transfer <1000 ms Slow but functional

// Causes high latency

1. Physical distance (geographic)

Speed of light : ~200,000 km/sec (fiber optic)

Distance NYC → London : 5,500 km
Minimum latency : 5,500 / 200,000 = 0.0275 sec = 27.5 ms
Round-trip : 55 ms (theoretical minimum)

Actual latency : 80-120 ms (routing overhead, processing)

→ Distance unavoidable (physics limit)

2. Congestion (queuing delay)

Normal : Ping 10 ms
During congestion : Ping 200 ms

→ Packets queued buffers (waiting transmission)
→ Bufferbloat (excessive buffering adds delay)

3. Satellite links

Geostationary satellite : 35,786 km altitude
Round-trip distance : 4 × 35,786 = 143,144 km
Latency : 143,144 / 200,000 = 0.716 sec = 716 ms

→ Satellite unavoidably high latency (>500 ms typical)

4. Too many hops (routing)

Each router hop : +1-10 ms processing delay
Inefficient routing : 20 hops = 200 ms added
Optimal routing : 5 hops = 50 ms added

// Diagnostic high latency

C:\> ping 8.8.8.8

Reply from 8.8.8.8: time=250ms TTL=54 (high!)
Reply from 8.8.8.8: time=245ms TTL=54

Traceroute identify delay hops :
C:\> tracert 8.8.8.8

1    1 ms    192.168.1.1
2    10 ms   10.0.0.1
3    180 ms  203.0.113.1 (delay spike here)
4    185 ms  8.8.8.8

→ Hop 3 adds 170 ms (problem hop)

// Résolution high latency

  • Reduce congestion : Upgrade bandwidth, QoS
  • Optimize routing : Shorter path (BGP tuning)
  • Use CDN : Content delivery network (geographically closer servers)
  • Satellite : Replace with fiber/cable (if possible)
  • Compression : Reduce data volume (less transmission time)
Virtual Private Network

VPN qui ne monte pas — VPN connection fails

VPN fails to connect = Encrypted tunnel cannot establish → remote access denied.

// Causes VPN connection failure

1. Firewall blocking VPN protocols

VPN protocols/ports :
IPsec : UDP 500 (IKE), UDP 4500 (NAT-T), Protocol 50 (ESP)
L2TP/IPsec : UDP 1701 (L2TP) + IPsec ports
OpenVPN : UDP/TCP 1194 (configurable)
PPTP : TCP 1723, GRE Protocol 47

Problem :
Firewall blocks UDP 500 → IPsec IKE handshake fails
→ VPN cannot establish tunnel

2. NAT traversal (NAT-T) problem

IPsec ESP (Protocol 50) : Doesn't work through NAT (no ports)

Solution : NAT-T (NAT Traversal)
→ Encapsulates ESP in UDP 4500
→ Works through NAT ✅

Problem : NAT-T disabled/blocked
→ VPN fails behind NAT router

3. Authentication failure

  • Wrong credentials : Username/password incorrect
  • Expired certificate : Client/server cert expired
  • Pre-shared key mismatch : IPsec PSK different both sides
  • RADIUS server down : Authentication server unavailable

4. Phase 1/Phase 2 mismatch (IPsec)

IPsec Phase 1 (ISAKMP) :
Client : Encryption AES-256, Hash SHA-256, DH Group 14
Server : Encryption 3DES, Hash SHA-1, DH Group 14

→ Encryption/Hash mismatch
→ Phase 1 fails → VPN doesn't establish

Must match :
Encryption algorithm, Hash, DH group, Authentication method

// Diagnostic VPN failure

Step 1 : Check basic connectivity
C:\> ping vpn-server.company.com
→ Reply ? = Server reachable ✅
→ Timeout ? = Network/DNS problem

Step 2 : Verify ports open
C:\> telnet vpn-server.company.com 500
→ Connection ? = Port 500 open
→ Timeout ? = Firewall blocking

Step 3 : Check VPN client logs
Windows Event Viewer → Applications and Services Logs → Microsoft → Windows → VPN Client
→ "Authentication failed" = Credentials/cert
→ "Phase 1 negotiation failed" = Crypto mismatch
→ "No response from server" = Firewall/network

// Résolution VPN failure

  • Firewall : Allow UDP 500, 4500, Protocol 50 (IPsec)
  • NAT-T : Enable NAT Traversal both sides
  • Authentication : Verify credentials, renew certificates
  • Phase mismatch : Match encryption, hash, DH group both ends
  • Update VPN client : Latest version (bug fixes, compatibility)
Maximum Transmission Unit

MTU / Fragmentation IPsec — MTU problems VPN

MTU (Maximum Transmission Unit) = Largest packet size link supports. IPsec adds overhead → exceeds MTU → fragmentation → slow/broken VPN.

// MTU standard values

Link TypeMTU (bytes)
Ethernet1500
PPPoE (DSL)1492
VPN tunnel (IPsec)1400-1440 (reduced overhead)
GRE tunnel1476

// IPsec overhead

Original packet :
Data : 1400 bytes
IP header : 20 bytes
TCP header : 20 bytes
Total : 1440 bytes

IPsec ESP adds :
ESP header : 8 bytes
ESP trailer : 2+ bytes (padding)
ESP auth : 12-16 bytes (HMAC)
New IP header : 20 bytes (tunnel mode)
Total overhead : ~60 bytes

Result :
1440 + 60 = 1500 bytes
→ Exactly MTU (edge case)
→ If original 1460 bytes → 1520 total → Exceeds MTU 1500 → Fragmentation

// Problème fragmentation

Packet 1520 bytes > MTU 1500

Router fragments :
Fragment 1 : 1500 bytes
Fragment 2 : 20 bytes

Problems :
→ IPsec encrypts entire packet (fragments can't be reassembled until decrypted)
→ Fragment loss = entire packet lost (both fragments needed)
→ Firewalls drop fragments (security policy)
→ VPN slow or broken

// Symptômes MTU problem

  • VPN connects BUT slow : Tunnel establishes, traffic slow/intermittent
  • Small packets work, large fail : Ping small (64 bytes) works, ping large (1400 bytes) fails
  • Web browsing broken : Can ping, can't load websites (large HTTP responses)
  • Email attachments fail : Small emails OK, large attachments timeout

// Diagnostic MTU problem

Test MTU (ping with Don't Fragment) :

Windows :
C:\> ping vpn-server.com -f -l 1472
→ Reply = MTU OK (1472 + 28 headers = 1500)

C:\> ping vpn-server.com -f -l 1500
→ "Packet needs to be fragmented but DF set" = MTU too large

Linux :
$ ping -M do -s 1472 vpn-server.com

→ Test progressively smaller sizes find maximum

// Résolution MTU problem

Solution 1 : Reduce MTU client/server

Windows :
> netsh interface ipv4 set subinterface "VPN Connection" mtu=1400 store=persistent

Linux :
$ sudo ip link set dev tun0 mtu 1400

VPN server config :
→ Set tunnel MTU 1400 (accommodates IPsec overhead)

Solution 2 : MSS clamping (TCP)

MSS (Maximum Segment Size) = TCP payload size
Normal MSS : 1460 (MTU 1500 - 40 headers)

Router/firewall :
→ Clamp MSS to 1360 (accounts IPsec overhead)
→ TCP sessions auto-adjust smaller packets
→ Prevents fragmentation ✅

Solution 3 : Path MTU Discovery (PMTUD)

Client sends packets DF (Don't Fragment) flag
Router : "Too big" ICMP message (MTU exceeded)
Client reduces packet size
→ Auto-discovers optimal MTU

Problem : Firewalls block ICMP → PMTUD fails → Black hole
→ Must manually set MTU if PMTUD broken
⚠️
MTU black hole = mysterious VPN failure VPN connects (small control packets), traffic fails (large data packets).
Symptom : Ping works, web/email doesn't.
Always test MTU VPN troubleshooting (ping -f -l test).
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : Packet loss always means bad cable
FAUX. Packet loss = multiple possible causes.

Cable : Possible (interface errors, CRC).
Also : Congestion (buffers full), firewall (ACL blocking), routing loops (TTL).

Diagnostic : Check interface errors. If none → Not cable (congestion/firewall).
Examen : Loss + no interface errors = congestion/routing (not physical).
❌ Piège #2 : High latency = slow bandwidth
CONFUSION. Latency ≠ Bandwidth.

Latency : Delay (ms) — how long packet takes reach destination.
Bandwidth : Capacity (Mbps) — how much data per second.

Can have : High bandwidth (1 Gbps) + high latency (satellite 500 ms).
OR : Low bandwidth (10 Mbps) + low latency (fiber 5 ms).
Examen : "Slow downloads" = bandwidth. "Delay/lag" = latency.
❌ Piège #3 : VPN connects = MTU not a problem
FAUX. VPN connect ≠ MTU OK.

VPN handshake : Small packets (IKE, authentication) — Works fine.
Data transfer : Large packets → MTU exceeded → Fragmentation → Broken.

Symptom : VPN connected, can't browse web (large HTTP responses fragmented/dropped).
Examen : "VPN connects but slow/broken" = suspect MTU fragmentation.
Résumé

🔑 Ce que tu dois retenir absolument

→Perte paquets : >5% serious. Causes: Congestion (buffers full), interface errors (CRC), firewall, routing loops. Diagnostic: Ping loss %, traceroute locate hop. Fix: Upgrade bandwidth, fix cable, QoS
→Congestion : Traffic exceeds capacity (>80% utilization). Causes: Insufficient bandwidth, bulk transfers, DDoS. Symptoms: Slow peak hours, loss, high latency. Fix: Upgrade link, QoS prioritize, schedule backups off-hours
→Latence élevée : Delay >150 ms VoIP bad. Causes: Distance (physics), congestion (queuing), satellite (500+ ms), many hops. Diagnostic: Ping RTT, traceroute identify delay hop. Fix: Reduce congestion, optimize routing, CDN
→VPN fails : Causes: Firewall blocks UDP 500/4500, NAT-T disabled, authentication (creds/cert), Phase 1/2 mismatch (crypto). Diagnostic: Ping server, telnet ports, check logs. Fix: Allow ports, enable NAT-T, match crypto
→MTU fragmentation : IPsec overhead (~60 bytes) exceeds MTU 1500. Symptoms: VPN connects, large packets fail. Test: ping -f -l 1472. Fix: Reduce MTU 1400, MSS clamping 1360, PMTUD (if ICMP allowed)
→Tools : ping (loss/latency), traceroute (hop-by-hop), pathping (combines both), show interfaces (utilization/errors), SNMP/NetFlow (monitoring). VPN logs (Event Viewer Windows). Latency vs Bandwidth: Separate concepts
