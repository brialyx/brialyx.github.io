Public Key Infrastructure

PKI — Fondations infrastructure

🏛️
Analogie — passeports gouvernement PKI = système gouvernemental délivrant passeports. CA (Certificate Authority) = gouvernement émet passeports. Certificat = passeport prouve identité. Trust = monde entier accepte passeports gouvernement légitime.

PKI (Public Key Infrastructure) = Framework gérant création, distribution, stockage, révocation certificats numériques. Permet chiffrement asymétrique + authentification échelle Internet.

// Problème résolu par PKI

Question : Comment vérifier public key légitime (pas attaquant MitM) ?

Sans PKI (vulnerable) :
Browser → "Quelle public key google.com ?"
Attacker MitM → "Voici MY public key (fake)"
Browser → Chiffre données avec fake key → Attacker déchiffre

Avec PKI (secure) :
Browser → "Quelle public key google.com ?"
Server → "Voici certificat signé CA trusted (Let's Encrypt)"
Browser → Vérifie signature CA → Accepte public key légitime
Attacker → Pas certificat signé CA → Browser rejette

// Composants PKI

  • CA (Certificate Authority) : Entité émet/signe certificats (DigiCert, Let's Encrypt)
  • RA (Registration Authority) : Vérifie identité demandeurs (optionnel, proxy CA)
  • Certificats : Documents numériques liant public key → identité (X.509 format)
  • CRL/OCSP : Mécanismes révocation certificats compromis
  • Certificate store : Stockage certificats trusted (OS, browser)
🔐
PKI = fondation trust Internet HTTPS, email signé (S/MIME), code signing, VPN certificates → TOUT repose PKI.
Sans PKI : Impossible vérifier identités échelle globale.
PKI = infrastructure invisible MAIS critique sécurité moderne.
Digital certificates

Certificats X.509 — Format standard

Certificat X.509 = Standard ITU-T définissant format certificats numériques. Utilisé SSL/TLS, S/MIME, code signing, 802.1X.

// Contenu certificat X.509

Certificate:
  Version: 3
  Serial Number: 04:d2:ac:7f:9e:...
  Subject: CN=www.google.com (identité titulaire)
  Issuer: CN=GTS CA 1C3 (CA émetteur)
  Public Key: RSA 2048 bits
    Modulus: 00:c4:7a:...
    Exponent: 65537
  Validity:
    Not Before: Jan 15 08:21:00 2024
    Not After: Apr 08 08:20:59 2024 (90 jours)
  Signature Algorithm: SHA256-RSA
  Signature: 5a:3f:9c:... (CA private key signé)
  Extensions:
    Subject Alternative Name: *.google.com, *.youtube.com

// Champs critiques

Subject (CN - Common Name)

  • Identité titulaire certificat (domain, organization, person)
  • Ex: CN=www.example.com

Issuer

  • CA ayant émis/signé certificat
  • Ex: CN=Let's Encrypt Authority X3

Public Key

  • Clé publique titulaire (RSA 2048/4096, ECC 256)
  • Usage : Chiffrement, vérification signatures

Validity Period

  • Durée validité (Not Before → Not After)
  • Moderne : 90 jours (Let's Encrypt), 1-2 ans max (commercial)
  • Certificat expiré = browser warning

Signature

  • CA hash certificat → signe avec CA private key
  • Verification : Hash certificat → déchiffre signature CA public key → compare

Extensions (v3)

  • SAN (Subject Alternative Name) : Multiple domains (*.example.com, example.com)
  • Key Usage : Restrictions usage clé (digitalSignature, keyEncipherment)
  • Extended Key Usage : Server Authentication, Client Authentication, Code Signing
🎯
Question fréquente examen "Champ certificat X.509 identifiant CA émetteur ?"
Réponse : Issuer

"Champ certificat contenant clé publique titulaire ?"
Réponse : Public Key
Certificate Authority

CA racine vs CA intermédiaire — Hiérarchie

Certificate Authority (CA) = Entité trusted émettant/signant certificats numériques.

// CA Racine (Root CA)

Définition : CA sommet hiérarchie, auto-signée (self-signed), trustée implicitement OS/browsers.

Caractéristiques Root CA :

  • Auto-signé : Issuer = Subject (signe propre certificat)
  • Trust anchor : Pré-installée OS (Windows, macOS, Linux), browsers (Chrome, Firefox)
  • Long-lived : Validité 20-30 ans (rarement renouvelée)
  • Offline storage : Private key gardée OFFLINE (HSM - Hardware Security Module)
  • Rarement utilisée direct : Signe CA intermédiaires (pas certificats end-entity)

Exemples Root CAs publiques :

  • DigiCert Global Root CA
  • Let's Encrypt Root CA (ISRG Root X1)
  • GlobalSign Root CA
  • Sectigo (Comodo) Root CA

// CA Intermédiaire (Intermediate CA)

Définition : CA milieu hiérarchie, signée par Root CA, émet certificats end-entity (serveurs, users).

Caractéristiques Intermediate CA :

  • Signée Root CA : Issuer = Root CA, Subject = Intermediate CA
  • Operational : Utilisée quotidiennement signer certificats
  • Shorter-lived : Validité 5-10 ans
  • Online : Private key accessible (automated signing)
  • Compromission limitée : Si compromised → révoquer intermediate, Root intact

// Hiérarchie PKI (3 niveaux)

Root CA (ISRG Root X1)
  │ Validité: 2035 (20 ans)
  │ Storage: Offline HSM
  │ Signs: Intermediate CAs uniquement
  └─> Intermediate CA (Let's Encrypt Authority X3)
        │ Validité: 2025 (5 ans)
        │ Storage: Online (automated)
        │ Signs: End-entity certificates
        └─> End-Entity Certificate (www.example.com)
              Validité: 90 jours
              Usage: SSL/TLS server

→ Chain of Trust : Root → Intermediate → End-Entity

// Pourquoi CA intermédiaire ?

  • Sécurité Root CA : Root offline → si compromised = catastrophe (tout PKI invalidé)
  • Compartmentalization : Intermediate compromised → révoquer, Root intact
  • Scaling : Multiples intermediates (géographie, business units)
  • Operational flexibility : Renouveler intermediate sans toucher root
Trust model

Chaîne de confiance — Certificate Chain

Chaîne de confiance (Certificate Chain) = Séquence certificats Root CA → Intermediate CA(s) → End-Entity validant authenticité.

// Workflow validation chaîne

  1. Browser reçoit certificat server (www.example.com)
  2. Browser vérifie signature certificat :
    - Hash certificat
    - Déchiffre signature avec Issuer public key (Intermediate CA)
    - Compare hash → Match = signature valide
  3. Browser vérifie certificat Intermediate CA :
    - Signature Intermediate signée par Root CA ?
    - Hash + déchiffre signature Root public key → Valid
  4. Browser vérifie Root CA :
    - Root CA présente trust store OS/browser ?
    - Yes → Chaîne trusted, connexion proceed
    - No → Warning "Certificate not trusted"
Certificate chain example :

[0] End-Entity: www.example.com
    Issuer: Let's Encrypt Authority X3
    Signature: [signed by X3 private key]

[1] Intermediate: Let's Encrypt Authority X3
    Issuer: ISRG Root X1
    Signature: [signed by Root private key]

[2] Root: ISRG Root X1
    Issuer: ISRG Root X1 (self-signed)
    Trusted: Yes (pre-installed OS)

→ Browser validates [0] → [1] → [2] → Trust established

// Certificate chain envoi

Server responsabilité : Envoyer certificate chain COMPLET (end-entity + intermediates).

Problème configuration courante : Server envoie end-entity certificate uniquement (pas intermediates) → Browser error "Incomplete chain".

🔗
Incomplete chain = erreur SSL fréquente Server DOIT envoyer : [End-entity] + [Intermediate(s)]
Root CA PAS envoyée (déjà trust store client)

Test : openssl s_client -connect example.com:443 -showcerts
→ Vérifier 2-3 certificates retournés (pas 1 seul)
Certificate request

CSR — Certificate Signing Request

CSR (Certificate Signing Request) = Fichier demande signature certificat envoyé CA. Contient public key + infos identité.

// Workflow obtention certificat

  1. Server génère paire clés : Private key + Public key (RSA/ECC)
  2. Server créé CSR :
    - Inclut : Public key, domain name (CN), organization, country
    - Signe CSR avec private key (prouve possession clé)
  3. Admin soumet CSR → CA (Let's Encrypt, DigiCert)
  4. CA valide demande :
    - Domain ownership (HTTP challenge, DNS TXT record)
    - Organization validation (business documents, phone verification)
  5. CA signe CSR → émet certificat
  6. Server installe certificat (+ private key existante)
Génération CSR (OpenSSL) :

$ openssl req -new -newkey rsa:2048 -nodes \
  -keyout server.key -out server.csr

→ Prompts :
Country: US
State: California
Organization: Example Inc
Common Name: www.example.com

→ Output :
server.key (private key - KEEP SECRET)
server.csr (send to CA)

// Contenu CSR

  • Subject : CN=www.example.com, O=Example Inc, C=US
  • Public Key : RSA 2048 bits modulus + exponent
  • Signature : CSR signé private key requester (proof of possession)

⚠️ IMPORTANT : CSR contient public key uniquement. Private key JAMAIS envoyée CA (reste serveur).

Certificate revocation

CRL vs OCSP — Révocation certificats

Révocation = Invalider certificat AVANT expiration (private key compromised, organization change, erreur émission).

// CRL (Certificate Revocation List)

Définition : Liste signée CA contenant serial numbers certificats révoqués.

Workflow CRL :

  1. CA publie CRL (URL dans certificat extension)
  2. Client télécharge CRL périodiquement (ex: daily)
  3. Client vérifie : Serial number certificat présent CRL ?
    - Yes → Certificat révoqué (reject)
    - No → Valid (proceed)

Problèmes CRL :

  • ❌ Taille : CRL grossit (millions certificats = MB)
  • ❌ Latency : Updates périodiques (certificat révoqué aujourd'hui, client update demain = window vulnerability)
  • ❌ Bandwidth : Télécharger CRL complet chaque check (inefficient)

// OCSP (Online Certificate Status Protocol)

Définition : Protocole real-time vérification statut certificat individuel (RFC 6960).

Workflow OCSP :

  1. Client query OCSP responder : "Serial 0x123abc révoqué ?"
  2. OCSP responder répond signed response :
    - Good : Certificat valid
    - Revoked : Certificat révoqué (date, raison)
    - Unknown : OCSP ne connaît pas certificat
  3. Client vérifie signature response (OCSP responder certificate)

Avantages OCSP :

  • ✅ Real-time : Statut current (pas stale CRL)
  • ✅ Efficient : Query 1 certificat (pas télécharger CRL complet)
  • ✅ Scalable : Bandwidth minimal

Problème OCSP :

  • ⚠️ Privacy : OCSP responder voit quels sites user visite (query leak)
  • ⚠️ Availability : OCSP responder down = connexions bloquées

// OCSP Stapling

Solution : Server query OCSP, cache response, envoie (staple) clients avec certificat.

Avantages stapling :

  • ✅ Privacy (client pas contact OCSP directly)
  • ✅ Performance (server cache, clients pas query chaque fois)

⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : CA racine signe certificats end-entity directement
FAUX (best practice). Root CA signe CA intermédiaires (pas end-entity).

Root CA offline → signe intermediates uniquement
Intermediate CA online → signe end-entity certificates

Pourquoi : Sécurité root (compromission intermediate ≠ catastrophe)
❌ Piège #2 : CSR contient private key
FAUX. CSR contient PUBLIC key uniquement.

CSR = Certificate Signing Request (public key + identity info)
Private key reste serveur (JAMAIS envoyée CA)

CA signe CSR → retourne certificat (public key + signature)
❌ Piège #3 : OCSP remplace complètement CRL
NUANCE. OCSP moderne standard MAIS CRL toujours utilisé fallback.

Browsers : Essaient OCSP first, fallback CRL si OCSP unavailable
Certificats : Contiennent souvent BOTH CRL + OCSP URLs

OCSP preferred, CRL legacy backup
Résumé

🔑 Ce que tu dois retenir absolument

→PKI : Infrastructure gérant certificats numériques. Résout problème vérification public keys (MitM). Composants : CA, certificats X.509, CRL/OCSP, trust stores
→Certificats X.509 : Subject (titulaire), Issuer (CA), Public Key, Validity, Signature (CA private key), Extensions (SAN, Key Usage). Standard SSL/TLS
→Root CA : Self-signed, trust anchor (OS pre-installed), offline, 20-30 ans. Intermediate CA : Signée Root, operational, online, 5-10 ans. Hiérarchie 3 niveaux
→Chain of Trust : Root → Intermediate(s) → End-Entity. Browser validate chaque maillon (hash + decrypt signature). Server envoie end-entity + intermediates (pas root)
→CSR : Certificate Signing Request (public key + identity). Server génère, soumet CA, reçoit certificat signé. Private key JAMAIS envoyée
→CRL : Liste certificats révoqués (large, stale, legacy). OCSP : Real-time query (efficient, privacy issue). OCSP Stapling = server cache response
