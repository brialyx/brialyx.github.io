Répartition charge

Load Balancer — haute disponibilité et scalabilité

⚖️
Analogie — caisses supermarché Load Balancer = manager qui dirige clients vers caisses disponibles. Si caisse 1 bondée → envoie vers caisse 2. Si caisse 3 en panne → plus de clients envoyés là. Distribution équitable = tout le monde avance vite.

Load Balancer = équipement répartissant trafic entre plusieurs serveurs identiques.

// Objectifs

  • Haute disponibilité : Si serveur tombe → trafic redirigé vers serveurs sains
  • Scalabilité : Ajouter serveurs = plus de capacité (scaling horizontal)
  • Performances : Répartir charge uniformément (pas de serveur surchargé)
  • Health checks : Surveiller serveurs, retirer ceux en panne

// Algorithmes de répartition

1. Round-Robin

  • Distribution tour à tour : Serveur 1 → 2 → 3 → 1 → 2 → 3...
  • ✅ Simple, équitable
  • ⚠️ Ignore charge réelle serveurs (certains peuvent être plus lents)
  • Usage : Serveurs identiques (même puissance)

2. Least-Connection

  • Envoie vers serveur avec moins de connexions actives
  • ✅ Adaptatif (charge réelle)
  • ⚠️ Plus complexe
  • Usage : Sessions longues durées (websockets, streaming)

3. IP Hash / Source IP Affinity

  • Hash IP client → toujours même serveur
  • ✅ Persistence automatique (client → toujours serveur X)
  • ⚠️ Distribution inégale si peu de clients
  • Usage : Applications avec sessions (paniers e-commerce)

4. Weighted Round-Robin

  • Comme Round-Robin MAIS serveurs puissants reçoivent plus de trafic
  • Exemple : Serveur 1 (2x puissance) = 2 requêtes, Serveur 2 (1x) = 1 requête
  • Usage : Serveurs hétérogènes (hardware différent)

// Comparaison algorithmes

AlgorithmeDistributionPersistenceUsage
Round-RobinTour à tour❌ NonServeurs identiques, stateless
Least-ConnectionMoins chargé❌ NonSessions longues
IP HashHash IP client✅ OuiApplications stateful
WeightedPoids serveurs❌ NonServeurs hétérogènes
🎯
Question type examen "Application e-commerce nécessite que client reste sur même serveur pour conserver panier. Quel algorithme ?"

Réponse : IP Hash (Source IP Affinity) OU Session Persistence (sticky sessions)
Cohérence sessions

Session Persistence (Sticky Sessions)

Session Persistence = garantir que client se connecte TOUJOURS au même serveur backend pendant sa session.

// Problème sans persistence

User ajoute article panier → Serveur 1 (panier stocké localement)
User clique paiement → Load Balancer envoie vers Serveur 2
→ Serveur 2 ne connaît PAS le panier !
→ Panier perdu 🛒❌

// Solutions

1. Sticky Sessions (Cookie-based)

  • Load Balancer insère cookie identifiant serveur
  • Requêtes suivantes avec ce cookie → redirigées vers même serveur
  • ✅ Simple
  • ⚠️ Si serveur tombe → sessions perdues

2. Session Replication

  • Serveurs synchronisent données sessions entre eux
  • ✅ Résilience (si serveur tombe → autre serveur a session)
  • ⚠️ Overhead réseau/mémoire

3. Centralized Session Store

  • Sessions stockées dans base externe (Redis, Memcached)
  • ✅ Résilience maximale, scalabilité
  • ⚠️ Latence légère (requête externe)
  • Best practice moderne
Proxy sortant

Forward Proxy — intermédiaire clients → Internet

Forward Proxy = proxy sortant. Clients → Proxy → Internet.

// Fonctions

  • Filtrage contenu : Bloquer sites interdits (réseaux sociaux, streaming)
  • Cache web : Stocker pages fréquentes localement (accélération + économie bande passante)
  • Anonymisation : Masquer IP client (serveur web voit IP proxy)
  • Contrôle accès : Authentification utilisateurs
  • Logs : Audit navigation employés

// Architecture

[Client PC] → [Forward Proxy] → [Internet]
↓ ↓ ↓
"Je veux Cache/Filtre Serveur web
site.com" règles (voit IP proxy)

Usage : Entreprises (contrôle employés), écoles, bibliothèques.

🏢
Forward Proxy typique entreprise Employé navigue web → passe par proxy Squid/Blue Coat
→ Proxy bloque YouTube, Facebook
→ Cache pages corporate fréquentes
→ Logs toute navigation (audit)
Proxy entrant

Reverse Proxy — intermédiaire Internet → serveurs

Reverse Proxy = proxy entrant. Internet → Proxy → Serveurs internes.

// Fonctions

  • Load balancing : Répartir requêtes entre serveurs backend
  • SSL termination : Déchiffrer HTTPS côté proxy (serveurs reçoivent HTTP)
  • Cache statique : Images, CSS, JS en cache (accélération)
  • Protection : Masquer IPs serveurs, WAF (Web Application Firewall)
  • Compression : Gzip responses avant envoi client

// Architecture

[Internet] → [Reverse Proxy] → [Serveur Web 1]
↓ [Serveur Web 2]
↓ [Serveur Web 3]
Load balance
SSL decrypt
Cache

Usage : Sites web haute disponibilité, datacenters, CDN.

Exemples : Nginx, HAProxy, Apache mod_proxy, Cloudflare.

🚀
Reverse Proxy = standard moderne Presque TOUS les sites web haute charge utilisent reverse proxy.
Exemple Netflix, Amazon, Google : Reverse proxy devant serveurs → load balancing + cache + SSL termination.

Mode transparent

Transparent Proxy

Transparent Proxy = proxy interceptant trafic SANS configuration client.

// Fonctionnement

  • Routeur/Firewall redirige automatiquement port 80/443 vers proxy
  • Client ne sait PAS qu'il passe par proxy (transparent)
  • Pas de config navigateur nécessaire

// vs Explicit Proxy

CritèreExplicit ProxyTransparent Proxy
Config client✅ Requise (navigateur)❌ Aucune
Client aware✅ Sait qu'il passe par proxy❌ Ne sait pas
Bypass⚠️ Possible (désactiver proxy)✅ Difficile
UsageEntreprises (contrôle)ISPs, hotspots Wi-Fi
⚠️
Transparent Proxy = moins secure pour HTTPS HTTPS chiffré bout-en-bout → Transparent proxy ne peut PAS inspecter contenu (sauf SSL interception avec certificat custom, complexe et controversé).
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : Round-Robin = toujours équitable
FAUX. Round-Robin ignore charge réelle.

Serveur lent reçoit autant de requêtes que serveur rapide → déséquilibre.
Solution : Least-Connection ou Weighted
❌ Piège #2 : Forward Proxy = Load Balancer
FAUX.
Forward Proxy : Clients → Internet (filtrage, cache)
Load Balancer : Répartition entre serveurs backend
Reverse Proxy : Peut faire load balancing MAIS rôle différent de Forward
❌ Piège #3 : Transparent Proxy = invisible totalement
PAS TOUJOURS. Détectable par :
- Latence accrue
- Logs proxy visibles (X-Forwarded-For header)
- Tests traceroute
"Transparent" = pas de config client, pas forcément indétectable
Résumé

🔑 Ce que tu dois retenir absolument

→Load Balancer : Round-robin (tour à tour), Least-connection (moins chargé), IP Hash (persistence), Weighted (serveurs hétérogènes)
→Session Persistence : Sticky sessions (cookies), Replication, Centralized store (Redis - best practice)
→Forward Proxy : Clients → Internet. Filtrage, cache, anonymisation (entreprises)
→Reverse Proxy : Internet → Serveurs. Load balancing, SSL termination, protection (datacenters)
→Transparent Proxy : Pas config client, interception automatique trafic
