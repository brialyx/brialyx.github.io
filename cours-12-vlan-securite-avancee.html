VLAN attacks

VLAN Hopping — Bypass segmentation

RAPPEL Cours S04 : VLAN Hopping détaillé. Ici = focus défenses examen.

VLAN Hopping = Attaquant accède VLANs interdits (bypass segmentation Layer 2).

// Méthode 1 : Switch Spoofing

Principe : Attaquant simule switch → établit trunk 802.1Q → reçoit trafic TOUS VLANs.

Attaque :
1. Attaquant port access VLAN 10
2. Envoie DTP (Dynamic Trunking Protocol) frames
3. Switch croit attaquant = switch → établit trunk
4. Attaquant reçoit frames ALL VLANs tagged 802.1Q
5. Accède VLAN 20 Finance, VLAN 30 Servers

✅ Mitigation :
switchport mode access (force access, disable DTP)
switchport nonegotiate (disable DTP completely)

// Méthode 2 : Double Tagging

Principe : Exploiter 802.1Q native VLAN double-tagging frame saute VLANs.

Attaque :
1. Attaquant VLAN 10 (native VLAN trunk)
2. Craft frame double 802.1Q tags :
   [Outer tag: VLAN 10] [Inner tag: VLAN 20] [Payload]
3. Switch 1 strips outer tag VLAN 10 (native) → forward trunk
4. Switch 2 voit inner tag VLAN 20 → deliver VLAN 20
5. Attaquant injecte trafic VLAN 20 Finance

✅ Mitigation :
- Native VLAN unused (VLAN 999 inutilisé)
- vlan dot1q tag native (force tag native VLAN)
🎯
Question fréquente examen "Défense VLAN Hopping switch spoofing ?"
Réponse : switchport mode access + switchport nonegotiate

"Défense VLAN Hopping double tagging ?"
Réponse : Native VLAN unused + vlan dot1q tag native
DHCP security

DHCP Snooping — Rogue DHCP protection

DHCP Snooping = Feature switch filtrant DHCP messages → empêche rogue DHCP servers + attaques DHCP-based.

// Problème : Rogue DHCP Server

Scénario attaque : Attaquant installe DHCP server malveillant → clients reçoivent config falsifiée.

Attaque Rogue DHCP :

Client broadcasts : DHCP Discover

Legitimate DHCP : "IP 192.168.1.50, Gateway 192.168.1.1"
Rogue DHCP : "IP 192.168.1.51, Gateway 192.168.1.66 (attacker)"

→ Client accepte Rogue DHCP (first response wins)
→ Gateway = attacker IP
→ MitM complet (tout trafic passe attacker)

// DHCP Snooping fonctionnement

Principe : Switch classifie ports trusted (DHCP server légitime) vs untrusted (clients, potential rogue).

Configuration :

ip dhcp snooping
ip dhcp snooping vlan 10

interface GigabitEthernet0/1
  description Uplink to Legitimate DHCP Server
  ip dhcp snooping trust

interface GigabitEthernet0/2-24
  description Client Ports
  (untrusted by default)

// Règles DHCP Snooping

  • Trusted ports : DHCP Offer/Ack autorisés (server légitimes)
  • Untrusted ports : DHCP Offer/Ack BLOQUÉS (prevent rogue servers)
  • DHCP Discover/Request : Permis tous ports (clients légitimes)

// DHCP Snooping Binding Table

Database : Switch stocke mappings IP ↔ MAC ↔ Port ↔ VLAN (learned DHCP transactions).

show ip dhcp snooping binding :

MAC Address         IP Address      Lease   VLAN  Interface
00:11:22:33:44:55  192.168.1.50  86400  10    Gi0/2
AA:BB:CC:DD:EE:FF  192.168.1.51  86400  10    Gi0/3

→ Utilisé par DAI (Dynamic ARP Inspection)
→ Utilisé par IP Source Guard

// Bénéfices DHCP Snooping

  • ✅ Bloque rogue DHCP servers (MitM prevention)
  • ✅ Database IP-MAC bindings (foundation DAI, IP Source Guard)
  • ✅ Rate limiting : Limite DHCP packets/sec (DoS protection)
⚠️
DHCP Snooping = prerequisite DAI/IP Source Guard Sans DHCP Snooping = pas binding table.
Sans binding table = DAI/IP Source Guard pas fonctionnels.
Toujours enable DHCP Snooping AVANT DAI.
MAC address control

Port Security — Limiter MACs par port

Port Security = Restreindre nombre/identité MAC addresses autorisées port switch → empêche MAC flooding, unauthorized devices.

// Principe Port Security

Problème résolu : Attaquant branche hub/switch → multiple devices 1 port. MAC flooding attack. Unauthorized devices.

// Configuration Port Security

interface GigabitEthernet0/5
  switchport mode access
  switchport access vlan 10
  switchport port-security
  switchport port-security maximum 2
  switchport port-security violation restrict
  switchport port-security mac-address sticky

// Paramètres Port Security

Maximum MACs

  • switchport port-security maximum X
  • Default : 1 MAC (single device)
  • Exemple : 2 MACs (PC + IP phone)

MAC Address Learning

  • Static : switchport port-security mac-address aa:bb:cc:dd:ee:ff (manual)
  • Dynamic : Learn automatiquement (oublié reboot switch)
  • Sticky : Learn automatiquement + sauvegardé config (persiste reboot)

Violation Actions

ModeActionTrafficLogPort state
Protect Drop frames violation Légitime MACs = OK ❌ Pas log ✅ UP
Restrict Drop + log Légitime MACs = OK ✅ SNMP trap + syslog ✅ UP
Shutdown (default) Disable port (err-disabled) ❌ TOUT bloqué ✅ Log ❌ err-disabled (manual recovery)

// Exemple violation

Config : Maximum 1 MAC, violation shutdown

1. PC-A (aa:bb:cc:dd:ee:ff) connects → Learned, allowed
2. Attacker unplugs PC-A → plugs Laptop (11:22:33:44:55:66)
3. Violation detected !
   → Port err-disabled
   → Syslog : "Port-security violation Gi0/5"

Recovery :
shutdown
no shutdown
(manual intervention required)

// Usages Port Security

  • MAC flooding defense : Limite MACs = empêche CAM table overflow
  • Unauthorized devices : Rogue APs, laptops visiteurs
  • MAC spoofing : Attaquant change MAC = violation
ARP protection

DAI — Dynamic ARP Inspection

DAI (Dynamic ARP Inspection) = Validate ARP packets contre DHCP Snooping binding table → empêche ARP poisoning/spoofing.

// Rappel ARP Poisoning (Cours S03)

Problème : ARP = pas authentification. Attaquant envoie gratuitous ARP falsifié → MitM.

ARP Poisoning attack :

Attacker → Broadcast ARP Reply :
"192.168.1.1 gateway → MAC 11:22:33:44:55:66 (attacker)"

Clients update ARP table :
192.168.1.1 → 11:22:33:44:55:66

→ Trafic gateway envoyé attacker MAC → MitM

// DAI fonctionnement

Principe : Switch vérifie ARP packets (IP-MAC binding) contre DHCP Snooping database.

Configuration DAI :

ip arp inspection vlan 10

interface GigabitEthernet0/1
  description Uplink to Router
  ip arp inspection trust

interface GigabitEthernet0/2-24
  description Client Ports
  (untrusted by default)

// Règles DAI

  • Trusted ports : ARP packets passent sans validation (routers, switches)
  • Untrusted ports : ARP packets validés contre DHCP Snooping table

// Validation DAI

Workflow : ARP packet reçu port untrusted →

ARP Reply : "192.168.1.50 → aa:bb:cc:dd:ee:ff"

DAI checks DHCP Snooping binding table :
192.168.1.50 → aa:bb:cc:dd:ee:ff → Gi0/2 ?

✅ Match found :
→ ARP packet forwarded

❌ No match (spoofed) :
→ ARP packet DROPPED
→ Log : "ARP packet dropped, invalid binding"
→ Port err-disabled (optional)

// DAI + Static entries

Problème : Devices static IP (pas DHCP) = pas binding table.

Solution : ARP ACL manual bindings.

arp access-list STATIC_HOSTS
  permit ip host 192.168.1.1 mac host aa:bb:cc:dd:ee:ff
  permit ip host 192.168.1.10 mac host 11:22:33:44:55:66

ip arp inspection filter STATIC_HOSTS vlan 10

// Bénéfices DAI

  • ✅ Empêche ARP poisoning (MitM defense)
  • ✅ Empêche ARP spoofing (fake MAC addresses)
  • ✅ Complément DHCP Snooping (uses binding table)
🛡️
DAI = Layer 2 security essentielle ARP poisoning = attaque trivial (Ettercap, arpspoof).
Sans DAI = network vulnerable MitM.
DAI + DHCP Snooping = standard enterprise modern.
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : DHCP Snooping bloque clients DHCP
FAUX. DHCP Snooping bloque rogue DHCP servers (pas clients).

Clients DHCP Discover/Request = permis tous ports.
DHCP Offer/Ack untrusted ports = BLOQUÉS.

Trusted ports (legitimate server) = Offer/Ack autorisés
❌ Piège #2 : Port Security violation mode protect = log events
FAUX. Protect = PAS log.

Protect : Drop silencieux (no log)
Restrict : Drop + log (SNMP trap + syslog)
Shutdown : Disable port + log

Restrict = meilleur balance (traffic + visibility)
❌ Piège #3 : DAI fonctionne sans DHCP Snooping
FAUX. DAI NÉCESSITE DHCP Snooping binding table.

DAI valide ARP packets contre bindings IP-MAC.
Sans DHCP Snooping = pas bindings = DAI pas fonctionnel.

Ordre : 1. Enable DHCP Snooping, 2. Enable DAI
Résumé

🔑 Ce que tu dois retenir absolument

→VLAN Hopping : Switch spoofing (DTP frames) mitigation = switchport mode access + nonegotiate. Double tagging mitigation = native VLAN unused + vlan dot1q tag native
