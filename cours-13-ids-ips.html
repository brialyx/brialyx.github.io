Concept

IDS vs IPS — détection et prévention

🚨
Analogie — caméra vs videur IDS = caméra de sécurité : observe, enregistre, alerte sécurité si problème détecté. N'intervient PAS physiquement.
IPS = videur : observe ET arrête immédiatement intrus (bloque entrée).

// IDS (Intrusion Detection System)

Système PASSIF de détection d'intrusions. Écoute copie trafic, alerte si attaque détectée.

  • Mode : Out-of-band (mirroring/SPAN port)
  • Action : Alerte, logs → SIEM
  • Bloque : ❌ NON
  • Impact : Aucun (passif)
  • Risque : Attaque réussit avant intervention humaine

// IPS (Intrusion Prevention System)

Système ACTIF de prévention. Inline dans flux, bloque attaques temps réel.

  • Mode : In-band (inline entre firewall et réseau)
  • Action : Bloque + alerte + logs
  • Bloque : ✅ OUI (temps réel)
  • Impact : Latence légère
  • Risque : False positives bloquent trafic légitime

Méthodes détection

Signature-based vs Anomaly-based

// Signature-based (Pattern Matching)

Compare trafic avec base signatures connues (empreintes attaques).

  • ✅ Précis pour attaques connues
  • ✅ Peu de false positives
  • ❌ Ne détecte PAS attaques inconnues (zero-day)
  • ⚠️ Nécessite mises à jour régulières signatures

// Anomaly-based (Behavior Analysis)

Apprend comportement normal réseau, alerte si déviation.

  • ✅ Détecte attaques inconnues (zero-day)
  • ✅ Pas besoin signatures
  • ❌ Beaucoup de false positives (comportement légitime inhabituel)
  • ⚠️ Phase apprentissage nécessaire (baseline)

Erreurs

False Positives vs False Negatives

// False Positive (Fausse alerte)

IDS/IPS alerte sur trafic légitime (erreur : pense que c'est attaque).

Problème IPS : Bloque trafic business légitime → interruption service.

Cause : Signatures trop génériques, baseline anomaly incorrecte.

// False Negative (Attaque manquée)

IDS/IPS NE détecte PAS une vraie attaque.

Problème : Attaquant passe inaperçu → compromission.

Cause : Signatures obsolètes, evasion techniques, chiffrement.

⚠️
Tuning IDS/IPS = essentiel Trop sensible → false positives (fatigue alertes)
Pas assez sensible → false negatives (attaques manquées)
Équilibre = ajustements continus, analyse logs, whitelisting
Centralisation

SIEM — Security Information & Event Management

SIEM = plateforme centralisée collectant/corrélant/analysant logs sécurité (IDS/IPS, firewalls, serveurs, endpoints).

// Fonctions SIEM

  • Collecte logs : IDS/IPS, firewalls, AV, AD, serveurs
  • Corrélation : Détecte patterns multi-sources (attaque coordonnée)
  • Alerting : Notifications temps réel
  • Dashboards : Visualisation sécurité globale
  • Forensics : Investigation incidents
  • Compliance : Rapports réglementaires (PCI-DSS, HIPAA)

Exemples SIEM : Splunk, IBM QRadar, ArcSight, Microsoft Sentinel, LogRhythm.

📊
IDS/IPS + SIEM = SOC Security Operations Center (SOC) utilise SIEM comme hub central.
IDS/IPS envoient alertes → SIEM corrèle avec autres sources → Analyste SOC investigate → Réponse incident.
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : IDS bloque attaques
FAUX. IDS = alerte uniquement (passif).

IPS = bloque (actif inline)
❌ Piège #2 : Anomaly-based meilleur que signature
PAS TOUJOURS.
Anomaly-based = détecte zero-day MAIS beaucoup false positives.
Signature-based = précis MAIS manque attaques inconnues.
Mieux : Combiner les deux (hybrid)
❌ Piège #3 : IPS = zéro false positives
FAUX. IPS a false positives MAIS conséquences graves (bloque trafic légitime).

IDS false positives = juste alerte inutile (moins grave)
Résumé

🔑 Ce que tu dois retenir absolument

→IDS : Passif (out-of-band), alerte uniquement, pas de blocage. IPS : Actif (inline), bloque attaques temps réel
→Signature-based : Patterns connus, précis, manque zero-day. Anomaly-based : Déviations, détecte zero-day, false positives élevés
→False Positive : Alerte sur trafic légitime. False Negative : Attaque manquée
→SIEM : Collecte/corrèle logs sécurité, hub central SOC
→IPS inline = latence légère + risque false positives bloquants. IDS passif = zéro impact réseau
