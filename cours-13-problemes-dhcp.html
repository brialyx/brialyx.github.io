Address depletion

Pool DHCP épuisé — No addresses available

Pool DHCP épuisé = Toutes IP addresses pool allouées → nouveaux clients cannot get IP → APIPA 169.254.x.x.

// Principe DHCP pool (rappel Cours F28)

DHCP pool configuration :

Network : 192.168.1.0/24
Pool : 192.168.1.100 - 192.168.1.200 (101 addresses)
Lease time : 24 hours

First 101 clients : Get IP from pool ✅
102nd client : No IP available → APIPA 169.254.x.x ❌

// Symptômes pool épuisé

  • APIPA addresses : New clients get 169.254.x.x (DHCP failed)
  • Some users work : Existing leases OK, new connections fail
  • Intermittent : Works if other users disconnect (IP freed)
  • DHCP server logs : "No addresses available in pool" warnings
  • No Internet : APIPA = no gateway, no routing

// Causes pool épuisé

1. Pool trop petit (undersized)

Pool : 192.168.1.100 - 192.168.1.150 (51 addresses)
Devices network : 100+ PCs, phones, printers

→ Pool insufficient devices count
→ Design problem (capacity planning error)

2. Lease time trop long (stale leases)

Lease time : 30 days (default Windows Server)
Laptop users : Connect, disconnect daily

Problem :
→ Laptop gets IP, disconnects
→ Lease held 30 days (not released)
→ IP unavailable even device offline
→ Pool fills with inactive leases

3. Devices multiples interfaces (consume multiple IPs)

Laptop : WiFi + Ethernet + VPN = 3 IPs
Desktop : Ethernet + WiFi (unused) = 2 IPs

50 devices × 2-3 IPs each = 100-150 IPs consumed
→ Pool exhausted faster than expected

// Diagnostic pool épuisé

User PC :
C:\> ipconfig

IP Address : 169.254.45.123 (APIPA)
Subnet Mask : 255.255.0.0
Default Gateway : (blank)

→ APIPA = DHCP failed

DHCP server check :
Windows DHCP Server :
→ DHCP console → Address Pool
→ Addresses in use : 101 / 101 (100%)
→ Addresses free : 0

Linux (ISC DHCP) :
$ cat /var/lib/dhcp/dhcpd.leases | grep "lease 192.168.1" | wc -l
→ 101 (pool size 101 = 100% allocated)

// Résolution pool épuisé

Option 1 : Expand pool (add IPs)

Old pool : 192.168.1.100 - 192.168.1.200 (101 IPs)
New pool : 192.168.1.100 - 192.168.1.250 (151 IPs)

→ Add 50 IPs capacity
→ Requires IPs not used static assignments

Option 2 : Reduce lease time (free IPs faster)

Old lease : 30 days
New lease : 8 hours (workday duration)

→ Laptops disconnect end-of-day → Lease expires overnight
→ IP freed next morning (reused)
→ Reduces pool consumption

Option 3 : Release stale leases (manual cleanup)

DHCP server :
→ Delete inactive leases (devices offline >7 days)
→ Reconcile scopes (Windows DHCP)
→ Frees IPs immediately

Caution : Device returns, may get different IP (breaks static mappings apps)
📊
Best practice : Pool sizing Formula : Pool size ≥ (Active devices × 1.5) + Growth margin 20%.
Example : 100 devices → Pool 150-180 IPs.
Lease time : 8-12 hours office (daily turnover), 24 hours datacenter (stable).
Lease lifecycle

Bail (Lease) expiré — Lease expiration

Bail expiré = DHCP lease timeout → client perd IP → must renew OR get APIPA.

// DHCP lease process (rappel)

DORA process :
Discover : Client broadcasts "Need IP"
Offer : DHCP server offers IP + lease time
Request : Client requests offered IP
Ack : DHCP server acknowledges

Lease renewal :
T1 (50% lease) : Client tries renew (unicast DHCP server)
T2 (87.5% lease) : If T1 fails, broadcast renew (any DHCP server)
Lease expires : If T2 fails, release IP → APIPA 169.254.x.x

// Scénario bail expiré

Timeline lease 8 hours :

00:00 : Client gets IP 192.168.1.150 (lease 8h)
04:00 (T1 50%) : Client tries renew → DHCP server unreachable (fails)
07:00 (T2 87.5%) : Client broadcast renew → No response (fails)
08:00 : Lease expires
      → Client releases 192.168.1.150
      → Client self-assigns 169.254.x.x (APIPA)
      → Connectivity lost

// Causes lease expiration problems

  • DHCP server down during renewal : T1/T2 attempts fail
  • Network connectivity loss : Client can't reach DHCP server
  • Firewall blocking : DHCP renewal packets (UDP 67/68) dropped
  • Lease time too short : Frequent renewals, higher failure risk

// Symptômes lease expiration

  • Sudden connectivity loss : Was working, suddenly APIPA
  • Timing pattern : Fails exactly lease duration (8h, 24h)
  • Event logs : "DHCP renewal failed" warnings
  • Manual renewal works : ipconfig /renew gets new IP

// Résolution lease expiration

Immediate fix (user) :
C:\> ipconfig /release
C:\> ipconfig /renew

→ Forces new DHCP request
→ Gets IP if DHCP server available ✅

Long-term fix :
1. Ensure DHCP server high availability (redundant servers)
2. Increase lease time (reduce renewal frequency)
3. Monitor DHCP server uptime (alerting)
4. Check network connectivity (VLAN, routing)
🎯
Question fréquente examen "User had IP, now APIPA. Works after ipconfig /renew. MOST likely cause ?"
A) Pool exhausted
B) Lease expired, renewal failed
C) Rogue DHCP
D) Cable defect

Réponse : B) Lease expired, renewal failed
Had IP → lost → manual renew works = lease expiration problem.
Address collision

Adresse en conflit — DHCP IP conflict

Conflit IP DHCP = DHCP server assigns IP already in use → duplicate address → both devices connectivity problems.

// Cause conflit DHCP

Scenario commun : DHCP + Static overlap

DHCP pool : 192.168.1.100 - 192.168.1.200
Static assignment : Printer 192.168.1.150 (manual config)

Problem :
DHCP server doesn't know about static 192.168.1.150
DHCP assigns 192.168.1.150 → Laptop
→ Conflict (Printer vs Laptop same IP)

// DHCP conflict detection

DHCP server behavior (conflict detection enabled) :

1. DHCP server selects IP from pool
2. Ping IP before assigning (ICMP echo request)
3a. No reply → Assign IP (safe) ✅
3b. Reply received → IP in use → Mark conflict ❌
4. If conflict, try next IP from pool

If all IPs conflict ?
→ DHCP server cannot assign (client gets APIPA)

// Symptômes conflit DHCP

  • Windows pop-up : "IP address conflict detected"
  • Intermittent connectivity : Works sometimes, fails randomly
  • DHCP server logs : "Conflict detected for IP X.X.X.X"
  • Both devices affected : Static device + DHCP client

// Résolution conflit DHCP

Best practice : Exclude static IPs from DHCP pool

Network design :

Subnet : 192.168.1.0/24
Static range : 192.168.1.1 - 192.168.1.99 (routers, servers, printers)
DHCP pool : 192.168.1.100 - 192.168.1.250 (workstations)

DHCP exclusion range :
Windows DHCP Server :
→ Add Exclusion Range : 192.168.1.1 - 192.168.1.99

Linux ISC DHCP (dhcpd.conf) :
subnet 192.168.1.0 netmask 255.255.255.0 {
  range 192.168.1.100 192.168.1.250;
}

→ DHCP never assigns 1-99 (reserved static)

Enable conflict detection (DHCP server)

Windows DHCP Server :
→ Server properties → Advanced
→ Conflict detection attempts : 2 (ping 2 times before assign)

→ Prevents assigning IPs already in use ✅

Immediate fix (release conflict IP)

DHCP client :
C:\> ipconfig /release
C:\> ipconfig /renew

→ Gets different IP (no conflict) ✅
Unauthorized server

DHCP rogue — Rogue DHCP server

DHCP rogue = Unauthorized DHCP server network → clients get wrong IP/gateway/DNS → connectivity broken OR man-in-the-middle attack.

// Scénario DHCP rogue

Legitimate DHCP server :
IP pool : 192.168.1.100 - 192.168.1.200
Gateway : 192.168.1.1 (correct router)
DNS : 192.168.1.100 (correct DNS)

Rogue DHCP server :
User plugs personal router (home router, accidental)
IP pool : 192.168.1.2 - 192.168.1.50
Gateway : 192.168.1.2 (rogue router - wrong!)
DNS : 8.8.8.8 (bypasses internal DNS)

Client behavior :
PC boots → Sends DHCP Discover (broadcast)
Both servers reply (legitimate + rogue)
PC accepts first response (race condition)

If rogue wins :
→ PC gets IP from rogue
→ Gateway points rogue router (broken routing)
→ No Internet access

// Symptômes DHCP rogue

  • Random connectivity failures : Some PCs work, others don't (luck DHCP race)
  • Wrong gateway : ipconfig shows unexpected gateway IP
  • Wrong subnet : IPs from different range than expected
  • Intermittent : Depends which DHCP server replies first
  • Started suddenly : After user plugged device (router, AP)

// Diagnostic DHCP rogue

Affected PC :
C:\> ipconfig

IP Address : 192.168.1.25 (unexpected range)
Subnet Mask : 255.255.255.0
Default Gateway : 192.168.1.2 (not legitimate 192.168.1.1)

→ Wrong gateway = suspect rogue DHCP

Identify rogue DHCP server :
Method 1 : ARP table
C:\> arp -a | findstr 192.168.1.2
192.168.1.2       aa-bb-cc-dd-ee-ff

→ Look up MAC vendor (aa-bb-cc = TP-Link home router)

Method 2 : DHCP snooping logs (managed switches)
Switch# show ip dhcp snooping binding
→ Shows DHCP servers MAC addresses, ports

Method 3 : Wireshark packet capture
→ Filter : bootp
→ See DHCP Offer packets (multiple servers = rogue present)

// Résolution DHCP rogue

Immediate : Locate and disconnect rogue

1. Identify rogue MAC address
2. Find switch port (MAC address table)
3. Physically disconnect device
4. Release/renew affected clients :
   C:\> ipconfig /release
   C:\> ipconfig /renew
   → Get IP from legitimate DHCP ✅

Prevention : DHCP Snooping (Cours S12)

Managed switch configuration :

Switch(config)# ip dhcp snooping
Switch(config)# ip dhcp snooping vlan 10

Switch(config)# interface GigabitEthernet0/1
Switch(config-if)# ip dhcp snooping trust
→ Port Gi0/1 : DHCP server port (trusted)

Switch(config)# interface range Gi0/2 - 24
Switch(config-if-range)# ip dhcp snooping (untrusted default)
→ Ports Gi0/2-24 : User ports (untrusted)

→ Untrusted ports : DHCP server messages BLOCKED
→ Rogue DHCP prevented ✅
⚠️
Rogue DHCP = security threat + availability problem Accidental : User plugs home router → breaks network (downtime).
Malicious : Attacker DHCP server → Man-in-the-middle (traffic interception, DNS poisoning).
DHCP Snooping = defense layer (trusted ports only).
Multi-subnet DHCP

Relay Agent mal configuré — IP Helper misconfiguration

DHCP Relay Agent (IP Helper) = Forwards DHCP broadcasts across subnets → centralized DHCP server multiple VLANs.

// Problème : DHCP broadcasts don't cross routers

Topology :

VLAN 10 : 192.168.10.0/24 (PCs)
VLAN 20 : 192.168.20.0/24 (DHCP server 192.168.20.100)

Without relay :
PC VLAN 10 → DHCP Discover (broadcast 255.255.255.255)
Router → Drops broadcast (doesn't forward different subnet)
DHCP server VLAN 20 → Never receives request
→ PC gets APIPA 169.254.x.x

// Solution : IP Helper-Address (DHCP Relay)

Router configuration :

Router(config)# interface GigabitEthernet0/1
Router(config-if)# description VLAN 10 - User PCs
Router(config-if)# ip address 192.168.10.1 255.255.255.0
Router(config-if)# ip helper-address 192.168.20.100

Workflow with relay :
1. PC VLAN 10 → DHCP Discover (broadcast)
2. Router receives (local subnet)
3. Router forwards as unicast → 192.168.20.100 (DHCP server)
4. DHCP server → DHCP Offer (unicast back via router)
5. Router → Forwards to PC VLAN 10
6. PC gets IP ✅

// Problèmes relay agent

1. IP helper-address missing

interface GigabitEthernet0/1
  ip address 192.168.10.1 255.255.255.0
  ! No ip helper-address configured

→ Broadcasts not forwarded
→ DHCP fails → APIPA

2. IP helper-address wrong IP

DHCP server : 192.168.20.100
Configured : ip helper-address 192.168.20.101 (typo)

→ Forwarded to wrong IP (nonexistent)
→ No response → APIPA

3. IP helper-address unreachable

ip helper-address 10.0.0.100
No route to 10.0.0.0/24

→ Routing problem (DHCP server unreachable)
→ Packets dropped → APIPA

// Diagnostic relay agent

Symptom : PCs VLAN 10 get APIPA

Check relay configuration :
Router# show running-config interface GigabitEthernet0/1

interface GigabitEthernet0/1
 ip address 192.168.10.1 255.255.255.0
 ip helper-address 192.168.20.100 ✅

Verify DHCP server reachable :
Router# ping 192.168.20.100
→ Success ✅ (DHCP server reachable)

Test DHCP from router :
Router# debug ip dhcp server packet
→ Shows DHCP relay forwarding (if working)

// Résolution relay agent

Add/fix ip helper-address :

Router(config)# interface GigabitEthernet0/1
Router(config-if)# ip helper-address 192.168.20.100

Verify :
Router# show running-config interface Gi0/1 | include helper
 ip helper-address 192.168.20.100 ✅

Test client :
C:\> ipconfig /release
C:\> ipconfig /renew
→ Gets IP from DHCP server ✅
🌐
Best practice : Centralized DHCP + relay Instead DHCP server per VLAN (costly, complex management).
Use : 1-2 DHCP servers (redundant) + ip helper-address all VLANs.
Simplified : Single pool management, consistent config, easier troubleshooting.
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : APIPA = DHCP server down always
FAUX. APIPA = DHCP failed, multiple causes.

DHCP server down : One cause (among many).
Other causes : Pool exhausted, no relay agent, wrong VLAN, network unreachable, firewall blocking.

APIPA = symptom (DHCP didn't work). Diagnose root cause.
Examen : APIPA → investigate DHCP infrastructure (not assume server down).
❌ Piège #2 : Rogue DHCP = always malicious attack
NUANCE. Rogue DHCP = often accidental, not intentional attack.

Common scenario : User plugs home router/AP (forgot disable DHCP).
Malicious : Possible (attacker DHCP for MitM), but rare.

Impact same (network broken) whether accidental or malicious.
Examen : Rogue DHCP = unauthorized DHCP (not necessarily attack).
❌ Piège #3 : IP helper-address = DHCP server IP only
FAUX. ip helper-address forwards multiple services, not just DHCP.

Forwarded (default Cisco) : DHCP (67/68), DNS (53), TFTP (69), NetBIOS (137/138), TACACS (49).
DHCP most common usage, but not exclusive.

Can configure : ip helper-address forwards any UDP service (configurable).
Examen : IP helper = broadcast relay mechanism (primarily DHCP).
Résumé

🔑 Ce que tu dois retenir absolument

→Pool épuisé : All IPs allocated → APIPA. Causes: Undersized pool, long lease time, multiple interfaces. Fix: Expand pool, reduce lease (8-12h), release stale leases. Formula: Pool ≥ devices × 1.5
→Bail expiré : Lease timeout → APIPA. T1 (50%) renew fails, T2 (87.5%) fails → expires. Symptom: Sudden loss after lease duration. Fix: ipconfig /renew, ensure DHCP availability, longer lease
→Conflit IP : DHCP assigns IP already used (static). Prevention: Exclude static range from pool, enable conflict detection (ping before assign). Fix: ipconfig /release/renew, proper IP planning
→DHCP rogue : Unauthorized DHCP server (home router accidental). Symptom: Wrong gateway, random failures. Diagnostic: Check gateway IP, MAC vendor. Prevention: DHCP Snooping (trusted ports only)
→Relay agent : ip helper-address forwards DHCP across subnets. Missing/wrong → APIPA different VLANs. Config: ip helper-address [DHCP-server-IP] on VLAN interface. Centralized DHCP + relay = best practice
→Commands : ipconfig /release, ipconfig /renew, show ip dhcp binding (server leases), show running-config interface (relay), show ip dhcp snooping binding (rogue detection). APIPA = 169.254.x.x signature DHCP failure
