VLAN configuration

VLAN mal configuré — Isolation involontaire

VLAN misconfiguration = Port assigné wrong VLAN → device isolé, pas communication expected.

// Scénario typique

Network design :

VLAN 10 : Data (workstations)
VLAN 20 : Voice (IP phones)
VLAN 99 : Management (switch mgmt)

Problème :
PC-A port Gi0/5 : Configured VLAN 20 (Voice) — ERREUR
Server port Gi0/10 : Configured VLAN 10 (Data) — Correct

→ PC-A VLAN 20 ≠ Server VLAN 10
→ Pas communication (isolated Layer 2)

// Symptômes VLAN misconfigured

  • No connectivity : Ping fails, no access network resources
  • DHCP fails : DHCP server different VLAN → no IP address (APIPA 169.254.x.x)
  • Selective connectivity : Can reach some devices (same VLAN), not others
  • Link UP : LED green, physical OK, but Layer 2 isolation

// Diagnostic VLAN assignment

User report : "PC can't access file server"

Step 1 : Check VLAN assignment
Switch# show vlan brief

VLAN Name                 Ports
10   Data                Gi0/1, Gi0/2, Gi0/10
20   Voice              Gi0/5 (PC-A port — wrong!)
99   Management        Gi0/24

→ PC-A (Gi0/5) assigned VLAN 20
→ Server (Gi0/10) assigned VLAN 10
→ Mismatch detected

Step 2 : Fix assignment
Switch(config)# interface GigabitEthernet0/5
Switch(config-if)# switchport access vlan 10

→ PC-A now VLAN 10 (Data) — matches server ✅

// Vérification VLAN membership

Switch# show interfaces GigabitEthernet0/5 switchport

Administrative Mode: access
Operational Mode: access
Access Mode VLAN: 10 (Data)
Trunking Native Mode VLAN: 1 (default)

→ Verify correct VLAN assignment
🎯
Question fréquente examen "User can't ping server. Link UP. DHCP fails. MOST likely cause ?"
A) Cable defect
B) Wrong VLAN assignment
C) DNS failure
D) Routing problem

Réponse : B) Wrong VLAN assignment
Link UP + DHCP fail + no connectivity = Layer 2 isolation (VLAN mismatch).
Trunk misconfiguration

Native VLAN Mismatch — Trunk problems

Native VLAN = VLAN untagged frames 802.1Q trunk. Mismatch between switches → CDP/DTP warnings, connectivity issues.

// Principe Native VLAN

802.1Q Trunk :

Tagged frames : VLAN tag inserted (10, 20, 30...)
Untagged frames : Native VLAN (default VLAN 1)

Example :
Frame VLAN 10 → 802.1Q tag added → Send trunk
Frame Native VLAN (1) → No tag → Send trunk

Receiving switch :
Tagged frame → Extract VLAN ID, forward appropriate VLAN
Untagged frame → Assume Native VLAN

// Native VLAN Mismatch scenario

Switch A :
interface GigabitEthernet0/1
  switchport mode trunk
  switchport trunk native vlan 1 (default)

Switch B :
interface GigabitEthernet0/1
  switchport mode trunk
  switchport trunk native vlan 99 (misconfigured)

→ Native VLAN mismatch : 1 vs 99

// Conséquences mismatch

  • CDP/DTP warnings : Syslog messages "native VLAN mismatch detected"
  • Untagged traffic misplaced : Switch A sends untagged (VLAN 1), Switch B receives as VLAN 99
  • Connectivity issues : Native VLAN traffic wrong destination VLAN
  • Security risk : VLAN hopping possible (double tagging attack)

// Diagnostic Native VLAN Mismatch

Syslog message :
%CDP-4-NATIVE_VLAN_MISMATCH: Native VLAN mismatch discovered on Gi0/1

Verify trunk config :
Switch-A# show interfaces trunk

Port       Mode         Native VLAN
Gi0/1      on          1

Switch-B# show interfaces trunk

Port       Mode         Native VLAN
Gi0/1      on          99

→ Mismatch confirmed (1 vs 99)

Fix (standard : use VLAN 99 unused both) :
Switch-A(config-if)# switchport trunk native vlan 99
Switch-B : Already 99 ✅

→ Both now native VLAN 99 (match)
⚠️
Security best practice : Native VLAN unused Default native VLAN 1 = security risk (management traffic, DTP).
Recommendation : Change native VLAN unused ID (ex: 999).
VLAN 999 exists, but no ports assigned → traffic isolated.
Spanning Tree

STP Loops — Broadcast storms

STP (Spanning Tree Protocol) = Prevents Layer 2 loops. STP failure/misconfiguration → broadcast storm catastrophique.

// Rappel Switching Loop (Cours TR03)

Loop sans STP : Redundant paths → broadcast frames loop infinitely → network meltdown.

// Causes STP loop production

1. STP disabled

Bad config :
Switch(config)# no spanning-tree vlan 10

→ STP disabled VLAN 10 → Loops possible
→ NEVER disable STP production

2. Unmanaged switch inserted

  • User branche personal unmanaged switch (no STP support)
  • Creates redundant path bypass STP
  • Loop immediate (broadcast storm)

3. PortFast on trunk

PortFast : Bypass STP states (Listening, Learning) → Forwarding immediately
Usage correct : Access ports (PCs, printers) — no loops possible
ERREUR : PortFast on trunk port → Bypass STP protection → Loop

spanning-tree portfast trunk (Cisco)
→ DANGEROUS (creates loops)

4. BPDU Guard triggered

  • BPDU Guard = Security feature (disable port si reçoit BPDU)
  • Expected BPDUs port → BPDU Guard disables → STP topology change
  • Peut causer loops si mal configuré

// Symptômes STP loop

→ Instant network meltdown (everything stops)
→ Switch CPU 100% (processing broadcast storm)
→ MAC address table flapping (same MAC multiple ports)
→ LED ports rapid blinking (massive traffic)
→ Users : Total connectivity loss (timeouts everything)

// Diagnostic & Resolution

Step 1 : Identify loop
Switch# show spanning-tree
→ Check topology changes (TC count high ?)
→ Check blocked ports (none = potential loop)

Switch# show mac address-table
→ Same MAC multiple ports ? (flapping = loop indicator)

Step 2 : Break loop immediately
→ Unplug ONE redundant cable → Network recovers
→ Identify which cable/port caused loop

Step 3 : Root cause fix
→ Enable STP (if disabled)
→ Remove unmanaged switch (rogue device)
→ Disable PortFast trunk ports
→ Verify STP convergence
🔁
STP variants modern RSTP (802.1w) : Rapid STP, convergence <5 sec (vs STP 30-50 sec).
MSTP (802.1s) : Multiple STP instances (per-VLAN optimization).
Modern switches : RSTP default (faster recovery loops).
CAM table attack

MAC Flooding — CAM table overflow

MAC Flooding = Attaque remplir CAM table (MAC address table) switch → switch behaves comme hub (broadcast all ports).

// CAM Table (Content Addressable Memory)

Switch MAC address table :

VLAN  MAC Address         Port
10    aa:bb:cc:dd:ee:ff  Gi0/1
10    11:22:33:44:55:66  Gi0/2
20    99:88:77:66:55:44  Gi0/5

Switch learns : Source MAC → Port association
Forward decision : Destination MAC → Lookup table → Forward specific port

// Attaque MAC Flooding

Attacker tool : macof (dsniff suite), yersinia

Attack process :
1. Attacker sends thousands frames fake source MACs
   → aa:aa:aa:aa:aa:01, aa:aa:aa:aa:aa:02, ...

2. Switch learns each MAC → CAM table fills
   → Typical table : 8,000-16,000 entries

3. CAM table FULL → New MACs can't learn
   → Switch floods unknown unicast all ports (failopen behavior)

4. Switch behaves like hub
   → ALL traffic broadcast all ports
   → Attacker sniff all network traffic (confidentiality breach)

// Symptômes MAC Flooding

  • Performance degradation : Network slow (excessive broadcasts)
  • High CPU switch : Processing flood traffic
  • Syslog messages : "MAC address table full"
  • Sniffing possible : Attacker captures unintended traffic

// Défense : Port Security

Port Security config (Cours S12) :

interface GigabitEthernet0/5
  switchport mode access
  switchport port-security
  switchport port-security maximum 2
  switchport port-security violation shutdown
  switchport port-security mac-address sticky

→ Limit MACs per port (max 2)
→ Violation = port shutdown (err-disabled)
→ Prevents MAC flooding attack

// Diagnostic MAC Flooding

Switch# show mac address-table count

MAC Entries for VLAN 10:
Dynamic Address Count: 15,987 (near max 16,384)
Static Address Count: 5

→ Abnormally high dynamic MACs = flooding attack

Switch# show mac address-table dynamic | include Gi0/5
→ Hundreds MACs same port = attacker port
Port shutdown

Err-Disabled — Port automatiquement désactivé

Err-Disabled = Switch détecte erreur/violation → disable port automatiquement (protection network).

// Causes Err-Disabled communes

CauseDescriptionPrevention
Port Security violation Trop MACs port (exceed maximum) Configure correct max, use sticky MACs
BPDU Guard BPDU reçu port PortFast (rogue switch) Don't connect switches access ports
Duplex mismatch Excessive late collisions Enable auto-negotiation both ends
Link flap Port UP/DOWN cycles rapides Fix cable, transceiver issues
UDLD Unidirectional Link Detection (fiber) Check fiber TX/RX paths
Loopback detected Port connected self (physical loop) Remove loop cable

// Diagnostic Err-Disabled

User report : "Network connection stopped working"

Switch# show interfaces status

Port      Status           VLAN      Duplex  Speed
Gi0/5    err-disabled    10       auto    auto

→ Port Gi0/5 err-disabled (shutdown protection)

Check reason :
Switch# show interfaces Gi0/5 status err-disabled
→ Err-disabled reason: port-security violation

Switch# show port-security interface Gi0/5
→ Security Violation Count: 1
→ Last Source Address: 11:22:33:44:55:66

// Recovery Err-Disabled

Option 1 : Manual recovery (immediate)

1. Fix root cause first
→ Port security : Remove unauthorized device
→ BPDU Guard : Disconnect rogue switch
→ Duplex : Configure auto-negotiation

2. Re-enable port manually :
Switch(config)# interface GigabitEthernet0/5
Switch(config-if)# shutdown
Switch(config-if)# no shutdown

→ Port recovers (if cause fixed)

Option 2 : Auto-recovery (timer)

Switch(config)# errdisable recovery cause psecure-violation
Switch(config)# errdisable recovery interval 300

→ Auto-recovery port security violations after 300 sec (5 min)
→ Useful if temporary issues (user unplugs device)

// Best Practices Err-Disabled

  • ✅ Monitor syslog : Alert err-disabled events (security incidents)
  • ✅ Investigate cause : Don't blindly re-enable (understand why)
  • ✅ Fix root cause : Address underlying issue, then recover
  • ✅ Auto-recovery selective : Enable auto-recovery benign causes (link flap), disable security violations (manual review)
  • ✅ Document incidents : Track patterns (recurring port-security violations = rogue device hunting)
🔒
Err-disabled = security feature, not bug Switch protects network shutting down problematic ports.
Port-security violations, BPDU Guard = intentional shutdown (prevent attacks/loops).
Manual recovery = forces investigation before re-enable (security by design).
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : Native VLAN mismatch = connection fails
NUANCE. Native VLAN mismatch = connectivity PARTIAL, pas total failure.

Tagged traffic (VLAN 10, 20...) : Still works (tags preserved).
Untagged traffic (native VLAN) : Misplaced wrong VLAN.

Symptom : Some VLANs work, native VLAN traffic issues.
Examen : Native VLAN mismatch ≠ complete outage (partial problems).
❌ Piège #2 : STP loop = Layer 3 routing loop
FAUX. STP loop = Layer 2 switching loop.

Layer 2 : Switches, broadcast frames, MAC addresses, STP.
Layer 3 : Routers, IP packets, routing protocols, TTL.

STP prevents switching loops (broadcast storms).
Routing loops prevented TTL (Time To Live) packets.
Examen : "Broadcast storm" = L2 STP loop, not L3 routing loop.
❌ Piège #3 : Err-disabled = hardware failure
FAUX. Err-disabled = software protection, not hardware fault.

Err-disabled = Switch intentionally disables port (security/stability).
Hardware failure = Port physically broken (LED OFF, no link possible).

Err-disabled recoverable (shutdown/no shutdown).
Hardware failure requires replacement (RMA port/switch).
Examen : Err-disabled = configuration/security issue, not physical damage.
Résumé

🔑 Ce que tu dois retenir absolument

→Native VLAN mismatch : Trunk ends different native VLAN (1 vs 99). Symptom : CDP warnings, untagged traffic misplaced. Diagnostic : show interfaces trunk. Fix : match native VLAN both ends

→MAC flooding : Attacker fills CAM table fake MACs → switch floods all ports (hub behavior). Defense : Port Security (limit MACs per port). Symptom : CAM table full, sniffing possible
→Err-disabled : Switch auto-disable port (protection). Causes : port-security violation, BPDU Guard, duplex mismatch, link flap. Recovery : Fix cause + shutdown/no shutdown. Auto-recovery configurable
