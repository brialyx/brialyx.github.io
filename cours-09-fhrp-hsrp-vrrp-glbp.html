Single Point of Failure

Problème — Default gateway SPOF

🚪
Analogie — porte unique immeuble Default gateway = porte unique sortie immeuble. Si porte casse → TOUS residents bloqués dedans (pas sortie possible). FHRP = installer porte secours backup automatique.

Default gateway = Routeur par défaut permettant PCs sortir subnet local (accès Internet/autres réseaux).

// Architecture traditionnelle (PROBLÈME)

VLAN 10 — Subnet 10.0.10.0/24

[PC-A 10.0.10.10] ──┐
[PC-B 10.0.10.11] ──┤
[PC-C 10.0.10.12] ──┼── [Switch]
[PC-D 10.0.10.13] ──┘       │
                              │
            [Router 10.0.10.1] ← Default gateway
                      │
                  [Internet]

PROBLÈME :
→ Router crashe → TOUS PCs perdent accès Internet
→ Default gateway = SPOF (Single Point of Failure)
→ Pas redondance

// Impact panne gateway

  • ❌ Perte connectivité totale : 0 accès Internet/autres subnets
  • ❌ Downtime complet : Tous users affectés simultanément
  • ❌ RTO long : Attendre technicien remplacer/réparer router (heures)
⚠️
Default gateway unique = risque inacceptable Production critique JAMAIS 1 seul router. Redondance gateway OBLIGATOIRE environnements HA. FHRP = solution standard.
Redundancy protocol

FHRP — First Hop Redundancy Protocol

FHRP = Protocoles permettant multiples routeurs partager 1 IP virtuelle (VIP). PCs utilisent VIP comme gateway → failover transparent si router actif tombe.

// Principe FHRP

  • 2+ routeurs physiques : R1, R2 (ou plus)
  • 1 IP virtuelle (VIP) : 10.0.10.1 (gateway PCs)
  • 1 MAC virtuelle (VMAC) : MAC address partagée
  • Active router : 1 routeur forwarde trafic (ex: R1)
  • Standby router(s) : Backup(s) monitoring active (ex: R2)
  • Failover automatique : Si R1 tombe → R2 prend relais immédiatement

// Architecture FHRP

VLAN 10 — Subnet 10.0.10.0/24

[PC-A] ──┐
[PC-B] ──┼── [Switch]
[PC-C] ──┘      │
                ├──── [R1 — 10.0.10.2] ACTIVE
                │         │ Priority 110
                │         ├── Internet
                │
                └──── [R2 — 10.0.10.3] STANDBY
                          │ Priority 100
                          ├── Internet

IP Virtuelle (VIP) : 10.0.10.1
MAC Virtuelle (VMAC) : 0000.0c07.ac0a

→ PCs configured : Default gateway = 10.0.10.1 (VIP)
→ R1 ACTIVE forwarde trafic
→ R2 STANDBY surveille R1 (heartbeats)
→ Si R1 tombe → R2 devient ACTIVE automatiquement (failover ~3 sec)

// Transparence failover

Depuis PC :

  • PC connait uniquement VIP 10.0.10.1
  • PC ARP request VIP → Reçoit VMAC
  • R1 tombe → R2 répond MÊME VMAC
  • PC continue envoyer vers VMAC → Fonctionne (pas détecte changement)
✅
Failover transparent = zéro reconfiguration PCs PCs jamais savent quel router physique actif. VIP + VMAC masquent complexité. Failover subsecond possible → Downtime imperceptible users.
Cisco proprietary

HSRP — Hot Standby Router Protocol

HSRP = FHRP propriétaire Cisco. Standard de facto réseaux Cisco.

// Caractéristiques HSRP

  • Vendor : Cisco uniquement
  • Version : HSRPv1 (legacy), HSRPv2 (actuel)
  • Roles : Active, Standby, Listen
  • Virtual MAC : 0000.0c07.acXX (XX = group number)
  • Multicast hello : 224.0.0.2 (v1), 224.0.0.102 (v2)
  • Hello timer : 3 sec défaut
  • Hold timer : 10 sec défaut (3× hello)

// États HSRP

Active : Forwarde trafic, répond ARP VIP.

Standby : Backup ready, surveille Active (heartbeats).

Listen : Autres routeurs group HSRP (si 3+ routeurs).

// Priority & Preemption

Priority : Valeur 0-255 (défaut 100). Plus haute priority = Active.

Exemple :
R1 priority 110
R2 priority 100

→ R1 devient Active (110 > 100)
→ R2 devient Standby

Preemption : Si router priority haute boot après router priority basse déjà Active → Preemption force takeover.

Sans preemption :
1. R2 (priority 100) boot first → Devient Active
2. R1 (priority 110) boot 5 min après → Reste Standby
→ R2 reste Active (premier arrivé garde rôle)

Avec preemption :
1. R2 (priority 100) boot first → Devient Active
2. R1 (priority 110) boot 5 min après → Devient Active (takeover)
→ R1 priority supérieure force failover (preempt R2)

// Configuration HSRP (Cisco IOS)

R1 config (Active) :

interface GigabitEthernet0/1
 ip address 10.0.10.2 255.255.255.0
 standby 1 ip 10.0.10.1
 standby 1 priority 110
 standby 1 preempt
!

R2 config (Standby) :

interface GigabitEthernet0/1
 ip address 10.0.10.3 255.255.255.0
 standby 1 ip 10.0.10.1
 standby 1 priority 100
!

→ Group 1 (arbitrary number)
→ VIP 10.0.10.1
→ R1 priority 110 (Active)
→ R1 preempt enabled (takeover si reboot)

// Tracking

Interface tracking : Décrémente priority si interface critique down.

R1 config avec tracking :

interface GigabitEthernet0/0
 description Uplink vers Internet
!
interface GigabitEthernet0/1
 ip address 10.0.10.2 255.255.255.0
 standby 1 ip 10.0.10.1
 standby 1 priority 110
 standby 1 preempt
 standby 1 track 1 decrement 20
!
track 1 interface GigabitEthernet0/0 line-protocol

Comportement :
→ R1 priority normal = 110
→ Si Gi0/0 (uplink Internet) down → Priority 110 - 20 = 90
→ R2 (priority 100) > R1 (90) → R2 devient Active
→ Failover intelligent (évite Active sans uplink)
Standard IETF

VRRP — Virtual Router Redundancy Protocol

VRRP = FHRP standard IETF (RFC 5798). Équivalent HSRP multi-vendor.

// Caractéristiques VRRP

  • Vendor : Multi-vendor (Cisco, Juniper, HP, Arista...)
  • Standard : IETF RFC 5798
  • Roles : Master, Backup
  • Virtual MAC : 0000.5e00.01XX (XX = VRID)
  • Multicast : 224.0.0.18
  • Advertisement timer : 1 sec défaut

// VRRP vs HSRP — Différences

CritèreHSRPVRRP
VendorCisco propriétaireStandard IETF multi-vendor
RolesActive / StandbyMaster / Backup
Priority0-255 (défaut 100)1-254 (défaut 100)
PreemptionDésactivée défaut✅ Activée défaut
Virtual MAC0000.0c07.acXX0000.5e00.01XX
Multicast224.0.0.2 (v1), 224.0.0.102 (v2)224.0.0.18
TimersHello 3s, Hold 10sAdvertisement 1s

// Configuration VRRP (Cisco IOS)

R1 config (Master) :

interface GigabitEthernet0/1
 ip address 10.0.10.2 255.255.255.0
 vrrp 1 ip 10.0.10.1
 vrrp 1 priority 110
!

→ VRID 1 (Virtual Router ID)
→ VIP 10.0.10.1
→ Priority 110 → Master
→ Preemption activée défaut (pas besoin configurer)
🌐
VRRP = préférable environnements multi-vendor Réseau 100% Cisco → HSRP OK.
Mix Cisco + Juniper + HP → VRRP obligatoire (standard universel).
VRRP preemption défaut = comportement plus logique.
Load balancing

GLBP — Gateway Load Balancing Protocol

GLBP = FHRP Cisco avec load balancing actif. Multiples routeurs forwardent trafic SIMULTANÉMENT (vs HSRP/VRRP = 1 seul actif).

// Problème HSRP/VRRP

  • HSRP/VRRP : 1 Active, autres Standby (idle)
  • Router Standby = gaspillage ressources (attend panne)
  • Pas load balancing

// Solution GLBP

Roles GLBP :

  • AVG (Active Virtual Gateway) : 1 routeur contrôle group, assigne VMACs
  • AVF (Active Virtual Forwarder) : TOUS routeurs forwardent trafic (load balancing)

Principe :

  1. PCs ARP request VIP
  2. AVG répond avec MACs différentes selon algo (round-robin, weighted, host-dependent)
  3. PC-A reçoit VMAC1 → Trafic vers R1
  4. PC-B reçoit VMAC2 → Trafic vers R2
  5. PC-C reçoit VMAC1 → Trafic vers R1 (round-robin)
  6. → Load balancing automatique entre routeurs
GLBP Architecture :

[PC-A] ──┐                     VMAC1
[PC-B] ──┼── [Switch] ───────┬──── [R1] AVG + AVF1 ← 50% trafic
[PC-C] ──┘                     │         │
                              │         ├── Internet
                              │
                              └──── [R2] AVF2 ← 50% trafic
                                       │
                           VMAC2    ├── Internet

→ 1 VIP (10.0.10.1)
→ 2 VMACs (VMAC1, VMAC2)
→ R1 AVG distribue VMACs round-robin
→ PC-A → VMAC1 (R1), PC-B → VMAC2 (R2), PC-C → VMAC1 (R1)
→ Load balancing actif

// GLBP vs HSRP/VRRP

CritèreHSRP/VRRPGLBP
Routeurs actifs1 seul (Active/Master)TOUS (AVFs)
Load balancing❌ Non✅ Oui
Utilisation ressources⚠️ Standby idle✅ Tous utilisés
ComplexitéSimplePlus complexe
VendorHSRP Cisco, VRRP multi-vendorCisco uniquement
⚖️
GLBP = meilleure utilisation ressources HSRP/VRRP simple, largement supporté.
GLBP optimise utilisation bande passante (load balancing) MAIS Cisco uniquement, complexité accrue.
Production : HSRP/VRRP suffisant majorité cas (GLBP = niche).
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : HSRP standard multi-vendor
FAUX. HSRP = Cisco propriétaire.

HSRP : Cisco uniquement
VRRP : Standard IETF multi-vendor

Mix vendors → VRRP obligatoire
❌ Piège #2 : HSRP preemption activée défaut
FAUX. HSRP preemption désactivée défaut.

HSRP : Preemption OFF défaut (configurer standby preempt)
VRRP : Preemption ON défaut

Sans preemption HSRP → Router priority haute peut rester Standby
❌ Piège #3 : GLBP = 2 routeurs actifs forwarding
INCOMPLET. GLBP = TOUS routeurs actifs (pas juste 2).

HSRP/VRRP : 1 Active, autres Standby (idle)
GLBP : TOUS AVFs forwardent trafic simultanément (load balancing)

GLBP peut avoir 2, 3, 4+ routeurs ALL active
Résumé

🔑 Ce que tu dois retenir absolument

→FHRP : First Hop Redundancy Protocol. Multiples routeurs partagent VIP + VMAC. Failover transparent si Active tombe. Élimine SPOF default gateway
→HSRP : Cisco propriétaire. Active/Standby. Priority 0-255 (défaut 100). Preemption OFF défaut. Virtual MAC 0000.0c07.acXX. Multicast 224.0.0.2
→VRRP : Standard IETF multi-vendor. Master/Backup. Priority 1-254 (défaut 100). Preemption ON défaut. Virtual MAC 0000.5e00.01XX. Multicast 224.0.0.18
→GLBP : Cisco load balancing. TOUS routeurs AVFs actifs simultanément (vs 1 seul HSRP/VRRP). AVG distribue VMACs round-robin. Optimise ressources
→Tracking : Décrémente priority si interface critique down → Failover intelligent. Preemption : Force takeover si router priority haute boot après priority basse Active
