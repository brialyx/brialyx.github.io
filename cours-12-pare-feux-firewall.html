Concept

Firewall — filtrage et contrôle d'accès

🚪
Analogie — videur de boîte de nuit Firewall = videur qui contrôle qui entre/sort. Il vérifie : identité (IP), heure (règles temporelles), tenue (protocole autorisé). Si pas conforme → refus entrée. Liste VIP (whitelist) entre automatiquement.

Firewall = équipement ou logiciel filtrant trafic réseau selon règles de sécurité.

// Fonctions principales

  • Filtrage paquets : Autoriser/bloquer basé sur IP source/dest, ports, protocoles
  • Inspection état : Suivi connexions (stateful)
  • Zones sécurité : Séparer réseaux (Inside/Outside/DMZ)
  • NAT/PAT : Masquage IPs privées
  • Logs : Enregistrer tentatives accès
🎯
Pour l'examen Network+ CompTIA teste : Stateless vs Stateful (différence critique), zones (Inside/DMZ/Outside), ACL (ordre règles, implicit deny), règles directionnelles (inbound/outbound). Question TRÈS fréquente : "Firewall bloque connexion légitime, pourquoi ?"
Types de filtrage

Stateless vs Stateful — la différence CRITIQUE

// Stateless Firewall (Packet Filtering)

Stateless = filtre chaque paquet individuellement, SANS mémoire des connexions précédentes.

Caractéristiques :

  • Examine : IP source, IP dest, port source, port dest, protocole (TCP/UDP/ICMP)
  • ❌ Pas de contexte : Ne sait pas si paquet appartient à connexion établie
  • ❌ Règles bidirectionnelles requises : Autoriser trafic sortant ET entrant de retour
  • ⚠️ Vulnérable : Spoofing, fragmentation attacks

Exemple problème :

Règle : Autoriser Inside (192.168.1.0/24) → Outside (any) port 80

Problème : Trafic sortant OK, MAIS réponse serveur bloquée !
→ Besoin règle retour : Outside → Inside port >1024 (ports éphémères)
→ Risque sécurité : Ouvre plage ports large

// Stateful Firewall (Stateful Inspection)

Stateful = suivi connexions. Mémorise état sessions (établies, nouvelles, fermées).

Caractéristiques :

  • ✅ Table d'états : Enregistre connexions actives (IP, ports, état TCP)
  • ✅ Autorisation automatique retours : Si connexion sortante établie → réponses entrantes autorisées
  • ✅ Sécurité renforcée : Détecte paquets hors contexte
  • ✅ Règles simplifiées : 1 règle sortante suffit

Exemple solution :

Règle : Autoriser Inside → Outside port 80

Stateful automatic : Firewall mémorise connexion sortante
→ Réponse serveur reconnue comme partie connexion établie
→ Autorisée automatiquement, SANS règle retour explicite
→ Sécurisé : Ports éphémères pas ouverts globalement

// Comparaison Stateless vs Stateful

CritèreStatelessStateful
Contexte❌ Aucun (paquet par paquet)✅ Suivi connexions
Règles retour⚠️ Nécessaires (bidirectionnelles)✅ Automatiques
Sécurité⚠️ Faible (spoofing)✅ Forte
Performances✅ Rapide (simple)⚠️ Plus lent (table états)
Usage moderne❌ Obsolète✅ Standard
🎯
Question type examen "Firewall bloque réponses serveurs web légitimes. Cause probable ?"

Réponse : Firewall Stateless SANS règle retour

Solution : Passer Stateful OU ajouter règle entrante ports éphémères
Segmentation

Zones de sécurité — Inside / DMZ / Outside

Zones = segmentation logique réseau selon niveau de confiance.

// Zone Inside (Trusted)

Réseau interne — confiance élevée.

  • Équipements : PCs employés, serveurs internes, imprimantes
  • IPs : Privées (192.168.x.x, 10.x.x.x)
  • Niveau sécurité : Élevé (confiance maximale)
  • Règles : Accès sortant libre (Internet), accès entrant bloqué par défaut

// Zone Outside (Untrusted)

Internet public — confiance zéro.

  • Équipements : Internet, réseaux tiers, attaquants potentiels
  • IPs : Publiques
  • Niveau sécurité : Zéro (hostile par défaut)
  • Règles : Trafic entrant TRÈS restreint (uniquement services publics autorisés)

// Zone DMZ (DeMilitarized Zone)

Zone tampon — confiance limitée, exposition contrôlée.

  • Équipements : Serveurs publics (web, mail, DNS, FTP)
  • IPs : Publiques ou privées (selon NAT)
  • Niveau sécurité : Moyen (isolée Inside ET Outside)
  • Règles : Internet → DMZ autorisé (ports spécifiques), DMZ → Inside très restreint

// Architecture typique

[Internet] (Outside)
↓
[Firewall Interface 1]
↓
[DMZ] — Serveurs web, mail
↓
[Firewall Interface 2]
↓
[Inside] — Réseau interne

→ Internet peut accéder DMZ (port 80/443)
→ DMZ NE PEUT PAS initier connexions vers Inside
→ Inside peut accéder DMZ (admin, backups)
→ Inside peut accéder Internet
🛡️
Pourquoi DMZ ? Si serveur web compromis dans DMZ → attaquant NE PEUT PAS accéder Inside directement. DMZ = isolation serveurs publics pour limiter blast radius si compromission.
Règles

ACL (Access Control List) — règles de filtrage

ACL = liste ordonnée de règles définissant quel trafic autoriser/bloquer.

// Structure règle ACL

[Action] [Protocole] [IP Source] [Port Source] [IP Dest] [Port Dest]

Exemple :
PERMIT TCP 192.168.1.0/24 ANY ANY 80
→ Autorise réseau interne → any destination port 80 (HTTP)

DENY TCP ANY ANY 10.0.0.5 22
→ Bloque tout trafic SSH vers 10.0.0.5

// Ordre règles = CRITIQUE

Règles évaluées de haut en bas. Première correspondance = appliquée, reste ignoré.

❌ MAUVAIS ORDRE :
1. DENY ALL
2. PERMIT 192.168.1.0/24 → ANY port 80

→ Règle 1 bloque TOUT (y compris trafic légitime)
→ Règle 2 jamais atteinte !

✅ BON ORDRE :
1. PERMIT 192.168.1.0/24 → ANY port 80
2. DENY ALL

→ Règle 1 autorise trafic légitime
→ Règle 2 bloque reste (implicit deny)

// Implicit Deny

Implicit Deny = règle invisible à la fin de TOUTES les ACL : "DENY ALL".

Si aucune règle explicite ne correspond → paquet bloqué par défaut.

⚠️
Oublier Implicit Deny = erreur classique Admin ajoute règle DENY spécifique en fin d'ACL → pense "bloquer juste ça".
Oublie que l'Implicit Deny qui suit bloque TOUT le reste aussi !
Solution : Toujours ajouter PERMIT nécessaires AVANT DENY final.
Direction trafic

Règles directionnelles — Inbound vs Outbound

Règles firewall appliquées selon direction trafic par rapport à l'interface.

// Inbound (Entrant)

Trafic entrant VERS réseau protégé (Outside → Inside).

  • Exemple : Internet → Serveur web interne
  • Par défaut : DENY ALL (sécurité maximale)
  • Autoriser : Uniquement services publics nécessaires

// Outbound (Sortant)

Trafic sortant DEPUIS réseau protégé (Inside → Outside).

  • Exemple : Employés → Internet navigation web
  • Par défaut : Souvent PERMIT ALL (confiance interne)
  • Restreindre : Bloquer sites dangereux, P2P, protocoles interdits

// Exemple config

OUTBOUND (Inside → Outside) :
PERMIT TCP 192.168.1.0/24 → ANY port 80,443 (HTTP/S)
PERMIT UDP 192.168.1.0/24 → ANY port 53 (DNS)
DENY ALL (implicit)

INBOUND (Outside → Inside) :
PERMIT TCP ANY → 203.0.113.5 port 443 (HTTPS vers serveur web public)
PERMIT established connections (stateful return traffic)
DENY ALL (implicit)
Évolution

NGFW (Next-Generation Firewall)

NGFW = firewall avancé combinant filtrage traditionnel + fonctions sécurité modernes.

// Fonctions NGFW

  • ✅ Stateful inspection (base)
  • ✅ Application awareness : Identifie applications (Facebook, Skype) pas juste ports
  • ✅ IPS intégré : Détection/prévention intrusions inline
  • ✅ Deep packet inspection (DPI) : Analyse contenu paquets (malware, phishing)
  • ✅ SSL/TLS inspection : Décryptage HTTPS pour analyser contenu chiffré
  • ✅ User identity : Règles basées sur utilisateur (pas juste IP)
  • ✅ Threat intelligence : Feeds signatures malware/botnet temps réel
  • ✅ Sandboxing : Exécuter fichiers suspects dans environnement isolé

// Firewall traditionnel vs NGFW

FonctionFirewall traditionnelNGFW
Filtrage IP/port✅ Oui✅ Oui
Stateful inspection✅ Oui✅ Oui
Application awareness❌ Non✅ Oui
IPS intégré❌ Non✅ Oui
DPI❌ Non✅ Oui
SSL inspection❌ Non✅ Oui
User-based policies❌ Non✅ Oui
🚀
NGFW = standard moderne Menaces modernes (malware, APT, ransomware) nécessitent inspection avancée. NGFW = firewall + IPS + antivirus + DPI. Recommandé entreprises 2024+.
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : Stateless = pas de règles
FAUX. Stateless a des règles MAIS sans contexte connexions.

Stateless = règles existent mais appliquées paquet par paquet individuellement.
Stateful = règles + suivi état connexions.
❌ Piège #2 : DMZ = plus sécurisée que Inside
FAUX. DMZ = MOINS sécurisée que Inside.

DMZ exposée Internet (serveurs publics). Inside = isolée, confiance élevée.
DMZ → Inside bloqué par défaut pour protéger Inside.
❌ Piège #3 : Ordre ACL pas important
FAUX. Ordre = CRITIQUE.

Première règle correspondante appliquée, reste ignoré.
DENY ALL en haut → bloque TOUT (y compris légitime).
Résumé

🔑 Ce que tu dois retenir absolument

→Stateless : Paquet par paquet, pas contexte, règles bidirectionnelles. Stateful : Suivi connexions, retours automatiques (standard)
→Zones : Inside (confiance élevée), Outside (zéro confiance), DMZ (tampon serveurs publics)
→ACL : Ordre critique (haut→bas, première correspondance). Implicit Deny à la fin.
→Règles directionnelles : Inbound (entrant) restrictif, Outbound (sortant) permissif
→NGFW : Application awareness, IPS, DPI, SSL inspection, user-based policies
