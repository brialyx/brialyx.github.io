IP Address Management

IPAM — Pourquoi gérer les IPs ?

📋
Analogie — attribution places parking Réseau = parking. IPs = places numérotées. Sans gestion : 2 voitures même place (conflit IP), places vides perdues (gaspillage), chaos retrouver voiture. IPAM = registre central attribution places.

IPAM (IP Address Management) = processus/outils suivre et gérer espace adressage IP réseau. Évite conflits, optimise utilisation, maintient documentation.

// Problèmes sans IPAM

  • ❌ Conflits IP : 2 devices même IP → réseau casse
  • ❌ Épuisement non détecté : Subnet /24 saturé → pas d'IPs libres nouveaux devices
  • ❌ Gaspillage : Grandes plages réservées, sous-utilisées
  • ❌ Pas visibilité : "Combien d'IPs libres VLAN 10 ?" → personne sait
  • ❌ Troubleshooting lent : "Qui utilise 10.0.1.50 ?" → chercher partout

// Bénéfices IPAM

  • ✅ Prévention conflits : Base centralisée allocation IPs
  • ✅ Visibilité utilisation : Taux occupation subnets (80% utilisé → planifier expansion)
  • ✅ Optimisation : Récupérer IPs inutilisées
  • ✅ Automatisation : Intégration DHCP, DNS
  • ✅ Audit/Compliance : Historique allocations (qui, quand, pourquoi)
⚠️
Conflit IP = catastrophe réseau 2 devices même IP → comportement erratique, connexions intermittentes, downtime. IPAM = assurance anti-conflits.

Duplicate address

Conflits d'adresses IP

// Causes conflits IP

  • IP statique dupliquée : Admin assigne manuellement 10.0.1.50 déjà utilisée
  • DHCP pool overlap : Pool DHCP inclut IPs statiques → DHCP assigne IP déjà utilisée
  • Rogue DHCP server : Serveur DHCP non autorisé distribue IPs en conflit
  • IP statique dans pool DHCP : Device statique .100, pool DHCP contient .100
  • Device relié après longue absence : Laptop déconnecté 6 mois, IP réassignée entre-temps

// Symptômes conflit IP

  • Windows : Alerte "IP address conflict detected"
  • Linux/Mac : Logs ARP duplicate address
  • Connexion réseau intermittente (fonctionne puis casse)
  • Impossibilité communiquer certains hosts
  • Switch logs : Duplicate MAC same port flip-flop

// Détection conflits

Gratuitous ARP : Device envoie ARP request sa propre IP au boot. Si réponse → conflit détecté.

IPAM scanning : Outils scannent réseau périodiquement, détectent duplicates.

Switch CAM table : Surveiller flip-flop MAC addresses (indicateur conflit).

// Résolution conflit

  1. Identifier devices : Qui utilise IP en conflit ? (arp -a, switch MAC table)
  2. Déterminer légitime : Quel device devrait avoir IP ?
  3. Changer IP illégitime : Reassigner IP différente au device non autorisé
  4. Documenter IPAM : Mettre à jour base IPAM avec allocation correcte
  5. Prévention future : Séparer plages statiques/DHCP clairement

// Prévention conflits (Best practices)

  • ✅ IPAM centralisé : Base unique allocation IPs
  • ✅ Séparation stricte : Statiques (.1-.50), DHCP (.51-.250), Réservées (.251-.254)
  • ✅ DHCP exclusions : Exclure plage statiques du pool DHCP
  • ✅ DHCP Snooping : Bloquer rogue DHCP servers
  • ✅ Scanning régulier : Détecter IPs non documentées
Exemple séparation plages :

Subnet: 10.0.10.0/24 (254 IPs utilisables)

Statiques (.1-.50) :
- 10.0.10.1 : Gateway
- 10.0.10.2 : Switch management
- 10.0.10.10-40 : Serveurs, imprimantes
- 10.0.10.50 : Monitoring probe

DHCP Pool (.51-.250) :
- 10.0.10.51-250 : Workstations dynamiques
- Exclusion DHCP : .1-.50 (évite overlap)

Réservées (.251-.254) :
- 10.0.10.254 : Future expansion

→ Pas overlap possible → Zéro conflits
🎯
Question fréquente examen "Prévenir conflits IP entre allocations statiques et DHCP ?"

Réponse : DHCP exclusions (exclure plage statiques du pool DHCP)
Software

Outils IPAM

// Fonctionnalités IPAM tools

  • Gestion subnets : Créer/modifier/supprimer subnets, allocation hiérarchique
  • Tracking IPs : Base données chaque IP (utilisée, libre, réservée)
  • Scanning réseau : Découverte automatique devices, détection IPs non documentées
  • Intégration DHCP/DNS : Synchronisation automatique
  • Détection conflits : Alertes duplicates
  • Rapports utilisation : Graphiques occupation, prévisions épuisement
  • Historique : Qui assigné IP quand
  • API : Automatisation (scripts, intégration autres outils)

// Outils populaires

NetBox

  • Open-source (gratuit)
  • IPAM + DCIM (Data Center Infrastructure Management)
  • Web interface moderne, API REST complète
  • Documentation VLANs, racks, équipements
  • Communauté active

phpIPAM

  • Open-source PHP/MySQL
  • Simple, léger
  • Scanning automatique (ping/SNMP)
  • Gestion VLANs, subnets
  • PowerDNS integration

Infoblox

  • Commercial (entreprise)
  • IPAM + DNS + DHCP (DDI suite)
  • Automatisation avancée
  • Très cher (10K$/an+)

Microsoft IPAM (Windows Server)

  • Inclus Windows Server
  • Intégration Active Directory
  • Gestion DHCP/DNS Windows
  • Environnements Microsoft-centric

SolarWinds IP Address Manager

  • Commercial
  • Découverte automatique réseau
  • Alertes utilisation
  • Intégration suite SolarWinds

VLANs

Documentation des VLANs

IPAM gère aussi documentation VLANs (subnets Layer 3 mappés VLANs Layer 2).

// Informations documenter par VLAN

  • VLAN ID : 10, 20, 30...
  • Nom : Data, Voice, Servers, Guest
  • Subnet IP : 10.0.10.0/24
  • Gateway : 10.0.10.1 (SVI sur switch L3)
  • DHCP scope : Pool .51-.250
  • Usage/Purpose : Workstations employees, Téléphones IP, etc.
  • Sites : HQ, Paris, NY (si VLAN multi-sites)
  • ACLs appliquées : Restrictions accès inter-VLANs
  • Trunk ports : Quels switches transportent VLAN

// Conventions nommage VLANs

Cohérence noms CRITIQUE (pas "VLAN10", "vlan_10", "Data-VLAN" mélangés).

Exemples conventions :

  • Descriptive : Data, Voice, Servers, Guest, Management
  • Par fonction : Employees, Contractors, IoT, CCTV
  • Par site : HQ-Data, Paris-Servers, NY-Voice
📝
VLAN 1 = natif, ne PAS utiliser VLAN 1 = natif défaut switches Cisco. Best practice sécurité : Changer native VLAN (ex: VLAN 999), ne JAMAIS utiliser VLAN 1 pour data/voice. VLAN 1 = vulnérable VLAN hopping attacks.
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : IPAM = juste liste IPs
INCOMPLET. IPAM = gestion complète :

- Subnets/VLANs
- Allocation IPs (statiques, DHCP pools)
- Détection conflits
- Intégration DHCP/DNS
- Rapports utilisation
- Historique changements

Pas juste spreadsheet IPs
❌ Piège #2 : Conflit IP résolu redémarrer device
FAUX. Redémarrer = masque temporairement, PAS résout.

Résolution vraie :
1. Identifier QUEL device légitime pour IP
2. Reassigner IP à device illégitime
3. Documenter IPAM allocation correcte
4. Prévenir future (DHCP exclusions)
❌ Piège #3 : Pool DHCP peut inclure gateway IP
FAUX. Gateway (.1) = IP statique (router/SVI).

Pool DHCP doit EXCLURE gateway + IPs statiques.
Exemple : Subnet .0/24, gateway .1 → Pool DHCP .51-.250 (exclut .1-.50)
Résumé

🔑 Ce que tu dois retenir absolument

→IPAM : Gestion centralisée adresses IP. Prévient conflits, optimise utilisation, documente allocations. Tracking subnets/VLANs/IPs
→Suivi plages : Hiérarchie (supernet → sites → VLANs → sous-allocations). Séparer statiques (.1-.50), DHCP pool (.51-.250), réservées (.251-.254)
→Conflits IP : Causes (statique dupliquée, DHCP overlap). Prévention : DHCP exclusions, IPAM centralisé, séparation stricte plages. Détection : Gratuitous ARP
→Outils IPAM : NetBox (open-source, moderne), phpIPAM (simple), Infoblox (entreprise cher), MS IPAM (Windows). Fonctions : subnet mgmt, scanning, intégration DHCP/DNS
→Documentation VLANs : VLAN ID, nom, subnet, gateway, DHCP scope, usage. Conventions nommage cohérentes. VLAN 1 natif = ne PAS utiliser (sécurité)
