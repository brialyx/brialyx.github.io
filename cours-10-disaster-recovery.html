Disaster Recovery

DR — Reprise après sinistre

🔥
Analogie — assurance incendie maison DR = assurance datacenter. Maison brûle → assurance reconstruit. Datacenter brûle → DR rebuild business. Pas DR = faillite garantie (95% entreprises sans DR ferment dans 2 ans après disaster majeur).

Disaster Recovery (DR) = Plan/processus restaurer opérations IT après catastrophe majeure (incendie, inondation, cyberattaque, panne totale). Focus : Récupération rapide données/systèmes critiques.

// DR vs HA — Différence

HA (Haute Disponibilité) : Évite downtime pannes quotidiennes (hardware fails, maintenance).

DR (Disaster Recovery) : Récupération après catastrophe majeure (datacenter entier perdu).

CritèreHADR
ScopePannes locales (1 server, 1 switch)Catastrophes site entier
FailoverAutomatique (secondes)Manuel ou semi-auto (minutes-heures)
FréquenceRégulière (mensuels)Rare (espoir jamais)
CoûtModéré (redondance locale)Élevé (site backup complet)
RTOSecondes-minutesHeures-jours

// Types disasters

  • Naturels : Incendie, inondation, tremblement terre, ouragan
  • Techniques : Panne électrique prolongée, climatisation totale, explosion
  • Cyber : Ransomware crypte TOUT, cyberattaque destructive
  • Humains : Sabotage, erreur admin catastrophique (rm -rf /*)
💥
Stat industry : 95% PME sans DR ferment dans 2 ans post-disaster Disaster majeur SANS DR plan = faillite quasi-garantie. DR pas luxe → obligation survie business. Coût DR inférieur coût faillite.
Data protection

Stratégies backup — Full / Incrémentale / Différentielle

Backup = Copie données permettant restauration si perte/corruption. Base DR.

// 1. Full Backup (Sauvegarde complète)

Définition : Copie TOUTES données à chaque backup.

Processus :

Dimanche : Backup full 1 TB (TOUTES données)
Lundi : Backup full 1 TB (TOUTES données again)
Mardi : Backup full 1 TB (TOUTES données again)
...

→ Chaque backup = copie complète indépendante

Avantages :

  • ✅ Restauration simple : 1 seul fichier backup nécessaire
  • ✅ Restauration rapide : Pas besoin multiples sources

Inconvénients :

  • ⚠️ Stockage massif : 1 TB × 7 jours = 7 TB/semaine
  • ⚠️ Temps backup long : Copier 1 TB = heures
  • ⚠️ Bande passante : Backups remote = consomme lien WAN

// 2. Incremental Backup (Sauvegarde incrémentale)

Définition : 1 full initial, puis uniquement fichiers MODIFIÉS depuis dernier backup (full OU incrémental).

Processus :

Dimanche : Full backup 1 TB
Lundi : Incrémental 10 GB (modifs depuis dimanche)
Mardi : Incrémental 12 GB (modifs depuis lundi)
Mercredi : Incrémental 8 GB (modifs depuis mardi)
...

→ Seules modifications sauvegardées (efficace)

Avantages :

  • ✅ Stockage minimal : 1 TB + 10+12+8 GB = ~1.03 TB/semaine
  • ✅ Backup rapide : Uniquement modifs (minutes)
  • ✅ Bande passante faible : Petits volumes quotidiens

Inconvénients :

  • ⚠️ Restauration complexe : Besoin full + TOUS incrémentaux
  • ⚠️ Restauration lente : Restaurer mercredi = full + lun + mar + mer
  • ⚠️ Chaîne fragile : Si backup mardi corrompu → impossible restaurer mer

// 3. Differential Backup (Sauvegarde différentielle)

Définition : 1 full initial, puis fichiers modifiés depuis dernier FULL uniquement (pas depuis dernier backup).

Processus :

Dimanche : Full backup 1 TB
Lundi : Différentielle 10 GB (modifs depuis dimanche)
Mardi : Différentielle 22 GB (modifs depuis dimanche = lun + mar cumulé)
Mercredi : Différentielle 30 GB (modifs depuis dimanche = lun + mar + mer cumulé)
...

→ Chaque différentielle = cumul modifs depuis dernier full

Avantages :

  • ✅ Restauration simple : Besoin full + dernière différentielle UNIQUEMENT
  • ✅ Restauration rapide : 2 fichiers max (vs N incrémentaux)

Inconvénients :

  • ⚠️ Stockage croissant : Différentielles grossissent chaque jour
  • ⚠️ Backup ralentit : Mercredi backup 30 GB vs lundi 10 GB

Recovery sites

Sites DR — Chaud / Tiède / Froid

Site DR = Datacenter secondaire backup permettant continuité opérations si site primaire détruit.

// 1. Hot Site (Site chaud)

Définition : Datacenter backup COMPLET, équipé, données répliquées temps réel. Prêt immédiatement.

Caractéristiques :

  • ✅ Hardware identique site primaire (serveurs, switches, storage)
  • ✅ Données répliquées synchrone/asynchrone continu
  • ✅ Applications pré-configurées
  • ✅ Réseau/ISP redondant
  • ✅ Staff peut travailler immédiatement

RTO : Minutes-heures (failover quasi-instantané).

RPO : Minutes (réplication quasi-temps réel).

Coût : $$$ Très élevé (double infrastructure).

Usage : Systèmes ultra-critiques (banques, healthcare, trading).

// 2. Warm Site (Site tiède)

Définition : Datacenter backup partiellement équipé. Hardware présent MAIS données/apps nécessitent installation/restauration.

Caractéristiques :

  • ✅ Hardware installé (serveurs, réseau)
  • ⚠️ Données restaurées depuis backups (heures)
  • ⚠️ Applications installées/configurées manuellement
  • ⚠️ Tests réguliers nécessaires (vérifier fonctionnement)

RTO : Heures-jours (temps restaurer données + configurer apps).

RPO : Heures (selon fréquence backups).

Coût : $$ Moyen (hardware présent, pas données temps réel).

Usage : Entreprises moyennes, systèmes importants mais tolèrent quelques heures downtime.

// 3. Cold Site (Site froid)

Définition : Espace physique vide (bâtiment, électricité, climatisation). PAS hardware, PAS données.

Caractéristiques :

  • ✅ Bâtiment sécurisé (accès, power, cooling)
  • ❌ Zéro hardware (commander/installer après disaster)
  • ❌ Données backups off-site (restaurer après hardware installé)
  • ❌ Setup complet nécessaire (semaines)

RTO : Semaines-mois (temps commander hardware, installer, restaurer).

RPO : Jours-semaines (backups périodiques).

Coût : $ Faible (juste bâtiment).

Usage : Systèmes non-critiques, PME budget limité.

// Comparaison sites DR

SiteHardwareDonnéesRTORPOCoût

Warm site = compromis populaire (majorité entreprises).
Cold site = acceptable uniquement systèmes très non-critiques.
Décision = analyse coût downtime vs coût DR site.
Recovery operations

Failover et Failback

// Failover

Définition : Basculement opérations site primaire → site DR après disaster.

Processus failover :

  1. Disaster détecté : Site primaire inaccessible (incendie, panne totale)
  2. Décision failover : Management autorise basculement DR
  3. Activation site DR :
    - Hot site : Failover automatique (DNS/load balancer redirige trafic)
    - Warm site : Restaurer données backups, démarrer apps
    - Cold site : Installer hardware, restaurer données (semaines)
  4. Redirection users : DNS update, VPN redirect, communication users
  5. Opérations normales DR site : Business continue depuis site secondaire

// Failback

Définition : Retour opérations site DR → site primaire APRÈS réparation disaster.

Processus failback :

  1. Site primaire réparé : Reconstruction/réparation terminée
  2. Synchronisation données : Copier données DR site → site primaire (données produites pendant disaster)
  3. Tests validation : Vérifier site primaire fonctionnel (pas fail immédiatement après failback)
  4. Planification fenêtre : Failback pendant heures creuses (évite double downtime)
  5. Basculement retour : Rediriger trafic DR → primaire
  6. Monitoring intensif : Surveiller site primaire post-failback (24-48h)
Timeline disaster complet :

Jour 0 — 10:00 AM : Incendie datacenter primaire
Jour 0 — 10:30 AM : Décision failover DR (warm site)
Jour 0 — 11:00 AM : Début restauration données backups
Jour 0 — 16:00 PM : Applications restaurées, tests
Jour 0 — 18:00 PM : Failover complet → Site DR actif
→ Business opère depuis DR site (semaines)

Jour 45 : Datacenter primaire reconstruit
Jour 50 : Tests site primaire OK
Jour 52 — Weekend : Failback planifié
Jour 53 : Retour site primaire → Opérations normales
⚠️
Failback = opération risquée Failback précipité = risque double disaster (site primaire fail again).
Tests exhaustifs site primaire AVANT failback obligatoires.
Failback fenêtre maintenance (évite impact business si problème).
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : Backup incrémentale = cumul depuis dernier full
FAUX. C'est différentielle.

Incrémentale : Modifs depuis dernier backup (full OU incrémental)
Différentielle : Modifs depuis dernier FULL uniquement (cumul)

Restaurer incrémental = full + TOUS incrémentaux
Restaurer différentielle = full + dernière différentielle
❌ Piège #2 : Warm site = données temps réel
FAUX. Warm site = hardware présent, données backups.

Hot site : Hardware + Données temps réel (réplication)
Warm site : Hardware + Restaurer données backups
Cold site : Bâtiment vide uniquement

Warm ≠ données prêtes immédiatement
❌ Piège #3 : Failback = automatique après failover
FAUX. Failback = opération planifiée manuelle.

Failover peut être auto (hot site) MAIS failback TOUJOURS planifié :
- Réparer site primaire
- Synchroniser données
- Tests validation
- Fenêtre maintenance

Failback précipité = risque échec
Résumé

🔑 Ce que tu dois retenir absolument

→DR : Disaster Recovery (catastrophe majeure). RTO heures-jours, RPO minutes-heures. DR ≠ HA (HA = pannes quotidiennes, DR = site entier perdu)
→Backups : Full (tout à chaque fois, simple restauration), Incrémentale (modifs depuis dernier backup, stockage minimal), Différentielle (modifs depuis dernier full, compromis)
→Sites DR : Hot (hardware + données temps réel, RTO minutes, $$$), Warm (hardware + backups, RTO heures, $$), Cold (bâtiment vide, RTO semaines, $)
→Failover : Basculement primaire → DR après disaster. Failback : Retour DR → primaire après réparation (planifié, tests, fenêtre maintenance)
→Stat : 95% PME sans DR ferment 2 ans post-disaster. DR = obligation survie, pas luxe. Coût DR < coût faillite
