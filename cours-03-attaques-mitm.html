Attack concept

MitM — Man-in-the-Middle

📬
Analogie — facteur malveillant Alice envoie lettre Bob via Poste. Facteur intercepte, lit lettre, modifie contenu, puis livre Bob (qui croit lettre originale). Alice ↔ Bob croient communication directe MAIS facteur invisible milieu = Man-in-the-Middle.

Man-in-the-Middle (MitM) = Attaquant s'insère secrètement ENTRE deux parties communicantes → intercepte, écoute, modifie trafic.

// Principe MitM

Communication normale :
[Client] ←──────────────────> [Server]

Attaque MitM :
[Client] ←────> [Attacker] ←────> [Server]
                   ↓
             Intercept
             Eavesdrop
             Modify

→ Client croit parler Server directement
→ Server croit parler Client directement
→ Attaquant invisible, relay tout trafic

// Objectifs MitM

  • Eavesdropping : Écouter trafic (passwords, données sensibles)
  • Data modification : Changer contenu messages (injecter malware, falsifier transactions)
  • Session hijacking : Voler session cookies → impersonation
  • Credential theft : Capturer logins (HTTP Basic Auth, forms)
🔓
MitM = invisible si pas chiffrement HTTP, Telnet, FTP = texte clair → MitM parfait (passwords lisibles).
HTTPS, SSH, VPN = chiffré → MitM difficile MAIS possible (SSL stripping, certificate attacks).
Layer 2 attack

ARP Poisoning / ARP Spoofing

ARP Poisoning = Exploiter protocole ARP (Address Resolution Protocol) → rediriger trafic via attaquant.

// Rappel ARP normal

ARP : Résout IP → MAC address (Layer 3 → Layer 2).

PC-A (192.168.1.10) veut parler 192.168.1.1 (Gateway)

1. PC-A broadcast : "Qui a 192.168.1.1 ?" (ARP Request)
2. Gateway répond : "Moi ! MAC = aa:bb:cc:dd:ee:ff" (ARP Reply)
3. PC-A cache : 192.168.1.1 → aa:bb:cc:dd:ee:ff (ARP table)
4. PC-A envoie frames directement MAC gateway

// Attaque ARP Poisoning

Problème ARP : Pas authentification. N'importe qui peut envoyer ARP Reply (gratuitous ARP).

Workflow attaque :

  1. Attaquant envoie ARP Reply falsifiée PC-A :
    "192.168.1.1 (Gateway) → MAC attaquant (11:22:33:44:55:66)"
  2. PC-A update ARP table :
    192.168.1.1 → 11:22:33:44:55:66 (MAC attaquant, PAS gateway)
  3. Attaquant envoie ARP Reply falsifiée Gateway :
    "192.168.1.10 (PC-A) → MAC attaquant (11:22:33:44:55:66)"
  4. Gateway update ARP table :
    192.168.1.10 → 11:22:33:44:55:66 (MAC attaquant)
  5. Résultat :
    PC-A → Gateway trafic = envoyé MAC attaquant
    Gateway → PC-A trafic = envoyé MAC attaquant
    Attaquant = Man-in-the-Middle (relay frames)
Architecture post-ARP Poisoning :

[PC-A] ──frames──> [Attacker] ──forward──> [Gateway]
                        ↓
                Sniff packets
                Steal credentials
                Modify content

PC-A ARP table :
192.168.1.1 → 11:22:33:44:55:66 (attacker MAC, not gateway)

Gateway ARP table :
192.168.1.10 → 11:22:33:44:55:66 (attacker MAC, not PC-A)

// Détection ARP Poisoning

  • ARP table inspection : arp -a (Windows), arp -n (Linux) → check duplicates MAC
  • Gratuitous ARP monitoring : IDS alertes ARP replies non sollicitées
  • MAC address conflicts : Alertes 2 devices même MAC

// Mitigation ARP Poisoning

  • Dynamic ARP Inspection (DAI) : Switch validate ARP packets (Cisco feature)
  • Static ARP entries : Hardcode ARP table (pas pratique large scale)
  • Port security : Limit MACs per port
  • ARP monitoring tools : ARPwatch, XArp alertent anomalies
🎯
Question TRÈS fréquente examen "Attaque Layer 2 falsifiant ARP cache rediriger trafic attaquant ?"

Réponse : ARP Poisoning (ou ARP Spoofing)

"Mitigation switch Cisco validant ARP packets ?"
Réponse : Dynamic ARP Inspection (DAI)
Name resolution attack

DNS Spoofing / DNS Cache Poisoning

DNS Spoofing = Falsifier réponses DNS → rediriger users vers serveurs malveillants.

// DNS Cache Poisoning

Principe : Injecter fausses entrées DNS cache resolver → users reçoivent IPs malveillantes.

Workflow attaque :

  1. User query : "Quelle IP bank.com ?" → DNS resolver
  2. Resolver pas cache → query authoritative nameserver
  3. Attaquant envoie réponse DNS falsifiée AVANT réponse légitime :
    "bank.com → 203.0.113.66 (serveur attaquant)"
  4. Resolver accepte première réponse (race condition)
  5. Resolver cache fausse entrée (TTL 24h)
  6. TOUS users utilisant resolver redirigés serveur attaquant 24h
User: "bank.com IP ?"
Resolver query nameserver

Attacker response (first, fake) :
bank.com → 203.0.113.66 (malicious server)

Legitimate response (second, ignored) :
bank.com → 198.51.100.10 (real bank)

→ Resolver cache fake entry
→ User visits 203.0.113.66 (phishing site looks like bank)

// Local DNS Spoofing (hosts file)

Principe : Modifier fichier hosts local victime.

Fichiers hosts :

  • Windows : C:\Windows\System32\drivers\etc\hosts
  • Linux/Mac : /etc/hosts

Malware modifie :

# Hosts file (malware injected)
203.0.113.66    bank.com
203.0.113.66    www.bank.com

→ OS résout bank.com → 203.0.113.66 (AVANT query DNS)
→ User redirigé phishing site

// Mitigation DNS Spoofing

  • DNSSEC : Signatures cryptographiques DNS responses (validate authenticity)
  • Query randomization : Port source + Transaction ID random (difficile guess)
  • DNS over HTTPS (DoH) : Chiffre DNS queries (empêche MitM)
  • DNS over TLS (DoT) : TLS-encrypted DNS (port 853)
HTTPS downgrade

SSL Stripping

SSL Stripping = Downgrade connexion HTTPS → HTTP → attaquant lit trafic texte clair.

// Principe SSL Stripping

Scénario : User accède site banking via HTTP initial redirect HTTPS.

Workflow attaque :

  1. User browser : http://bank.com (HTTP initial)
  2. Server redirect : https://bank.com (HTTPS)
  3. Attacker MitM intercepte redirect :
    - Connexion User ↔ Attacker = HTTP (texte clair)
    - Connexion Attacker ↔ Server = HTTPS (chiffré)
  4. User croit connexion sécurisée (voit contenu correct)
  5. Attacker lit TOUT trafic HTTP (passwords, données)
Normal flow :
[User] ──HTTPS (encrypted)──> [Bank Server]

SSL Stripping attack :
[User] ──HTTP (plaintext)──> [Attacker] ──HTTPS (encrypted)──> [Bank]
                                ↓
                        Read passwords
                        Session cookies
                        CC numbers

User browser URL bar : http://bank.com (NO padlock 🔒)
→ User thinks secure (content looks right)
→ Actually plaintext to attacker

// Outil attaque

sslstrip : Outil MitM automatisant SSL stripping (Moxie Marlinspike).

// Mitigation SSL Stripping

  • HSTS (HTTP Strict Transport Security) :
    - Header server : Strict-Transport-Security: max-age=31536000
    - Browser force HTTPS toujours (ignore HTTP links)
    - Empêche downgrade attacks
  • HTTPS Everywhere : Extension browser force HTTPS
  • User awareness : Vérifier padlock 🔒 URL bar
  • Certificate pinning : App mobile hardcode expected certificate
🔐
HSTS = défense clé SSL stripping Sites bancaires/sensibles DOIVENT implémenter HSTS.
HSTS Preload List : Chrome/Firefox hardcode liste sites HTTPS-only (impossible downgrade).
Submit : hstspreload.org
VLAN attack

VLAN Hopping

VLAN Hopping = Attaquant accède VLANs interdits → bypass segmentation réseau.

// Méthode 1 : Switch Spoofing

Principe : Attaquant simule switch → établit trunk 802.1Q.

Workflow :

  1. Attaquant branché port access switch
  2. Envoie DTP (Dynamic Trunking Protocol) frames
  3. Switch croit attaquant = autre switch → établit trunk
  4. Attaquant reçoit trafic TOUS VLANs (tagged 802.1Q)
  5. Attaquant accède VLANs segmentés (ex: VLAN Finance, VLAN HR)

Mitigation :

  • Disable DTP : switchport mode access (force access, not auto-negotiate)
  • Disable unused ports : shutdown

// Méthode 2 : Double Tagging

Principe : Exploiter 802.1Q double-tagging → frame saute VLANs.

Workflow :

  1. Attaquant VLAN 10 (native VLAN trunk)
  2. Craft frame double 802.1Q tags :
    - Outer tag : VLAN 10 (native, stripped first switch)
    - Inner tag : VLAN 20 (target)
  3. First switch strips outer tag VLAN 10 → forward trunk
  4. Second switch voit inner tag VLAN 20 → deliver VLAN 20
  5. Attaquant injecte trafic VLAN 20 (bypass segmentation)

Mitigation :

  • Native VLAN unused : Set native VLAN = VLAN inutilisé (ex: VLAN 999)
  • Explicit tagging : vlan dot1q tag native (force tag even native)
Session attack

Replay Attack

Replay Attack = Capturer transmission légitime → rejouer (replay) plus tard → authentification/action non autorisée.

// Scénario classique

1. Légitime transaction :
User → Server : "Transfer $100 Bob" (packet captured attacker)
Packet contains : Auth token, transaction data

2. Attacker replay packet (later) :
Attacker → Server : [same packet replayed]
Server : "Valid auth token → Execute transfer $100 Bob"

→ Transaction exécutée 2× (user lose $200 instead $100)

// Exemples replay attacks

  • Authentication : Replay login packet (hash password) → gain access
  • Session cookies : Replay cookie → session hijacking
  • Kerberos tickets : Replay ticket → authentification reuse
  • Wireless (WEP) : Capture encrypted packet → replay injection

// Mitigation Replay Attacks

1. Timestamps

  • Inclure timestamp packet
  • Server rejette packets trop anciens (>5 min)
  • ⚠️ Nécessite clocks synchronisées (NTP)

2. Nonces (Numbers Used Once)

  • Server génère nonce unique chaque session
  • Client include nonce request
  • Server accepte nonce 1× only
  • ✅ Défense robuste

3. Sequence Numbers

  • Packets numérotés séquentiellement
  • Server rejette duplicates ou out-of-order
  • Ex: TCP sequence numbers, IPsec

4. Session Tokens Expiration

  • Tokens courte durée vie (15 min)
  • Replay ancien token = expiré (rejected)
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : ARP Poisoning = Layer 3 attack
FAUX. ARP Poisoning = Layer 2 attack (Data Link).

ARP résout IP (L3) → MAC (L2) MAIS protocole opère Layer 2.
Attack falsifie MAC addresses (Layer 2).

Mitigation : DAI (switch Layer 2 feature)
❌ Piège #2 : SSL Stripping = casser chiffrement HTTPS
FAUX. SSL Stripping = downgrade HTTPS → HTTP.

Pas "casser" chiffrement (impossible pratiquement).
Empêche établissement connexion chiffrée (man-in-the-middle).

User ↔ Attacker = HTTP, Attacker ↔ Server = HTTPS
❌ Piège #3 : Replay attack nécessite déchiffrer packet
FAUX. Replay attack = rejouer packet captured (même chiffré).

Pas besoin déchiffrer contenu.
Replay packet entier (ciphertext) → server accepte si valide.

Mitigation : Nonces, timestamps (pas chiffrement seul)
Résumé

🔑 Ce que tu dois retenir absolument

→MitM : Attaquant invisible entre deux parties. Intercept, eavesdrop, modify trafic. Objectifs : passwords, session hijacking, data modification
→ARP Poisoning : Layer 2 attack, falsifier ARP replies → redirect trafic attaquant. Gratuitous ARP. Mitigation : DAI (Dynamic ARP Inspection), port security
→DNS Spoofing : Falsifier DNS responses → redirect users sites malveillants. Cache poisoning (race condition). Mitigation : DNSSEC, DoH/DoT
→SSL Stripping : Downgrade HTTPS → HTTP. User ↔ Attacker HTTP (plaintext), Attacker ↔ Server HTTPS. Mitigation : HSTS (force HTTPS always)
→VLAN Hopping : Switch spoofing (DTP trunk) ou double-tagging (802.1Q). Bypass segmentation. Mitigation : Disable DTP, native VLAN unused
→Replay Attack : Capture packet légitime → rejouer. Mitigation : Nonces (use once), timestamps, sequence numbers, session expiration
