NAT review

NAT — Rappel fonctionnement

NAT (Network Address Translation) = Traduit IP privées (RFC 1918) → IP publiques (Internet routable).

// Principe NAT (rappel Cours I20)

Workflow NAT :

PC private : 192.168.1.50 (inside local)
Router NAT translates → 203.0.113.10 (inside global)
Internet sees : 203.0.113.10 (public IP)

Return traffic :
Internet → 203.0.113.10
Router NAT translates → 192.168.1.50
PC receives packet

// NAT types

TypeDescriptionMapping
Static NAT 1:1 permanent mapping 192.168.1.10 ↔ 203.0.113.10 (always)
Dynamic NAT 1:1 from pool (temporary) 192.168.1.50 → 203.0.113.15 (while active)
PAT / Overload Many:1 with port translation 192.168.1.x:port → 203.0.113.1:port

// NAT terminology

  • Inside local : Private IP (LAN side) — 192.168.1.50
  • Inside global : Public IP (translated) — 203.0.113.10
  • Outside local : Internet destination (as seen inside) — 8.8.8.8
  • Outside global : Internet destination (actual) — 8.8.8.8
Address depletion

Pool NAT épuisé — No addresses available

Pool NAT épuisé = Dynamic NAT pool : all public IPs allocated → new connections fail.

// Scénario pool épuisé

NAT configuration :

Pool : 203.0.113.10 - 203.0.113.15 (6 public IPs)
Inside devices : 100 PCs (192.168.1.0/24)

Problem :
First 6 PCs connect Internet → Pool fully allocated
7th PC tries connect → No IP available → Connection FAILS

// Symptômes pool épuisé

  • Selective connectivity : Some users work, others can't reach Internet
  • Intermittent failures : Works if other users disconnect (IP freed)
  • Syslog messages : "NAT pool exhausted" warnings
  • First users OK : Early connections succeed, later connections fail

// Diagnostic pool épuisé

Router# show ip nat statistics

Total active translations: 6 (6 dynamic, 0 static, 0 extended)
Peak translations: 6
Outside interfaces: GigabitEthernet0/0
Inside interfaces: GigabitEthernet0/1
Hits: 4523  Misses: 127

Dynamic mappings:
-- Inside Source
[Id: 1] access-list 1 pool PUBLIC refcount 6
pool PUBLIC: netmask 255.255.255.0
start 203.0.113.10 end 203.0.113.15
type generic, total addresses 6, allocated 6 (100%)

→ Pool 100% allocated (6/6 used) → No capacity

// Résolution pool épuisé

Option 1 : Expand pool (add IPs)

Router(config)# ip nat pool PUBLIC 203.0.113.10 203.0.113.30 netmask 255.255.255.0

→ Increase pool 6 → 21 IPs (203.0.113.10-30)

Option 2 : Use PAT/Overload (RECOMMENDED)

Router(config)# ip nat inside source list 1 interface GigabitEthernet0/0 overload

→ PAT (Port Address Translation) : Many:1 mapping
→ 65,535 ports available → Supports thousands simultaneous connections
→ Solves exhaustion problem ✅

Option 3 : Reduce timeout (aggressive)

Router(config)# ip nat translation timeout 300

→ Timeout 300 sec (5 min) instead default 24h
→ IPs freed faster (BUT can break long connections)
💡
PAT/Overload = solution moderne Dynamic NAT pool = legacy (requires many public IPs, expensive).
PAT (Overload) = standard modern (1 public IP → thousands connections).
Home routers : PAT default (single ISP IP serves whole household).
Interface direction

Inside/Outside inversé — Wrong NAT direction

Inside/Outside inversé = Interfaces configured wrong direction → NAT translation fails OR incorrect direction.

// Principe Inside vs Outside

Correct configuration :

Inside interface (LAN side) :
  interface GigabitEthernet0/1
    ip nat inside
  → Connected to private network (192.168.1.0/24)

Outside interface (WAN side) :
  interface GigabitEthernet0/0
    ip nat outside
  → Connected to Internet (public IP)

// Scénario inversé

INCORRECT configuration :

interface GigabitEthernet0/1 (LAN side)
  ip nat outside (WRONG - should be inside)

interface GigabitEthernet0/0 (WAN side)
  ip nat inside (WRONG - should be outside)

→ NAT translates wrong direction
→ Internet IPs translated → private (backward)
→ Private IPs sent untranslated → Internet (fail)

// Symptômes inside/outside inversé

  • No Internet connectivity : Private IPs sent Internet (not routable)
  • NAT table empty/wrong : No translations created OR backward entries
  • ISP rejects packets : RFC 1918 private IPs on Internet (dropped)
  • Configuration looks correct : NAT rules OK, BUT interfaces wrong

// Diagnostic inside/outside

Router# show ip nat statistics

Inside interfaces: GigabitEthernet0/0 (WAN - WRONG!)
Outside interfaces: GigabitEthernet0/1 (LAN - WRONG!)

→ Interfaces reversed

Router# show ip interface brief
Gi0/0    203.0.113.1 (public IP - should be OUTSIDE)
Gi0/1    192.168.1.1 (private IP - should be INSIDE)

→ Mismatch detected

// Résolution inside/outside inversé

Step 1 : Remove incorrect config
Router(config)# interface GigabitEthernet0/0
Router(config-if)# no ip nat inside

Router(config)# interface GigabitEthernet0/1
Router(config-if)# no ip nat outside

Step 2 : Apply correct direction
Router(config)# interface GigabitEthernet0/0
Router(config-if)# ip nat outside (WAN side)

Router(config)# interface GigabitEthernet0/1
Router(config-if)# ip nat inside (LAN side)

Step 3 : Clear NAT translations
Router# clear ip nat translation *

→ Reset NAT table, rebuild correct translations
🎯
Question piège examen "NAT configured, but private IPs appear on Internet. MOST likely cause ?"
A) NAT pool exhausted
B) Inside/Outside interfaces reversed
C) ACL blocking
D) Routing problem

Réponse : B) Inside/Outside interfaces reversed
Private IPs on Internet = NAT not translating (wrong direction).
Port Address Translation

Overload PAT mal configuré — Port translation issues

PAT (Port Address Translation) = Overload : Many inside IPs → 1 public IP using port translation.

// Principe PAT/Overload

PAT workflow :

PC-A : 192.168.1.50:5000 → Internet
PC-B : 192.168.1.51:5000 → Internet

Router NAT translates :
192.168.1.50:5000 → 203.0.113.1:10001
192.168.1.51:5000 → 203.0.113.1:10002

→ Same public IP (203.0.113.1)
→ Different ports (10001 vs 10002)
→ Unique mapping each connection ✅

// Problème #1 : Overload keyword manquant

INCORRECT :
Router(config)# ip nat inside source list 1 interface Gi0/0

→ No "overload" keyword
→ Dynamic NAT (1:1) instead PAT (many:1)
→ Only 1 connection works (single IP no port translation)

CORRECT :
Router(config)# ip nat inside source list 1 interface Gi0/0 overload

→ PAT enabled ✅
→ Thousands connections supported

// Problème #2 : Port exhaustion (rare)

Scenario :
Thousands simultaneous connections → Same destination IP:port
Example : 10,000 users → same web server 8.8.8.8:80

Ports available : 1024-65535 = ~64,000
If ALL users → same destination → Port exhaustion possible

→ Very rare production (distributed destinations)
→ Mitigation : Multiple public IPs, load balancing

// Problème #3 : ACL pas match traffic

Router(config)# access-list 1 permit 192.168.1.0 0.0.0.255
Router(config)# ip nat inside source list 1 interface Gi0/0 overload

Problem :
User subnet : 192.168.2.0/24 (different!)
ACL 1 : 192.168.1.0/24
→ ACL doesn't match user traffic
→ No NAT translation → Connection fails

Fix :
Router(config)# access-list 1 permit 192.168.2.0 0.0.0.255

→ ACL matches actual subnet ✅

// Diagnostic PAT/Overload

Router# show ip nat translations

Pro  Inside global      Inside local       Outside local      Outside global
tcp  203.0.113.1:10001  192.168.1.50:5000  8.8.8.8:80          8.8.8.8:80
tcp  203.0.113.1:10002  192.168.1.51:5000  8.8.8.8:80          8.8.8.8:80

→ Multiple connections same public IP (203.0.113.1)
→ Different ports (10001, 10002) = PAT working ✅

If table empty :
→ Check ACL matches traffic
→ Check "overload" keyword present
→ Check inside/outside interfaces correct
Troubleshooting

Connexions qui échouent — NAT troubleshooting workflow

// Symptômes généraux NAT failure

  • No Internet access : Ping external IPs fail (8.8.8.8 timeout)
  • Local network works : Can ping gateway, local devices OK
  • DNS may work : Can resolve names (if DNS server local)
  • Traceroute stops gateway : Doesn't leave local network

// Diagnostic workflow complet

Step 1 : Verify NAT configuration
Router# show running-config | section nat
→ Check inside/outside interfaces
→ Check ACL matches traffic
→ Check pool OR overload present

Step 2 : Check NAT statistics
Router# show ip nat statistics
→ Inside/outside interfaces correct ?
→ Pool exhausted (100% allocated) ?
→ Hits vs Misses ratio (high misses = problem)

Step 3 : Check active translations
Router# show ip nat translations
→ Translations created when user connects ?
→ If empty = NAT not triggering (ACL, interfaces)

Step 4 : Test connectivity
Router# ping 8.8.8.8 source GigabitEthernet0/1
→ Ping from inside interface
→ Creates NAT entry ? Check translations again

Step 5 : Debug (last resort)
Router# debug ip nat
→ Shows real-time NAT translations
→ WARNING : High CPU usage (disable quickly)

// Common NAT problems summary

ProblemSymptomFix
Pool exhausted Some users work, others fail Expand pool OR use PAT overload
Inside/Outside reversed Private IPs on Internet Swap interface directions
Missing overload Only 1 user can connect Add "overload" keyword PAT
ACL mismatch No translations created Fix ACL match actual subnet
No default route NAT OK, routing fails Add default route 0.0.0.0/0
🔍
Clear NAT translations after changes After NAT config changes : clear ip nat translation *
Removes stale entries, forces rebuild.
Active connections may drop (warn users before clearing production).
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : PAT = same as NAT pool
FAUX. PAT (Overload) ≠ Dynamic NAT pool.

Dynamic NAT : 1:1 mapping (1 inside IP → 1 public IP, requires pool).
PAT/Overload : Many:1 mapping (many inside IPs → 1 public IP + ports).

PAT requires "overload" keyword.
Examen : "Overload" = PAT (port translation), not just dynamic NAT.
❌ Piège #2 : NAT inside = LAN interface always
NUANCE. Inside/Outside = NAT perspective, not physical location.

Typically : Inside = LAN (private), Outside = WAN (public).
MAIS : Can reverse (ex: NAT servers outbound connections).

Inside = source of traffic needing translation.
Outside = destination side (Internet typically).
Examen : Follow NAT flow direction (where translation originates).
❌ Piège #3 : Pool exhausted = router problem
FAUX. Pool exhausted = design/capacity problem, not router fault.

Router working correctly (translating within pool limits).
Problem : Pool too small for user count (undersized).

Solution : Expand pool (more public IPs) OR use PAT (better scalability).
Examen : Pool exhausted = capacity planning issue, not technical failure.
Résumé

🔑 Ce que tu dois retenir absolument
