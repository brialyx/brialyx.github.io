Processus ITIL

Change Management — Pourquoi contrôler changements ?

🚦
Analogie — travaux routiers Changement réseau = travaux route. Pas planification = chaos (embouteillages, accidents). Change Management = autorisation travaux, fermeture voie heures creuses, plan détour, équipe standby si problème.

Change Management = processus formel contrôlant TOUS changements infrastructure IT (réseau, serveurs, apps). Minimiser risques pannes, downtime non planifié.

// Problème changements non contrôlés

  • ❌ Downtime imprévu : Ingénieur change config switch → casse réseau production
  • ❌ Conflits changements : 2 équipes modifient même firewall simultanément → chaos
  • ❌ Pas rollback plan : Changement échoue, personne sait comment revenir arrière
  • ❌ Pas communication : Users pas informés maintenance → tickets support explosent
  • ❌ Pas traçabilité : Réseau cassé, personne sait qui changé quoi

// Bénéfices Change Management

  • ✅ Risques minimisés : Revue changements AVANT exécution
  • ✅ Coordination : Évite conflits entre équipes
  • ✅ Communication : Users/business informés avance
  • ✅ Traçabilité : Historique complet changements (audit)
  • ✅ Rollback rapide : Plan B documenté si problème
⚠️
Stat industry : 80% pannes causées par changements Majorité incidents production = changements mal planifiés/exécutés. Change Management = réduire drastiquement downtime évitable.
Demande formelle

RFC — Request for Change

RFC (Request for Change) = document formel demandant autorisation effectuer changement infrastructure. Base processus Change Management.

// Contenu RFC typique

  • Numéro RFC : RFC-2024-0156 (tracking unique)
  • Demandeur : Nom ingénieur/équipe
  • Date soumission : 2024-01-15
  • Description changement : "Upgrade firmware core switches Cisco v17.3 → v17.6"
  • Justification business : "Corriger CVE-2023-12345 vulnérabilité critique"
  • Impact : "Redémarrage switches → downtime 15 min par switch"
  • Risques : "Boot failure si firmware corrompu, incompatibilité config"
  • Systèmes affectés : SW-CORE-01, SW-CORE-02
  • Fenêtre maintenance demandée : "Dimanche 2024-01-21, 02:00-06:00 AM"
  • Plan implémentation : Steps détaillés (pré-checks, backup config, upgrade, tests)
  • Rollback plan : "Restaurer firmware v17.3 depuis backup TFTP"
  • Tests validation : "Ping tests, vérifier logs, monitoring 2h post-change"
  • Communication : "Email users vendredi, notification maintenance"

// Exemple RFC simplifié

RFC-2024-0156
Demandeur: John Doe (Network Team)
Date: 2024-01-15

Changement: Upgrade firmware core switches
Justification: Patch CVE-2023-12345 (RCE critical)
Impact: Downtime 15 min/switch, redondance maintenue
Risques: Boot failure (1% probabilité)

Fenêtre: Dimanche 2024-01-21, 02:00-06:00 AM

Plan:
1. Backup configs (TFTP)
2. Upgrade SW-CORE-02 (standby) first
3. Tests validation 30 min
4. Upgrade SW-CORE-01 (active)
5. Monitoring 2h post-change

Rollback: TFTP restore firmware v17.3 (15 min)

// Types changements

Standard Change : Pré-approuvé, faible risque, répétitif (ex: ajout user, backup hebdomadaire). Pas besoin CAB approval à chaque fois.

Normal Change : Nécessite RFC + approbation CAB. Majorité changements.

Emergency Change : Urgent (fix incident critique). Processus accéléré, approbation post-mortem possible.

Validation

Processus approbation — CAB

CAB (Change Advisory Board) = comité évaluant/approuvant RFCs. Assure changements alignés business, risques acceptables.

// Membres CAB typiques

  • Change Manager : Préside CAB, coordonne processus
  • Ingénieurs techniques : Network, sysadmin, sécurité (expertise technique)
  • Service owners : Responsables apps/services affectés
  • Business representatives : Managers business (impact métier)
  • Risk/Compliance (optionnel) : Si changement sensible

// Workflow approbation

  1. Soumission RFC : Ingénieur soumet RFC (formulaire, ticket)
  2. Revue initiale : Change Manager vérifie RFC complet
  3. CAB meeting : Hebdomadaire (ex: mardi 10h), revue RFCs soumis
  4. Discussion : Questions techniques, risques, alternatives
  5. Décision :
    - Approved : Changement autorisé, planifié
    - Rejected : Refusé (risques trop élevés, justification insuffisante)
    - Deferred : Reporté (besoin info additionnelle)
  6. Planification : Si approuvé, schedule dans calendrier changements
  7. Exécution : Ingénieur implémente pendant fenêtre maintenance
  8. Post-implementation review : Vérifier succès, documenter leçons
Timeline exemple RFC-2024-0156 :

Lundi 2024-01-15 : Soumission RFC
Mardi 2024-01-16 10h : CAB meeting → Approved
Mercredi 2024-01-17 : Planning détaillé, communication users
Vendredi 2024-01-19 : Email notification maintenance dimanche
Dimanche 2024-01-21 02h-06h : Exécution changement
Lundi 2024-01-22 : Post-implementation review → Success
🎯
Question fréquente examen "Comité évaluant/approuvant changements infrastructure ?"

Réponse : CAB (Change Advisory Board)
Timing

Fenêtre de maintenance (Maintenance Window)

Maintenance Window = période planifiée AUTORISÉE effectuer changements production. Minimise impact business.

// Caractéristiques fenêtre maintenance

  • Heures creuses : Généralement nuit/weekend (faible activité business)
  • Pré-planifiée : Calendrier publié avance (users informés)
  • Durée définie : Ex: 02:00-06:00 AM (4h window)
  • Backup window : Si changement échoue, rollback avant fin fenêtre

// Fenêtres typiques entreprise

Type fenêtreTimingFréquenceUsage
Weekly standard Dimanche 02:00-06:00 AM Hebdomadaire Changements routine (patches, upgrades)
Monthly major 1er samedi mois 00:00-08:00 AM Mensuelle Changements majeurs (migrations, reconfigs)
Emergency On-demand (24/7) Ad-hoc Incidents critiques uniquement

// Blackout periods

Blackout period = période INTERDICTION changements (activité business critique).

Exemples :

  • Black Friday (retail)
  • Fin d'année fiscale (finance)
  • Périodes examens (éducation)
  • Événements marketing majeurs
🚫
Respecter fenêtres = discipline critique Changement HORS fenêtre maintenance = violation processus grave. Exception uniquement emergency approuvé. Changer production 14h mardi = carrière-limiting move.
Plan B

Rollback Plan — Retour arrière

Rollback Plan = procédure documentée restaurer état AVANT changement si problème. OBLIGATOIRE chaque RFC.

// Éléments rollback plan

  • Trigger conditions : Quand rollback ? (ex: perte connectivité >5 min, erreurs critiques logs)
  • Steps détaillés : Procédure exacte (commandes, ordre)
  • Durée estimée : Temps nécessaire rollback (ex: 15 min)
  • Backups requis : Configs, firmwares, data sauvegardés AVANT changement
  • Responsable : Qui décide/exécute rollback
  • Communication : Notification teams si rollback activé

// Exemple rollback plan

ROLLBACK PLAN — RFC-2024-0156

Trigger:
- Switch pas boot après upgrade (>10 min)
- Perte connectivité réseau production
- CPU >90% sustained 5 min post-upgrade

Procédure rollback (15 min) :
1. Console switch (out-of-band mgmt)
2. Boot mode ROMMON
3. TFTP restore firmware v17.3
  tftp://10.1.1.100/sw-core-01-v17.3.bin
4. Restore config backup
  copy tftp://10.1.1.100/sw-core-01-config.txt running-config
5. Reload switch
6. Verify boot, connectivity tests

Backups (pre-change) :
- Firmware v17.3 : TFTP server 10.1.1.100
- Config backup : sw-core-01-2024-01-20.txt

Responsable décision rollback: Network Lead on-call

// Go/No-Go decision

Pendant changement, décision critique points :

  • Go : Changement progresse bien → continuer
  • No-Go (rollback) : Problème détecté → activer rollback plan immédiatement
✅
Test rollback plan AVANT changement Best practice : Tester rollback plan environnement lab AVANT production. Rollback échoue = double catastrophe.
Base configuration

CMDB — Configuration Management Database

CMDB (Configuration Management Database) = base données centralisée stockant informations TOUS assets IT et leurs relations. Fondation Change Management.

// Contenu CMDB

  • Configuration Items (CIs) : Équipements, software, services (switches, serveurs, apps, liens WAN...)
  • Attributs CIs : Hostname, IP, modèle, OS version, localisation, propriétaire
  • Relations/Dépendances : App X dépend Serveur Y dépend Switch Z
  • Historique changements : Qui changé quoi quand sur chaque CI
  • État actuel : Production, dev, maintenance, decommissioned

// Rôle CMDB dans Change Management

Impact analysis : RFC modifie Switch A → CMDB montre quels serveurs/apps affectés.

Éviter conflits : CMDB indique si autre changement planifié même CI.

Traçabilité : Historique complet modifications chaque équipement.

Compliance : Audits vérifient changements autorisés vs CMDB.

// Exemple relation CMDB

CI: Application "SAP Production"
├─ Dépend: Serveur "SRV-SAP-01"
│  ├─ Dépend: Switch "SW-CORE-01" Port Gi0/12
│  │  └─ Dépend: Lien WAN "WAN-HQ-DC1"
│  └─ Dépend: Storage "SAN-PROD-01"
└─ Dépend: Database "DB-SAP-PROD"

RFC upgrade SW-CORE-01 firmware :
CMDB Impact Analysis →
- Switch SW-CORE-01 affecté
- Serveur SRV-SAP-01 affecté (connexion switch)
- Application SAP Production affectée
→ Notification équipe SAP nécessaire avant changement

// Outils CMDB

ServiceNow : Leader marché (enterprise)

Device42 : Auto-discovery réseau

NetBox : Open-source IPAM/DCIM (basique CMDB)

BMC Remedy : ITSM suite classique

🔗
CMDB ≠ inventaire simple Inventaire = liste équipements. CMDB = inventaire + relations/dépendances. Relations = valeur CMDB (impact analysis impossible sans).

Résumé

🔑 Ce que tu dois retenir absolument

→Change Management : Processus formel contrôler TOUS changements infra. Minimise risques pannes (80% incidents = changements mal gérés)
→RFC : Request for Change (doc formel). Contenu : description, justification, impact, risques, plan implémentation, rollback plan, tests validation
→CAB : Change Advisory Board (comité approbation). Revue RFCs, décision Approved/Rejected/Deferred. Types : Standard (pré-approuvé), Normal (CAB), Emergency (accéléré)
→Fenêtre maintenance : Période autorisée changements (heures creuses, nuit/weekend). Blackout periods = interdiction (activité business critique)
→Rollback plan : Procédure restaurer état avant changement. Obligatoire chaque RFC. Trigger conditions, steps, durée, backups. Test rollback lab AVANT prod
→CMDB : Configuration Management Database. CIs + attributs + relations/dépendances + historique. Impact analysis changements (qui affecté ?)
