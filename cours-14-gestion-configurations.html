Configuration Management

Pourquoi gérer les configurations ?

📄
Analogie — code source software Configs réseau = code source entreprise. Developer code sans Git = suicide (perte travail, pas rollback). Réseau sans config management = idem (changement casse réseau, pas backup → panic restore).

Configuration management = Processus contrôler/tracker/sauvegarder configurations équipements réseau. Essentiel stabilité/sécurité/compliance.

// Problèmes SANS config management

  • ❌ Perte config : Switch crash → config perdue → rebuild from scratch (heures)
  • ❌ Changements non documentés : "Qui changé ACL hier ?" → mystère
  • ❌ Pas rollback : Changement casse réseau → impossible revenir version précédente
  • ❌ Configuration drift : 100 switches, configs divergent lentement → chaos
  • ❌ Compliance violations : Audits exigent backup configs (PCI-DSS, SOX)
  • ❌ Troubleshooting impossible : "Config était quoi avant problème ?" → pas historique

// Bénéfices config management

  • ✅ Disaster recovery rapide : Device crashé → restore config backup (minutes)
  • ✅ Version control : Historique complet changements (diff between versions)
  • ✅ Rollback facile : Changement problématique → revert version précédente
  • ✅ Audit trail : Qui changé quoi quand (compliance)
  • ✅ Consistency : Templates → configs cohérentes multi-devices
  • ✅ Drift detection : Alertes si device diverge standard
⚠️
80% pannes réseau = changements config mal gérés Stat industry : Majorité downtime causée humains (config errors). Config management = assurance qualité changements + rollback rapide si problème.
Backup strategies

Backup des configurations

// Types configs équipements

Running-config : Configuration active RAM (volatile, perdue si reboot sans save).

Startup-config : Configuration NVRAM (persistante, chargée au boot).

Commandes Cisco :

show running-config    # Voir config active (RAM)
show startup-config     # Voir config boot (NVRAM)
copy running-config startup-config  # Sauvegarder RAM → NVRAM
write memory             # Alias "wr" (même fonction)

// Méthodes backup configs

1. TFTP (Trivial File Transfer Protocol)

  • Protocole simple (UDP 69)
  • Pas authentification (⚠️ sécurité faible)
  • Usage : Backup/restore configs, firmware uploads
Backup via TFTP :

copy running-config tftp:
  Address: 10.1.1.100
  Destination filename: SW01-config-2024-02-27.txt

→ Config sauvegardée serveur TFTP 10.1.1.100

2. SCP (Secure Copy Protocol)

  • SSH-based (chiffré)
  • ✅ Sécurisé (vs TFTP texte clair)
  • Authentification requise
Backup via SCP :

copy running-config scp:
  Address: 10.1.1.100
  Username: admin
  Password: ******
  Destination filename: SW01-backup.cfg

3. FTP/SFTP

  • FTP = texte clair (⚠️ pas sécurisé)
  • SFTP = SSH-based (✅ sécurisé)

4. Git / Version Control

  • ✅ Best practice moderne
  • Historique complet (commits)
  • Diff between versions
  • Branching (test configs isolés)
  • Collaboration (pull requests, reviews)

// Automatisation backups

Manuelle : Admin SSH device → copy running tftp (fastidieux, oublis).

Automatique : Scripts cron/Ansible backup TOUS devices nightly.

Exemple script backup automatique (pseudo-code) :

FOR each device IN inventory:
  SSH connect device
  Execute "show running-config"
  Save output → Git repo
  Git commit "Backup DEVICE 2024-02-27"
  Git push

→ Backups quotidiens automatiques → Git historique complet
🔄
Backup automatique = NON-NÉGOCIABLE Backup manuel = oublis garantis. Automatisation (cron, Ansible, RANCID) = backups quotidiens sans effort. Coût automation << coût downtime config perdue.
Version control

Versioning et Rollback

Versioning = Tracking historique changements configs (comme Git code source).

// Git pour configs réseau

Workflow Git configs :

  1. Repository initial : Créer Git repo (ex: network-configs/)
  2. Structure :
    network-configs/
    ├─ switches/
    │  ├─ SW-CORE-01.cfg
    │  ├─ SW-CORE-02.cfg
    ├─ routers/
    │  ├─ RTR-WAN-01.cfg
    ├─ firewalls/
       ├─ FW-DMZ-01.cfg
  3. Backup quotidien : Script pull configs → commit Git
  4. Commits : Historique changements (qui, quand, diff)
  5. Branches : Test configs isolés (merge après validation)

// Avantages Git versioning

  • ✅ Historique complet : Voir TOUTES versions précédentes
  • ✅ Diff visualisation : git diff montre changements exacts
  • ✅ Rollback facile : git checkout commit-abc123 → version précédente
  • ✅ Blame tracking : git blame → qui changé ligne config
  • ✅ Branching test : Tester changements branch avant merge production

// Exemple diff Git

$ git diff SW-CORE-01.cfg

- access-list 100 permit ip 10.0.10.0 0.0.0.255 any
+ access-list 100 deny ip 10.0.10.0 0.0.0.255 any
+ access-list 100 permit ip 10.0.20.0 0.0.0.255 any

→ Changement visible : ACL modifié (deny VLAN 10, permit VLAN 20)

// Rollback config

Scénario : Changement config casse réseau → rollback version précédente.

Rollback Git :

$ git log SW-CORE-01.cfg  # Voir historique commits
commit abc123 - "Added VLAN 50" (2024-02-26)
commit def456 - "Working config" (2024-02-25) ← Version stable

$ git checkout def456 SW-CORE-01.cfg  # Restore version stable
$ scp SW-CORE-01.cfg admin@SW-CORE-01:/startup-config

→ Config restaurée version 2024-02-25 (avant problème)
📦
Tools config management RANCID (Really Awesome New Cisco confIg Differ) : Open-source, backup + diff automatique
Oxidized : Moderne alternative RANCID, multi-vendor
Ansible : Automation + config management
NetBox : IPAM + config storage (API-driven)
Standardization

Templates et Standardisation

Config templates = Configurations standards réutilisables (base commune multi-devices).

// Principe templates

Éviter réinventer roue chaque device → Template base + variables spécifiques.

Exemple template switch access :

Template : switch-access-base.j2 (Jinja2)

hostname {{ hostname }}
!
vlan 10
 name Data
vlan 20
 name Voice
vlan 99
 name Management
!
interface range GigabitEthernet1/0/1-48
 switchport mode access
 switchport access vlan 10
 switchport voice vlan 20
 spanning-tree portfast
!
interface Vlan99
 ip address {{ mgmt_ip }} 255.255.255.0
 no shutdown

Variables par device :
SW-01: hostname=SW-ACCESS-01, mgmt_ip=192.168.99.11
SW-02: hostname=SW-ACCESS-02, mgmt_ip=192.168.99.12

// Avantages templates

  • ✅ Consistency : Tous devices même baseline (évite errors typo)
  • ✅ Rapidité déploiement : 100 switches configurés minutes (vs heures manuel)
  • ✅ Best practices intégrés : Template = config validated (sécurité, performance)
  • ✅ Changements globaux faciles : Update template → redeploy tous devices

// Outils templating

Jinja2 : Moteur templating Python (utilisé Ansible).

Ansible : Automation déploiement templates multi-devices.

Terraform : Infrastructure as Code (IaC) configs réseau cloud.

Déploiement Ansible + Jinja2 :

ansible-playbook deploy-switches.yml

→ Ansible :
  1. Lit inventaire devices (100 switches)
  2. Génère configs depuis template (variables per-device)
  3. Déploie configs SSH automatiquement
  4. Vérifie deployment OK

→ 100 switches configurés identiquement en 10 minutes
Drift detection

Configuration Drift — Dérive de configuration

Configuration drift = Divergence config device vs standard attendu (configs dérivent lentement au fil du temps).

// Causes drift

  • Changements manuels : Admin SSH device, modifie config ad-hoc (pas via template)
  • Troubleshooting temporaire : "shutdown interface test" → oubli réactiver
  • Changements non documentés : Config modifiée, pas commit Git
  • Multiples admins : Chacun modifie configs différemment

// Problèmes drift

  • ❌ Inconsistency : 100 switches, 50 configs différentes → chaos troubleshooting
  • ❌ Vulnérabilités : Device pas patché, config dépassée → exploit
  • ❌ Compliance violations : Audits échouent (configs pas standards)
  • ❌ Bugs subtils : "Switch 42 behave differently" → config divergé

// Détection drift

Automated compliance checks : Comparer running-config vs baseline/template.

Workflow drift detection :

1. Baseline expected : Template standard config
2. Actual config : Pull running-config device
3. Diff comparison : Baseline vs Actual
4. Alert si différences détectées

Exemple diff drift :
Expected : logging host 10.1.1.100
Actual :    [missing line]
→ DRIFT DETECTED : Syslog server config manquante

// Remédiation drift

Options :

1. Alerte admin : Notification drift détecté → admin investigate/fixe manuel.

2. Auto-remediation : Système automatiquement restore baseline config (risky, besoin validation).

3. Enforcement : Immutable infrastructure (configs read-only prod, changements via CI/CD uniquement).

// Tools drift detection

Ansible : Playbooks vérifient compliance, remedient drift.

Chef / Puppet : Infrastructure as Code (enforce desired state).

RANCID / Oxidized : Détectent changements configs, alertent drift.

⚠️
Drift = entropy naturelle infrastructure Sans monitoring actif, configs dérivent inévitablement (loi entropie). Automated drift detection = hygiene check régulier → évite divergence catastrophique.
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : Running-config persistant après reboot
FAUX. Running-config = RAM (volatile).

Reboot SANS save → running-config perdu.
Startup-config = NVRAM (persistent).

Sauvegarder : copy running-config startup-config
❌ Piège #2 : TFTP sécurisé (chiffré)
FAUX. TFTP = texte clair (UDP 69, pas chiffrement).

TFTP simple MAIS pas sécurisé (configs exposées sniffing).
Sécurisé : SCP (SSH-based) ou SFTP

Production : Préférer SCP vs TFTP
❌ Piège #3 : Configuration drift = bug équipement
FAUX. Drift = divergence config vs baseline (pas bug hardware).

Causes drift : Changements manuels, modifications non documentées
Pas = bug device firmware

Remédiation drift = restore baseline config (pas RMA hardware)
