Avant de commencer

Le problème — limites des VLANs traditionnels

🏙️
Analogie — réseau postal VLAN traditionnel = codes postaux limités à 4096 valeurs (12 bits). VxLAN = nouveau système avec 16 millions de codes (24 bits). Tu peux maintenant isoler des millions de clients dans le même datacenter au lieu de seulement 4000.

VLANs traditionnels (802.1Q) ont des limitations critiques pour les datacenters modernes :

// Limitation 1 : Seulement 4096 VLANs

Le tag VLAN = 12 bits → 2^12 = 4096 VLANs maximum (0-4095).

  • Datacenter cloud avec 10,000 clients → impossible d'isoler chacun
  • Multi-tenancy (SaaS, IaaS) → besoin millions de segments isolés
  • 4096 VLANs épuisés rapidement dans grands réseaux

// Limitation 2 : Pas de mobilité entre sites

VLANs limités au même domaine Layer 2 (même switch ou switches interconnectés directement).

  • VM migre d'un datacenter à l'autre → perd son VLAN
  • Stretching Layer 2 entre datacenters = complexe et fragile
  • Pas de multi-site transparent

// Limitation 3 : Dépendance au réseau physique

VLANs = configuration switch par switch, pas flexible, pas programmable.

⚠️
Datacenters modernes = impossible avec VLANs Cloud providers (AWS, Azure, GCP), VMware NSX, OpenStack, conteneurs Kubernetes → tous nécessitent plus de 4096 segments isolés. VLANs traditionnels = inadaptés.
Concept

VxLAN — réseau overlay Layer 2 sur Layer 3

VxLAN (Virtual Extensible LAN) = technologie d'encapsulation qui créé des réseaux virtuels Layer 2 (overlay) par-dessus infrastructure IP Layer 3 (underlay).

// Principes clés

  • 24 bits VNI (VXLAN Network Identifier) → 16 millions de segments (2^24)
  • Encapsulation MAC-in-UDP → trame Ethernet encapsulée dans paquet UDP
  • Port UDP 4789 (standard IANA)
  • VTEP (VXLAN Tunnel Endpoint) → équipement qui encapsule/désencapsule
  • Overlay network → réseau virtuel indépendant du réseau physique

// VxLAN vs VLAN

Architecture

Overlay vs Underlay — deux réseaux superposés

// Underlay network (réseau physique)

Infrastructure IP Layer 3 physique qui transporte le trafic encapsulé.

  • Routeurs, switches Layer 3
  • Routage IP classique (OSPF, BGP...)
  • Ne connaît PAS les VxLAN, voit juste paquets UDP
  • Fourni connectivité entre VTEPs

// Overlay network (réseau virtuel)

Réseau Layer 2 virtuel créé par encapsulation VxLAN par-dessus underlay.

  • VMs, conteneurs communiquent comme si même VLAN
  • Totalement indépendant du réseau physique
  • Programmable, flexible, mobile
  • Géré par SDN controller (VMware NSX, Cisco ACI...)
Architecture VxLAN :

[VM A] ←→ [VTEP 1] ←→ [ Réseau IP Underlay ] ←→ [VTEP 2] ←→ [VM B]
↑ ↑ (routeurs) ↑ ↑
Overlay Encap Décap Overlay
Layer 2 VxLAN VxLAN Layer 2

VM A et VM B pensent être sur même VLAN
→ En réalité séparées par réseau IP routé
💡
Overlay = abstraction Underlay = réseau physique (comment paquets sont routés physiquement)
Overlay = réseau virtuel (comment VMs/conteneurs communiquent logiquement)
VxLAN découple les deux → flexibilité maximale
Composants

VTEP — VXLAN Tunnel Endpoint

VTEP (VXLAN Tunnel Endpoint) = équipement qui encapsule/désencapsule trames VxLAN. C'est le pont entre overlay et underlay.

// Rôle VTEP

  1. Encapsulation (envoi) : Trame Ethernet originale → ajoute en-tête VxLAN → encapsule dans UDP/IP → envoie sur underlay
  2. Désencapsulation (réception) : Paquet UDP reçu → extrait trame Ethernet originale → forward vers VM/conteneur
  3. Apprentissage MAC : Associe MACs VMs avec VTEPs distants (comme switch apprend MACs)
  4. Tunneling : Maintient tunnels VxLAN vers autres VTEPs

// Types de VTEP

TypeLocalisationExemples
Hardware VTEP Switch Top-of-Rack (ToR) Cisco Nexus, Arista, Juniper QFX
Software VTEP Hyperviseur (serveur) VMware ESXi, Linux VXLAN kernel, OVS

// Exemple : Envoi trame VM A → VM B

1. VM A envoie trame Ethernet classique
Trame : [MAC dest: VM B] [MAC src: VM A] [Data]

2. VTEP 1 reçoit trame, encapsule
→ Ajoute en-tête VxLAN (VNI = 5000)
→ Encapsule dans UDP (port 4789)
→ Encapsule dans IP (dest = VTEP 2)
Paquet : [IP: VTEP1→VTEP2] [UDP: 4789] [VxLAN: VNI 5000] [Trame originale]

3. Réseau underlay route paquet IP vers VTEP 2
(Routage classique, ne voit que paquet UDP)

4. VTEP 2 reçoit, désencapsule
→ Extrait trame Ethernet originale
→ Forward vers VM B

5. VM B reçoit trame comme si même switch
Identifiant

VNI — VXLAN Network Identifier

VNI (VXLAN Network Identifier) = identifiant 24 bits qui identifie le segment VxLAN (équivalent VLAN ID mais plus grand).

// Caractéristiques VNI

  • 24 bits → 2^24 = 16,777,216 VNIs possibles (0 à 16,777,215)
  • Équivalent VLAN ID mais beaucoup plus scalable
  • Chaque VNI = segment Layer 2 isolé
  • VMs dans même VNI communiquent, VMs dans VNIs différents = isolées

// Mapping VNI ↔ VLAN

Souvent, VNI mappé sur VLAN traditionnel au niveau VTEP :

VLAN 10 (local switch) ←→ VNI 5000 (VxLAN overlay)
VLAN 20 (local switch) ←→ VNI 5001 (VxLAN overlay)

VM dans VLAN 10 → VTEP encapsule avec VNI 5000
→ Tunnel VxLAN → VTEP distant désencapsule → VLAN 10 local

// Comparaison VLAN ID vs VNI

CritèreVLAN IDVNI
Taille12 bits24 bits

Protocole

Encapsulation VxLAN — MAC-in-UDP

Encapsulation VxLAN = trame Ethernet originale encapsulée dans UDP/IP avec en-tête VxLAN.

// Structure paquet VxLAN

┌─────────────────────────────────────────────┐
│ Outer Ethernet (14B) │
│ MAC src = VTEP 1, MAC dest = next hop │
├─────────────────────────────────────────────┤
│ Outer IP (20B) │
│ IP src = VTEP 1, IP dest = VTEP 2 │
├─────────────────────────────────────────────┤
│ Outer UDP (8B) │
│ Source port = random, Dest port = 4789 │
├─────────────────────────────────────────────┤
│ VxLAN Header (8B) │
│ Flags, VNI (24 bits), Reserved │
├─────────────────────────────────────────────┤
│ Inner Ethernet Frame (original) │
│ Trame Ethernet complète de la VM │
└─────────────────────────────────────────────┘

Overhead total : ~50 bytes

// En-tête VxLAN (8 bytes)

ChampTailleDescription
Flags8 bitsBit I = 1 (VNI valide)
Reserved24 bitsRéservé (0)
VNI24 bitsVXLAN Network Identifier
Reserved8 bitsRéservé (0)

// Port UDP 4789

  • Port destination : 4789 (standard IANA)
  • Port source : Aléatoire (éphémère), basé sur hash inner frame
  • Hash inner frame → load balancing ECMP sur underlay
⚠️
MTU et fragmentation Overhead VxLAN (~50 bytes) → MTU effectif réduit.
MTU underlay 1500 → MTU overlay ~1450
Solution : Jumbo frames sur underlay (MTU 9000) ou ajuster MTU VMs
Applications

Cas d'usage VxLAN — quand l'utiliser ?

// 1. Datacenters multi-tenants (cloud providers)

  • AWS, Azure, GCP : millions de clients isolés
  • Chaque client = VNI unique
  • Isolation totale entre tenants

// 2. Virtualisation / SDN

  • VMware NSX, Cisco ACI, OpenStack Neutron
  • Mobilité VM entre hosts/sites
  • Réseaux programmables via API

// 3. Datacenters distribués

  • Stretching Layer 2 entre sites géographiques
  • DCI (Data Center Interconnect)
  • Active-active datacenters

// 4. Conteneurs (Kubernetes)

  • CNI plugins (Calico, Flannel...)
  • Isolation entre namespaces
  • Networking pods cross-hosts

// Quand NE PAS utiliser VxLAN ?

  • ❌ Petits réseaux (<100 VMs) → VLANs suffisent
  • ❌ Pas de multi-tenancy → overhead inutile
  • ❌ Équipements ne supportent pas → mise à jour nécessaire
  • ❌ Latence critique → overhead encapsulation peut impacter
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : VxLAN = 4096 segments
FAUX. VxLAN = 16,777,216 segments (2^24).

4096 = VLAN (12 bits)
16 millions = VxLAN (24 bits VNI)
❌ Piège #2 : VxLAN port TCP 4789
FAUX. VxLAN utilise UDP port 4789 (pas TCP).

UDP = sans connexion, approprié pour encapsulation
❌ Piège #3 : VxLAN = Layer 3 uniquement
FAUX. VxLAN = Layer 2 over Layer 3.

Overlay = Layer 2 (Ethernet)
Underlay = Layer 3 (IP routé)
VxLAN encapsule Layer 2 dans Layer 3
❌ Piège #4 : VTEP = routeur
Pas exactement. VTEP = point d'encapsulation/désencapsulation.

Peut être switch ToR (hardware) ou hyperviseur (software).
Pas forcément un routeur.
