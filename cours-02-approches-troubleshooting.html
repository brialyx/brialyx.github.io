OSI Model review

Modèle OSI — Framework troubleshooting

Rappel modèle OSI (Cours Module 1.0) : 7 couches logiques communication réseau.

7. Application : HTTP, DNS, SMTP, FTP
6. Presentation : SSL/TLS, encryption, compression
5. Session : Session establishment, maintenance
4. Transport : TCP, UDP (ports, segments)
3. Network : IP, routing, routers (packets)
2. Data Link : Ethernet, MAC, switches (frames)
1. Physical : Câbles, connecteurs, signaux électriques (bits)

Utilité troubleshooting : OSI = roadmap localiser problèmes. Chaque couche = tests spécifiques.

💡
OSI = outil diagnostic, pas théorie abstraite Problème réseau ? → Identifier couche OSI défaillante → Résolution ciblée.
Exemple : "Pas Internet" = vague. "Couche 3 routing problème" = précis actionable.
Approches troubleshooting = stratégies naviguer couches OSI efficacement.
Layer 7 → Layer 1

Top-Down Approach — Application vers Physique

Top-Down = Commencer couche Application (L7) → descendre vers Physical (L1) jusqu'à trouver couche défaillante.

// Workflow Top-Down

Problème : User "Pas accès site web company.com"

L7 (Application) : Browser error ?
→ Test : Navigate http://company.com → "DNS resolution failed"
→ Hypothèse : DNS problem (L7)

L7 (Application - DNS) : DNS fonctionne ?
→ Test : nslookup company.com → "Server failure"
→ Test : nslookup company.com 8.8.8.8 (Google DNS) → Success
→ Cause trouvée : DNS server local défaillant (L7)
→ Solution : Configure DNS 8.8.8.8 temporaire, fix DNS server

// Quand utiliser Top-Down ?

  • ✅ Problèmes application-specific : "Email fonctionne pas" "Web lent" "VoIP quality mauvaise"
  • ✅ User reports vagues : "Internet marche pas" (symptôme haut-niveau → investiguer depuis application)
  • ✅ Software/config suspect : Changement récent application, firewall, proxy

// Avantages Top-Down

  • ✅ User-centric (commence où user interagit)
  • ✅ Rapide si problème applicatif (pas besoin descendre L1)
  • ✅ Naturel user perspective

// Inconvénients Top-Down

  • ⚠️ Inefficace problèmes physiques (cable unplugged → tester L7,L6,L5,L4,L3,L2 AVANT trouver L1 = perte temps)
  • ⚠️ Complexité croissante descendant (L7 = multiples apps/protocols)
Layer 1 → Layer 7

Bottom-Up Approach — Physique vers Application

Bottom-Up = Commencer couche Physical (L1) → monter vers Application (L7) jusqu'à trouver couche défaillante.

// Workflow Bottom-Up

Problème : PC pas connectivité réseau

L1 (Physical) : Câble branché ? LED switch ?
→ Test : Vérifier LED NIC PC → OFF
→ Test : Vérifier LED switch port → OFF
→ Cause trouvée : Câble Ethernet débranché (L1)
→ Solution : Rebrancher câble → LED ON → Connectivity restored

// Exemple Bottom-Up complet

Problème : Laptop pas Internet

L1 (Physical) :
→ LED NIC : ✅ ON (link established)
→ Câble : ✅ Branché
→ L1 OK → Monter L2

L2 (Data Link) :
→ ipconfig /all → MAC address shown : ✅ OK
→ Switch port : ✅ UP
→ L2 OK → Monter L3

L3 (Network) :
→ ipconfig → IP : 169.254.x.x (APIPA)
→ Cause trouvée : Pas IP DHCP (L3 problem)
→ Ping gateway : ❌ Timeout (expected, pas IP valide)
→ Solution : Troubleshoot DHCP (server down ? Client port VLAN wrong ?)

// Quand utiliser Bottom-Up ?

  • ✅ Nouvelle installation : Configuration initiale équipements (vérifier fondations first)
  • ✅ Problèmes intermittents : Connexions drop random (souvent L1/L2 : câbles, collisions)
  • ✅ Symptômes totale perte connectivité : "Rien marche du tout" (likely L1/L2)
  • ✅ After physical changes : Déménagement équipements, recâblage

// Avantages Bottom-Up

  • ✅ Élimine problèmes physiques rapidement (quick wins)
  • ✅ Logique progression (fondations solides avant complexité)
  • ✅ Recommandé débutants (apprentissage systématique)

// Inconvénients Bottom-Up

  • ⚠️ Perte temps si problème applicatif (tester L1-L6 alors que L7 DNS failure)
  • ⚠️ Overkill problèmes simples user-level
Middle-out approach

Divide-and-Conquer — Stratégie optimale

Divide-and-Conquer = Commencer couche MILIEU OSI (typiquement L3 Network) → diviser problème moitié chaque test.

// Principe Divide-and-Conquer

Stratégie : Tester couche milieu → Résultat élimine 50% couches possibles.

Problème : PC pas ping serveur remote

Commencer L3 (Network) :
→ Test : Ping gateway local (192.168.1.1)

Scénario A : Ping gateway SUCCESS
→ L1, L2, L3 local = ✅ OK
→ Problème : L3 routing OU L4+ (upper layers)
→ Focus : Routing, firewall, application
→ Eliminated : L1, L2 (50% couches éliminées 1 test)

Scénario B : Ping gateway FAIL
→ L1, L2, OU L3 local = ❌ Problem
→ Focus : Physical, Data Link, IP config
→ Eliminated : L4-L7 (50% couches éliminées 1 test)

// Workflow Divide-and-Conquer typique

Étape 1 : Test L3 (ping gateway)
→ Success ? → Problème upper layers (L4-L7)
→ Fail ? → Problème lower layers (L1-L3)

Étape 2A : Si upper layers suspect
→ Test : Ping remote server IP (not hostname)
  → Success ? → DNS problem (L7)
  → Fail ? → Routing/Firewall (L3/L4)

Étape 2B : Si lower layers suspect
→ Test : Check IP config (ipconfig)
  → IP valid ? → L2 problem (switch, VLAN)
  → IP APIPA ? → L3 DHCP problem
→ Test : Check link (LED, ipconfig /all media disconnected ?)
  → No link ? → L1 problem (cable, NIC)

// Quand utiliser Divide-and-Conquer ?

  • ✅ Général purpose : Approche par défaut (efficient most scenarios)
  • ✅ Problèmes complexes : Symptômes ambigus (pas évident L1 vs L7)
  • ✅ Time-critical : Production down, besoin résolution rapide
  • ✅ Expérience technicien : Permet intuition choisir starting point optimal

// Avantages Divide-and-Conquer

  • ✅ Efficacité maximale : Réduit espace recherche 50% chaque test (logarithmic O(log n))
  • ✅ Flexible : Starting point ajustable selon symptômes
  • ✅ Scalable : Fonctionne réseaux simples → complexes

// Inconvénients Divide-and-Conquer

  • ⚠️ Requires knowledge : Débutants pas intuition où commencer
  • ⚠️ Can miss obvious : Starting milieu peut skip simple L1 fix (cable unplugged)
⚡
Divide-and-Conquer = approche professionnelle Experts utilisent Divide-and-Conquer par défaut.
Raison : Minimise temps diagnostic (critical production environments).
CompTIA examen : Reconnaître quand Divide-and-Conquer optimal vs Top-Down/Bottom-Up.
Choosing approach

Quelle approche choisir ? — Decision matrix

Symptôme / ContexteApproche recommandéeRaison
"Email marche pas, web OK" Top-Down Application-specific (L7 SMTP problem likely)
"Rien marche, tout down" Bottom-Up Total failure = likely L1/L2 (cable, switch)
"Nouvelle installation équipement" Bottom-Up Verify foundations first (physical correct?)
"Intermittent connectivity" Bottom-Up Flapping = often L1/L2 (bad cable, collisions)
"Pas ping server remote" Divide-and-Conquer Ambiguous (could be L1-L7) → Start L3
"DNS résolution lente" Top-Down DNS = L7, commence application layer
"After datacenter move" Bottom-Up Physical changes = verify cabling first
"VoIP quality dégradée" Top-Down Application QoS (L7/L4 codec, jitter, latency)
"Production critical down" Divide-and-Conquer Time-critical → Fastest diagnosis (L3 start)

A) Top-Down
B) Bottom-Up
C) Divide-and-Conquer

Analyse :
"Cannot access website" = application-specific (browser, DNS, HTTP).
Commence L7 (application) logical → Réponse : A) Top-Down
⚠️ Pièges d'examen

Ce qui trompe les candidats

❌ Piège #1 : Divide-and-Conquer = toujours meilleure approche
FAUX. Dépend contexte.

Divide-and-Conquer = optimal problèmes ambigus, time-critical.
MAIS : Application-specific symptoms → Top-Down faster.
Total connectivity loss → Bottom-Up faster (likely L1).

Examen : Lire contexte carefully. "Cannot access web" ≠ "No connectivity at all".
❌ Piège #2 : Bottom-Up commence toujours Physical Layer
NUANCE. Bottom-Up = généralement commence L1.

MAIS : Si évidence L1 OK (LED link UP, cable confirmed good) → Skip L1, commence L2.
Bottom-Up = philosophie (foundations first), pas rigide "must start L1".

Pragmatic : "Question obvious" (Méthodologie CompTIA Étape 2) → Si LED ON évident, pas besoin re-vérifier.
❌ Piège #3 : OSI troubleshooting = tester TOUTES 7 couches systématiquement
FAUX. OSI = framework, pas checklist exhaustive.

Goal : Identifier couche défaillante RAPIDEMENT (pas tester toutes).
Approches (Top/Bottom/Divide) = stratégies naviguer OSI efficiently.

Exemple : Ping gateway success → L1-L3 OK confirmé (pas besoin tester chaque individuellement).
Examen : "Most efficient approach" ≠ "Test all 7 layers sequentially".
Résumé

🔑 Ce que tu dois retenir absolument

→Top-Down : L7 → L1 (Application → Physical). Use : Application-specific problems, user-centric symptoms. Exemple : "Email down, web OK" → Start L7 SMTP
→Bottom-Up : L1 → L7 (Physical → Application). Use : New installations, total connectivity loss, physical changes, intermittent issues. Exemple : "Nothing works" → Start L1 cable
→Divide-and-Conquer : Start L3 (Network) milieu, divide 50% each test. Use : Ambiguous problems, time-critical, general purpose. Most efficient approach experienced techs
→Decision factors : Symptom specificity (app-specific vs total loss), context (new install vs existing), urgency (production critical), evidence (LED status, recent changes)
→OSI framework : L1 Physical (cables LED), L2 Data Link (MAC switch), L3 Network (IP routing ping), L4 Transport (TCP/UDP ports), L5-7 Application (DNS HTTP SMTP)
→Key tests : L1 (LED check), L2 (MAC table), L3 (ping gateway, ipconfig), L4 (telnet port, netstat), L7 (nslookup, curl, browser). Each test confirms/eliminates layers
